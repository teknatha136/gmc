How the Internet Works

The internet is not magic — it's infrastructure. Once you understand how it works, you can debug problems, read error messages, and build things that work reliably.

Foundation

The Internet & IP Addresses

The internet is a global network of billions of computers that are all connected together. Not wirelessly through the air — physically, through underground and undersea cables, via satellites, and through large data centres in cities around the world.

For two computers to communicate, they each need an address. That address is called an IP address (Internet Protocol address).

Analogy

Think of your house address. The postman can't deliver a letter to you without knowing where you live. In the same way, the internet can't deliver data to your device without knowing your IP address.

IPv4 addresses look like this: 192.168.1.1 — four numbers, each between 0 and 255, separated by dots. There are about 4 billion possible IPv4 addresses, which is no longer enough for all the devices in the world. IPv6 was invented to solve this problem.

  • Your laptop, phone, and the selise.ch server all have IP addresses
  • Public IP: visible to the whole internet (your router's address)
  • Private IP: only visible on your local network (e.g., 192.168.x.x)
Domain Name System

DNS — The Internet's Phonebook

You remember names — like selise.ch or google.com. But computers connect using numbers — IP addresses. DNS is the system that translates between the two.

When you type a domain name in your browser:

  • Your browser first checks its local cache — has it looked this up recently?
  • If not, it asks a DNS resolver (usually provided by your ISP or Google/Cloudflare)
  • The resolver looks up the domain name and returns the matching IP address
  • Your browser then connects to that IP address
Key fact

A DNS lookup takes less than 100 milliseconds — faster than you can blink. And once your browser has looked it up, it caches the result so it doesn't have to do it again for a while.

If DNS breaks (which happens occasionally), websites appear to be offline even though the servers are perfectly fine — your browser just can't find the address.

URLs

Anatomy of a URL

Every URL (Uniform Resource Locator) is an address that points to a specific resource on the internet. Each part tells the browser something different.

httpsProtocol
://
seliseDomain
.chTLD
/careersPath
?lang=enQuery
  • Protocol — https: Use encrypted HTTPS (never plain HTTP for sensitive data)
  • Domain — selise: The registered name of the organisation
  • TLD — .ch: Top-level domain. .ch = Switzerland, .bt = Bhutan, .com = commercial
  • Path — /careers: Which specific page or file on the server
  • Query string — ?lang=en: Extra parameters sent to the server (like a filter)
Protocols & Status Codes

HTTP, HTTPS & Status Codes

HTTP (HyperText Transfer Protocol) is the rulebook for how browsers and servers communicate. It defines how requests are formatted, how responses are structured, and what the different types of requests mean.

HTTPS is HTTP with encryption added (using TLS — Transport Layer Security). All data sent between your browser and the server is scrambled so that nobody else on the network can read it.

Practical rule

Always check for the padlock icon in the browser address bar before typing a password or card number. No padlock = no HTTPS = your data is travelling in plain text.

Status codes are 3-digit numbers the server sends back with every response. They tell you instantly whether the request succeeded or failed:

200
OK
Request succeeded
301
Moved
New permanent URL
404
Not Found
Page doesn't exist
403
Forbidden
No permission
500
Server Error
Server-side problem

Memory shortcut: 4xx = your fault (bad request) · 5xx = server's fault · 2xx = success

Class 04 — Key Takeaways

  • Every device has an IP address — the internet uses these numbers to route data to the right place
  • DNS translates names to numbers — you type selise.ch, DNS gives your browser the IP address to connect to
  • A URL has 5 parts — protocol, domain, TLD, path, and optional query string. Each part tells the browser something different
  • HTTPS is mandatory for sensitive data — look for the padlock. HTTP sends data in plain text; HTTPS encrypts it
  • Status codes summarise every response — 200 = success, 404 = not found, 500 = server broke
  • Every web request travels: Browser → Internet → Server → back — DNS, TCP connection, HTTP request, server processing, HTTP response, browser rendering

APIs, Data & Databases

APIs are how software systems talk to each other. Almost every app you use — from weather to maps to banking — works by calling APIs. Understanding them is foundational to every role in this programme.

Application Programming Interface

What is an API?

An API (Application Programming Interface) is a way for two pieces of software to communicate with each other. It defines what requests you can make, what format to use, and what responses you'll get back.

The Restaurant Analogy

You're the client. The waiter is the API. The kitchen is the server. The pantry is the database.

You tell the waiter what you want (your request). The waiter goes to the kitchen (without you). The kitchen prepares it using ingredients from the pantry. The waiter brings it back (the response). You never enter the kitchen — the waiter is the only interface.

  • APIs let apps share data without exposing their internal systems
  • When you "Log in with Google", your app calls Google's OAuth API
  • When you check the weather, your app calls a weather service API
  • Claude Code calls Anthropic's API every time you ask it something
  • SELISE's products all expose APIs that client systems connect to
Data Format

JSON — How APIs Send Data

JSON (JavaScript Object Notation) is the most common format for sending data between an API and a client. It's human-readable and works in every programming language.

trainee.json
{
  "name": "Karma Dorji",
  "city": "Gelephu",
  "age": 24,
  "enrolled": true,
  "skills": ["Python", "SQL"],
  "vertical": "Software Dev"
}
  • Keys — always a string in double quotes. The label for the field.
  • String values — text, always in double quotes: "Gelephu"
  • Number values — no quotes needed: 24
  • Boolean values — true or false, no quotes
  • Arrays — a list of values inside square brackets: ["Python", "SQL"]
  • Nested objects — an object inside an object, using curly braces inside curly braces
REST & HTTP Methods

REST APIs & HTTP Methods

REST (Representational State Transfer) is the most popular way to design APIs. REST APIs use URLs to represent resources (things like users, posts, products) and HTTP methods to describe what to do with them.

REST URL pattern

/trainees → all trainees
/trainees/42 → trainee with ID 42
/trainees/42/classes → all classes that trainee 42 attended

The HTTP method in the request tells the server what action to perform:

Method Action CRUD Example
GET Read / fetch data Read GET /trainees
POST Create new record Create POST /trainees
PUT Update existing record Update PUT /trainees/42
DELETE Remove a record Delete DELETE /trainees/42
Remember: CRUD

Every application ever built does four things: Create, Read, Update, Delete — called CRUD. REST maps these directly to POST, GET, PUT, DELETE. Learn these four and you understand the foundation of every API you'll ever work with.

Data Storage

Databases — Organised Storage for Data

A database is an organised system for storing data persistently (meaning it survives when you turn off the server). Think of it like a collection of very powerful, interconnected spreadsheets.

🗄️ trainees — example SQL table
id name city vertical cohort
1 Karma Dorji Gelephu Software Dev 3
2 Tenzin Wangmo Thimphu UI/UX Design 3
3 Pema Choden Paro QA Testing 3
… … … … …
  • Table — one type of data (like one spreadsheet). A database has many tables.
  • Row — one record. In the table above, one row = one trainee.
  • Column — one field. Every row has the same columns (name, city, etc.).
  • Primary key — a unique ID for each row. The id column — no two rows share the same value.
  • Foreign key — a column that points to a row in another table (e.g., trainee_id in an Attendance table).

🗃️ SQL (Relational)

  • →Tables with fixed columns
  • →Great for structured data
  • →PostgreSQL, MySQL, SQLite
  • →Used in most SELISE products

📄 NoSQL (Document)

  • →Stores JSON documents
  • →Flexible — fields can vary
  • →MongoDB, Firestore
  • →Good for fast-changing data

Class 05 — Key Takeaways

  • An API is the interface between client and server — you never access a database directly. The API controls what you can ask for and what you get back.
  • JSON is the language of APIs — key-value pairs, strings in quotes, numbers without quotes, true/false for booleans, [ ] for arrays
  • REST uses URLs + methods — the URL describes the resource (/trainees/42), the method describes the action (GET, POST, PUT, DELETE)
  • CRUD = the four operations of every app — Create (POST), Read (GET), Update (PUT), Delete (DELETE)
  • A database table = rows and columns — each row is one record, each column is one field, the primary key uniquely identifies each row
  • SQL vs NoSQL — SQL for structured relational data, NoSQL for flexible document-style data. Most real apps use both.

✍ Practice at Home

  1. Open your browser and go to jsonplaceholder.typicode.com/users. Read the JSON. Try changing the URL to /users/3 and /posts?userId=1. What pattern do you notice?
  2. Visit any 3 websites. For each one, write down the full URL and label each part: protocol, domain, TLD, path, query string (if any).
  3. Open DevTools on any website (F12 or Cmd+Option+I). Go to the Network tab and refresh. Look at the list of requests — what status codes do you see? Click on one and look at its Headers tab.
  4. Schema design challenge: Design a database for a small library. What tables do you need? What columns? How do they connect? Draw it on paper.
