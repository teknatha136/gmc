Part 1

Apps that talk to each other

When you open an app, most of what you see doesn't actually live on your phone. Your phone is just asking other computers for it. The way it asks is called an API.

The idea

An API is how one app asks another app for something

Open mBoB and tap "Check Balance." Your phone doesn't know your balance. Bank of Bhutan's computer does. Your phone sends a polite message: "Hi, this is Karma. What's my balance?" The bank's computer checks, and sends a message back: "Nu. 4,250."

That polite-messages system is an API. Your phone and the bank never touch each other directly — they only pass messages through this agreed-upon channel.

💡 In plain words

An API is the invisible messenger between two apps. One app says "please give me this," the other app says "here you go" — or "no, you're not allowed."

🍽️ The restaurant picture

Think of a restaurant. You are the app on your phone. The waiter is the API. The kitchen is the other company's computer. You never walk into the kitchen. You just tell the waiter what you want, the waiter brings it back. If you ask for something the kitchen doesn't serve, the waiter says no. That's exactly how APIs work.

Where you already use APIs

You've been using APIs every day — without knowing

Every time any of these happen, an API call is happening quietly behind the scenes:

  • You tap "Log in with Google" on a website — the site is calling Google's API to check who you are.
  • You open Google Maps to find Gelephu — Maps is calling a location API for map tiles and directions.
  • You check the weather — the weather app is calling a weather service's API.
  • You pay through eSewa or mBoB — the shop's app is calling the payment provider's API.
  • You ask Claude Code a question — it's calling Anthropic's API.

Apps don't build everything themselves. They borrow from each other through APIs. That's why an app can be built in weeks instead of years.

Why you should care

Every role in this programme touches APIs

  • Developers write code that calls APIs, and also build APIs that other apps will call.
  • Designers design screens that show what an API sends back (loading, empty, error).
  • Testers check that the API returns the right thing, every time.
  • Business Analysts decide which APIs the product needs and what they should return.

SELISE products are mostly APIs. When a bank or government office uses SELISE software, they're talking to our APIs.

Part 2

The neat little box — JSON

When two apps pass information through an API, they need to agree on the shape of the message. Almost every API uses the same shape, called JSON.

The idea

JSON is a clean way to write down information

JSON stands for JavaScript Object Notation — but forget the name. What matters is: JSON is how an app writes down facts about a thing so another app can read them.

💡 In plain words

JSON is like a small form with labels on the left and values on the right. "Name: Karma. City: Gelephu. Age: 24." Neat, labeled, easy for any computer anywhere to read.

trainee.json
{
  "name": "Karma Dorji",
  "city": "Gelephu",
  "age": 24,
  "enrolled": true,
  "skills": ["Python", "SQL"],
  "vertical": "Software Dev"
}
The rules

Reading JSON, piece by piece

JSON looks scary because of the punctuation. It's actually simple once you know the six pieces:

  • Curly braces { } — wrap one "thing" (like one trainee).
  • Label in quotes — the name of a field, e.g. "name". Always in double quotes.
  • Colon : — separates the label from its value.
  • Text values — go in double quotes: "Gelephu".
  • Number values — no quotes: 24.
  • True/false — no quotes: true or false.
  • Square brackets [ ] — wrap a list of things: ["Python", "SQL"].
  • Comma , — separates one field from the next.
⚠️ Three common mistakes

1. Forgetting a comma between fields. 2. Using 'single quotes' — JSON only accepts "double quotes". 3. A stray comma after the last field, before the closing }. If your JSON breaks, check these three first.

Part 3

The four things every app does

Think of any app you use. Behind the scenes, it only ever does four things to information: show it, add it, change it, remove it. This idea has a name: CRUD. And the way APIs implement CRUD is called REST.

CRUD

Four actions. That's it.

Open WeChat. Everything you can do falls into one of four buckets:

  • Create — send a new message, post a status.
  • Read — see your chats, read someone's profile.
  • Update — edit your name, change your photo.
  • Delete — remove a message, leave a group.
💡 In plain words

Every app you've ever used — mBoB, Gmail, Facebook, Druk Air booking, eSewa — is just CRUD on different kinds of information. Once you see this, apps stop feeling magical.

REST + HTTP methods

How an API names these four actions

When your app sends a request to an API, it uses one of four HTTP methods. Each one maps to one CRUD action:

Method What it does CRUD Example
GET Read / fetch information Read GET /trainees
POST Add a new record Create POST /trainees
PUT Change an existing record Update PUT /trainees/42
DELETE Remove a record Delete DELETE /trainees/42

REST is the most common style of API. It says: use a URL to name the thing (/trainees), and use an HTTP method to say what you want to do with it.

📬 Another picture

Think of the URL as the address on an envelope (which trainee?), and the HTTP method as the instruction written on the envelope ("read this," "replace this with what's inside," "throw this away"). The API reads both and acts.

URLs that mean something

REST URLs read like sentences

A well-designed REST URL tells you exactly what it's about:

  • /trainees — all trainees.
  • /trainees/42 — the one trainee whose id is 42.
  • /trainees/42/classes — the classes that trainee 42 attended.
  • /trainees/42/classes/05 — trainee 42's record for class 05.

Notice the pattern: bigger group → one item → something about that item. Once you know this pattern, you can guess what an API does just by looking at its URLs.

Part 4

Where apps keep their memory

APIs handle conversation. JSON handles the shape of the message. But where does the information actually live when no one is asking? In a database.

The idea

A database is an app's long-term memory

Turn off your phone, come back tomorrow, open Facebook — all your photos are still there. Your phone didn't store them. Facebook's database did. A database is a place where information is kept safely, organised neatly, and can be found again fast.

💡 In plain words

A database is like a very large, very organised set of spreadsheets that a computer can search in a fraction of a second. When an API needs information, it goes to the database.

SQL databases

The spreadsheet style — SQL

Most databases look like spreadsheets: rows and columns, in tables. Each row is one record (one trainee, one payment, one booking). Each column is one fact about that record (name, age, city).

The language you use to ask questions of a SQL database is also called SQL (say it "sequel"). It reads almost like English:

  • SELECT name FROM trainees WHERE cohort = 3; — give me the names of all cohort-3 trainees.
  • SELECT * FROM trainees WHERE city = 'Gelephu'; — give me everything about trainees in Gelephu.

Examples of SQL databases: PostgreSQL, MySQL, SQL Server, SQLite. SELISE mostly uses PostgreSQL.

NoSQL databases

The JSON style — NoSQL

Some information doesn't fit neatly into rows and columns. A chat message might have 1 reply or 50 replies. A product might have 3 photos or 20. For this, there's NoSQL — databases that store information as JSON-like documents instead of tables.

Examples: MongoDB, Firestore, DynamoDB. You don't need to pick a side today — just know both styles exist, and each has its place.

📍 From your life

WhatsApp's chat history is NoSQL-shaped (every message is a little document with varying fields). mBoB's account ledger is SQL-shaped (every transaction is a neat row: date, amount, balance). Same app can use both.

Putting it all together

The full journey of one tap

You tap "Check Balance" in mBoB. Here's what actually happens in about 200 milliseconds:

  1. Your phone sends a GET request to an API URL like /accounts/12345/balance.
  2. The API asks the bank's database: SELECT balance FROM accounts WHERE id = 12345;
  3. The database finds one row, returns the balance.
  4. The API wraps it in JSON: { "balance": 4250 }.
  5. Your phone receives the JSON and shows "Nu. 4,250" on screen.

That's the whole story. Every app you use is doing some version of this, millions of times a day. You now understand the machinery.

Part 5

Practice at home

These are optional, but doing even one will make tomorrow's class much easier. None require coding.

1

Pick any app on your phone. Write its CRUD list.

Pick WeChat, mBoB, Instagram, or anything else. On a piece of paper, write one example each for Create, Read, Update, Delete. This trains your eye to see CRUD everywhere.

2

Write yourself as JSON.

On paper: open brace, "name" colon your name in quotes comma, "age" colon your age comma, "skills" colon a list in square brackets, close brace. Ask a friend to spot your mistakes.

3

Draw one SQL table.

Pretend you're building a bookings app for Druk Air. Draw a table called bookings. What columns would it have? (Hint: at least 5.) This is exactly how BAs start requirements.

4

Re-read this note tomorrow morning.

Five minutes, with coffee. The second pass is when things actually stick. This is the single most useful thing you can do.
