Class 04 · 9:30 – 10:45

How the Internet Works

Cover IP addresses, DNS, URLs, the client-server model, HTTP/HTTPS, and status codes. The goal is conceptual understanding — they don't need to memorise anything, just be able to describe what happens when they visit a website.

Explain what the internet is Describe IP addresses & DNS Read and break down a URL Distinguish HTTP from HTTPS Recognise common status codes Trace a full browser request end-to-end
🎤 Opening Discussion
9:30 – 9:35 · 5 min
Ask the class

"You open WhatsApp and send a message to your friend in Thimphu. What happens in that 0.3 seconds?"

Take 4–5 answers. Don't correct them yet — write key words on the board (if available). You'll revisit these at the end and confirm who was right.

⚡

Trainer tip: This sets up curiosity. Most trainees will say "it goes through the internet" — which is fine. The goal is to surface what they already know before filling in the gaps.

🌍 The Internet & IP Addresses
9:35 – 9:50 · 15 min
Key teaching points
  • The internet is a global network of computers connected by physical cables, fibre, satellites, and routers
  • Every device has an IP address — like a postal address but for computers. Without it, packets of data don't know where to go
  • IPv4: four numbers 0–255 separated by dots (192.168.1.1). Only ~4 billion combinations → IPv6 exists to solve this
  • Bhutan angle: Gelephu connects via fibre from India; Druknet is the national ISP. The same undersea cables that connect continents are why your WhatsApp message reaches London in milliseconds
⚡

Analogy to use: "Your house has a physical address so the postman can find you. Your laptop has an IP address so the internet can find it. Same idea, different scale."

Quick check

"If every device needs an IP address, and there are 8 billion people on Earth — do we have enough IPv4 addresses?" → No! That's why IPv6 was invented.

📋 DNS — Domain Name System
9:50 – 10:00 · 10 min
Key teaching points
  • Humans remember names (selise.ch). Computers need numbers (185.34.33.33). DNS is the system that translates between them
  • When you type a domain: browser asks DNS resolver → resolver looks up hierarchy → returns IP → browser connects
  • DNS lookup happens in under 100ms. Results are cached so you don't pay the cost every time
  • If DNS is down → websites are unreachable even though the servers are fine (DNS outage != internet down)
⚡

Analogy: "DNS is like the phonebook. You know your friend's name, not their phone number. The phonebook looks it up for you. DNS does the same — you know 'google.com', DNS finds '142.250.x.x'."

⚡

Common confusion: Trainees often think the domain name IS the address. Clarify: it's just a human-friendly label. The real address is always an IP number.

🔗 URLs & Client-Server
10:00 – 10:15 · 15 min
URL anatomy (go through this slowly)
https://selise.ch/careers?lang=en
https → protocol (use HTTPS, not HTTP)
selise → domain name
.ch → TLD — top-level domain (Switzerland)
/careers→ path — which page on the server
?lang=en→ query string — extra parameters
Client-Server model
  • Client = the one asking (browser, mobile app, Claude Code, Postman)
  • Server = the one answering, runs 24/7, can handle thousands of requests simultaneously
  • Use the diagram on slide 9: You in Gelephu (client) → Internet → selise.ch server in Zürich (server)
⚡

Make it tangible: Ask a trainee to open selise.ch on their phone and navigate to a few pages. Point out how the URL changes. Every different URL = different request to the same server.

🔒 HTTP, HTTPS & Status Codes
10:15 – 10:30 · 15 min
HTTP
  • HyperText Transfer Protocol — the rulebook for how browsers and servers communicate
  • HTTP sends data in plain text — anyone watching the network can read it (like a postcard)
  • HTTPS = HTTP + TLS encryption — data is scrambled, only client and server can read it (like a sealed envelope)
  • Always check for the padlock before typing passwords or payment info
Status codes — the server's reply code
  • 200 OK — success, here's your data
  • 301 Moved Permanently — page is at a new URL, follow this redirect
  • 404 Not Found — the path you requested doesn't exist on this server
  • 403 Forbidden — the resource exists, but you don't have permission
  • 500 Internal Server Error — something broke on the server side
⚡

Memory trick: 4xx errors = client's fault (you asked for something wrong or forbidden). 5xx errors = server's fault (it broke). 2xx = success.

Ask the class

"If you see a 404 on a website you built — whose fault is it? What would you check first?" → Check the path in the URL, check your route definitions on the server.

🚀 The Full URL Journey (Reveal)
10:28 – 10:32 · 4 min — use slide 12

Walk through slide 12 step by step — each press of the arrow reveals the next step. Pause after each step and ask "what's happening here?" before revealing the next.

  • ① Cache check — browser has a local copy? Use it, skip DNS
  • ② DNS lookup → get IP address for selise.ch
  • ③ TCP connection on port 443 (HTTPS)
  • ④ TLS handshake — agree on encryption keys
  • ⑤ Browser sends the actual HTTP GET request
  • ⑥ Server finds the requested page files
  • ⑦ Server sends 200 OK + HTML content
  • ⑧ Browser parses HTML, loads linked CSS/JS/images, renders
⚡

Come back to the opener: After this slide, return to the WhatsApp question from the start. Ask: "Now does your original answer make more sense? What would you add?" This closes the loop and rewards the trainees who guessed correctly.

Exercise 1 URL Hunt & DevTools 15 min · 10:30 – 10:45

Trainees work individually. Use slide 14 timer.

  1. Open browser and go to selise.ch. Write the full URL in the notebook.
  2. Break it into parts: protocol, domain, TLD, path. Label each one.
  3. Open DevTools (F12 Windows / Cmd+Option+I Mac). Go to the Network tab.
  4. Refresh the page. Count requests. Find the status code of the first request.
  5. Navigate to 2 more pages. Notice how the URL path changes each time.
Watch for: Trainees who can't find DevTools — help them quickly. The DevTools experience will be useful for the entire program. Common issue: DevTools opens undocked — show them how to dock it to the side or bottom.

Debrief (2 min): Ask 2–3 trainees to share their answers. Target: most pages return 200. Some resources may return 304 (Not Modified — cached). Point out how many sub-requests a single page makes (images, fonts, CSS files).

Class 05 · 11:15 – 13:30

APIs, Data & Databases

The focus is on understanding what APIs are conceptually, what JSON looks like, and how data is stored. Trainees should leave able to read a REST endpoint URL and understand what it does, read basic JSON, and describe the difference between a database table and a document store.

Explain what an API is and why it exists Read and interpret a JSON response Identify the 4 REST HTTP methods Navigate a live REST API in the browser Describe tables, rows, and columns Compare SQL and NoSQL databases Trace data flow through a full application
🍽️ Opening Discussion
11:15 – 11:18 · 3 min
Ask the class

"When you order food at a restaurant, you don't walk into the kitchen yourself — why not? Who do you talk to instead?"

The answer (the waiter) sets up the entire API analogy. The kitchen doesn't want random people walking in. The waiter is the interface — takes your order in a standard way, brings back what you asked for.

🔌 What is an API?
11:18 – 11:30 · 12 min
Key teaching points
  • API = Application Programming Interface — the contract between two pieces of software
  • The client (browser, app) never touches the database directly. The API sits between them, enforcing rules
  • APIs let different systems talk: your weather app calls a weather service API; you log in with Google via an OAuth API; Claude Code calls Anthropic's API
  • Walk through the restaurant diagram on slide 18: You → Waiter (API) → Kitchen (Server) → Pantry (Database)
⚡

Real example for Bhutan: When you use the Bhutan Payment Gateway for an online purchase, the website calls the payment API. The API talks to the bank. You never get direct access to the bank's database — the API is the only way in.

Ask the class

"Can you name an app on your phone that you think uses an API?" → Almost every app. Weather, maps, news, social media, ride-hailing. Everything is APIs.

📄 JSON — How APIs Send Data
11:30 – 11:45 · 15 min
Key teaching points
  • JSON = JavaScript Object Notation. The most common format for API responses (and request bodies)
  • Key-value pairs: the key is always a string in double quotes, the value can be a string, number, boolean, array, or another object
  • Walk through the JSON on slide 19 — point to each type with colour coding
  • Arrays use [ ] and can hold multiple values. Objects use { } and group related fields
"name": "Karma Dorji" ← string (text, needs quotes)
"age": 24 ← number (no quotes)
"enrolled": true ← boolean (true/false, no quotes)
"skills": ["Python", "SQL"] ← array (list of values)
⚡

Common mistake trainees make: Forgetting quotes around keys or string values. JSON is strict — missing a comma or quote breaks the whole thing. This is why parsers exist.

⚡

Practical point: In most of your jobs, you'll be reading JSON from APIs or writing JSON to send to APIs. Understanding this format is foundational — even more important than knowing specific programming syntax.

🔗 REST APIs & HTTP Methods
11:45 – 12:10 · 25 min
REST — the most common API design style
  • REST uses URLs to describe resources (things) and HTTP methods to describe actions (what to do)
  • Walk through the endpoint examples on slide 20 — the URL structure is predictable and self-documenting
  • /trainees → all trainees · /trainees/42 → trainee with ID 42
HTTP Methods — use slide 21 table
  • GET — read data. Safe: doesn't change anything. Most common.
  • POST — create new data. Sends data in the request body.
  • PUT — replace/update existing data. Also uses request body.
  • DELETE — remove data. Usually just a URL + DELETE method.
⚡

CRUD mapping: Every application ever built does Create, Read, Update, Delete. REST maps these 1:1 to POST, GET, PUT, DELETE. This is the universal vocabulary.

Ask the class

"Instagram lets you post a photo, view a feed, edit a caption, and delete a post. Which HTTP method is each of those?" → POST, GET, PUT, DELETE. Perfect 1:1 mapping.

🧠 Quick Quiz 2
12:08 – 12:10 · 2 min — slide 22

Show slide 22. Correct answer is C — GET. If trainees choose POST: explain POST is for creating new data, not reading. GET is always for reading.

Exercise 2 Try a Live API 30 min · 12:10 – 12:40

Trainees open browser and explore a real public API. No login or Postman needed — just a browser address bar. Use slide 23 timer.

  1. Go to jsonplaceholder.typicode.com/users — this is a real mock REST API
  2. Read the JSON. How many users? What fields does each user have?
  3. Change URL to /users/1 — one user vs all users
  4. Explore /posts and /users/1/posts
  5. Bonus: /users?username=Bret — query parameter filtering
Watch for: Trainees who struggle to read the raw JSON. This is expected — it's dense. Show them how to install a JSON formatter browser extension (JSONVue / JSON Viewer) to make it readable, or point them to the browser's pretty-print button if available. The goal is pattern recognition, not perfection.

Debrief (5 min): Ask: "What's the URL pattern for getting all posts vs one post? What would you guess /comments returns?" This tests whether they've internalised the REST pattern.

🗄️ Databases
12:45 – 13:05 · 20 min
Key teaching points
  • A database is organised, persistent storage for data. Unlike files, databases are optimised for searching, filtering, and relating data
  • SQL (relational) databases: data in tables with fixed columns. Rows = records. Every row in a table has the same columns.
  • Use slide 24's table visual — the trainee table. Point out: id = primary key (unique, never repeats), name, city, vertical, cohort
  • NoSQL: stores JSON documents. No fixed schema — different records can have different fields. Great for speed and flexibility.
⚡

SQL vs NoSQL — when to use which: SQL when data is structured and relations matter (users, orders, invoices). NoSQL when data is varied or high-volume (logs, user sessions, product catalogues with varying attributes). Most real systems use both.

⚡

Don't go deep on SQL syntax — that's for the specialisation phase. Today is just "this is what a database is and what problem it solves." Dev vertical will learn PostgreSQL. Testing will learn to query it for verification.

🔄 Full App Data Flow
13:05 – 13:10 · 5 min — slide 25

Walk through the data flow diagram. This is the "big picture" slide — how everything they learned today connects into one picture.

  • Browser → HTTP request → API → Server logic → SQL query → Database → returns data → API → JSON response → Browser renders it
  • Emphasise: every app you've ever used follows this pattern. Instagram, WhatsApp, banking apps — same architecture.
  • Their job in different verticals: Dev builds the API & server. Design creates the browser UI. QA tests each layer. BA writes the requirements that define what the API should do.
Exercise 3 Design a Database Schema 20 min · 13:10 – 13:30

Pairs or groups of 3. Paper & pen preferred — drawing tables by hand reinforces the concept. Use slide 26 timer.

  1. Design the Pelsung programme database with 3 tables: Trainees, Classes, Attendance
  2. For each table: list all columns needed. Mark the primary key (unique ID).
  3. How does the Attendance table link to both Trainees and Classes? (hint: foreign keys)
  4. Draw the tables on paper. Show arrows between linked columns.
Expected answers: Trainees(id, name, city, cohort, vertical) · Classes(id, title, date, phase, duration_min) · Attendance(id, trainee_id, class_id, attended). The Attendance table uses trainee_id and class_id as foreign keys pointing to the other tables — this is the core concept of relational databases.

Debrief (3 min): Draw one solution on the whiteboard. Ask: "Why do we have a separate Attendance table instead of just adding 'attended_classes' to Trainees?" → Because one trainee attends many classes and one class has many trainees. Many-to-many relationships need a junction table.

Use the last 5 minutes before the wrap-up slide to reinforce the key message and connect the two sessions.

Key messages to reinforce

📄"JSON is the universal language of APIs. You will read and write it every single day as a developer."
🗄️"Databases sit at the bottom of every app — understanding what's in them, and how to get data in and out, is the foundation of everything you'll build."
