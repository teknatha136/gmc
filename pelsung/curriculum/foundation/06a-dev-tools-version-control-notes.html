Section 1

VS Code — Your Workspace

Visual Studio Code (VS Code) is a free code editor made by Microsoft. It's the most-used editor among developers worldwide, and it's what you'll use every day in this programme.

The Editor

What is VS Code?

At its core, VS Code is just a text editor — you open files and type code into them. But it's designed for programmers, so it adds a lot of helpful features on top: syntax highlighting (colour-coded code), auto-complete, error warnings, and a huge library of extensions.

Three panels you'll see all day

Left: file explorer — every file in your project.
Middle: the editor — where you actually type.
Bottom: the integrated terminal — run commands without leaving VS Code.

  • Free and open-source — nothing to pay for, ever
  • Runs on Windows, macOS, and Linux the same way
  • Built-in Git support — you can see what changed, commit, and push with buttons instead of commands
  • Integrated terminal means you don't need a separate terminal app
Extensions

Extensions — Power-Ups for VS Code

Extensions are small add-ons that teach VS Code new tricks. You install them from the Extensions panel (square icon on the left sidebar). There are tens of thousands of free extensions.

🎨 Language support

Adds syntax highlighting, auto-complete, and error checking for specific languages (e.g. Python, Java, Go).

🌈 Themes

Change how the editor looks — dark themes, light themes, colour schemes. Picks what's comfortable for your eyes.

🧹 Linters & formatters

Tools that check for bugs or auto-format your code. Prettier and ESLint are the most common for web projects.

🤖 AI assistants

Claude Code and GitHub Copilot live here. They read your code and help you write, fix, or understand it.

AI in this programme

Claude Code is the primary AI tool for Pelsung. It can read your entire project, write new code, edit existing files, and explain what's happening. You'll start using it from Class 03 onward and it'll be with you through to the capstone.

GitHub Copilot is complementary — it offers line-by-line autocomplete as you type. Many developers use both.

Section 2

Git — Version Control

Git is the tool developers use to track changes to their code and work together without stepping on each other's work. Every professional team on earth uses Git.

What It Is

Version Control, Explained

Imagine you're writing a report and save many versions: report.docx, report_v2.docx, report_final.docx, report_final_REAL.docx. It works — sort of — but it's chaotic, and it breaks down completely when three people are editing the same file at the same time.

Git solves this. It keeps an invisible history of every change ever made to every file, lets you label each change with a message, and lets multiple people work in parallel without losing anyone's work.

The Google Docs analogy

Think of Google Docs' "version history" — you can always go back to how the document looked yesterday, or last week. Git is the same idea, but far more powerful, and for code instead of documents. It works offline too — your history lives right on your laptop.

Three Core Ideas

Commits, Branches, Merges

Three words you'll hear every day. Learn these and you understand 85% of what Git is doing.

  • Commit — a snapshot of your project at a specific moment. Each commit has a short message ("Add login page") and records exactly what changed. You can always go back to any previous commit.
  • Branch — a parallel line of work. You can experiment on a branch without affecting the main version. Teams usually have a main branch for stable code, and short-lived branches for each new feature or bug fix.
  • Merge — combining two branches. When your feature branch is done and reviewed, you merge it into main, and your work joins the team's shared history.
Where Your Code Lives

Local vs. Remote

There are always (at least) two copies of your project: the one on your laptop (local) and the one on GitHub (remote). Git commands either change the local copy, or sync the two.

💻 Local (your laptop)

git add — stage files
git commit — save a snapshot
git branch — new line of work

☁️ Remote (GitHub)

git clone — copy a repo down
git push — send commits up
git pull — get others' commits

Everyday Commands

A Typical Git Session

This is what a normal day using Git looks like. VS Code's Git panel does all this with buttons — but knowing the commands helps when things go wrong.

terminal
# Get the latest from GitHub before you start
$ git pull

# See what files you've changed
$ git status

# Stage the files you want in the next commit
$ git add index.html style.css

# Commit with a message describing the change
$ git commit -m "Add homepage heading"

# Push to GitHub so teammates can see it
$ git push
Commit message tip

Write commit messages in the imperative — "Add login page", "Fix typo in header", "Update README". Short, specific, present tense. Future-you (reading the history in six months) will thank present-you.

Section 3

GitHub — Where Code Lives

Git runs on your laptop. GitHub is the cloud service that hosts remote copies of Git repositories, and adds a huge set of tools for working with other people.

Cloud Home for Code

What GitHub Adds to Git

GitHub is owned by Microsoft and used by over 100 million developers. Almost every open-source project in the world lives on GitHub — along with most companies' private projects.

  • Hosted repositories — your code is safely stored in the cloud, not only on your laptop
  • Collaboration — teammates can clone, push, and pull from the same repo no matter where they are
  • Pull requests — the standard way to propose and review changes
  • Issues — a built-in bug & task tracker
  • Projects — Kanban-style boards for planning work
  • Actions — automation that runs tests and deploys code when things are pushed
The Standard Flow

From Idea to Merged Code

This is the workflow every team at SELISE follows, and most companies in the world as well. You'll do this dozens of times during the programme.

📥 Clone
→
🌱 Branch
→
✏️ Commit
→
📤 Push
→
👀 PR + Review
→
✅ Merge
  • Clone — copy the repo from GitHub to your laptop (once, the first time you work on a project)
  • Branch — create a branch for your task, e.g. add-login-button
  • Commit — make changes in VS Code; save snapshots as you go
  • Push — send your commits to GitHub
  • PR + Review — open a pull request; a teammate reviews your changes, suggests improvements, and approves
  • Merge — your branch is merged into main; the work is now part of the shared project
Code Review

Pull Requests — A Conversation About Code

A Pull Request (PR) is how you say to your team: "I've made these changes. Please have a look before they go into the main project." It's less about the code itself and more about the conversation around it.

A good PR has

A clear title ("Add trainee search to dashboard"), a short description of what changed and why, screenshots if it's a UI change, and a link to the Issue it solves. Small PRs are easier to review than big ones.

  • Reviewers comment line-by-line, asking questions or requesting changes
  • You address the feedback by pushing more commits — the PR automatically updates
  • When the reviewer approves, the PR can be merged — often automated tests must pass first
  • No code goes to production without a PR. This is how SELISE maintains quality across Bhutan, Zurich, and everywhere else the team works
Section 4

GitHub Projects

Code is only half the story. Teams also need to plan what to build, track bugs, and know who's working on what. GitHub Projects handles that — all in the same place where your code lives.

Tasks as Issues

Issues — Everything Starts Here

An Issue is any single unit of work — a bug to fix, a feature to build, a question to answer, a piece of documentation to write. Think of it like a digital sticky note that stays around until the work is done.

  • Every issue has a title, description, and optionally a person assigned to it
  • Labels categorise issues (bug, feature, docs, etc.)
  • Discussion happens in the comments — anyone on the team can chime in
  • When you link a PR to an Issue with words like Closes #42, the Issue auto-closes when the PR is merged
Kanban Boards

Projects — Your Team's Board

A Project is a Kanban board showing every Issue in your team, grouped by status. The most common columns are To Do, In Progress, and Done. Issues move from left to right as work progresses.

The full loop

Create an Issue → assign it → it appears in To Do → someone opens a branch + PR → Issue moves to In Progress → PR is reviewed & merged → Issue auto-closes → card moves to Done.

You now have a clear, honest picture of what the team is doing, what's blocked, and what's ready to ship.

Class 06a — Key Takeaways

  • VS Code is a free editor from Microsoft with a file explorer, built-in terminal, Git panel, and thousands of extensions — including Claude Code and GitHub Copilot for AI assistance.
  • Git is version control — it tracks every change to your code, lets you go back in time, and lets multiple people work in parallel without overwriting each other.
  • Commit, branch, merge are the three ideas you'll use every day. A commit is a snapshot; a branch is a parallel line of work; merging combines branches together.
  • Local vs remote — your laptop has one copy, GitHub has the shared copy. push and pull keep them in sync.
  • GitHub hosts your Git repos in the cloud and adds collaboration tools: pull requests for review, issues for tracking work, projects for planning.
  • A pull request is a conversation — no code goes to production without one being reviewed and approved. This is how teams anywhere in the world maintain quality together.
  • GitHub Projects is a Kanban board over your Issues. Link PRs to Issues and cards move to Done automatically when the PR is merged.

✍ Practice at Home

  1. If you haven't already — create a GitHub account at github.com using your real name. Install VS Code from code.visualstudio.com and sign in with GitHub.
  2. In VS Code, open the Extensions panel (square icon on the left sidebar) and install one each of: a language extension (e.g. Python or Live Server), a theme you like, and Prettier.
  3. Clone the starter repo we used in class again at home. Pull the latest changes with git pull. See if your classmates' PRs have been merged in.
  4. Make a small change to a new file (anything — a poem, a list of your favourite songs, a paragraph about Gelephu). Stage it, commit with a message, and push it on a new branch.
  5. Open a Pull Request for your change, write a short description, and wait for a review. When merged, refresh GitHub and see your work in the main branch.
  6. Explore the Issues tab of any public repo on GitHub (e.g. facebook/react or microsoft/vscode). See how real teams discuss bugs and features. Notice the labels, the back-and-forth, the linked PRs.
