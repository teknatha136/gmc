Part 1

Why this class matters

Imagine two different Monday mornings for a developer working on mBoB.

❌ Request A

"Users are complaining about bill payments. Make it better. Also add some favourites thing. Make it nice. Thanks."

✅ Request B

"62% of users pay the same 3 billers each month, re-typing 12-digit account numbers. Let them save billers as favourites — show on home, pre-fill on tap, MPIN still required, max 10."

Both ask for the same feature. Only one lets the developer start coding — or lets an AI build a useful first draft.

The big idea today

A great product starts with a sharp problem, not a feature. A dev-ready spec is the bridge between your thinking and AI's speed. Learn to write one, and you can ship 10× faster than someone who can't.

Part 2

The product thinking funnel

Every great product passes through five stages — in this order. Teams that skip the first three end up building features nobody uses.

1. Problem

What hurts? Whose day gets worse without this?

2. Users

Who exactly has this problem? Name real people — not "everyone".

3. Solution

What approach fits their reality? What's the simplest version?

4. Features

Which concrete pieces (screens, buttons, behaviours) deliver the solution?

5. Spec

A dev-ready document. The bridge to AI and engineers.

Worked example: mBoB favourites

  1. Problem: Tashi pays the same 3 billers every month and re-types their 12-digit account numbers each time. 36 re-entries a year.
  2. Users: Repeat-biller payers — shopkeepers, parents paying school fees, people on standing utility accounts.
  3. Solution: Let users save billers they've paid before, show them on the Bill Payments home, pre-fill on tap.
  4. Features: (a) "Save to favourites" action, (b) Favourites section on home, (c) Tap-to-prefill, (d) Edit/remove.
  5. Spec: A written document with problem, user story, acceptance criteria, UI notes, data sketch, edges, out-of-scope. (See Part 6.)
Part 3

Users & personas

"Everyone" is not a user. Good products are designed for specific people. If it works for them, it works for millions. If you think about "users" as a blob, you end up with a grey, average feature that fits nobody.

Aum Pema
Trashigang · 58

Big buttons. Dzongkha. Hates typing numbers. Her son showed her mBoB once. She uses it for exactly two things.

Tashi
Thimphu · 34 · shopkeeper

Power user. Pays 30+ bills a month. Wants speed. Will learn shortcuts. Annoyed by extra taps.

Karma
Student abroad · 22

On Wi-Fi. Wants to send money to family fast. Forex and limits matter. Might not have a Bhutan SIM.

Rule of thumb

A feature that works for Aum Pema, Tashi, and Karma is a feature worth building. If it only works for one of them, either narrow your audience (fine) or go back and rethink (more often).

Part 4

Solution & MVP thinking

An MVP (Minimum Viable Product) is not a half-done product. It's the smallest complete thing that proves your idea works.

❌ Too big

"Save favourites, categorise them, share with family, auto-pay, schedule, get reminders, export to Excel…"

Takes 6 months. Most of it won't get used.

✅ Right-sized MVP

"Save a biller as a favourite. See it on the Bill Payments home. Tap it to pay. That's it."

Ships in 2 weeks. If people use it, add more. If not, you saved 5 months.

Test

If you can't describe your MVP in one sentence, it's not an MVP — it's a wish list. Cut until it fits in a sentence.

Part 5

The AI-era hourglass

Traditional software development was a straight line: requirements → design → code → test → ship. AI bends that line into an hourglass.

  • Top (wide): humans doing product thinking — problem framing, user research, writing specs. This part grows in the AI era.
  • Middle (narrow): AI writes code in minutes, not weeks. The typing-speed part of software collapsed.
  • Bottom (wide): humans reviewing, testing, validating. This part also grows — because AI is fast but not always right.

The punchline

AI didn't kill the engineer. It killed the vague requirement. The person who can write a sharp spec is now the most valuable person in the room. The person who can review AI output is the second most valuable. Typing speed matters less. Thinking quality matters more.

This is why this class sits where it does in the Pelsung foundation. Before you touch code, you need to be able to think like a product person — because that's where your leverage is.

Part 6

Anatomy of a dev-ready spec

A dev-ready spec is one where a mid-level frontend or backend engineer can read it and start coding — with no follow-up meeting. Every such spec has these 7 parts:

1. Problem / Why

One paragraph. Whose pain? How big is it?

2. User story

"As a ___, I want ___ so that ___"

3. Acceptance criteria

Given / When / Then. Testable. No ambiguity.

4. UI notes / sketch

Where it lives. A rough wireframe is fine.

5. Data sketch

What fields? What gets stored? Where?

6. Edge cases

Offline? Max reached? Duplicate? Deleted item?

7. Out of scope

What this spec does NOT cover. Prevents scope creep.

Bad spec vs good spec — same feature

❌ The spec that kills projects
Title: Add favourites

Make users able to save favourites.
Put them somewhere in the app.
Should be easy to use.
Make it fast.
✅ The spec that ships
Title: Biller favourites · mBoB · v1

Problem:
  62% of users pay same 3+ billers/month,
  re-typing 12-digit account numbers.

User story:
  As a repeat-biller payer,
  I want to save billers as favourites
  so I can pay in 2 taps.

Acceptance criteria:
  • Given I've just paid a biller,
    When I tap "⭐ Save",
    Then it appears in Favourites.
  • Given I'm on Bill Payments home,
    When I tap a favourite,
    Then amount + account pre-filled;
    MPIN still required to confirm.
  • Max 10 favourites; sorted by last-used.

UI: Favourites row under "Quick Send Money"
    on Bill Payments home. ⭐ button on
    receipt screen after payment.

Data: favourites[] per user in DB —
    {billerId, alias, lastUsedAt, amount?}

Edge cases:
  • Already 10 saved → prompt to remove one
  • Offline at save → queue & sync later
  • Biller deactivated → grey out, show toast
  • Duplicate tap → idempotent (no dupes)

Out of scope (v1):
  - Sharing favourites between users
  - Categories / custom icons
  - Auto-pay from favourite
Part 7

User stories & acceptance criteria

The universal user story template

As a who,

I want what

so that why.

Example: "As a shopkeeper who pays repeat billers, I want to save billers as favourites so I don't re-type account numbers each month."

The "so that" is the most important part

It tells the engineer — and the AI — the real goal. That's what lets them make smart tradeoffs when the "what" meets reality. A user story without a "so that" is a feature request, not a story.

Acceptance criteria · Given / When / Then

Acceptance criteria are testable statements. Each one becomes a QA test case later. The format:

Given  [the starting situation]
When   [the user does a specific action]
Then   [a specific thing happens]

Example:

Given the user has just completed a bill payment,
When they tap the "⭐ Save as favourite" button,
Then the biller appears in their Favourites list within 1 second.

Good acceptance criteria are

  • Specific — anyone reading gets the same picture
  • Testable — QA can verify yes/no
  • Independent — each one stands alone
  • About behaviour, not implementation — "appears in list" not "adds a row to favourites_table"

Edge cases — junior vs senior thinking

Juniors spec the happy path. Seniors spec what happens when things go wrong. For every feature, ask yourself:

  • What if the user is offline?
  • What if a limit is hit (max items, rate limit, daily cap)?
  • What if the user taps twice quickly?
  • What if the underlying data changes (biller deactivated, card expired)?
  • What if the user logs out and back in?
  • What about Dzongkha input / long names / unusual characters?

Every edge case you list in the spec = one less bug in production.

Part 8

The Spec → AI → Code loop

This is how real teams work in 2026:

✍️ Write spec

In Markdown, with the 7 parts.

🤖 Feed to AI

Claude Code reads it.

💻 AI writes code

Minutes, not weeks.

🔍 You review

Does it match the spec?

When output is weird, fix the spec — not the code

Ambiguity in the spec → weird code. The temptation is to patch the code. Resist it. Fix the spec, regenerate, and you end up with a source of truth that's still correct next month when you need to change it.

Treat the spec as the source of truth. The code is just one of its outputs.

Key takeaways

  • Great products come from problems, not feature wish-lists.
  • The funnel: Problem → Users → Solution → Features → Spec.
  • The AI-era hourglass: specs up top, AI in the middle, review at the bottom.
  • A dev-ready spec has 7 parts — learn them, use them, reuse them.
  • User story + Given/When/Then acceptance criteria = testable behaviour.
  • If you can spec it, you can ship it — 10× faster.
