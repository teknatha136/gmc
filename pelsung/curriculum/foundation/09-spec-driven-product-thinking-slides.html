From a fuzzy idea to a buildable feature

Every great product passes through these five stages — in this order.

What hurts? Whose day gets worse without this?

👥

💡
▼

Which concrete pieces deliver that solution?
🧩
▼

A dev-ready document. The bridge to AI & engineers.

Most teams skip stages 1–3 and start at 4. That's why you get mBoB features nobody uses.

07 Stage 1 — Problem

Favourites didn't start as a feature

It started as a problem.

"I pay my BPC electricity bill, my B-Mobile bill, and my school fees every month. Every time I open mBoB I have to type in the 12-digit account number again. I know it by heart by now. Why does the app make me do it?"

— Tashi, shopkeeper, Thimphu. 34 years old. Pays same 3 billers × 12 months = 36 re-entries a year.

  • 🎯 A problem is pain in someone's day — not "we should build X"
  • 🎯 If you can't describe the pain in one sentence, you don't understand it yet
  • 🎯 The sharper the problem, the sharper everything downstream
08 Stage 2 — Users

"Everyone" is not a user

Design for specific people. If it works for them, it works for millions.

Aum Pema
Trashigang · 58
Big buttons. Dzongkha. Hates typing numbers. Her son showed her mBoB once. She uses it for exactly 2 things.
Tashi
Thimphu · 34 · shopkeeper
Power user. Pays 30+ bills a month. Wants speed. Will learn shortcuts. Annoyed by extra taps.
Karma
Student abroad · 22
On Wi-Fi. Wants to send money to family fast. Forex & limits matter. Might not have Bhutan SIM.

A feature that works for Aum Pema, Tashi, and Karma is a feature worth building.

09 Stage 3 — Solution & MVP

The smallest thing that solves it

An MVP (Minimum Viable Product) is not a half-done product. It's the smallest complete thing that proves the idea works.

❌ Too big

"Save favourites, categorise them, share with family, auto-pay, schedule, get reminders, export to Excel…"

Will take 6 months. Most of it won't get used.

✅ MVP

"Save a biller as a favourite. See it on Bill Payments home. Tap it to pay. That's it."

Ships in 2 weeks. If it's used, add more. If not, you saved 5 months.

Rule of thumb: if you can't describe your MVP in one sentence, it's not an MVP — it's a wish list.

10 Stage 4 — Features

From solution to concrete pieces

A feature is a specific, nameable piece of the app. It has a screen, a button, a behaviour.

Solution:

Let users save repeat billers so they don't re-type.

Features that deliver it:

  • "Save to favourites" action on a paid biller
  • Favourites section on Bill Payments home
  • Tap-to-prefill from a favourite
  • Edit / remove favourite

One solution → several features. Each feature → one spec.

16 Bad vs Good Spec

Same feature. Two specs. Very different outcomes.

Feature: "Add biller favourites to mBoB"

❌ The spec that kills projects
Title: Add favourites

Make users able to save favourites.
Put them somewhere in the app.
Should be easy to use.
Make it fast.

Every reviewer will ask the SAME 20 questions. Answers change every meeting. Built feature ≠ what anyone wanted.

✅ The spec that ships
Title: Biller favourites · mBoB · v1

Problem:
  62% of users pay same 3+ billers/month,
  re-typing 12-digit account numbers.

User story:
  As a repeat-biller payer,
  I want to save billers as favourites
  so I can pay in 2 taps.

Acceptance criteria (Given/When/Then):
  • Given I've just paid a biller, When I tap
    "⭐ Save", Then it appears in Favourites.
  • Given I'm on Bill Payments home, When I
    tap a favourite, Then amount + account
    are pre-filled, MPIN still required.
  • Max 10 favourites; sorted by last-used.

Out of scope (v1):
  - Sharing favourites, categories, icons.
17 The 7 Parts of a Spec

Every dev-ready spec has these 7 parts

1. Problem / Why

One paragraph. Whose pain? How big?

2. User story

"As a ___, I want ___ so that ___"

3. Acceptance criteria

Given / When / Then. Testable. No ambiguity.

4. UI notes / sketch

Where does it live? Rough wireframe works.

5. Data sketch

What fields? What gets stored? Where?

6. Edge cases

Offline? Max reached? Duplicate? Deleted biller?

7. Out of scope

What this spec does NOT cover. Prevents scope creep.

A mid-level FE or BE engineer should read this and start coding — with no follow-up meeting.

18 User Stories

The universal user story template

As a   who  ,

I want   what  

so that   why  .

✅ "As a shopkeeper who pays repeat billers, I want to save billers as favourites so that I don't re-type account numbers each month."

The "so that" is the most important part. It tells the developer — and AI — the real goal, so they can make smart tradeoffs.

19 Acceptance Criteria

Given / When / Then

Acceptance criteria are testable statements. Each one becomes a test case later.

Given the user has just completed a bill payment,
When they tap the "⭐ Save as favourite" button,
Then the biller appears in their Favourites list within 1 second.

Good acceptance criteria are:
✅ Specific — anyone reading gets the same picture
✅ Testable — QA can verify yes/no
✅ Independent — each one stands alone
✅ About behaviour, not implementation ("appears in list" — not "adds a row to favourites_table")

20 Edge Cases

The difference between junior and senior thinking

Juniors spec the happy path. Seniors spec what happens when things go wrong.

Happy path

User taps ⭐ → biller saved → appears on home.

Edge cases to list

  • User already has 10 favourites (max)
  • User is offline when tapping save
  • Biller is deactivated by the bank later
  • User taps save twice quickly (duplicates?)
  • User logs out and back in — still there?
  • Dzongkha biller name — rendering?

Every edge case you list in the spec = one less bug in production.
