🎤 Opening Discussion
· 5 min
Ask the class

"You open WhatsApp and send a message to your friend in Thimphu. What happens in that 0.3 seconds?"

Take 4–5 answers. Don't correct them yet — write key words on the board (if available). You'll revisit these at the end and confirm who was right.

⚡

Trainer tip: This sets up curiosity. Most trainees will say "it goes through the internet" — which is fine. The goal is to surface what they already know before filling in the gaps.

🌍 The Internet & IP Addresses
· 15 min
Key teaching points
  • The internet is a global network of computers connected by physical cables, fibre, satellites, and routers
  • Every device has an IP address — like a postal address but for computers. Without it, packets of data don't know where to go
  • IPv4: four numbers 0–255 separated by dots (192.168.1.1). Only ~4 billion combinations → IPv6 exists to solve this
  • Bhutan angle: Gelephu connects via fibre from India; Druknet is the national ISP. The same undersea cables that connect continents are why your WhatsApp message reaches London in milliseconds
⚡

Analogy to use: "Your house has a physical address so the postman can find you. Your laptop has an IP address so the internet can find it. Same idea, different scale."

Quick check

"If every device needs an IP address, and there are 8 billion people on Earth — do we have enough IPv4 addresses?" → No! That's why IPv6 was invented.

📋 DNS — Domain Name System
· 10 min
Key teaching points
  • Humans remember names (selisegroup.com). Computers need numbers (185.34.33.33). DNS is the system that translates between them
  • When you type a domain: browser asks DNS resolver → resolver looks up hierarchy → returns IP → browser connects
  • DNS lookup happens in under 100ms. Results are cached so you don't pay the cost every time
  • If DNS is down → websites are unreachable even though the servers are fine (DNS outage != internet down)
⚡

Analogy: "DNS is like the phonebook. You know your friend's name, not their phone number. The phonebook looks it up for you. DNS does the same — you know 'google.com', DNS finds '142.250.x.x'."

⚡

Common confusion: Trainees often think the domain name IS the address. Clarify: it's just a human-friendly label. The real address is always an IP number.

🔗 URLs & Client-Server
· 15 min
URL anatomy (go through this slowly)
https://selisegroup.com/careers?lang=en
https → protocol (use HTTPS, not HTTP)
selise → domain name
.ch → TLD — top-level domain (Switzerland)
/careers→ path — which page on the server
?lang=en→ query string — extra parameters
Client-Server model
  • Client = the one asking (browser, mobile app, Claude Code, Postman)
  • Server = the one answering, runs 24/7, can handle thousands of requests simultaneously
  • Use the diagram on slide 9: You in Gelephu (client) → Internet → selisegroup.com server in Zürich (server)
⚡

Make it tangible: Ask a trainee to open selisegroup.com on their phone and navigate to a few pages. Point out how the URL changes. Every different URL = different request to the same server.

🔒 HTTP, HTTPS & Status Codes
· 15 min
HTTP
  • HyperText Transfer Protocol — the rulebook for how browsers and servers communicate
  • HTTP sends data in plain text — anyone watching the network can read it (like a postcard)
  • HTTPS = HTTP + TLS encryption — data is scrambled, only client and server can read it (like a sealed envelope)
  • Always check for the padlock before typing passwords or payment info
Status codes — the server's reply code
  • 200 OK — success, here's your data
  • 301 Moved Permanently — page is at a new URL, follow this redirect
  • 404 Not Found — the path you requested doesn't exist on this server
  • 403 Forbidden — the resource exists, but you don't have permission
  • 500 Internal Server Error — something broke on the server side
⚡

Memory trick: 4xx errors = client's fault (you asked for something wrong or forbidden). 5xx errors = server's fault (it broke). 2xx = success.

Ask the class

"If you see a 404 on a website you built — whose fault is it? What would you check first?" → Check the path in the URL, check your route definitions on the server.

🚀 The Full URL Journey (Reveal)
· 4 min — use slide 12

Walk through slide 12 step by step — each press of the arrow reveals the next step. Pause after each step and ask "what's happening here?" before revealing the next.

  • ① Cache check — browser has a local copy? Use it, skip DNS
  • ② DNS lookup → get IP address for selisegroup.com
  • ③ TCP connection on port 443 (HTTPS)
  • ④ TLS handshake — agree on encryption keys
  • ⑤ Browser sends the actual HTTP GET request
  • ⑥ Server finds the requested page files
  • ⑦ Server sends 200 OK + HTML content
  • ⑧ Browser parses HTML, loads linked CSS/JS/images, renders
⚡

Come back to the opener: After this slide, return to the WhatsApp question from the start. Ask: "Now does your original answer make more sense? What would you add?" This closes the loop and rewards the trainees who guessed correctly.

Exercise 1 URL Hunt & DevTools 15 min ·

Trainees work individually. Use slide 14 timer.

  1. Open browser and go to selisegroup.com. Write the full URL in the notebook.
  2. Break it into parts: protocol, domain, TLD, path. Label each one.
  3. Open DevTools (F12 Windows / Cmd+Option+I Mac). Go to the Network tab.
  4. Refresh the page. Count requests. Find the status code of the first request.
  5. Navigate to 2 more pages. Notice how the URL path changes each time.
Watch for: Trainees who can't find DevTools — help them quickly. The DevTools experience will be useful for the entire program. Common issue: DevTools opens undocked — show them how to dock it to the side or bottom.

Debrief (2 min): Ask 2–3 trainees to share their answers. Target: most pages return 200. Some resources may return 304 (Not Modified — cached). Point out how many sub-requests a single page makes (images, fonts, CSS files).
