How the Internet Works

The internet is not magic — it's infrastructure. Once you understand how it works, you can debug problems, read error messages, and build things that work reliably.

Foundation

The Internet & IP Addresses

The internet is a global network of billions of computers that are all connected together. Not wirelessly through the air — physically, through underground and undersea cables, via satellites, and through large data centres in cities around the world.

For two computers to communicate, they each need an address. That address is called an IP address (Internet Protocol address).

Analogy

Think of your house address. The postman can't deliver a letter to you without knowing where you live. In the same way, the internet can't deliver data to your device without knowing your IP address.

IPv4 addresses look like this: 192.168.1.1 — four numbers, each between 0 and 255, separated by dots. There are about 4 billion possible IPv4 addresses, which is no longer enough for all the devices in the world. IPv6 was invented to solve this problem.

  • Your laptop, phone, and the selisegroup.com server all have IP addresses
  • Public IP: visible to the whole internet (your router's address)
  • Private IP: only visible on your local network (e.g., 192.168.x.x)
Domain Name System

DNS — The Internet's Phonebook

You remember names — like selisegroup.com or google.com. But computers connect using numbers — IP addresses. DNS is the system that translates between the two.

When you type a domain name in your browser:

  • Your browser first checks its local cache — has it looked this up recently?
  • If not, it asks a DNS resolver (usually provided by your ISP or Google/Cloudflare)
  • The resolver looks up the domain name and returns the matching IP address
  • Your browser then connects to that IP address
Key fact

A DNS lookup takes less than 100 milliseconds — faster than you can blink. And once your browser has looked it up, it caches the result so it doesn't have to do it again for a while.

If DNS breaks (which happens occasionally), websites appear to be offline even though the servers are perfectly fine — your browser just can't find the address.

URLs

Anatomy of a URL

Every URL (Uniform Resource Locator) is an address that points to a specific resource on the internet. Each part tells the browser something different.

httpsProtocol
://
seliseDomain
.chTLD
/careersPath
?lang=enQuery
  • Protocol — https: Use encrypted HTTPS (never plain HTTP for sensitive data)
  • Domain — selise: The registered name of the organisation
  • TLD — .ch: Top-level domain. .ch = Switzerland, .bt = Bhutan, .com = commercial
  • Path — /careers: Which specific page or file on the server
  • Query string — ?lang=en: Extra parameters sent to the server (like a filter)
Protocols & Status Codes

HTTP, HTTPS & Status Codes

HTTP (HyperText Transfer Protocol) is the rulebook for how browsers and servers communicate. It defines how requests are formatted, how responses are structured, and what the different types of requests mean.

HTTPS is HTTP with encryption added (using TLS — Transport Layer Security). All data sent between your browser and the server is scrambled so that nobody else on the network can read it.

Practical rule

Always check for the padlock icon in the browser address bar before typing a password or card number. No padlock = no HTTPS = your data is travelling in plain text.

Status codes are 3-digit numbers the server sends back with every response. They tell you instantly whether the request succeeded or failed:

Memory shortcut: 4xx = your fault (bad request) · 5xx = server's fault · 2xx = success

Class 04 — Key Takeaways

  • Every device has an IP address — the internet uses these numbers to route data to the right place
  • DNS translates names to numbers — you type selisegroup.com, DNS gives your browser the IP address to connect to
  • A URL has 5 parts — protocol, domain, TLD, path, and optional query string. Each part tells the browser something different
  • HTTPS is mandatory for sensitive data — look for the padlock. HTTP sends data in plain text; HTTPS encrypts it
  • Status codes summarise every response — 200 = success, 404 = not found, 500 = server broke
  • Every web request travels: Browser → Internet → Server → back — DNS, TCP connection, HTTP request, server processing, HTTP response, browser rendering
