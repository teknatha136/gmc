06 Walkthrough Example

Running Example: mBoB (Mobile Banking)

The app you use to send money, pay bills, recharge your phone — how did Bank of Bhutan build it?

1. Plan

  • Who: BoB customers with smartphones
  • Problem: Long queues at branches for basic transactions
  • Goal: Let users send money & pay bills from phone

2. Analysis

  • User stories: login, transfer, pay bills, check balance
  • Acceptance criteria per story
  • Data: accounts, transactions, beneficiaries

3. Design

  • Screens: Login, Home, Transfer, Bills, Profile
  • Flow: Login → PIN → Action → Confirm → Receipt
  • Architecture: Mobile + Secure API + Core Banking

4. Build

  • Frontend: Android + iOS native apps
  • Backend: Secure API + core banking integration
  • Integrations: SMS OTP, biometric login, BFS/RMA

06 Walkthrough (cont.)

mBoB · Stages 5–7

5. Test

  • Happy path: login, transfer Nu. 100, get receipt
  • Edge cases: wrong PIN 3x, network drop mid-transfer
  • Security: can someone steal your PIN? SSL valid?

6. Deploy

  • Soft launch: BoB staff first
  • Roll-out: selected branches, then all customers
  • Published on Play Store & App Store

7. Maintain

  • 24/7 uptime, fraud monitoring, patches
  • New features: QR pay, scheduled transfers
  • Bug fixes, customer support tickets

Maintenance often takes more time than everything else combined — a product's real life begins after launch.

08 The Big Three

Waterfall · Agile · Scrum

💧 Waterfall

Finish each stage before the next. One big plan up front.

  • Predictable timeline
  • Hard to change mid-way
  • Use for: fixed scope, legal/regulated projects

⚡ Agile

Build in small chunks. Get feedback. Adjust. Repeat.

  • Welcomes change
  • Needs active customer
  • Use for: most modern products

🏃 Scrum

A flavor of Agile: 2-week sprints, standups, retros, clear roles.

  • Steady rhythm — you know what happens each week
  • Lots of meetings (the "ceremonies")
  • Use for: stable teams, clear backlog

Scrum is not equal to Agile. Agile is the philosophy. Scrum is one way to do it.

09 Other Methodologies

You'll also hear about...

Definitions only — don't memorise. Know they exist.

Kanban

Visual board of "To Do → Doing → Done" with a limit on work-in-progress.

Use for: support teams, continuous ticket flow
Extreme Programming (XP)

Pair programming, test-first, frequent releases. Very disciplined.

Use for: critical systems where quality is non-negotiable
Lean

Eliminate waste. Build only what the user needs. Measure everything.

Use for: startups validating an idea with minimum effort
Spiral

Repeat Plan→Build→Evaluate in widening spirals. Risk-driven.

Use for: large, high-risk systems (aerospace, medical devices)
RAD (Rapid Application Development)

Build prototypes fast with the user in the room. Refine until they approve.

Use for: internal tools, demos, MVPs
DevOps

Developers + Operations as one team. Automate build, test, deploy.

Use for: products with frequent releases (every day or hour)
Scrumban

Scrum's ceremonies + Kanban's flow. Sprints without strict commitments.

Use for: teams moving from Scrum → continuous delivery
SAFe

Scaled Agile Framework — Scrum for huge organisations (100s of people).

Use for: enterprises coordinating many teams on one product

03
12 The Hourglass Model
AI Era

Specs expand. Implementation shrinks. Review expands.

Traditional

Plan / Spec
Design
Build (humans type every line)
Test / Review
Deploy

AI Era (Hourglass)

Plan / Spec (bigger, clearer)
Design (clearer handoffs)
Build (AI drafts, humans steer)
Test / Review (bigger, more critical)
Deploy (automated)

AI is fast at typing code. It's bad at deciding what to build. So the work shifts to the ends.

13 What Actually Changes

The AI Era Developer

✅ Spend more time on

  • Writing clear specs & user stories
  • Reading & reviewing AI-generated code
  • Testing edge cases
  • Architectural decisions
  • Understanding the user's real problem

❌ Spend less time on

  • Boilerplate code
  • Looking up syntax on Stack Overflow
  • Writing the same CRUD for the 10th time
  • Manual test data generation
  • Formatting & refactoring by hand

Key insight: AI is a power tool, not a replacement. A power drill doesn't replace the carpenter — it lets one carpenter do the work of three. Your taste & judgement become more valuable, not less.
