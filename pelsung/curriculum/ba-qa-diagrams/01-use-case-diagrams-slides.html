02 What is a use case diagram?

The scope-in-one-picture diagram

  • Part of UML — invented by Ivar Jacobson in 1986, absorbed into UML in the late 1990s. Still in daily use 35 years later.
  • Shows the system's outside — who uses it and what each user can do. It does not show flow, sequence, or internal detail.
  • High-signal, low-detail — deliberately a summary. The detail goes in use case descriptions, not the diagram.
  • The "scoping conversation" tool — drawing one in front of stakeholders is the fastest way to surface "you forgot X" or "that's out of scope".
  • Every use case becomes downstream work — a user story, a feature spec, an acceptance test. The diagram is the index.

If you can't draw the use case diagram for your system on one page, you don't yet understand the scope.

03
Why bother

Five things it tells you in 30 seconds

  • 👥 Who uses the system — every actor, named
  • 🎯 What each actor can do — every goal, listed
  • 🧱 What's in scope vs. out — the system boundary makes it explicit
  • 🔌 Which external systems integrate — payment gateway, SMS provider, core banking
  • 🕳 What's missing — the moment you draw it, gaps become obvious

Worth 1,000 words of scope debate.

04 The four core elements

Just four things — that's the entire vocabulary

Actor

A person or external system. Stick figure. Always sits outside the boundary.

Use case

A goal an actor can achieve. Oval. Always a verb phrase.

System boundary

Labeled rectangle around use cases. Says "this is what's in scope".

Association

Plain line connecting an actor to a use case they perform.

No arrows on associations. No conditions. No ordering. The simplicity is the point.

05 Actors · more than just stick figures

Two distinctions worth knowing

Primary actor

The one who initiates the use case to achieve their own goal. The use case exists to serve them.

e.g. Customer on "Place order" — the customer wants to buy.

Secondary actor

An actor the system uses to fulfill the primary actor's goal. They provide a service.

e.g. Payment Gateway on "Place order" — the gateway is asked to process payment.

Human actor

A real person in a role: Customer, Doctor, Trainee, Admin. Always a role, not a specific person ("Sonam" is wrong; "Customer" is right).

System actor

An external software system. Drawn as a stick figure too, often labeled «system» or as a rectangle box.

e.g. SMS Gateway, Stripe, Google Maps API, mBoB Core Banking.

How to find your actors

  • Walk through every use case and ask: "Who initiates this?" — that's a primary actor.
  • Then ask: "Does the system call out to anyone?" — those are secondary actors.
  • Don't forget time-based actors — if something runs on a schedule (nightly billing, weekly report), add a Timer or Scheduler actor.
06 Use case granularity · the Goldilocks rule

Not too small · not too big · just right

Too small ❌
Click-level
  • "Click login button"
  • "Enter password"
  • "Press submit"
  • "Open menu"

These are interactions, not goals. They belong in a flow diagram or a UI spec, not here.

Just right ✅
Goal-level
  • "Log in"
  • "Place order"
  • "Schedule appointment"
  • "Generate monthly report"

A meaningful goal. Takes minutes, has a clear outcome, fits in a one-page description.

Too big ❌
System-level
  • "Use the system"
  • "Manage everything"
  • "Run the business"
  • "Do banking"

Tells you nothing. Everything fits inside, so no scope is communicated.

Quick test: if a stakeholder asks "what does that include?" and your answer takes more than 30 seconds, the use case is too big. If it takes less than 5 seconds, it's probably too small.

10 Relationships · include · extend · generalization

Three stereotypes — use sparingly

«include»

Always uses another use case as a sub-step. Factor out shared behavior.

Arrow: A → B (A includes B)

e.g. "Transfer funds" «include» "Authenticate". Always.

«extend»

Optionally adds to another use case under a specific condition.

Arrow: B → A (B extends A)

e.g. "Upload medical cert" «extend» "Request leave" — only if reason is illness.

Generalization

A specialized actor / use case inherits from a general one.

Arrow: child → parent (hollow triangle)

e.g. "Trainer" and "Coordinator" both generalize "Staff" — both inherit "View dashboard".

Warning: beginners scatter these everywhere to "look more UML". Don't. If removing the relationship makes the diagram clearer, do that. Most simple systems need none.

11 How to draw one from scratch

A six-step process you can run in 30 minutes

1
Name the system. Draw the boundary box and label it. "Hospital Appointment System", not "the app". This forces clarity early.
2
List actors first. Brainstorm who interacts with the system. Both humans and external systems. Aim wide — you can prune later. Place them around the boundary.
3
For each actor, ask "what goals do they have here?" Each goal becomes a use case (oval). Name with verb + object. Add associations.
4
Walk through scenarios. Pick a real one ("a patient calls to reschedule") and trace it across the diagram. Missing? Add it. Confusing? Rename it.
5
Add «include» / «extend» only if needed. Look for sub-steps shared by 3+ use cases (Authenticate is the classic). Resist the urge to over-decorate.
6
Walk it past stakeholders. Ask "what's missing? what's wrong?" — and watch them point at things. That's the whole point of the diagram.
12 The diagram is an index · descriptions are the spec

Each oval needs a one-page description behind it

USE CASE · UC-03 · BOOK APPOINTMENT

Primary actor: Patient · Secondary actors: SMS Gateway

Goal: Reserve a time slot with a doctor for a future date.

Preconditions: Patient is registered and authenticated. Doctor's calendar exists.

Main flow:

  1. Patient selects a department.
  2. System shows doctors with availability in the next 14 days.
  3. Patient picks a doctor and time slot.
  4. Patient confirms with reason for visit.
  5. System books the slot, generates a reference number.
  6. System sends SMS confirmation via SMS Gateway.

Alternate flows: (a) no slots available → suggest waitlist. (b) SMS gateway down → log for retry, show "confirmation will be sent shortly".

Postconditions: Slot reserved. Patient sees appointment in "My appointments". Reminder scheduled for 24h before.

The diagram says "Patient can Book Appointment" in 3 seconds. The description says "…and here's exactly how" on one page. You need both.

13 When to reach for one

The scoping and alignment moments

✅ Use a use case diagram when…
  • You're scoping a new system or major feature
  • You want stakeholders to validate "everything the system does"
  • You need a one-page summary for a review
  • You're identifying actors before writing user stories
  • You want to show which external systems integrate
  • You're starting a discovery / requirements workshop
⚠️ It's the wrong tool when…
  • You need to show flow — use activity / BPMN / flowchart
  • You're documenting detailed behavior — write a use case description
  • You want to show data structure — use ERD / class diagram
  • You're modeling a process with handoffs — use swimlane or BPMN
  • You're showing data movement — use a DFD
14 Common mistakes

Mostly about treating it as a UI mockup or a spec

  • Use cases as clicks — "Click login button" is not a use case. "Log in" is.
  • Too fine-grained — 80 tiny ovals is a backlog, not a use case diagram. Collapse to user goals.
  • Too coarse — "Use the system" tells you nothing. Aim for meaningful goals.
  • Noun use cases — always verb + object. "Login" okay, "Login page" wrong, "Profile" wrong.
  • Specific people as actors — actor is a role. "Sonam" is wrong, "Customer" is right.
  • Missing system boundary — without the box, the reader can't tell what's in scope.
  • Forgetting external-system actors — payment gateway, SMS provider, core banking are actors too.
  • Over-using «include» / «extend» — most diagrams need none. Reach for them only when they clarify.
  • Treating the diagram as the spec — every oval needs a description behind it.
15 Wrap-up
🎯 Key Takeaways

Use case diagrams · what to carry forward

  • ✅ The scoping diagram — who uses the system and what they can do
  • ✅ Four elements: actor · use case · system boundary · association
  • ✅ Use cases are verb phrases. Not features. Not clicks. Not nouns.
  • ✅ Granularity is goal-level — not too small, not too big
  • ✅ Actors come in flavors: primary / secondary · human / system
  • ✅ External systems are actors too — not just humans
  • ✅ «include» / «extend» exist but are over-used — keep diagrams simple
  • ✅ The diagram is a summary — the descriptions behind each oval are the real spec
