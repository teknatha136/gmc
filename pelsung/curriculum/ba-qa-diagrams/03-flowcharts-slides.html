02
Why flowcharts exist

"Walk me through how this works."

The simplest universal diagram for "if-this-then-that" thinking.
Invented in 1921, standardized by ISO/ANSI in the 1960s, used everywhere from kindergarten classrooms to airline manuals.

Anyone, anywhere, in any field, will read a flowchart correctly the first time.

If you can't draw a flowchart for it, you don't understand it yet.

03 The simplest possible flowchart

Boil water · 4 shapes · you can already read it

A complete, valid flowchart. Nothing missing. The shapes mean exactly what they look like.
  • Oval = start / end. Always.
  • Rectangle = a step. Something gets done.
  • Arrow = then. Direction of flow.

No legend. No vocabulary. No notation cheat-sheet. That's the whole point of flowcharts.

04 Reading any flowchart · the rule

Find Start · follow the arrows · stop at End

  1. Find the oval labeled "Start" — that's where you begin.
  2. Follow the arrow to the next shape and do whatever it says.
  3. If you hit a diamond — answer the question, take the matching exit.
  4. If you hit a parallelogram — read or write data (depending on direction).
  5. Stop when you reach an oval labeled "End".

Trick: if a path doesn't end in End, the chart is broken. Every path must terminate.

A flowchart is not a multi-actor process. It describes what happens, not who does it. (For "who does it" we use swimlanes — next class.)

05 The five shapes

Master these five · cover 95% of real work

Terminator

Start / End. Oval / pill.

Process

A single step. Rectangle.

Decision

Yes / No branch. Diamond.

Input / Output

Data in / out. Parallelogram.

Arrow

Flow direction. Always labeled on decisions.

Extras (subroutine, document, on-page connector) exist. You'll rarely need them. When you reach for shape number 6, ask whether a different diagram type would serve better.

06 Decisions · the diamond rules

Always ask · always label · always merge

  • Always a question. "PIN correct?" "Stock available?" "Age > 18?". Never a verb.
  • Always exactly two paths (Yes / No). For more options, chain multiple diamonds — don't try to fit 5 branches onto one.
  • Both arrows labeled. Reader shouldn't guess which way "yes" goes.
  • Eventually merge. Branches usually rejoin downstream. If they truly never meet, both must reach an End.
Both arrows labeled. Both paths lead somewhere. Reader has zero ambiguity.

10 How to draw one from scratch

A six-step recipe

1
Write the trigger. What makes this start? "Customer presents card" — that's your Start oval. One per chart.
2
List the steps in order. One per rectangle. Verb + object. "Insert card", "Enter PIN".
3
Insert decisions where they belong. Anywhere the path can go differently — diamond. Always a question. Always two arrows out.
4
Trace the failure paths. Each "no" must go somewhere. Either rejoin the main flow or terminate in End. Don't leave dangling arrows.
5
Add I/O parallelograms only if data flow matters. "Read card data" or "Print receipt". Optional — don't force it.
6
Walk through it as the user. Read your own diagram, doing each step. Stuck? Confused? Fix it now — it's about to confuse a stranger.
11 When a flowchart beats other diagrams

Simple wins · until it doesn't

✅ Flowchart is the right tool
  • Algorithms, decisions, validation logic
  • One-page quick reference for a single role
  • Troubleshooting guides ("is it plugged in?")
  • Eligibility / qualification rules
  • Whiteboarding with non-technical people
  • Specifying small piece of software logic
⚠️ Reach for something else when…
  • Multiple actors involved → swimlane or BPMN
  • Parallel work that re-syncs → activity diagram
  • Cross-organization messaging → BPMN
  • You're modeling data movement → DFD
  • Process will be automated by an engine → BPMN
12 Common flowchart mistakes

Easy to draw · easy to draw badly

  • Arrows without arrowheads — direction must always be unambiguous.
  • Unlabeled decision branches — every diamond exit needs Yes / No / specific condition.
  • No Start or no End — every flow starts with a terminator and ends with at least one.
  • Dead-end shapes — a step that leads nowhere = reader is stuck.
  • Loops without exits — "go back and try again" must have a way out (retry count, escape hatch).
  • Mixing shape meanings — if rectangle = process once, rectangle = process always.
  • Three branches from one diamond — chain diamonds instead.
  • "Who does this?" sneaking in — if actor matters, upgrade to swimlane.
13 Wrap-up
🎯 Key Takeaways

Flowcharts · what to carry forward

  • ✅ The oldest, simplest, most universally read diagram in software
  • ✅ Five shapes: oval · rectangle · diamond · parallelogram · arrow
  • ✅ Reading rule: find Start, follow arrows, stop at End
  • ✅ Best for algorithms, decisions, eligibility — not multi-actor processes
  • ✅ Decisions are questions, with both branches labeled
  • ✅ Every path reaches an End. No dangling arrows.
  • ✅ If "who does it" matters — upgrade to swimlane (next class)

When in doubt, sketch a flowchart first. Upgrade only if you actually need more.
