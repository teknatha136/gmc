Part 1

What is a use case diagram

A use case diagram is a UML diagram that shows what a system does from the outside: the actors who interact with the system, the system's boundary, and the goals each actor can achieve. It's a scoping diagram — it answers "who uses this and what can they do?" on one page.

Invented by Ivar Jacobson in 1986 as part of his Objectory methodology, absorbed into UML 1.0 in 1997. It pre-dates agile, but it complements user stories beautifully: each use case in the diagram typically becomes a cluster of user stories.

What it doesn't show

Use case diagrams deliberately don't show: flow, sequence, data, internal structure, business rules, or UI. That's what they're good at — they strip everything away except the answer to "what can users do?"

Part 2

Core elements

Actor

A role that interacts with the system. Drawn as a stick figure. An actor is not a specific person — it's a role. Saying "Sonam uses the app" is wrong; the actor is "Customer" (and Sonam is an instance). Actors can be:

  • Human — customer, admin, manager, doctor, reviewer.
  • External systems — payment gateway, core banking system, sms provider. Drawn as a rectangle with «system» or similar stereotype.
  • Time — if the system does things on a schedule without a human trigger ("Timer" actor, nightly billing).

Use case

A goal an actor can achieve with the system. Drawn as an oval, named with a verb phrase: "Log in", "Transfer funds", "Print statement". Not a feature name and not a click.

Granularity test: a good use case typically takes minutes, has a clear outcome, and can be described in a one-page use case description.

System boundary

A labeled rectangle drawn around all the use cases. Everything inside = in scope. Everything outside = actors or out of scope. Without this box, the diagram is ambiguous — add it always.

Association

A plain line connecting an actor to a use case they can perform. No arrowhead needed unless you want to show initiation direction.

Part 3

Worked example · online banking

Online banking. Three actors (Customer, Bank Admin, Core Banking system), six system use cases, one shared sub-use-case (Authenticate) that "Transfer funds" and "Pay bills" both «include».

Notice several things:

  • The Core Banking system is an actor, not an internal component — it's outside the boundary.
  • "Authenticate" is a shared sub-use-case — drawn inside, connected by «include» from the use cases that always need it.
  • No flow is shown. We don't see that "Log in" leads to "View balance" — that's in the use case descriptions, not the diagram.
Part 4

Relationships · include / extend / generalization

RelationshipMeaningArrow direction
«include»A always uses B as a sub-step (factor out common behavior)A → B (dashed, "A includes B")
«extend»B optionally adds behavior to A, under a specific conditionB → A (dashed, "B extends A")
GeneralizationA specialized actor or use case inherits from a more general onechild → parent (solid line with hollow triangle)

Warning — these are often abused

New UML learners scatter «include» and «extend» everywhere to make their diagram look "proper". Resist. Most simple systems don't need them. Rule of thumb: if removing the relationship doesn't make the diagram clearer, don't add it. A simpler diagram with duplicated behavior is often easier to read than a tidier one with five stereotypes.

Part 5

The use case diagram is not the spec — descriptions are

The diagram alone won't get a developer or QA very far — it just says "Customer can Transfer Funds." The real detail lives in the use case description (sometimes called "use case narrative") behind each oval. A typical template:

Use case · UC-03 · Transfer Funds

Actor(s): Customer (primary), Core Banking (secondary)

Goal: Move money from one customer account to another account (same bank or external).

Preconditions: Customer is authenticated. Source account has sufficient balance.

Main flow:

  1. Customer selects "Transfer".
  2. System displays source-account selector and recent recipients.
  3. Customer chooses source, recipient, amount, note.
  4. System shows confirmation with fee and exchange rate (if applicable).
  5. Customer confirms with MPIN.
  6. System posts the transfer to Core Banking.
  7. System shows success receipt with reference number.

Alternate flows: (a) insufficient balance → show error, return to step 3. (b) MPIN wrong three times → lock session.

Postconditions: Balance reduced. Transaction recorded in statement.

Non-functional: Confirmation shown within 3s. Fully logged for audit.

This description is what a developer and QA actually implement against. The diagram tells you the description exists.

Part 6

When to reach for a use case diagram

✅ Use a use case diagram when…

  • Scoping a new system or major feature
  • Getting stakeholders to validate "everything the system does"
  • Identifying actors before writing user stories
  • Showing external-system integrations
  • Creating a one-page summary for status reviews

⚠️ Don't use it for…

  • Flow / sequence / behavior — use activity, flowchart, BPMN, or sequence diagram
  • Detailed specs — use case descriptions / feature specs do that
  • Data structure — ER diagram / class diagram
  • Process across organizations — swimlane or BPMN
Part 7

Common mistakes

MistakeFix
Use case as a click ("Click login button")Use a user goal ("Log in")
80 tiny use cases on one diagramConsolidate — aim for 5–20 per diagram, group when bigger
Noun use cases ("Login page")Always verb + object ("Log in")
Over-using «include» / «extend»Use them only when they clarify — most diagrams don't need any
No system boundaryAlways draw the rectangle — without it, scope is ambiguous
Forgetting external systems as actorsExternal systems (payment gateway, SMS provider) are actors too
Using a specific person as an actor ("Sonam")Use the role ("Customer", "Admin")
Treating the diagram as the specEach oval needs a use case description underneath — the diagram is an index
Takeaways

Key takeaways

What to carry forward

  • A use case diagram is the one-page scope diagram — who uses the system and what they can do.
  • Four core elements: actor · use case · system boundary · association.
  • Use cases are user goals, written as verb phrases. Not features. Not clicks.
  • External systems are actors too — not just humans.
  • «include» / «extend» exist but are often over-used — keep diagrams simple.
  • The diagram alone is not the spec — each oval needs a use case description.
  • The fastest diagram in the BA toolkit for scoping conversations.

Recommended tools: PlantUML, draw.io, Lucidchart, Visual Paradigm.
