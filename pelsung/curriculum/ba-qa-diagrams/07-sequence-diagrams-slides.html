02
Why sequence diagrams exist

"What does the system actually say to the payment gateway?"

Use Case Diagram says "Customer can Place Order."
Activity Diagram shows the steps inside.
BPMN shows handoffs across departments.

None of them shows the back-and-forth — who calls whom, with what request, in what order, and what comes back.

Sequence diagrams are how you specify a conversation.

03 The simplest possible sequence diagram

Login flow · 3 actors · 5 messages

A complete sequence diagram. Time runs top to bottom. Five labeled messages, in order. You can already read this without any further explanation.
  • Participants at the top — humans drawn as stick figures, systems as boxes, databases as cylinders.
  • Lifelines hang down — dashed vertical lines representing each participant's existence over time.
  • Time goes downward — the higher the message, the earlier it happens.
  • Solid arrow = call (request), dashed arrow = return (response).
  • Blue strip on a lifeline = activation bar — that participant is "busy" handling something.
04 Reading rule · top to bottom · left to right

A simple recipe for any sequence diagram

  1. Start at the top. Read the actor names — that's everyone in the conversation.
  2. Find the first arrow. Always the highest one. That's the trigger.
  3. Read each arrow as a sentence. "Sender calls Receiver with this message."
  4. Solid arrow = call (synchronous request). Dashed arrow = return (response).
  5. Activation bars (the strips) show when each actor is "busy" handling something.
  6. Move down through messages — each one happens after the one above.
  7. Stop at the last arrow. The conversation is over.

Key intuition: a sequence diagram is a screenplay. Each line is a character speaking. The script runs top to bottom.

05 Core elements

Six things · master these and you can read 95%

Actor / Object

Stick figure for human. Box for system / class / service.

Lifeline

Dashed vertical line. "This actor exists for the duration of the diagram."

Activation bar

Thin blue strip on a lifeline. "Actor is busy / executing".

Synchronous message

Solid arrow with closed head. "Call and wait for return."

Asynchronous message

Solid arrow with open V head. "Send and don't wait."

Return message

Dashed arrow with open head. "Response coming back."

In simple BA-facing diagrams you can skip async vs sync — just use solid for calls and dashed for returns. The distinction matters when modeling messaging architectures (queues, events).

06 Combined fragments · alt · opt · loop

Boxed regions for if-else, optional, and repeat

When a sequence has branches or loops, you wrap that part of the diagram in a labeled rectangle:

  • alt — alternative branches (if / else). Two regions separated by a dashed line. One condition wins.
  • opt — optional. Just one region; runs only if the guard condition is true.
  • loop — a region that repeats. Label includes the iteration condition.
  • par — parallel regions running concurrently (rare in BA work).

Don't reach for these on the first sketch. Add them only when the conversation genuinely branches or repeats.

An alt fragment with two branches. Top: balance is enough → withdraw. Bottom (after dashed separator): else → show error.

10 Sequence vs activity vs use case · what each adds

Three diagrams · same scenario · three different lenses

Place an order — three views of the same thing

  • Use Case Diagram says: "Customer can Place Order. Place Order includes Authenticate."
    (Goal-level. Says what's possible.)
  • Activity Diagram says: "User selects items → enters address → confirms payment → if in stock then ship else notify, end."
    (Step-level. Says what happens, with branches.)
  • Sequence Diagram says: "User → Web → OrderService.create() → InventoryService.reserve() → PaymentService.charge() → return success → Email.send()."
    (Message-level. Says who calls whom, in order, with what arguments.)

All three are correct. Each answers a different question. Use the one that matches your audience and what you need to communicate.

11 How to draw one from scratch

Six steps · stay focused on one scenario

1
Pick one scenario. Don't try to cover all cases on one diagram. "Successful login" is one diagram; "failed login due to wrong password" is another (or an alt fragment if both fit).
2
List the participants. Place them across the top, ordered roughly by who calls whom (initiator on the left). Stick figures for humans, boxes for systems / services.
3
Drop their lifelines. Dashed vertical lines hanging down from each participant.
4
Add the first message at the top. Then each next message goes below. Solid arrow with closed head for calls, dashed for returns. Always label the message.
5
Add activation bars (the blue strips) where each participant is "busy". Optional but very helpful for showing nesting.
6
Wrap branches in alt / opt fragments only if the scenario truly branches. Otherwise: one happy-path diagram is much clearer than one "everything possible" diagram.
12 When to use a sequence diagram

When the conversation matters

✅ Use a sequence diagram when…
  • You're specifying an API or integration contract
  • You're realizing a use case ("what messages flow?")
  • QA needs to derive integration test scenarios
  • You're designing or reviewing a microservice interaction
  • Order of events matters — and you want it explicit
  • Async vs sync messaging needs to be visible
⚠️ Don't use one for…
  • High-level scope — use case diagram
  • Multi-actor business processes — BPMN
  • Decisions / pure logic — flowchart
  • Data movement — DFD
  • Internal data structure — ERD / class diagram
  • "Everything that could happen" — pick one scenario per diagram
13 Common mistakes

Mostly · trying to cram too much in

  • One diagram for all branches — usually unreadable. Make a "happy path" diagram and one or more "alternative" diagrams.
  • Unlabeled messages — every arrow needs a name. "1: getUser(id)" not just "1:".
  • Wrong arrow direction for returns — returns are dashed arrows, going back to the caller. Don't draw them as solid.
  • Forgetting activation bars — they make nesting obvious; without them, who's busy when becomes a guess.
  • Time going wrong direction — UML standard is top-down. Don't try to flip it sideways.
  • Mixing in process flow shapes — no diamonds, no rectangles for actions, no swimlanes. Just lifelines and messages.
  • Too many participants — >6 lifelines on one page = unreadable. Split.
  • Using it for stakeholders who don't read UML — they need a use case diagram or BPMN, not this.

🎯 Key Takeaways

Sequence diagrams · what to carry forward

  • ✅ A sequence diagram is a screenplay — actors talking to actors, top to bottom
  • ✅ Six elements: actor · lifeline · activation bar · sync message · async message · return
  • ✅ Reading rule: top-down · solid = call · dashed = return
  • ✅ alt / opt / loop fragments handle branches and repeats — use sparingly
  • ✅ One scenario per diagram. Don't cram all branches into one.
  • ✅ Best for API contracts, use-case realization, integration test design
  • ✅ Pair with use case diagram (what) and activity diagram (steps) — three lenses on the same thing
