02
The problem BPMN solves

"Can someone explain how this actually works?"

A loan takes 5 days. A leave request gets stuck for a week. A customer complaint disappears between three teams.

Everyone has a piece of the truth. Nobody has the whole picture.

BPMN is a shared language for that whole picture.

03 The simplest possible BPMN diagram

Five shapes. One line. Order a coffee.

That's a complete, valid BPMN diagram. Nothing missing. You can already read this.
  • Thin green circle = start. Where it begins.
  • Rounded rectangles = activities. Things that get done.
  • Arrows = sequence flow. "And then…"
  • Thick black circle = end. Done.
04 Reading any BPMN diagram · the rule

Find the start · follow the arrows · stop at end

That's it. Every BPMN diagram, no matter how complex, reads the same way:

  1. Find the thin circle — that's where you start.
  2. Follow the arrows, one shape at a time, doing whatever each shape says.
  3. If you hit a diamond — answer the question, take the matching exit.
  4. If two arrows go out at once — both happen, in parallel.
  5. Stop when you reach a thick circle. That's the end.

Trick: if you can't tell where to start, the diagram is broken. There must be exactly one start per pool.

If you can read a flowchart, you can already read 70% of any BPMN diagram. The rest is just more shapes for special situations.

05 Notation · Events (the circles)

Three flavors of "something happens"

Start event

Thin single line. Where the process begins. Exactly one per pool.

Intermediate event

Double line. Something happens mid-process. Often: timer ("wait 2 days"), message received.

End event

Thick line. A path terminates. You can have several — one per distinct outcome.

Mental model: circles = things that happen to the process (passive). Rectangles = things the process does (active).

06 Notation · Activities (the rounded rectangles)

Tasks and subprocesses · the work itself

Task

One unit of work. Atomic — can't break down further on this diagram. Always verb + object.

Subprocess

A collapsed group of activities. The + means "drill in for details". Hides complexity, keeps the parent diagram readable.

Naming rule (don't skip this)

  • ✅ "Verify documents" — verb + object, specific, scannable
  • ✅ "Send confirmation email"
  • ❌ "Documents" — just a noun, no action
  • ❌ "Processing" — what's being processed?
  • ❌ "Step 3" — useless
07 Notation · Gateways (the diamonds)

Where the process branches or merges

Exclusive · ×

Pick one path. "Approved? Yes → ship. No → reject." Only one branch is taken.

Parallel · +

Do all paths. "Run credit check AND verify ID — at the same time." All branches happen at once.

Inclusive · ○

Pick one or more. "Notify by SMS, email, or both — based on preference."

Rules to remember

  • A gateway never does work — it asks a question or splits/merges paths. Label it with a question, not a verb. ("Approved?" not "Approve".)
  • Every split needs a matching merge of the same type downstream. Parallel split → parallel merge.
  • Label each outgoing arrow on an exclusive split with its condition (yes / no / specific value).
08 Notation · Flows (the arrows)

Two kinds of arrow · don't mix them up

Sequence flow · solid line

"And then this happens next." Used inside one pool. Cannot cross a pool boundary.

99% of arrows you'll draw.

Message flow · dashed line

"I'm sending something to another organization." Used between pools.

Email · API call · paper form moving.

The pool rule: sequence flow stops at the pool boundary. To talk between pools, you must use message flow. Never cross pools with a solid arrow.

09 Pools & lanes · who does what

The "who" dimension · stacked horizontal stripes

One pool (Tourism Council). Two lanes (Officer · System). The Officer reviews and decides. The System sends the confirmation. Each shape sits in the lane of whoever does it.

Pool = an organization. Lanes = roles within. Cross a lane = handoff. Cross a pool = different company / different system entirely.

10 Example 1 · coffee shop · with a decision

One pool · two lanes · one decision

Customer orders, decides cash or card, pays, receives. The barista takes the order and makes the coffee in parallel. One pool (the shop), two lanes (the actors inside).
11 Example 2 · Bhutan e-visa · two pools · parallel work

Two organizations · message flow between · parallel review

Two pools — Tourist and Tourism Council. The dashed lines (message flow) carry the application across pools. Inside the Council, the parallel gateway (+) splits work between Officer review and Background check; both must finish before the exclusive gateway (×) routes to approve or reject.
12 Example 3 · SELISE leave request

Two pools · timer event · message flow

Two pools (Employee · HR System). The form crosses pools as a message. Inside HR, the System notifies the Manager, who reviews and decides. Approval triggers a record-and-deduct task; either outcome sends a decision email back to the employee.
13 How to draw one from scratch

Six steps · same recipe every time

1
Identify the actors. List every person, role, and external system involved. Each independent organization gets a pool. Each role inside gets a lane.
2
Pick the trigger. What kicks the process off? "Customer submits form" — that's your start event. Place it in the lane of whoever first acts.
3
Walk through the happy path. Add activities (rounded rectangles), one per real step. Connect with sequence flow inside a pool, message flow across pools.
4
Add decisions. Anywhere the process can branch — exclusive gateway. Anywhere parallel work happens — parallel gateway. Label each branch with its condition.
5
End every path. Each terminating path needs a thick end circle. A diagram with no end is broken.
6
Walk it past someone who lives the process. They will instantly point at "you forgot the part where…". That correction is the entire reason you drew it.

Tools: bpmn.io (free, in browser), Camunda Modeler (free desktop), Lucidchart, draw.io.

14 When to reach for BPMN

Powerful — but not always the right tool

✅ Use BPMN when…
  • Process crosses departments or organizations
  • Hand-offs between humans and systems matter
  • You're documenting as-is / to-be for a BRD
  • The process will be automated by a workflow engine
  • Regulators or auditors need to see the flow
  • A BA or process owner will maintain it over years
❌ Skip BPMN when…
  • You're sketching an algorithm — flowchart is simpler
  • Single actor, three steps — bullet list wins
  • You need to show data movement — use a DFD
  • You're scoping features — use a use case diagram
  • Audience has never seen BPMN — start with flowchart, upgrade later
15 Common BPMN mistakes

Fix early · these kill readability

  • No pools when there should be — two organizations? Two pools.
  • Sequence flow crossing pools — not allowed. Use dashed message flow.
  • Gateway labeled with a verb — gateways ask questions, they don't do work. "Approved?" not "Approve".
  • Mismatched split / merge — a parallel split needs a parallel merge.
  • Unlabeled exclusive branches — every outgoing arrow on an × needs its condition.
  • No end events — every path must terminate in a thick circle.
  • Mixing abstraction levels — "receive email" next to "update database index" = confusing.
  • 20+ tasks on one diagram — collapse repeatable chunks into subprocesses.
  • Specific people as lane labels — use roles ("Manager"), not names ("Sonam").
16 Wrap-up
🎯 Key Takeaways

BPMN · what to carry forward

  • ✅ BPMN is a shared visual language for processes — OMG industry standard, BPMN 2.0
  • ✅ Reading rule: find the start, follow arrows, stop at end
  • ✅ Four element families: events · activities · gateways · flows
  • ✅ Three gateways: × exclusive · + parallel · ○ inclusive
  • ✅ Two flows: solid (sequence, in-pool) · dashed (message, cross-pool)
  • ✅ Pools = organizations, lanes = roles — handoffs become visible
  • ✅ Best for multi-actor processes — not for pure algorithms
  • ✅ Six-step recipe: actors · trigger · happy path · decisions · ends · walk it past someone

A well-drawn BPMN diagram replaces 3 pages of prose and 10 meetings of confusion.
