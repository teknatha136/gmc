02
Why DFDs exist

"Where does the customer's PII actually go?"

None of the other diagrams answer this.
Flowcharts show steps. BPMN shows handoffs. Use cases show goals.

DFDs answer one different question: what data exists, where does it live, and where does it move?

Compliance · integration · threat modeling — DFDs are the right tool.

03 A DFD is NOT a flowchart

Different question · different rules · different shapes

Flowchart shows…

  • Sequence — "first this, then that"
  • Decisions — "if-then-else"
  • Control flow — what the program does

DFD shows…

  • Data flowing — "this data moves there"
  • Stores — "this data lives here"
  • No sequence · no decisions · no time

The trap: beginners draw DFDs that look like flowcharts. If you find yourself adding diamonds or thinking "and then…", you've drifted. Stop and ask: what data is moving?

04 The simplest possible DFD

Library checkout · Level 0 (context diagram)

A complete Level 0 (context) DFD. One process (the whole system), two external entities, four labeled data flows. No stores at this level — those appear when we explode.
  • Plain rectangle = external entity. Someone or some system outside our system.
  • Rounded rectangle = process. A function that transforms data.
  • Arrow = data flow. Always labeled with the name of the data being moved.
05 Reading rule · the four-thing question

For each arrow, ask the same four things

  1. What's flowing? Read the arrow label. If unlabeled, the diagram is broken.
  2. Where is it coming from? External entity, process, or data store?
  3. Where is it going? Same three options.
  4. Is the connection legal? (See connection rules — slide 8.)

Don't ask "when?" A DFD has no time. Don't ask "what if?" — no decisions. Just data, source, destination.

06 The four shapes

Each has exactly one job

External Entity

Source / destination outside our system. Customer, supplier, gov agency.

Process

Transforms data. Verb phrase. Numbered hierarchically (1, 1.1, 1.2…).

Data Store

Where data is kept. Database, file. Labeled D1, D2, D3...

Data Flow

Arrow with the data name written on it. Always labeled.

Two notations exist: Yourdon/DeMarco uses circles for processes; Gane/Sarson uses rounded rectangles. We use Gane/Sarson — more common in modern tools.

07 Levels · DFDs decompose

Start broad · drill into detail

  • Level 0 · Context Diagram. The whole system as one process. External entities surround it. High-level data flows. One page · one picture.
  • Level 1. Decompose the single process into 3–7 sub-processes. Data stores appear. Externals stay on the edges.
  • Level 2, 3… Each sub-process can explode into its own diagram if needed. Most BA work stops at Level 1 or 2.

The balancing rule

When you explode a process, the inputs and outputs of the child diagram must exactly match the inputs and outputs of the parent process. No data appears from nowhere; no parent data disappears. This is what makes decomposition reliable.

08 Connection rules · what's allowed

Three legal connections · three illegal ones

✅ Legal data flows
  • Entity ↔ Process — external party gives data to / receives data from your system
  • Process ↔ Process — internal data passing
  • Process ↔ Data store — read or write to storage
❌ Illegal flows
  • Entity ↔ Entity — externals don't talk to each other on YOUR DFD; your system mediates
  • Store ↔ Store — stores don't talk; a process must read from one and write to the other
  • Entity ↔ Store — externals never touch your storage directly; always via a process

If you draw an illegal connection, it's not a stylistic preference — it's a sign you missed a process. Add the missing process between them.

12 How to draw one from scratch

Five steps · stop early

1
Start with Level 0. One process (the whole system). List external entities. Draw the highest-level flows in and out.
2
Label every flow. Be specific: "credit card number" not "data". This is the most-skipped step and the one that breaks DFDs.
3
Explode to Level 1 if needed. Decompose the single process into 3–7 sub-processes. Add data stores. Keep external entities on the edges.
4
Check the balance. Every input flow on Level 0 must appear as an input on Level 1. Same for outputs. If they don't match, fix.
5
Stop when the next level adds no new insight. Most BA work stops at Level 1 or 2. Anything deeper is usually a developer's job.
13 When to use a DFD

Data questions, not process questions

✅ Use a DFD when…
  • You need to know what data moves and where it lives
  • Privacy / compliance review (GDPR, Bhutan DPA, HIPAA)
  • Documenting integrations between systems
  • Legacy system analysis — "where is the data today?"
  • Threat modeling (STRIDE) — DFDs are the starting point
  • You want a diagram stable across process changes
⚠️ Don't use a DFD for…
  • Step-by-step process — flowchart / BPMN / activity
  • Decisions / branching — DFDs have no decisions
  • Scoping features — use case diagram
  • Data structure / schema — ER diagram
  • Concurrent / parallel work — activity diagram
14 Common mistakes

Mostly · drifting back into flowchart thinking

  • Unlabeled arrows — the #1 mistake. Every arrow says what data it carries.
  • Adding decisions — DFDs have no diamonds, no "if-then". Just data, source, destination.
  • Sequence implied — DFDs don't have time. The order of processes left-to-right is layout, not chronology.
  • Entity ↔ Entity — externals don't talk to each other on your DFD. Your system mediates.
  • Store ↔ Store — never. A process must mediate.
  • "Miracle" process — one with output but no input, or input but no output. Something is missing.
  • Unbalanced child diagrams — Level 1 inputs/outputs must match Level 0.
  • Decomposing too far — stop when extra detail stops adding value.
15 Wrap-up · and the diagram series
🎯 Key Takeaways

DFDs · what to carry forward

  • ✅ DFDs show data movement, not flow, not decisions
  • ✅ Four shapes: external entity · process · data store · data flow
  • ✅ Every flow is labeled. No exceptions.
  • ✅ Decomposed by levels (0, 1, 2…) with a strict balancing rule
  • ✅ Three legal connections: entity↔process · process↔process · process↔store
  • ✅ Excellent for compliance, integration mapping, threat modeling
  • ✅ Stop drawing levels when extra detail stops adding insight

Six diagrams · six different questions. You now have the toolkit.
