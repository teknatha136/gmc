Part 1

What is a UML activity diagram

A UML activity diagram is a diagram type from the Unified Modeling Language (UML), an OMG-maintained standard used widely in software engineering. It's a flowchart's big sibling: same core idea (steps + decisions + flow), but with formal extensions that make it better for modeling software workflows.

Key differences from a plain flowchart:

  • Explicit parallelism via fork and join bars — things that happen at the same time, not just one after another.
  • Object flow — data or artifacts flowing from one action to another (e.g., an "Order" object flowing from "Validate" to "Process").
  • Partitions (swimlanes) — you can still show actor / system boundaries.
  • Formal semantics — UML specifies exactly what each shape means, so a tool can check correctness.

Who is the audience?

Activity diagrams are most useful when software engineers are the primary readers. If your audience is business-side (sponsors, business owners, operations managers), prefer BPMN or a plain swimlane — both are easier for non-technical readers to scan.

Part 2

Core shapes

Initial node

Solid filled circle. Where the flow begins. A diagram has exactly one.

Activity final node

Bullseye. Flow terminates. A diagram may have several, one per distinct outcome.

Action

Rounded rectangle (more rounded than BPMN — almost pill-shaped). A single step. Named with verb + object.

Decision / Merge

Diamond. A choice split (one path taken) or the matching merge point. Outgoing arrows have guard conditions like [approved] or [score > 700].

Fork / Join bar

Thick horizontal (or vertical) bar. A parallel split (all outgoing paths run concurrently) or the matching rejoin.

Object node

Sharp-cornered rectangle. Represents a data object flowing between actions (e.g. "Order[pending]", "Invoice[signed]").

Part 3

Choice vs. parallel — the crucial distinction

This is the one thing people mix up when moving from flowcharts to activity diagrams. A diamond represents a choice: exactly one branch is taken based on a guard condition. A fork bar represents parallelism: all branches run at once.

Diamond · choice flow

Exactly one outgoing arrow is followed. Each arrow is labeled with a guard condition in square brackets: [yes], [no], [order total > 1000].

Matches BPMN exclusive gateway (×).

Fork bar · parallel flow

All outgoing arrows are followed simultaneously. No conditions needed. All the parallel paths eventually merge at a matching join bar before the flow continues.

Matches BPMN parallel gateway (+).

Rule: every fork needs a join

A fork splits one flow into N parallel flows. They must all reach a join bar before the main flow continues. Forget the join, and the diagram is incorrect (different parallel branches might reach a final node independently — which is a legitimate special case but rarely what you mean).

Part 4

Worked example · online order fulfillment

A customer places an online order. The system checks stock. If out of stock, it notifies the customer and ends. If in stock, three things happen in parallel: charging payment, picking the item from the warehouse, and sending an order-confirmation email. All three must complete before the order is shipped.

Order fulfillment. The diamond handles choice (stock yes/no). The fork bars handle parallel execution of three activities before ship.
Part 5

Partitions (UML swimlanes)

An activity diagram can be divided into partitions — UML's term for swimlanes. Each partition represents an actor, role, or subsystem. Every action sits in the partition responsible for it. Crossing a partition = a handoff, exactly like in a regular swimlane diagram.

Use partitions when both parallelism and actor-responsibility matter — for example, modeling a software workflow where the UI, the service layer, and the database each have their own activities, and some happen in parallel.

Part 6

When to use activity diagrams

✅ Use an activity diagram when…

  • You're modeling a software workflow for developers
  • Parallel steps exist and must be shown
  • Your team already uses UML (sequence, class, use case)
  • You're "realizing" a use case — showing its step-by-step flow
  • Object flow matters (what data moves between actions)

⚠️ Prefer another diagram when…

  • Audience is business-side — BPMN or swimlane reads better
  • Process crosses organizations — BPMN pools handle this properly
  • No parallelism and no object flow — a flowchart is simpler
  • You need a diagram that will drive automation — BPMN 2.0 is the right choice
Part 7

Common mistakes

MistakeFix
Using a diamond to mean parallel flowParallel = fork bar. Diamond = choice.
Fork without matching joinEvery fork must have a join downstream (or be an intentional parallel termination)
Unlabeled guard conditions on diamondsEvery outgoing arrow from a diamond needs a guard like [condition]
Multiple initial nodesOnly one solid dot. Flow has a single start.
Mixing activity-diagram notation with BPMN notationPick one. Don't put a BPMN circle event on a UML activity diagram (or vice versa)
Using it for non-technical readersConvert to BPMN or plain swimlane for business audiences
Takeaways

Key takeaways

What to carry forward

  • Activity diagrams are UML's workflow diagram — a flowchart with parallelism and object flow.
  • Initial node · action · decision · fork/join · final node are the core shapes.
  • Diamond = choice (one path). Fork bar = parallel (all paths).
  • Every fork needs a matching join. Every decision branch needs a guard condition.
  • Partitions (UML swimlanes) show actor responsibility while keeping parallelism visible.
  • Best audience: developers and architects. For business audiences, use BPMN or a plain swimlane.

Recommended tools: PlantUML (text-based, version-controllable), draw.io, Lucidchart, Enterprise Architect, Visual Paradigm.
