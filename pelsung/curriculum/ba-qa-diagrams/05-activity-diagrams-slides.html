02
Why activity diagrams exist

"Three things happen at once. Then we wait for all three."

A flowchart can't show that. Its arrows are strictly sequential — one thing, then the next.

But real software workflows fork: charge the card AND reserve inventory AND email the customer — at the same time, then re-sync.

Activity diagrams are flowcharts that handle parallelism properly.

03 The simplest possible activity diagram

Same as flowchart · just rounder rectangles

A complete activity diagram. Filled black dot = start. Bullseye = end. Pill-shaped rectangles = actions.
  • Filled dot = initial node (start). Always exactly one per diagram.
  • Bullseye = activity final node (end). Can have multiple — one per outcome.
  • Pill rectangles = actions. More rounded than BPMN/flowchart rectangles — that's the visual signal "this is UML".
04 Reading rule · same as flowchart, plus one twist

Find the dot · follow arrows · respect the bars

  1. Find the filled dot — that's where you start.
  2. Follow the arrows through actions, doing each one.
  3. If you hit a diamond — answer the guard condition, take that exit.
  4. If you hit a thick bar with multiple arrows out — all those things happen at the same time. (This is the new shape.)
  5. If you hit a thick bar with multiple arrows in — wait for ALL of them to finish before continuing.
  6. Stop at a bullseye. Done.

The big idea: diamonds = "pick one path", bars = "do all paths". That distinction is what activity diagrams add over flowcharts.

05 Core shapes · five plus an object

Master five · meet object node sometimes

Initial node

Filled dot. Start. Exactly one.

Final node

Bullseye. End. Can be several.

Action

Pill. A single step.

Decision

Diamond. Pick one path.

Fork / Join bar

Thick bar. Parallel split / re-sync.

There's also Object node (sharp-cornered rectangle) for showing data flowing between actions. Useful sometimes; skip it when you don't need it.

06 Diamond vs. bar · the crucial distinction

Choice flow vs. parallel flow

Diamond · choice flow

Exactly one outgoing arrow is followed. Each arrow has a guard condition in square brackets: [yes], [score > 700].

"Is the user premium or free?" — only one branch.

Fork bar · parallel flow

All outgoing arrows are followed simultaneously. No conditions. They re-sync at a matching join bar downstream.

"Charge AND reserve AND email" — all three at once.

Iron rule · forks need joins

Every fork bar must have a matching join bar downstream where the parallel branches re-sync. Without the join, the diagram is incorrect — the main flow continues without waiting for parallel branches to finish.

07 Example 1 · online order fulfillment

One choice + one parallel block

Diamond at top = choice (in stock yes/no). Top fork bar splits into 3 parallel actions; bottom join bar waits for all 3 to finish before shipping.
08 Example 2 · ATM withdrawal · parallel finishing tasks

Same ATM as flowchart · but with parallel "finishing" steps

After "funds enough?" passes, the ATM does three things in parallel: dispenses cash, prints receipt, logs the transaction. Only when ALL three finish does the session end. Flowchart can't show this — activity diagram can.
09 Example 3 · CI build pipeline

Three checks in parallel · merge result · deploy or fail

A modern CI/CD pipeline. Three quality gates run in parallel after the build. The diamond waits for all three results, then deploys or fails. Two final nodes — one for success, one for failure.
10 Partitions · UML's name for swimlanes

When parallelism AND actor responsibility both matter

An activity diagram can be divided into partitions — UML's term for swimlanes. Each partition represents an actor or subsystem. Every action sits in the partition of whoever does it.

Use partitions when:

  • You need to show parallel work AND who is doing each parallel branch
  • You're modeling a software workflow with UI / service / database layers, and some of the work is concurrent
  • Your team is already in UML — partitions read naturally to UML-fluent readers

If you don't need both, prefer a plain swimlane diagram (no partitions) or a plain activity diagram (no swimlanes). Keep it simple.

11 How to draw one from scratch

Five steps · adapt as the workflow grows

1
Pick the trigger. The dot. What kicks this off? "API receives request", "User clicks Submit".
2
Sketch the happy path as a sequence of actions. Pill rectangles, verb + object, one per atomic step.
3
Mark the choices. Anywhere the path branches based on a value — diamond. Label each exit with a guard [condition].
4
Mark the parallels. Anywhere multiple things run at once — fork bar. Anywhere they re-sync — join bar. Forks must have joins.
5
Add ends. Each terminating path needs a bullseye. A failed path can have its own bullseye distinct from the success bullseye.
12 When to reach for activity diagrams

Software workflows for technical readers

✅ Use an activity diagram when…
  • You're modeling a software workflow for developers
  • Parallel steps exist and must be shown
  • Your team already uses UML
  • You're "realizing" a use case — its detailed flow
  • Object flow matters (what data moves between actions)
⚠️ Pick something else when…
  • Audience is business-side → BPMN or swimlane
  • Process crosses organizations → BPMN with pools
  • No parallelism, no object flow → flowchart
  • You need to drive automation → BPMN 2.0
13 Common mistakes

Mostly fork/join discipline

  • Diamond used for parallel — diamond = pick one, fork bar = do all. Not interchangeable.
  • Fork without matching join — main flow continues without waiting for parallel branches.
  • Unlabeled guard conditions on diamonds — every outgoing arrow needs [condition].
  • Multiple initial nodes — only one filled dot. Single start.
  • Mixing notations — don't put a BPMN circle event on a UML activity diagram. Pick one.
  • Using it for non-technical readers — convert to BPMN or plain swimlane.
14 Wrap-up
🎯 Key Takeaways

Activity diagrams · what to carry forward

  • ✅ UML's flowchart · adds parallelism and object flow
  • ✅ Initial node · action · decision · fork/join · final node
  • ✅ Diamond = pick one. Fork bar = do all.
  • ✅ Every fork needs a join. Every diamond exit needs a [guard].
  • ✅ Best audience: developers and architects
  • ✅ For business audiences, prefer BPMN or swimlane

Right tool when parallelism and object flow matter and the reader is technical.
