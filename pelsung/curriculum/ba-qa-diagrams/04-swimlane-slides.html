02
Why swimlanes exist

"Why does this take 5 days when the actual work is 90 minutes?"

Because the rest of the time, the work is waiting in someone else's queue.

A flowchart hides that. It shows the steps but not who's doing them — so you can't see the handoffs, can't see the queues, can't see the bottlenecks.

A swimlane diagram makes those invisible delays visible.

03 The simplest possible swimlane

Pizza ordering · two lanes · two handoffs

Two lanes (Customer · Pizza Shop). Red dashed arrows are handoffs — work crossing from one actor to another. Two handoffs visible at a glance.
  • Two horizontal lanes = two actors. Each step sits in the lane of whoever does it.
  • Same shapes as flowchart — just organized into rows by actor.
  • Lane crossing = handoff. Watch for these — they are where time and information get lost.
04 Reading rule · trace the path · count the handoffs

Same as flowchart · plus pay attention to lane changes

  1. Find the start. Usually top-left, in the lane of whoever initiates.
  2. Follow the arrows, doing each step.
  3. Each time the arrow crosses a lane boundary — work has been handed to someone else. Count these.
  4. Long arrows between lanes often mean long waits in real life. Add timing labels for evidence.
  5. Stop at the End, usually in the lane of whoever closes the loop.

Key intuition: the lanes themselves are just labels. The information is in the shape of the path across them — that's where the process truth lives.

05 Structure · lanes & orientation

Two layout choices · same content

Horizontal · most common

Lanes are horizontal stripes stacked top-to-bottom. Process flows left to right. Best for long, linear processes.

Reads like a timeline.

Vertical

Lanes are vertical stripes side-by-side. Process flows top to bottom. Better when activity labels are long.

Reads like a checklist with assignment.

Lane rules

  • 2–5 lanes is the sweet spot. Beyond 5 → unreadable.
  • Use roles, not names. "Manager", not "Sonam".
  • Add a "System" lane if anything is automated. Don't attribute system work to a human.
  • Order lanes intentionally. Adjacent lanes should usually exchange the most handoffs.
06 Example 1 · loan application

Four lanes · five handoffs

Five red dashed arrows = five handoffs. Customer's only steps are the very first and very last — everything in between is the bank passing the work around. Each red arrow could be a 1–2 day wait.
07 Example 2 · SELISE expense reimbursement

Three lanes · annotate the waiting time

With timing annotations on the handoffs: 1–2 days waiting on Manager, 2–3 days on Finance verification, weekly batched payment. Total: ~7 working days. Active work: ~30 minutes.

The improvement question: can manager approval and finance verification run in parallel? Could the weekly batch run daily? Each timing label is a place to attack the cycle time.

08 What swimlanes reveal at a glance

Four questions answered without calculation

  • Who is overloaded? The lane with the most boxes.
  • Where do things wait? Long arrows between lanes = waiting time.
  • Is anyone idle? A lane with almost nothing means under-use or missed responsibility.
  • Where can we automate? A step always in the System lane is already automated; one repeatedly in a human lane is a candidate.

Add per-step duration and per-handoff waiting time and a swimlane becomes a bottleneck-finder.

09 Swimlane vs. BPMN · when to upgrade

Same family · BPMN is the formal cousin

Stay with simple swimlane when…

  • Process is mostly linear · no parallel/inclusive gateways needed
  • One organization · multiple roles within
  • Audience is internal / non-technical
  • You're sketching for understanding, not automation

Upgrade to BPMN when…

  • Process crosses organizations · need pools + message flow
  • You need parallel work, inclusive gateways, timer events
  • The process will be automated by a workflow engine
  • Regulators or auditors require formal notation

A swimlane is BPMN with the volume turned down. BPMN is a swimlane with all the dials at 11.

10 How to draw one from scratch

Five steps · 20 minutes start to finish

1
List the actors. Roles, not names. Add a System lane if anything is automated. Order them so adjacent lanes exchange the most work.
2
Draw the lanes. Stack them horizontally, label each on the left. The boundary between lanes is where handoffs happen.
3
Walk through the happy path. Each step goes in the lane of whoever does it. Connect with arrows — short within a lane, long crossing lanes.
4
Annotate timings. Per step: typical duration. Per handoff: typical waiting time. This is where the diagram becomes powerful.
5
Review with the people in the lanes. "Is this what actually happens?" Watch them point at the missing step or the secret handoff. Update.
11 When to reach for a swimlane

Sweet spot: 2–5 actors · mostly linear

✅ Use a swimlane when…
  • 2–5 distinct actors are involved
  • Handoffs are the thing you want to highlight
  • You want to spot bottlenecks & waiting time
  • You're doing a "current-state" analysis
  • Audience is non-technical managers
  • Process is mostly linear, few parallel branches
⚠️ Prefer something else when…
  • Single actor → flowchart
  • Need parallel gateways, message flow → BPMN
  • Cross-organization process → BPMN with pools
  • Diagram will drive an automation engine → BPMN
  • Regulated / formal documentation → BPMN
12 Common swimlane mistakes

Same flowchart traps · plus a few new ones

  • More than 5 lanes — group related roles or split into multiple diagrams.
  • Activity in the wrong lane — put a step in the lane of whoever actually does it, not whoever asked for it.
  • No System lane for automated steps — automation gets attributed to the human clicking a button.
  • Specific people as labels — use roles. "Manager", not "Sonam".
  • No timing annotations — for process improvement, this is the whole point.
  • Single-actor swimlane — if only one lane is busy, just use a flowchart.
  • Ignoring failure paths — what happens when manager rejects? When verification fails?
13 Wrap-up
🎯 Key Takeaways

Swimlanes · what to carry forward

  • ✅ A flowchart with actor lanes — one lane per role / system
  • ✅ Lane crossing = handoff, and handoffs are where processes fail
  • ✅ Sweet spot: 2–5 actors, mostly linear
  • ✅ Reveals who's overloaded · where things wait · who's idle · what can be automated
  • ✅ Add timing annotations to turn it into a bottleneck-finder
  • ✅ Upgrade to BPMN for parallel work / multi-org / automation
  • ✅ Most useful diagram for process-improvement work

Reducing handoffs usually beats optimizing any single step.
