Part 1

What is a data flow diagram

A Data Flow Diagram (DFD) is a diagram that models how data moves through a system: where it comes from, what processes it flows through, where it's stored, and where it ends up. DFDs were introduced in the 1970s by Larry Constantine and Tom DeMarco, then popularized by Ed Yourdon. A parallel notation was developed around the same time by Chris Gane and Trish Sarson. Both are still in use.

The DFD mindset

A DFD is not about what happens. It's about what data moves. There are no decisions, no sequence, no branching. An arrow on a DFD means "this data moves from here to there" — not "this step happens next". Mixing this up is the most common mistake.

DFDs remain highly relevant today — especially for data privacy compliance (GDPR, Bhutan's Data Protection Act, HIPAA), integration design, legacy system analysis, and as the foundation for STRIDE threat modeling.

Part 2

The four symbols

We'll use the Gane/Sarson notation (rounded-rectangle processes) in this note — it's the more common one in modern BA tools. Yourdon/DeMarco uses circles for processes; everything else is the same.

External entity

A source or destination of data that lives outside the system. A customer, a supplier, a government agency, a partner system. Drawn as a plain rectangle (Gane/Sarson) or a shadowed square (Yourdon).

Process

A function or procedure that transforms incoming data into outgoing data. Named with a verb phrase. Numbered hierarchically (1.0, 1.1, 1.2, 2.0 …) so you can refer to them from the level above.

Data store

A repository of data. A database, a file, a spreadsheet, a paper folder. Open-ended rectangle (Gane/Sarson) or parallel lines (Yourdon/DeMarco). Labeled D1, D2, D3 … with a name.

Data flow

Arrow showing data moving from one element to another. Always labeled with the name of the data being moved. Never unlabeled.

Connection rules

  • Data flows go: entity ↔ process, process ↔ process, process ↔ store.
  • Never: entity ↔ entity (external parties don't talk on your DFD — your system must mediate).
  • Never: store ↔ store (data stores can't talk directly — a process must read from one and write to the other).
  • Never: entity ↔ store (an external party doesn't touch your store directly — it goes through a process).
Part 3

Levels · decomposition

DFDs are hierarchical. You start broad and drill in.

  1. Level 0 · Context diagram. The entire system is a single process. External entities surround it. High-level data flows go in and out. One page, one picture — the 10-second-elevator view.
  2. Level 1. The single process is decomposed into its major sub-processes (typically 3–7). Data stores appear. External entities stay on the edges. All the inputs/outputs that crossed the boundary at Level 0 must still cross it here (that's the balancing rule).
  3. Level 2. Any Level 1 process that's still too complex gets its own Level 2 diagram. Its external interface again balances with the Level 1 view.
  4. Levels 3, 4, … Only if needed. Most BA work stops at Level 1 or 2.

The balancing rule

When you explode a process into a child diagram, the inputs and outputs of the child must exactly match the inputs and outputs of the parent process. No new data appears from nowhere; no parent data disappears. This is a strict rule — it's what makes DFD decomposition reliable.

Part 4

Worked example · payroll system

Level 0 · Context diagram

One process (the whole payroll system). Three external entities: employees submit timesheets and receive payslips; HR managers configure pay rates and receive reports; the bank receives salary transfers.

Level 0. One process. Three entities. Five labeled data flows. No stores at this level.

Level 1 · Decomposed

The single "Payroll System" process becomes three: Capture timesheets, Calculate payroll, Disburse salaries. Three data stores appear. All the Level 0 external flows still cross the boundary — the diagram balances.

Level 1. External inputs/outputs balance Level 0. Three internal processes, three stores.

If needed, each Level 1 process could explode further. For example, "Calculate payroll" (2.0) might get a Level 2 with sub-processes 2.1 Compute gross, 2.2 Apply tax, 2.3 Apply deductions. The balancing rule applies again: the inputs and outputs of the Level 2 diagram must match process 2.0's interface on Level 1.

DFD vs. flowchart · the confusing cousin

DFDs are often confused with flowcharts. They look similar but answer completely different questions.

Part 6

When to use a DFD

✅ Use a DFD when…

  • You need to know what data moves and where it's stored
  • You're doing privacy / compliance analysis (GDPR, Bhutan DPA, HIPAA) — "where does PII live and flow?"
  • You're documenting integrations — which systems exchange what data
  • You're analyzing a legacy system — where does the data live today?
  • You're doing threat modeling (STRIDE) — DFDs are the starting point
  • You want a diagram that's stable across process changes — data flows change slower than process flow

⚠️ Don't use a DFD for…

  • Step-by-step process — use flowchart, BPMN, or activity
  • Decisions / branching — DFDs don't model decisions
  • Scoping features — use a use case diagram
  • Data schema / relationships — use an ER diagram
  • Concurrent / parallel behavior — use an activity diagram

Common mistakes

MistakeFix
Unlabeled data flow arrowsEvery arrow says what data it carries — no exceptions
Treating DFD like a flowchartNo sequence, no decisions. If you're tempted to add a diamond, you want a flowchart or BPMN
Flow directly between two external entitiesExternal entities don't talk to each other on your DFD — your system mediates
Flow directly between two data storesStores don't talk to each other — a process must read from one and write to the other
External entity writes to store directlyAn external entity always goes through a process
"Miracle" process (output with no input, or input with no output)Every process transforms something — check for missing data flows
Unbalanced levelsThe external interface of a parent process must exactly match its child diagram
Decomposing everything to Level 3+Stop when further detail no longer adds value — most work stops at Level 1 or 2
Confusing data flows with message flowsA DFD flow is raw data (e.g. "timesheet"), not a message event
Takeaways

Key takeaways

What to carry forward

  • DFDs show data movement, not process flow and not decisions.
  • Four symbols: external entity · process · data store · data flow.
  • Every data flow is labeled. No exceptions.
  • Decomposed by level — Level 0 (context) · Level 1 · Level 2, with a balancing rule between them.
  • Never entity ↔ entity. Never store ↔ store. Never entity ↔ store. A process must mediate.
  • Excellent for compliance, integration mapping, legacy analysis, and threat modeling.

Six diagrams · six different questions

You've now covered the six diagrams every BA should know. A quick lookup guide:

  • BPMN — multi-actor business process, especially for automation
  • Flowchart — algorithm, decision tree, eligibility logic
  • Swimlane — process improvement, handoffs, bottleneck analysis
  • Activity diagram — software workflow with parallelism, for developers
  • Use case diagram — scope and actors, one-page system summary
  • Data Flow Diagram — data movement, compliance, integration mapping

Pick the diagram that answers the question you're trying to answer. The tool follows the question — never the other way around.
