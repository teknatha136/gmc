Part 1

What is a sequence diagram

A sequence diagram is a UML diagram that shows the messages exchanged between actors and systems for a single scenario, in time order. It is the right diagram when the question is "who calls whom, in what order, with what message?"

It looks like a screenplay. Participants stand at the top of the page. Time runs downward. Each horizontal arrow is a line of dialogue — sender, receiver, message.

Where it fits among UML diagrams

UML offers three behavioral diagrams that often describe the same scenario from different angles:

  • Use case diagram — names the scenario (e.g. "Place Order")
  • Activity diagram — shows the steps (with branches and parallel work)
  • Sequence diagram — shows the messages between participants

You don't need all three for every feature. Pick the one that matches the question you're trying to answer.

Part 2

Core elements

Actor / object

Stick figure for a human actor. Box for a system, service, or class. Drawn at the top of the diagram. Each one represents a role, not a specific person.

Lifeline

The dashed vertical line hanging from each actor. Represents the actor's existence over time. Time flows downward.

Activation bar

A thin rectangle on a lifeline. Represents the period when that participant is "busy" handling something. Optional but very useful for showing nesting.

Synchronous message

Solid arrow with a closed (filled) arrowhead. Means "call this and wait for it to return". The most common arrow in BA-facing diagrams.

Asynchronous message

Solid arrow with an open V arrowhead. Means "send and don't wait" — fire-and-forget. Used for messaging, queues, events.

Return message

Dashed arrow with an open arrowhead. Means "this is the response coming back". Always travels right-to-left or up-to-down (back to whoever called).

Message labels

Every message must be labeled with the operation being called (and ideally its arguments). Conventions:

  • 1: getOrders() — sequence number plus operation name
  • 2: charge(amount, currency) — operation with arguments
  • 3: order data — for a return, just describe what's coming back

If you can't name the message, you don't yet understand the interaction.

Part 3

Combined fragments · alt · opt · loop

Real interactions branch and repeat. UML wraps those parts of the diagram in a labeled rectangle called a combined fragment. The label in the top-left tells you what kind.

LabelMeaningCommon use
altAlternative — multiple branches separated by dashed lines, only one runs based on the guardIf/else logic ("if balance > amount" else)
optOptional — runs only if the guard is true; nothing happens otherwise"If the customer opted into emails, send the receipt"
loopLoop — runs repeatedly while a guard is true"For each item in the cart, reserve inventory"
parParallel — multiple regions run concurrentlyAsynchronous messaging, two services contacted at once. Rare in BA work.
breakBreak — if condition true, run this and exit the enclosing fragmentError handling. Rare.
refReference — placeholder for another sequence diagram (modular reuse)"See sequence diagram for Authenticate"

Use sparingly

It's tempting to wrap everything in fragments to show every possible variation. Don't. A clean "happy path" sequence diagram with one or two alt fragments at decision points reads beautifully. A diagram with five nested alt/opt/loop blocks is unreadable. When in doubt, split into multiple diagrams — one per scenario.

Part 5

Worked example · mBoB bill payment

A more realistic Bhutanese-context scenario. A customer pays a recurring bill from the mBoB mobile app. The app talks to a backend Payment Service, which validates the bill against the external Biller (e.g. Bhutan Power Corporation), deducts the amount from the customer's balance, and returns success.

Four participants. Notice the self-call on Payment Service (message 5) — the same service calling itself for an internal step. Drawn as an arrow that loops back to its own lifeline.

This diagram doubles as an API spec. Each arrow becomes:

  • A method to implement (processPayment, validateBill, deductBalance)
  • A test case for QA (happy path; insufficient balance; biller down; wrong mPin)
  • A piece of the Payment Service contract (input parameters + return shape)

Part 8

Sequence vs activity vs use case

The same scenario described three ways:

DiagramSays…Best for
Use case diagram"Customer can Place Order"Scoping and stakeholder alignment
Activity diagram"Select items → enter address → confirm payment → if in stock, ship; else notify, end"Showing flow steps with branches and parallels
Sequence diagram"User → Web.placeOrder() → OrderService.create() → InventoryService.reserve() → PaymentService.charge() → Email.send()"Specifying the conversation between participants

Three lenses, three audiences. A non-technical sponsor reads the use case diagram. A developer reads the activity and sequence. A QA engineer derives test cases from all three.

Part 9

How to draw one from scratch

  1. Pick one scenario. Don't try to cover everything. "Successful login" is one diagram; "failed login" is a different one (or an alt fragment if both fit cleanly).
  2. List the participants. Place them across the top, ordered roughly by who calls whom (initiator on the left). Stick figures for humans, boxes for systems / services / classes.
  3. Drop their lifelines. Dashed vertical lines hanging down from each participant.
  4. Add the first message at the top. Then each next message goes below. Solid arrow with closed head for calls, dashed arrow for returns. Always label the message with name + arguments.
  5. Add activation bars. Strips on the lifelines showing when each participant is "busy". Optional but very helpful — they make nesting obvious.
  6. Wrap branches in alt / opt fragments only if the scenario truly branches. Otherwise: one happy-path diagram is much clearer than one diagram trying to cover every case.
  7. Walk through it as the receiver of message 1. Read each message in order — does the diagram make sense? Is anything missing? Are returns to the right caller?
Part 10

When to use a sequence diagram

✅ Use a sequence diagram when…

  • You're specifying an API or integration contract
  • You're realizing a use case ("what messages flow?")
  • QA needs to derive integration test scenarios
  • You're designing or reviewing microservice interactions
  • Order of events matters and must be explicit
  • Sync vs async messaging needs to be visible

⚠️ Don't use one for…

  • High-level scope — use case diagram
  • Multi-actor business processes — BPMN
  • Decisions / pure logic — flowchart
  • Data movement — DFD
  • Internal data structure — ERD or class diagram
  • "Everything that could happen" — pick one scenario per diagram

Common mistakes

MistakeFix
One diagram for all branches and edge casesMake a happy-path diagram. Cover edge cases in separate diagrams or with one alt fragment per branch point.
Unlabeled messagesEvery arrow must have a name. Ideally with arguments and return type.
Returns drawn as solid arrowsReturns are dashed arrows with open arrowheads.
No activation barsAdd them. They make nesting obvious and the diagram easier to read.
Time going sidewaysUML standard is top-down. Don't flip it.
Process flow shapes mixed in (diamonds, rectangles)Sequence diagrams have lifelines and messages — that's it. No flowchart shapes.
Too many participants on one diagram>6 lifelines = unreadable. Split.
Specific people as actorsUse roles. "Customer", "Manager" — not "Sonam", "Pema".
Drawing one for a non-technical audienceUse a use case diagram or BPMN instead.
Takeaways

Key takeaways

What to carry forward

  • Sequence diagrams are screenplays — actors talking to actors, top to bottom.
  • Six core elements: actor · lifeline · activation bar · sync message · async message · return.
  • Reading rule: top-down · solid arrow = call · dashed arrow = return.
  • alt / opt / loop fragments handle branches and repetition. Use sparingly.
  • One scenario per diagram. Don't cram every branch into one.
  • Best for API contracts, use-case realization, integration test design.
  • Pair with use case (what is possible) and activity (steps with branches) — three lenses on the same scenario.

Recommended tools: PlantUML (text-based, version-controllable, free), Mermaid (works in Markdown, GitHub renders it natively), draw.io, Lucidchart, Visual Paradigm.
