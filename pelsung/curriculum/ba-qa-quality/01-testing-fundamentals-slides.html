02
Why testing exists

A bug found in production costs 100× more than one found in a spec review

Testing isn't a phase at the end. It's a discipline that runs alongside development, catching mistakes as early and cheaply as possible.

BAs catch them in spec review. Devs catch them in unit tests. QAs catch them in integration and system tests. Users find what slipped through.

Every role is part of testing. Today: shared vocabulary.

03 Verification vs Validation · the two questions

"Did we build it right?" vs "Did we build the right thing?"

Verification

"Does the software match the spec?"

A check against documents and requirements. Static reviews, walkthroughs, inspections, unit tests against contracts.

Example: "The spec says password must be 8+ characters. Does the code reject 7-char passwords?"

Validation

"Does the software solve the user's problem?"

A check against real-world need. UAT, beta tests, observation, usability sessions.

Example: "The 8-char rule meets the spec — but real users still write passwords on sticky notes. Did we solve the security problem?"

The trap: teams that only verify ship technically-correct products that fail in the real world. Teams that only validate ship products that work in demos but break under load. Both matter.

04 The test pyramid · where tests live

Many cheap tests at the bottom · few expensive ones at top

More tests at the bottom (cheap, fast, many — runs in seconds). Fewer at the top (slow, brittle, expensive — runs in hours). When the pyramid inverts, the test suite becomes a bottleneck.
05 Test levels in detail

Each level catches different bugs

Unit testing

One function or one class, in isolation. Tests a single behavior. Runs in milliseconds. Mocks the rest of the world.

Owner: Developer · catches: logic bugs, edge cases in single functions

Integration testing

Two or more components working together. Real DB, real API calls, real file system. Catches contracts that don't match.

Owner: Developer + QA · catches: API mismatches, DB schema drift, broken integrations

System testing

The whole assembled application as one. End-to-end flows tested through the UI or top API. Often slow.

Owner: QA · catches: full-flow regressions, UI bugs, deployment-config issues

Acceptance / UAT

Business users (or proxies) run real-world scenarios. Confirms the system solves the problem, not just passes specs.

Owner: BA + business stakeholder · catches: spec gaps, "this isn't what we asked for"

Regression testing

Re-running prior tests to confirm nothing that worked before is now broken. Cuts across all levels.

Owner: QA · catches: unintended breakage from new changes

Smoke / sanity testing

Quick "is this build basically alive?" check before running deeper tests. Five minutes, not five hours.

Owner: QA · catches: broken builds before wasting time on detailed tests

06 Test types · functional vs non-functional

Two big families · both matter

Functional testing

"Does it do what it should?"

  • Login with correct credentials → success
  • Transfer 1000 BTN → balance reduced by 1000
  • Submit blank form → validation error
  • Reject expired card

Most acceptance criteria are functional tests in disguise.

Non-functional testing

"How well does it do it?"

  • Performance — response time under load
  • Security — resistant to common attacks
  • Usability — can a real user get through it
  • Accessibility — keyboard / screen reader access
  • Reliability — uptime, error rate, recovery
  • Compatibility — works on the browsers/devices we promised

BA hint: if every AC starts with "the user can…", you've only specified functional. Add at least one non-functional ("response within 2 seconds at 100 concurrent users").

07 Black-box vs white-box thinking

Two ways to think about what to test

Black-box testing

Tester treats the system as a sealed box. Inputs go in, outputs come out. No knowledge of internals required.

"What should happen if I do X?"

Used for: System / UAT / acceptance tests · most QA testing · all BA validation

White-box testing

Tester sees inside. Tests specific code paths, branches, conditions. Requires knowing the implementation.

"Have I exercised this if-branch?"

Used for: Unit tests · code coverage analysis · reviews

Most BA/QA work is black-box. White-box belongs to developers. Knowing both terms means you can talk to your dev team without confusion.

08 What "good" looks like · ISO 25010

Eight quality characteristics that define software quality

1 · Functional Suitability

Does it do what it claims? Completeness, correctness, appropriateness.

2 · Performance Efficiency

Time, resources, capacity. Fast enough? Doesn't melt the server?

3 · Compatibility

Co-existence with other software. Browser, OS, device.

4 · Usability

Learnability, operability, accessibility, error protection, aesthetics.

5 · Reliability

Availability, fault tolerance, recoverability. Uptime + error rate.

6 · Security

Confidentiality, integrity, non-repudiation, accountability, authenticity.

7 · Maintainability

Modular, reusable, analyzable, modifiable, testable.

8 · Portability

Adaptable, installable, replaceable across environments.

BA's role: spec which of these 8 dimensions matter most for the feature, and to what level. QA's role: verify each one with the appropriate test type. The 8 categories ARE the checklist for non-functional requirements.

09 Accessibility · usability with teeth

Accessibility lives inside Usability — but it's also law

WCAG (Web Content Accessibility Guidelines) is the de-facto standard. Three levels: A (basic), AA (standard), AAA (gold). Most laws require AA.

Four POUR principles

  • Perceivable — alt text on images, captions on videos, sufficient contrast
  • Operable — keyboard-only navigation, no time traps, skip links
  • Understandable — predictable behavior, error messages that help, plain language
  • Robust — works with screen readers (proper semantic HTML, ARIA where needed)

Where it shows up legally

  • EU — European Accessibility Act (mandatory June 2025 for many products)
  • US — Section 508 (federal procurement); ADA (private sector via lawsuits)
  • Global trend — accessibility lawsuits doubled 2018→2023

BA spec'ing "the form should be accessible" without saying "WCAG 2.1 AA" is meaningless. Spec the level. QA tests against it.

10 Worked example · mBoB bill payment

One feature · all the test types

Feature: pay a recurring bill from mBoB

  • Unit test · the calculateFee() function returns 0.5% of amount, rounded to 2 decimals
  • Integration test · Payment Service correctly calls Biller API and parses response
  • System test · end-to-end: user opens app → selects bill → pays → sees confirmation
  • Acceptance test · a real customer pays a real BPC bill and sees correct deduction in her balance the next day
  • Regression test · all of the above run again every time the team changes anything
  • Performance test · 1000 concurrent payments, p99 response < 3 seconds
  • Security test · MPIN brute-force lockout works after 3 attempts; injection inputs sanitized
  • Accessibility test · the entire flow operable with screen reader, keyboard-only
  • Compatibility test · works on iOS 14+, Android 9+, low-end devices common in rural Bhutan

Same feature, nine test types. The BA's job is to spec which ones matter. The QA's job is to design tests for each.

11 Where BA and QA fit in testing

Different points of the lifecycle · same goal

BA's testing contributions

  • Spec testable AC (Given/When/Then)
  • Spec which quality attributes matter and to what level
  • Define what UAT looks like
  • Review test coverage with QA
  • Sign off on go-live based on test results
  • Validate the right thing got built

QA's testing contributions

  • Design test cases from AC
  • Cover all levels (smoke, integration, system, regression)
  • Cover non-functional types (perf, security, a11y, etc.)
  • Identify edge cases and risks
  • Maintain regression suite
  • Verify the thing got built right

The handoff: BA's AC becomes QA's test cases. If the AC is sloppy, the test cases are sloppy. If the BA can't write a test for it, the AC isn't done.

12 Common testing mistakes (everyone makes these)

Avoid these and you're ahead of most teams

  • Testing only the happy path — easy to demo, dangerous in production. Always test rejection, error, edge.
  • "It works on my machine" — that's not testing, that's wishful thinking. Test in the actual deployed environment.
  • No regression strategy — every fix introduces a new bug somewhere. Without regression, you're patching with one hand and breaking with the other.
  • Manual-only testing — fine at first; suicidal at scale. The pyramid inverts and the suite becomes the bottleneck.
  • Functional-only testing — system passes all functional tests, falls over at 100 users. NFRs matter.
  • Inverted pyramid — too many slow E2E tests, too few unit tests. CI takes hours, no one trusts the suite.
  • "QA will catch it" — testing is everyone's job. Devs writing only happy-path code = bugs that QA can't catch.
  • Skipping accessibility — it's law in many places, and the lawsuit risk grows yearly.
