Part 1

Why testing exists — the cost curve

Software testing isn't an end-of-project activity. It's a discipline that runs alongside development from the first spec review to the last production hotfix. The reason is simple: the later a defect is caught, the more it costs to fix.

Caught at…Approximate costWhy
Spec review1×Just edit the document. Minutes of work.
Development5×Rework code that's already partly written.
QA / system testing10×Bug filed, dev fixes, QA re-tests, regression runs.
UAT30×Stakeholders involved, decisions revisited, schedule pressure.
Production100×+Hotfixes, customer support tickets, lost trust, possible compliance fines.

The mindset shift

Testing is not "QA's job." It's a layered defense where BAs catch defects in spec reviews, devs catch them in unit tests, QAs catch them in integration and system testing, and users find what slipped through. The earlier the layer, the cheaper the catch.

Part 2

Verification vs Validation

Two questions, often confused, both essential.

Verification — "Did we build it right?"

A check against the spec. Does the software match the documented requirements?

Methods: code review, static analysis, unit tests, contract tests, walkthroughs, inspections.

Example: the spec says "password must be at least 8 characters." Verification asks: does the code reject 7-character passwords?

Validation — "Did we build the right thing?"

A check against the real-world need. Does the software actually solve the user's problem?

Methods: UAT, beta tests, field observation, usability testing, customer interviews.

Example: the 8-character rule was met. Validation asks: did this actually make accounts more secure, or are users still writing passwords on sticky notes?

The trap teams fall into

Teams that only verify ship technically-correct products that fail in the real world ("we built exactly what was specified"). Teams that only validate ship products that work in demos but break under real load ("looked great in user testing, fell over at 100 users"). Both verification and validation must happen for software to be truly tested.

Part 3

The test pyramid

Mike Cohn's 2009 model that became the industry default. It describes how a healthy test suite should be shaped:

  • Wide base of unit tests — fast, cheap, many. Run in milliseconds. Cover individual functions in isolation. The team should have hundreds or thousands.
  • Middle layer of integration tests — slower, fewer. Real database, real API. Cover how components fit together. The team should have dozens to hundreds.
  • Narrow top of system / E2E tests — slow, expensive, brittle. Run through the UI or top API. The team should have a handful — only the most critical user journeys.
  • Tip of UAT / manual — humans involved. Slowest. Most expensive. Reserved for what humans must judge.

When the pyramid inverts

Some teams accumulate hundreds of slow E2E tests and almost no unit tests. The CI pipeline takes hours, tests are flaky, no one trusts them, and feedback to developers takes days. This is "the ice cream cone anti-pattern" — fix it by pushing tests downward (replace E2E with integration where possible, replace integration with unit tests where possible).

Part 5

Test types · functional vs non-functional

Levels describe where a test runs. Types describe what a test verifies. Two big families:

Functional testing — "Does it do what it should?"

Direct check of features against acceptance criteria.

  • Login with correct credentials succeeds
  • Transfer of 1000 BTN reduces balance by 1000
  • Submitting a blank form shows the right validation error
  • An expired card is rejected

Most of QA's daily work and most of BA's acceptance criteria live here.

Non-functional testing — "How well does it do it?"

The system might pass every functional test but still be unusable in production. Non-functional tests check the dimensions that determine that.

  • Performance — response time under load (p50, p95, p99 latency, throughput)
  • Security — resistant to common attacks (OWASP Top 10)
  • Usability — can a real user actually complete the task
  • Accessibility — keyboard navigation, screen reader, color contrast
  • Reliability — uptime SLA, error rate, recovery from failures
  • Compatibility — works on the browsers, OS versions, devices we promised
  • Localization — works in target languages, character sets, formats

BA hint

If every acceptance criterion in your story starts with "the user can…", you've only specified functional behavior. Add at least one non-functional criterion for any feature that matters: "Response within 2 seconds at 100 concurrent users." "Operable with keyboard alone." "Form data not logged in plain text."

Part 6

Black-box vs white-box thinking

Two perspectives on what to test:

Black-box

Treat the system as a sealed box. Inputs go in, outputs come out. Tester needs no knowledge of internals.

Question: "What should happen if I do X?"

Used for: system / acceptance / UAT testing, most QA testing, all BA validation.

White-box

Tester sees inside. Tests specific code paths, branches, conditions. Requires knowing the implementation.

Question: "Have I exercised this if-branch? Is this loop covered?"

Used for: unit testing, code coverage analysis, code review.

Most BA/QA work is black-box. White-box belongs to developers. Knowing both terms means you can talk to your dev team without confusion when a code coverage report shows up in a meeting.

Part 7

ISO 25010 · the 8 quality characteristics

ISO/IEC 25010 (2011) is the international standard for software quality. It defines 8 high-level characteristics that together describe what "good" software is. Each one breaks down into sub-characteristics. As BAs and QAs, this is your complete checklist for non-functional requirements — every feature should be examined through these lenses.

CharacteristicWhat it meansSub-characteristics
1 · Functional SuitabilityDoes it do what it claims?Completeness, correctness, appropriateness
2 · Performance EfficiencyIs it fast enough? Resource-efficient?Time behaviour, resource utilization, capacity
3 · CompatibilityWorks with other software / systems?Co-existence, interoperability
4 · UsabilityCan humans actually use it?Learnability, operability, user-error protection, accessibility, aesthetics
5 · ReliabilityStays up and recovers?Maturity, availability, fault tolerance, recoverability
6 · SecurityProtects data and users?Confidentiality, integrity, non-repudiation, accountability, authenticity
7 · MaintainabilityCan it evolve?Modularity, reusability, analysability, modifiability, testability
8 · PortabilityAdaptable to environments?Adaptability, installability, replaceability

How to use this as a BA

For every feature you spec, walk through the 8 characteristics and ask: does this matter here, and to what level? Most features need 2-4 of them explicitly specified. Capture the answers as non-functional requirements. Don't say "must be fast" — say "p95 response under 500ms at 100 concurrent users" (Performance Efficiency · Time behaviour).

How to use this as a QA

For every test plan, ensure all 8 characteristics that the BA marked as relevant have test coverage. If the spec only addresses Functional Suitability, push back — that's a half-tested feature.

Part 8

Accessibility · WCAG and the law

Accessibility lives inside Usability (ISO 25010 #4) — but unlike most other quality attributes, it's increasingly enforceable by law. As a BA or QA in 2026, you cannot treat it as optional.

WCAG · Web Content Accessibility Guidelines

The de-facto international standard, maintained by W3C. Three conformance levels:

  • Level A — bare minimum. Critical issues only.
  • Level AA — the standard most laws require. Sufficient for "broadly accessible."
  • Level AAA — gold standard. Often impossible for entire products; achievable for specific content.

Default expectation: WCAG 2.1 AA (or 2.2 AA, the latest version).

The four POUR principles

  • Perceivable — alt text on images, captions on videos, sufficient color contrast (4.5:1 for normal text)
  • Operable — fully usable by keyboard alone, no time traps, skip-to-main-content links
  • Understandable — predictable navigation, error messages that explain how to fix the error, plain language
  • Robust — works with screen readers (proper semantic HTML, ARIA where genuinely needed)

Legal landscape

  • European Union — European Accessibility Act (EAA) became fully enforceable June 2025. Many products must comply.
  • United States — Section 508 covers federal procurement. The ADA covers private sector via lawsuits — over 4,000 web accessibility lawsuits filed in 2023 alone.
  • United Kingdom — Public Sector Bodies Accessibility Regulations 2018.
  • Bhutan — Persons with Disabilities Bill (in development); follows WCAG implicitly via international agreements.

SELISE-relevant: any product touching EU users must consider the EAA. Any product procured by a US federal agency must hit Section 508. Cost of retrofitting is far higher than cost of building accessible from the start.

Spec hint

Specifying "the form should be accessible" is meaningless. Spec the level: "Conforms to WCAG 2.1 AA." QA tests against that target with tools like axe, WAVE, and keyboard / screen-reader walkthroughs.

Part 9

Worked example · mBoB bill payment

One feature, all the test types in action.

Feature: Pay a recurring bill (e.g., to BPC for electricity) from the mBoB mobile app. Customer selects biller, enters amount, authenticates with mPin, sees confirmation.

Test typeExample test
UnitcalculateFee(amount) returns 0.5% of amount, rounded to 2 decimals
IntegrationPayment Service correctly calls Biller API and parses the response when amount and biller ID are valid
System / E2ECustomer opens app → selects bill → enters amount → enters mPin → sees confirmation. End-to-end through real backend.
UAT / AcceptanceA real customer pays a real BPC bill and confirms the deduction is correct in her statement the next day
RegressionAll of the above run again every time a new build is released
Performance1,000 concurrent payment requests, p99 response time under 3 seconds
SecuritymPin lockout triggers after 3 wrong attempts; SQL injection inputs sanitized; rate-limiting on the payment endpoint
Usability5 customers can complete the payment without assistance in under 2 minutes
AccessibilityEntire flow operable with screen reader (TalkBack on Android, VoiceOver on iOS); usable with keyboard alone for desktop
CompatibilityWorks on Android 9+ and iOS 14+, including the low-end devices common in rural Bhutan (e.g., 2GB RAM)
LocalizationCurrency formats correctly in BTN; date formats follow Bhutan convention; UI strings translate to Dzongkha
ReliabilityRecovers gracefully if Biller API is down — shows clear error, doesn't double-deduct, doesn't lose the request

Same feature, twelve distinct test approaches. The BA's job is to spec which ones matter (probably all of them for a payment feature in a regulated industry). The QA's job is to design specific test cases for each.

Part 10

BA and QA roles in testing

What BAs do for testing

  • Write testable acceptance criteria (Given/When/Then)
  • Spec which ISO 25010 attributes matter and to what level
  • Define what UAT looks like and who runs it
  • Review test coverage with QA before sign-off
  • Sign off on go-live decisions based on test evidence
  • Catch validation issues — "we built the right thing" checks

What QAs do for testing

  • Design test cases from acceptance criteria
  • Cover all relevant levels (unit awareness, integration, system, regression, smoke)
  • Cover all relevant non-functional types (perf, security, a11y, etc.)
  • Identify edge cases the BA missed
  • Maintain the regression suite
  • Catch verification issues — "built right" checks

The BA→QA handoff

A BA's acceptance criteria become the QA's test cases. If the AC is sloppy or missing edge cases, the test cases are too. The fundamental rule: if you can't write a clear test for an AC, the AC isn't done. Either tighten the AC or have an explicit "Three Amigos" conversation with the dev to figure out what's missing.

MistakeWhy it hurtsFix
Testing only the happy pathEasy to demo; falls over in production at the first edge caseFor every positive test, write at least one negative and one boundary test

Takeaways

Key takeaways

What to carry forward

  • Testing isn't a phase — it's a discipline that runs alongside development from spec to production.
  • Verification = built right (against the spec). Validation = right thing built (against the user need). Both matter.
  • The test pyramid: many fast unit tests, fewer integration, very few E2E. Inverted pyramid = slow CI = no one trusts the suite.
  • Six levels: unit, integration, system, acceptance/UAT, regression, smoke.
  • Two type families: functional (does it work?) and non-functional (how well?).
  • ISO 25010 defines the 8 characteristics that make software "good." Use it as your NFR checklist.
  • Accessibility (WCAG 2.1 AA) is increasingly law. Spec the level explicitly.
  • BA writes testable AC; QA designs test cases from them. Same artifact, two views.
  • If you can't write a test for an AC, the AC isn't done.
