02
Why this matters

"Login is broken" wastes 4 hours of dev time. A good report wastes 10 minutes.

A bug report is a tiny spec.
It tells a developer exactly what's wrong, exactly how to reproduce it, and exactly what should happen instead.

A bad bug report triggers ping-pong messages, missed releases, and "can't reproduce → reopened" cycles.

Bug reporting is a writing skill. Today: how to do it well.

03 Bad bug vs good bug

Same defect · two reports · totally different outcomes

❌ Bad bug

Title: Login is broken

Desc: Can't log in. Please fix urgently!

Dev's reaction: "Login? Which form? Which user? What error? What browser? What did you click? When did this start?" Result: 6 messages back and forth before any fixing happens.

✅ Good bug

Title: Login button unresponsive on mBoB v3.4 (Android 13) when MPIN starts with 0

Steps: 1) Open mBoB v3.4 on Pixel 7 (Android 13). 2) Enter username. 3) Enter MPIN starting with 0 (e.g. 0123). 4) Tap Login. Expected: dashboard appears. Actual: button greys out, no error, no progress. Reproduces 5/5.

Dev's reaction: opens code, finds where MPIN is validated, sees the bug in 10 minutes.

The difference: the second report tells the dev exactly where to look. The first triggers a chain of messages.

04 Anatomy of a great bug report

Eight fields · use the same template every time

Title: [What's wrong] [where] [under what condition] · short, scannable, searchable
Environment: app version · OS · browser · device · build number · server / staging / prod
Steps to reproduce: numbered list · short verbs · exact inputs · "click" not "interact"
Expected result: what should happen, per spec or common sense
Actual result: what happened instead · screenshots / video / logs
Reproducibility: always · sometimes · once · how many times of how many tries (e.g. 5/5, 1/10)
Severity: how bad is it for the user (Critical · High · Medium · Low)
Priority: how soon must we fix (P0 · P1 · P2 · P3)
Optional but valuable: attachments (screenshot, video, HAR file, logs) · related bugs · workaround if known · regressed-since-build

Most bug trackers (Jira, Linear, GitHub) let you save this as a template. Use it. Consistency saves hours per week.

05 Severity vs Priority · the eternal confusion

They're independent · stop using them as synonyms

Severity

How bad is it for the user?

  • Critical — system unusable, data loss, security breach
  • High — major function broken, no workaround
  • Medium — function broken with workaround, or minor function fully broken
  • Low — cosmetic, typo, edge case nobody hits

Set by QA / reporter. Objective.

Priority

How soon must we fix it?

  • P0 — drop everything · fix today
  • P1 — fix this sprint
  • P2 — fix next sprint or two
  • P3 — backlog · maybe never

Set by PM / business / triage. Subjective. Reflects business priority.

06 Why they're different · concrete examples

Severity ≠ priority · the four corners

  • High severity · High priority — Login completely broken. Users locked out. Drop everything. (Critical / P0)
  • High severity · Low priority — Bug crashes the app, but only triggers on a date in the year 2099. (Critical / P3)
  • Low severity · High priority — Typo in the legal disclaimer on the homepage during a regulator inspection tomorrow. (Low / P0)
  • Low severity · Low priority — Footer copyright says 2024 instead of 2025. (Low / P3)

The lesson: a critical-severity bug isn't always urgent. A trivial bug can be urgent. Set them independently. The combination drives the fix order.

07 The defect lifecycle · a state machine

Every bug travels through these states

Forward path (blue): New → Triaged → In Progress → Fixed → Verified → Closed. Backward paths (red) when something fails: re-opened, sent back to dev, or regressed in production. Rejected/Won't Fix is the dead-end for invalid or out-of-scope reports.
08 What each state means
  • New / Open — just filed. Awaits triage. The reporter (often QA or BA) wrote it; nobody has yet decided what happens.
  • Triaged — reviewed by the triage team. Severity confirmed, priority set, owner assigned. Now it's "real work."
  • In Progress — a developer is actively fixing it.
  • Fixed — code changed, dev believes it's resolved, ready for QA to verify.
  • Verified — QA confirmed the fix works in the staging / test environment. Did NOT regress anything.
  • Closed — done. Bug ticket archived. Often only happens after the fix ships to production.
  • Reopened — bug came back. Either QA's verification failed, OR the bug regressed after closure (rare but happens).
  • Rejected / Won't Fix / Duplicate — terminal states for invalid reports. Not a bug, by design, won't fix this version, duplicate of #1234.
09 Triage · the meeting that decides

Who's in the room · what gets decided

Who attends triage

  • QA lead — represents the reported bugs, advocates for severity
  • Product / BA — represents user impact and roadmap, owns priority
  • Engineering lead — represents capacity, owns assignment
  • Optional: support lead (if customer-reported), security lead (if security-flagged)

What gets decided per bug

  • Is this a bug? Reject / dedupe if not. Often 20–40% of reports get rejected here.
  • Severity: objectively, how bad is the user impact?
  • Priority: when must this be fixed? (P0/P1 → this sprint, P2/P3 → backlog)
  • Owner: which dev / team takes it?
  • Target release: ship in current build, hotfix, or next release?

Cadence

  • P0 bugs: triaged immediately on filing (within an hour)
  • Daily triage: for active sprints, 15–30 minutes
  • Weekly triage: for backlog grooming, 1 hour
10 Bug categories · not all bugs are created equal

Tag bugs by category — different categories, different handling

  • Functional — feature doesn't work as specified · most common · usually owned by feature team
  • Performance — slow, timeouts, high resource use · often surfaces under load · owned by perf or platform team
  • Security — vulnerabilities, leaks, broken auth · ALWAYS high priority · often confidential · special handling rules
  • Accessibility — fails WCAG, screen reader broken, keyboard trap · increasingly tracked separately for compliance
  • Usability / UX — works but is confusing or frustrating · "not technically a bug" but real impact
  • Data — wrong calculation, corrupted record, lost data · often urgent because data loss compounds
  • Localization — wrong language, wrong format, broken layout in target locale
  • Regression — something that worked before is now broken · special priority because trust is at stake

Tagging matters because: security bugs may follow a separate disclosure process. Accessibility bugs feed into compliance reports. Performance bugs may trigger SLA reviews. Don't lose this signal by tagging everything "bug".

11 Accessibility bugs · special handling

Two reasons to tag and track them separately

  • 1 · Compliance reporting. WCAG conformance is increasingly audited. You need to be able to answer "how many WCAG AA violations are open?" without scanning every bug.
  • 2 · Different remediation approach. Accessibility bugs often need design + dev collaboration (semantic HTML, focus order, ARIA). Pure dev fix often misses the accessibility goal.

Severity guidance for a11y bugs

  • Critical — feature completely unusable for users with assistive tech (e.g., login form not navigable by keyboard)
  • High — feature usable but with significant friction (e.g., screen reader announces button as "button" with no label)
  • Medium — minor friction (e.g., contrast slightly below WCAG AA on a label)
  • Low — best-practice violations, level AAA failures

Tools like axe DevTools, WAVE, and Lighthouse generate accessibility violations as JSON. You can pipe them straight into your bug tracker tagged "accessibility-wcag-AA".

12 Who files bugs and how BAs fit in

Anyone can file. BAs file too.

QA's bug-reporting work

  • File bugs found in test execution
  • Confirm reproducibility before filing
  • Assign initial severity
  • Verify fixes
  • Maintain regression test for any "Closed" bug
  • Track defect metrics (escape rate, age, density)

BA's bug-reporting work

  • File bugs found in spec review or UAT
  • Translate user complaints into well-formed reports
  • Set priority during triage (business impact lens)
  • Decide if a bug → spec change or pure fix
  • Communicate fix status to stakeholders
  • Watch for "bugs that are really missed requirements"

The hidden category: some "bugs" are actually missed requirements — the system does what was specified, but the spec was wrong. The BA owns sorting these from real defects.

13 Common bug-reporting mistakes

Avoid these · save hours per week

  • Vague title — "Login broken" tells nobody anything. Title should make it scannable in a list of 200.
  • No environment — same code, different OS = different result. Always specify.
  • "Sometimes happens" with no reproduction count — useless. State exact frequency (5/5, 2/10, 1/100).
  • Severity = priority — they're independent. Stop using them as synonyms.
  • Multiple bugs in one ticket — splits attention, hard to close. One bug per ticket.
  • Not attaching evidence — screenshots cost 5 seconds and save 30 minutes of "what do you mean by that?"
  • Filing without first searching for duplicates — wastes triage time and dilutes signal
  • Closing without verifying — "fixed" by the dev means nothing until QA verifies
  • "Won't fix" without explanation — the next person hitting this bug needs to understand why it stays open
