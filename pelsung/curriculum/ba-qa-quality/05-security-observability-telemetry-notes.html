Part 1

Why these three together

Once a system is in production, three different questions matter. They sound separate but share one prerequisite — you must instrument the system to see inside it.

QuestionDisciplineTools
Is the system being attacked?Security monitoringOWASP scanners, SIEM, intrusion detection
Is the system working correctly?ObservabilityLogs, metrics, traces, alerts
Is anyone using it, and how?Telemetry / product analyticsMixpanel, Amplitude, PostHog, GA

The unifying principle

Software that you can't see inside is software you can't defend, fix, or improve. As a BA, you must spec the instrumentation. As a QA, you must verify it works before launch.

Part 2

Security · OWASP Top 10

The Open Web Application Security Project (OWASP) Top 10 is the most widely recognized list of web application security risks. It's updated every few years; the 2021 list is the current standard. You don't need to memorize the list — but you need to know it exists, walk through it during BRD review, and verify your team has covered each item.

CodeCategoryWhat it meansCommon mitigation
A01Broken Access ControlUsers can act outside their permissionsRole-based access control · deny by default · test inside-the-fence
A02Cryptographic FailuresSensitive data exposed via weak/missing encryptionTLS everywhere · encrypt PII at rest · use modern algorithms
A03InjectionSQL, XSS, command injection — user input treated as codeParameterized queries · input validation · output encoding
A04Insecure DesignVulnerabilities baked into architectureThreat modeling · security review at design phase
A05Security MisconfigurationDefault passwords, exposed admin, debug in prodHardened defaults · automated config checks · least privilege
A06Vulnerable ComponentsOutdated libraries with known exploits (Log4Shell)SCA tools (Snyk, Dependabot) · regular updates
A07Identification & Auth FailuresWeak passwords, broken sessions, no MFAMFA · session timeout · password policy · rate limiting
A08Software & Data Integrity FailuresUnsigned updates, untrusted CI/CD, supply chain attacksSigned artifacts · SBOM · code-signing
A09Security Logging & Monitoring FailuresCan't detect attacks because nothing is logged or watchedAudit logs · SIEM · alerts on auth failures
A10Server-Side Request ForgeryApp tricked into making requests on attacker's behalfAllowlist destinations · validate URLs

How to use the Top 10 as a BA/QA

Walk through the 10 categories during spec review. For each one ask: "Does this feature have any exposure here? If yes, what's the spec'd mitigation?" Most features won't apply to most categories — but the 2-3 that do apply are now visible and addressed.

Part 3

BA / QA roles in security

You're not penetration testers. But you are first-line defenders — your spec quality and verification rigor shapes whether the right defenses get built and tested.

BA's security spec work

  • Authentication — MFA, session timeout, password policy, lockout rules
  • Authorization — who can do what (role-based matrix)
  • Rate limiting — how many attempts before lockout / throttle
  • Input validation — format, length, allowed characters per field
  • Failure behavior — error messages helpful but not too helpful
  • Audit logging — who did what when (without leaking PII)
  • Encryption — in transit, at rest, end-to-end (be explicit)

QA's security verify work

  • Try injection inputs: '; DROP TABLE users;--, <script>alert(1)</script>
  • Log in as user A, try to access user B's data
  • Test auth flow with empty / wrong / expired tokens
  • Verify rate limiting actually triggers (bombard the endpoint)
  • Verify error messages don't leak system internals (stack traces, table names)
  • Verify logs capture security events without leaking secrets
  • Run automated scans: OWASP ZAP, Burp Suite Community, Nikto on staging

What's NOT your job

Deep penetration testing, red team exercises, exploit development, code-level security review — these belong to security specialists. But the everyday discipline of "did we think about each OWASP category?" is BA/QA territory.

Part 4

Observability · the three pillars

Observability is your ability to understand what's happening inside the system from outside it. Three different signal types, three different questions, all three needed.

PillarQuestionVolumeBest forTools
LogsWhat happened?High (one event per action)Debugging incidents, audit trails, compliance evidenceELK, Loki, Datadog, Splunk
MetricsHow much / how often / how fast?Low (aggregated)Dashboards, alerts, SLAsPrometheus, Datadog, Grafana, CloudWatch
TracesHow did this one request flow?Sampled (1-10% typically)Distributed systems, finding the slow serviceJaeger, Tempo, Datadog APM, Honeycomb

How they work together

A typical incident response uses all three:

  1. A metric alert fires. Error rate on /payments crossed 1% threshold.
  2. Open a trace. A failing request shows the chain — Web → API → Payment Service → Biller. Latency jumps in Payment Service.
  3. Read the logs from Payment Service at that timestamp. See the actual exception.
  4. Fix → verify the metric returns to baseline.
Part 5

Good logs vs bad logs

Fields every log line should have

  • Timestamp — ISO 8601, with timezone (e.g. 2026-04-28T14:32:01.234+06:00)
  • Severity — DEBUG · INFO · WARN · ERROR · FATAL
  • Service / module — payment-service, login-handler
  • Trace ID / request ID — to correlate across services
  • User context — user ID (NOT name/email), tenant ID, session ID
  • Event — what happened, in structured form (JSON preferred)
  • Outcome — success/failure + reason

What never goes in a log

  • Passwords, MPINs, OTPs, session tokens — PCI/DPPA/GDPR violation
  • Full credit card numbers (PAN) — PCI violation
  • Sensitive PII unless absolutely needed (and then encrypted)
  • Large blobs (full request bodies) — fill disk and capture leaks
  • Unstructured free text where you'll need to parse later — use structured fields

QA's log-review check

When verifying a feature, also check the logs:

  • Can you tell what happened from the log line alone? If not, dev didn't log enough.
  • Do you see PII you shouldn't? Dev logged too much.
  • Both are bugs. File them.
Part 6

The metrics every system needs

Google's "four golden signals" (from the SRE book)

  • Latency — how long requests take. Watch p50, p95, p99 (the tails matter most — the average lies).
  • Traffic — how many requests per second / minute
  • Errors — % of requests that fail (4xx user errors, 5xx server errors — distinguish)
  • Saturation — how "full" the system is (CPU, memory, connection pools, queue depth)

Plus business metrics — the ones BAs care about

  • Active users — DAU (daily), WAU (weekly), MAU (monthly)
  • Conversion rate — % completing the funnel
  • Feature adoption — % using a new feature
  • Revenue per user
  • Customer satisfaction (NPS, CSAT) — gathered separately, not from app metrics

The "p" numbers explained

p50 = the value 50% of requests are below (the median). p95 = 95% are below. p99 = 99% are below.

Why not just use the average? Because averages hide the bad experiences. If most requests take 100ms but 5% take 10 seconds, the average looks fine but 5% of users are angry. p95 / p99 tell that story.

Part 7

SLI · SLO · SLA · alerts

Three terms that turn metrics into operational discipline.

TermMeaningExample
SLI · Service Level IndicatorA specific measured thing"p95 latency on /payments endpoint"
SLO · Service Level ObjectiveThe target value for an SLI"p95 latency < 500ms, 99.9% of the time, measured monthly"
SLA · Service Level AgreementAn SLO codified in a contract with consequences"99.9% uptime; 10% credit if missed"
AlertA rule that fires when an SLI breaches its threshold"Page on-call if error rate > 1% for 5 minutes"

Error budget

The inverse of an SLO. If 99.9% uptime is the SLO, you have 0.1% (~43 minutes per month) of "error budget" — time the system can be down before violating the SLO. When the budget is exhausted, no new releases until reliability is restored. This forces a tradeoff between feature velocity and reliability.

BA / QA implications

  • BA: for any new feature with non-trivial user impact, spec the SLI(s) and SLO(s) it should hit. Without them, "fast enough" is undefined.
  • QA: verify the SLI is actually measured before launch. If it's not instrumented, it doesn't exist.
Part 8

Telemetry · funnels and KPIs

Observability tells you the system works. Telemetry tells you whether it matters — whether anyone uses the feature you built and whether it solved their problem.

What telemetry captures

  • Page views / screen views — which screens are seen
  • Events — clicks, form submissions, completions
  • User properties — segment, plan, tenure (for grouping)
  • Funnels — step 1 → step 2 → step 3 — where do users drop off?
  • Cohorts — group users by signup week and track behavior over time
  • A/B test outcomes — which variant performed better

Common tools

  • Product analytics: Mixpanel, Amplitude, PostHog, Heap
  • Web analytics: Google Analytics, Plausible, Fathom
  • Event routing: Segment
  • Feature flags + experiments: LaunchDarkly, GrowthBook, Unleash

The funnel example · mBoB bill payment

  1. Open Bill Pay screen — 10,000 users
  2. Select biller — 8,200 (-18%)
  3. Enter amount — 7,400 (-10%)
  4. Enter MPIN — 5,100 (-31%)
  5. Success — 4,800 (-6%)

Biggest drop is at step 4 (MPIN entry). Without telemetry, you'd never know. With it, you know exactly where to focus UX research and improvement.

Part 9

BA / QA roles in telemetry

BA's telemetry work

  • Spec the success metrics for the feature (per BRD §13)
  • Spec which events to capture and their schema
  • Spec the funnel definition (steps 1-N)
  • Spec what segments / cohorts matter for analysis
  • Spec privacy: no PII in event payloads
  • Define "done" in measurable terms (e.g. "10% adoption in 60 days")
  • Review post-launch metrics; iterate or kill features that don't perform

QA's telemetry work

  • Verify each spec'd event actually fires
  • Verify event schemas match spec (correct field names, types, values)
  • Verify event timing — fires when expected, not before, not after
  • Verify no PII leaks into event payloads
  • Verify dashboards show events correctly end-to-end
  • Catch missing events before they're noticed in production

The trap

Features ship with broken telemetry. Two months later someone wants to know "how is the feature doing?" and the answer is "we don't know — the events never fired." Verify before launch. Fix telemetry bugs the same priority as functional bugs.

Part 10

Common mistakes

Security

  • "It's behind a login, so it's secure" — broken access control is the #1 OWASP issue. Test inside-the-fence too.
  • Helpful error messages — "Username doesn't exist" tells attackers which usernames are valid. Use generic "Invalid credentials".
  • No rate limiting — endpoint can be hammered; brute force becomes practical.
  • Outdated dependencies — Log4Shell-style attacks via libraries you depend on. Run dependency scans (Snyk, Dependabot).

Observability

  • Logs without trace IDs — can't correlate across services. Useless for distributed debugging.
  • Alerts on everything — alert fatigue. On-call ignores everything because of false positives. Alert on SLO breach, not raw metric noise.
  • Logs leak PII — compliance violation. Sanitize before logging.
  • No metrics on new features — can't tell if they work. Spec metrics in the BRD.
  • Average-only metrics — averages lie. Always look at percentiles.

Telemetry

  • Tracking everything — overwhelm; no one looks at 90% of events. Spec the few that matter.
  • Inconsistent event names — btn_click, buttonClicked, button_click — same event, three names. Pick a schema.
  • Events fire incorrectly — fire twice, miss the conversion, fire on hover. Verify before launch.
  • No segmentation — averaged-across-everyone metrics hide the truth. Always look at cohorts.
  • Vanity metrics — "10,000 page views!" tells you nothing about whether anyone got value. Track outcomes, not raw counts.
Takeaways

Key takeaways

What to carry forward

  • Security · OWASP Top 10 is your mental checklist. Walk through it during BRD review.
  • BAs spec defenses; QAs verify them. Not pen-testers — first-line defenders.
  • Three pillars of observability: logs (what?) · metrics (how much?) · traces (how did this request flow?)
  • Logs need structured fields, trace IDs, no PII.
  • Four golden signals: latency · traffic · errors · saturation. Watch p95/p99, not averages.
  • SLI / SLO / SLA · spec the targets, monitor against them, alert on breaches.
  • Error budget gives a real currency for the velocity-vs-reliability tradeoff.
  • Telemetry · usage, funnels, KPIs · the answer to "did the feature work?"
  • Telemetry events need their own QA — verify they fire correctly before launch.
  • If you can't see it, you can't defend it, fix it, or improve it.
