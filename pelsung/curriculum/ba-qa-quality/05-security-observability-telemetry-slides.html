02
Why these three together

"Is the system being attacked? Is it working? Is anyone using it?"

Three different post-launch questions. They share one answer: you must instrument the system to be able to see inside it.

Security needs visibility into attempted attacks. Observability needs visibility into errors and latency. Telemetry needs visibility into what users actually do.

If you can't see it, you can't defend it, fix it, or improve it.

Part One

Security awareness for BA & QA

You won't write the defenses. But you must spec them, verify them, and recognize when they're missing.

04 OWASP Top 10 · the application security canon

Updated every few years · the most common web app vulnerabilities

A01:2021
Broken Access Control

Users can act outside their permissions. The most common category.

A02:2021
Cryptographic Failures

Sensitive data exposed via weak or missing encryption.

A03:2021
Injection

SQL injection, XSS, command injection. User input treated as code.

A04:2021
Insecure Design

Vulnerabilities baked into the architecture from the start.

A05:2021
Security Misconfiguration

Default passwords, exposed admin interfaces, debug mode in production.

A06:2021
Vulnerable Components

Outdated libraries with known exploits. (Log4Shell, anyone?)

A07:2021
Identification & Auth Failures

Weak passwords, broken session management, no MFA.

A08:2021
Software & Data Integrity Failures

Unsigned updates, untrusted CI/CD pipelines, supply chain attacks.

A09:2021
Security Logging & Monitoring Failures

Can't detect attacks because nothing is logged or watched.

A10:2021
Server-Side Request Forgery

App can be tricked into making requests on the attacker's behalf.

You don't need to memorize the list. You need to know it exists, walk through it during BRD review, and verify your team has covered each item.

05 What BAs and QAs do about security

You're not pen-testers · you are first defenders

BA's security spec work

  • Spec authentication: MFA, session timeout, password policy
  • Spec authorization: who can do what (use a role-based matrix)
  • Spec rate limiting: how many attempts before lockout
  • Spec input validation: format, length, allowed characters
  • Spec what happens on failure (helpful but not too helpful)
  • Spec audit logging: who did what when
  • Spec encryption requirements: in transit, at rest, end-to-end

QA's security verify work

  • Try injection inputs: '; DROP TABLE users;-- etc.
  • Try logging in as one user, accessing another's data
  • Try the auth flow with empty / wrong / expired tokens
  • Verify rate limiting actually triggers
  • Verify error messages don't leak system internals
  • Verify logs capture security events without leaking PII
  • Run automated scans (OWASP ZAP, etc.) on staging

The deeper testing (penetration testing, red team) is for security specialists. But the everyday discipline of "did we think about each OWASP category?" is BA/QA's job.

Part Two

Observability · the three pillars

If you can't see what's happening inside the system in production, you can't fix it.

07 The three pillars · logs, metrics, traces

Different questions · different tools · all three matter

Logs

"What happened?"

Discrete events recorded as text or JSON. Most detailed but most volume.

Best for: debugging specific incidents, audit trails, compliance evidence.

Tools: ELK, Loki, Datadog, Splunk.

Metrics

"How much / how often / how fast?"

Aggregated numbers over time. Cheap to store, fast to query.

Best for: dashboards, alerts, SLAs ("p95 latency < 500ms").

Tools: Prometheus, Datadog, Grafana, CloudWatch.

Traces

"How did this one request flow through the system?"

End-to-end view of a single request across services.

Best for: distributed systems, finding the slow service, understanding cascades.

Tools: Jaeger, Tempo, Datadog APM, Honeycomb.

Together: a metric tells you something is broken (alert fires). Traces show you which service. Logs show you the exact error in that service. The pipeline.

08 What makes a log line useful

Structured · contextual · safe

The fields every log line should have

  • Timestamp — ISO 8601, with timezone
  • Severity — DEBUG · INFO · WARN · ERROR · FATAL
  • Service / module — payment-service, login-handler
  • Trace ID / request ID — to correlate across services
  • User context — user ID (NOT name/email), tenant ID, session ID
  • Event — what happened, in structured form (JSON preferred)
  • Outcome — success / failure + reason

What never goes in a log

  • Passwords, MPINs, OTPs, session tokens (PCI/DPPA/GDPR violation)
  • Full credit card numbers (PCI violation)
  • Sensitive PII unless absolutely needed (and then encrypted)
  • Large blobs (request bodies in full) — they fill disk and capture leaks
  • Unstructured free text where you'll need to parse later

QA: when verifying a feature, check the logs. If you can't tell what happened from the log line, the dev didn't log enough. If you see PII you shouldn't, the dev logged too much. Both are bugs.

09 The metrics every system needs

Four golden signals · plus business metrics

Google's "four golden signals" (SRE book)

  • Latency — how long requests take. Watch p50, p95, p99 (the tails matter most).
  • Traffic — how many requests per second / minute
  • Errors — % of requests that fail (4xx user errors, 5xx server errors)
  • Saturation — how "full" the system is (CPU, memory, connection pools)

Plus business metrics (the ones BAs care about)

  • Active users — DAU, WAU, MAU
  • Conversion rate — % completing the funnel
  • Feature adoption — % using a new feature
  • Revenue per user
  • Customer satisfaction (NPS, CSAT) — gathered separately

If a feature ships without metrics, you don't know if it worked. The BA's job is to spec which metrics matter before launch.

10 Alerts, SLIs, SLOs · turning metrics into action

Three terms that separate "data" from "operations"

  • SLI · Service Level Indicator — a specific measured thing. "p95 latency on /payments endpoint."
  • SLO · Service Level Objective — the target value for an SLI. "p95 latency < 500ms, 99.9% of the time, measured monthly."
  • SLA · Service Level Agreement — an SLO codified in a contract with consequences (refunds, etc.) if violated.
  • Alert — a rule that fires when an SLI breaches its threshold. "Page on-call if error rate > 1% for 5 minutes."
  • Error budget — the inverse of an SLO. If 99.9% uptime is the SLO, you have 0.1% (~43 min/month) of error budget. When it's exhausted, no new releases.

BA implication: for any new feature with non-trivial user impact, spec the SLI(s) and SLO(s) it should hit. QA implication: verify the SLI is actually measured before launch.

Part Three

Telemetry · what users actually do

Observability tells you the system works. Telemetry tells you whether it matters.

12 Telemetry · usage, funnels, KPIs

Beyond uptime · is the feature actually being used?

What telemetry captures

  • Page views / screen views — which screens are seen
  • Events — clicks, form submissions, completions
  • User properties — segment, plan, tenure
  • Funnels — step 1 → step 2 → step 3 — where do users drop off?
  • Cohorts — group users by signup week and track behavior over time
  • A/B test outcomes — which variant performed better

Common tools

  • Mixpanel · Amplitude · PostHog · Heap (product analytics)
  • Google Analytics · Plausible (web analytics)
  • Segment (event routing)
  • LaunchDarkly · GrowthBook (feature flags + experiments)

Telemetry events are like log events — they need a schema, they need to be consistent across the codebase, and they need to be safe (no PII).

14 BA spec · QA verifies events fire correctly

Telemetry has its own QA discipline

BA's telemetry work

  • Spec the success metrics for the feature (per BRD §13)
  • Spec which events to capture and their schema
  • Spec the funnel definition (steps 1-N)
  • Spec what segments / cohorts matter
  • Spec privacy: no PII in event payloads
  • Define "done" in terms of metrics (e.g. "10% adoption in 60 days")
  • Review post-launch metrics; iterate

QA's telemetry work

  • Verify each spec'd event actually fires
  • Verify event schemas match spec (correct field names / types)
  • Verify event timing (fires when expected, not before/after)
  • Verify no PII leaks into event payloads
  • Verify dashboards show events correctly
  • Catch missing events before they're noticed in production

The trap: features ship with broken telemetry. Two months later someone wants to know "how is the feature doing?" and the answer is "we don't know — the events never fired." Verify before launch.

15 Common mistakes

Across all three areas

Security

  • "It's behind a login, so it's secure" — broken access control is #1 OWASP issue. Test inside-the-fence too.
  • Helpful error messages — "Username doesn't exist" tells attackers which usernames are valid. Use generic "Invalid credentials."
  • No rate limiting — endpoint can be hammered. Brute force becomes practical.
  • Outdated dependencies — Log4Shell-style attacks via libraries you depend on. Run dependency scans.

Observability

  • Logs without trace IDs — can't correlate across services. Useless for distributed debugging.
  • Alerts on everything — alert fatigue. On-call ignores everything because of false positives. Alert on SLO breach, not raw metrics.
  • Logs leak PII — compliance violation. Sanitize before logging.
  • No metrics on new features — can't tell if they work. Spec metrics in the BRD.

Telemetry

  • Tracking everything — overwhelm; no one looks at 90% of events. Spec the few that matter.
  • Inconsistent event names — btn_click, buttonClicked, button_click — same event, three names. Pick one schema.
  • Events fire incorrectly — fire twice, miss the conversion, fire on hover. Verify before launch.
  • No segmentation — averaged-across-everyone metrics hide the truth. Look at cohorts.
