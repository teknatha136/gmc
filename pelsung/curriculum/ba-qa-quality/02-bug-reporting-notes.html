Part 1

Why bug reports matter

A vague bug report — "login is broken, please fix urgently" — triggers a chain of clarifying messages between QA, dev, support, and PM that can take days to resolve before any actual fixing happens. A clear bug report gets a developer straight to the problem.

The math is simple: a developer's time is expensive. A 5-minute investment in writing a clear report saves 30+ minutes on the receiving end and avoids the "can't reproduce → reopened" cycle that wastes everyone's day.

Mental model

Treat a bug report like a tiny specification. It has the same goal: tell the reader exactly what's wrong, exactly how to recreate it, and exactly what should happen instead. Anything less and you're outsourcing your own thinking to the developer.

Part 2

Anatomy of a great bug report

Every bug report should have these eight fields. Most bug trackers (Jira, Linear, GitHub Issues, Azure DevOps) let you save this as a template — use it.

Title: [What's wrong] [where] [under what condition] Example: "Login button unresponsive on mBoB v3.4 (Android 13) when MPIN starts with 0" Environment: - App / build version - OS + version - Browser + version (if web) - Device model - Server tier (dev / staging / prod) Steps to reproduce: 1. Numbered list 2. Short verbs 3. Exact inputs (e.g. "Enter MPIN: 0123" not "Enter some MPIN") 4. "Click Login button" not "interact with the form" Expected result: What should happen, per spec or common sense. Actual result: What happened instead. Attach screenshot / video / logs. Reproducibility: - Always (5/5) - Sometimes (e.g. 3/10) - Rarely (e.g. 1/50) Severity: Critical · High · Medium · Low Priority: P0 · P1 · P2 · P3 Optional but valuable: - Screenshot / video / HAR file / logs - Workaround if any - Related bugs (#) - Regressed-since-build (which build first showed it)

Title-writing rule

Imagine 200 bugs in a list. Your title should be enough to:

  • Tell a triage person whether to read the body
  • Help someone search for "is this already filed?"
  • Give a sense of severity at a glance

Bad: "Login broken." Good: "Login fails with 500 error when email contains apostrophe (Chrome 122, prod)."

Part 3

Severity vs Priority — the eternal confusion

The single most-misused pair of words in bug tracking. They are independent dimensions and you should set them separately.

DimensionQuestionSet byReflects
SeverityHow bad is the user impact?QA / reporterObjective technical impact
PriorityHow soon must we fix it?PM / BA / triageSubjective business decision

Severity scale (typical 4-level)

  • Critical (S1) — system unusable, data loss, security breach, complete outage
  • High (S2) — major function broken with no workaround
  • Medium (S3) — function broken with workaround, or minor function fully broken
  • Low (S4) — cosmetic issue, typo, edge case nobody hits, minor inconvenience

Priority scale (typical P0-P3)

  • P0 — drop everything, fix today (production outage, security, regulatory)
  • P1 — fix this sprint
  • P2 — fix next sprint or two
  • P3 — backlog, fix when convenient (or never)

Why they're different — the four corners

  • High sev · High pri — Login broken, users locked out. Drop everything. (S1/P0)
  • High sev · Low pri — Crash that only happens in the year 2099. (S1/P3)
  • Low sev · High pri — Typo in legal disclaimer during a regulator inspection tomorrow. (S4/P0)
  • Low sev · Low pri — Footer copyright shows 2024 instead of 2025. (S4/P3)

The lesson: a critical-severity bug isn't always urgent, and a trivial bug can be urgent. Set them independently. Their combination drives the fix order.

Part 4

The defect lifecycle

Every bug travels through a state machine. Most teams use a variation of this:

StateMeaningWho advances it
New / OpenJust filed. Awaits triage.Reporter (QA / BA / support)
TriagedReviewed. Severity confirmed, priority set, owner assigned.Triage team
In ProgressA developer is actively fixing it.Developer
FixedCode changed, dev believes resolved, awaits QA verification.Developer
VerifiedQA confirmed the fix works in staging / test env. No regression caused.QA
ClosedDone. Often only happens after the fix ships to production.QA / release manager
ReopenedBug came back. Either verification failed, or regressed in production.QA / support
Rejected / Won't Fix / DuplicateTerminal states for invalid reports.Triage team

The reopened cycle

A bug going from Fixed → Reopened twice is a sign of one of these problems:

  • The original report was vague — dev fixed something different
  • The fix only addresses the symptom, not the root cause
  • The bug has multiple triggers and only some are fixed
  • The verification environment doesn't match production

If a bug reopens twice, escalate. Don't just fix-and-close again.

Part 5

Triage · the meeting that decides

Triage is the regular ceremony where new bugs get classified, prioritized, and assigned. Without it, bugs pile up unread, severity drifts, and important issues get lost.

Who's in the room

  • QA lead — represents reported bugs, advocates for severity
  • Product / BA — represents user impact and roadmap, owns priority
  • Engineering lead — represents capacity, owns assignment
  • Optional: support lead (if customer-reported), security lead (if security-flagged)

What gets decided per bug

  • Is this actually a bug? Reject / dedupe / mark as feature request if not. ~20-40% of incoming reports get rejected here in well-run teams.
  • Severity — objectively, how bad is the user impact?
  • Priority — when must this be fixed? P0/P1 → this sprint; P2/P3 → backlog
  • Owner — which dev / team takes it
  • Target release — current build, hotfix, or next release?

Cadence

  • P0 bugs — triaged immediately on filing (within an hour of report)
  • Daily triage — for active sprints, 15–30 minutes each morning
  • Weekly triage — for backlog grooming, ~1 hour

Skip a triage cycle and the backlog becomes unreviewable. It's compounding debt.

Part 6

Bug categories

Tag every bug by category. Different categories often need different handling — and metrics filtered by category give you valuable signals.

CategoryExamplesSpecial handling
FunctionalWrong behavior vs spec, missing validation, wrong outputStandard flow. Most common type.
PerformanceSlow, timeouts, memory leak, high CPUOften surfaces under load. Owned by perf or platform team.
SecurityVulnerabilities, leaks, broken auth, injectionAlways P0 or P1. Often confidential — separate disclosure process.
AccessibilityWCAG failures, broken keyboard nav, no alt text, low contrastTracked separately for compliance reporting (WCAG AA).
Usability / UXWorks but confusing, hard to find, frustrating"Not technically a bug" but real user impact. Often becomes a feature request.
DataWrong calculation, corrupted record, lost data, double-entryOften urgent because data loss compounds. Recovery plan needed.
LocalizationWrong language, wrong format, broken layout in target localeAffects only specific markets — flag those markets explicitly.
RegressionSomething that worked before is now brokenSpecial priority — trust is at stake. Often P0/P1 even for low severity.
Part 7

Accessibility bugs · special handling

Accessibility bugs deserve separate tracking for two reasons:

  1. Compliance reporting. WCAG conformance is increasingly audited (EAA in EU, Section 508 in US, etc.). You need to be able to answer "how many WCAG 2.1 AA violations are open?" without scanning every bug.
  2. Different remediation approach. Accessibility bugs often need design + dev collaboration: semantic HTML, focus order, ARIA labels, contrast adjustments. Pure dev fixes often miss the accessibility goal.

Severity guidance for accessibility bugs

  • Critical — feature completely unusable for users with assistive tech (e.g., login form not navigable by keyboard)
  • High — feature usable but with significant friction (e.g., screen reader announces button as "button" with no label)
  • Medium — minor friction (e.g., contrast slightly below WCAG AA on a label)
  • Low — best-practice violations or AAA-level failures

Tools that auto-generate accessibility bugs

  • axe DevTools (Deque) — browser extension; outputs JSON
  • WAVE (WebAIM) — visual inspection
  • Lighthouse — built into Chrome DevTools
  • Pa11y — CLI for CI integration

These tools generate violations in structured form. Pipe them into your tracker tagged "accessibility-wcag-AA" for trend visibility.

Part 8

Who files bugs · BAs file too

QA's bug-reporting work

  • File bugs found in test execution
  • Confirm reproducibility before filing
  • Assign initial severity
  • Verify fixes after dev marks Fixed
  • Maintain regression test for any Closed bug
  • Track defect metrics

BA's bug-reporting work

  • File bugs found in spec review or UAT
  • Translate user complaints into well-formed reports
  • Set priority during triage (business impact lens)
  • Decide if a bug → spec change or pure fix
  • Communicate fix status to stakeholders
  • Watch for "bugs that are actually missed requirements"

The hidden category — missed requirements

Some "bugs" aren't defects at all — they're missed requirements. The system does what was specified, but the spec was wrong or incomplete. Examples:

  • "The form lets you submit without filling the phone number" — but the spec didn't say phone was required
  • "It uses the wrong date format" — but the spec didn't specify a format
  • "It doesn't work on Internet Explorer" — but the spec said browsers, not which ones

The BA owns sorting these from real defects. Real bug → fix it. Missed requirement → update the spec, then either build it or backlog it. Don't blame QA for "filing a non-bug" — they reported a real user problem; the system's response just happens to require a spec update rather than code.

Useful metrics for QA leads

Bug data is signal. A few metrics worth watching:

MetricDefinitionWhat it tells you
Defect escape rate% of bugs found in production vs caught before releaseQuality of pre-release testing. Aim < 5%.
Mean time to fix (MTTF)Average days from Open → ClosedTeam velocity on fixes. Watch trends.
Reopen rate% of bugs that get reopened after FixedQuality of fixes. High reopen = root-cause not addressed.
Defect densityBugs per 1000 lines of code (or per feature)Hot spots — which modules are bug-prone
Severity distributionCount by severity levelEarly warning: rising critical bugs = quality crisis
Age of open bugsDays each open bug has been openBacklog health. Bugs older than 6 months should be closed or fixed.
Bug categories trendBugs by category over timeWhere to invest — rising perf bugs = need perf testing

Takeaways

Key takeaways

What to carry forward

  • A bug report is a tiny spec — eight fields: title · env · steps · expected · actual · repro rate · severity · priority.
  • Severity = how bad the impact is. Priority = how soon to fix. They're independent dimensions.
  • The lifecycle: New → Triaged → In Progress → Fixed → Verified → Closed, with backward paths when things fail.
  • Triage is a regular ceremony — daily for active sprints, weekly for backlog. Without it, bugs rot.
  • Tag categories: functional · performance · security · accessibility · UX · data · localization · regression.
  • Accessibility bugs need separate tracking for compliance reporting.
  • BAs file bugs too — and watch for "bugs that are really missed requirements."
  • Useful metrics: escape rate · MTTF · reopen rate · defect density · severity distribution · age.
  • A clear bug report is a gift to your dev team. A vague one is an unpaid invoice for their time.
