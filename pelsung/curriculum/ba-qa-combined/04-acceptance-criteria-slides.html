04 Same story, two implementations

Same story. Two devs. Two very different apps.

Developer A built…
1. Enter email
2. Get reset link via email
3. Click link, set new password
4. Log in with new password

~30 min. Works. But: what if user has no email? What if link expires? What if they re-use their old password?

Developer B built…
1. Enter phone number
2. Get OTP via SMS (valid 5 min)
3. Enter OTP + set new password
   (must be ≥ 8 chars, not recent 5)
4. Auto-login + email confirmation

~2 hours. Works. But: phone-only excludes overseas users. What if SMS is down?

Both "finished" the story. Neither was wrong. They just filled the ambiguity with different assumptions — and neither version matches what a manager or QA actually expected.

AC is what closes that gap.

05
Part One

What is
Acceptance Criteria?

The shared contract between BA, Dev, and QA.

01
06 The 20-second definition

Acceptance criteria are the specific, testable conditions a user story must meet for everyone to agree it's "done".

Each AC is a pass/fail statement — either the system does this when that happens, or it doesn't. If a QA can't answer "yes" or "no" after testing, the AC isn't written well enough yet.

Where AC fits

User story = what we want and why. Acceptance criteria = how we'll know we got it right. One story has multiple AC. AC can't stand alone — they always belong to a story.

07 The shared artifact

One artifact. Three different readers. Each uses it differently.

📋 The BA

Writes AC. Uses them to clarify scope with stakeholders and to ensure nothing was assumed silently. If there's no AC, the story isn't ready.

💻 The Developer

Builds to AC. They're the "definition of done" from the implementer's side. No guessing required — the behaviour is specified.

🔍 The QA

Verifies AC. Each AC becomes at least one test case. QA signs off by walking every AC and confirming pass/fail.

Why this matters

Nowhere else in software do three different roles depend on the same sentences for their job. That's what makes AC the single strongest lever in the BA/QA/Dev relationship — and why bad AC ripples everywhere.

08
Part Two

Given
When
Then

The industry-standard format for AC.

02
09 The format

Three lines. Every AC. Every time.

Given   the starting situation  

When   the user does a specific action  

Then   a specific thing happens  

What each line does

  • Given — the context or precondition
  • When — the triggering action or event
  • Then — the expected, observable outcome

Why this shape works

  • Forces you to specify state before action
  • Makes the cause → effect explicit
  • Maps cleanly to a test case (precondition → step → expected)
  • Reads well in plain English even to a non-technical stakeholder
10 A filled-in example

Story: "Save a biller as a favourite"

Given the user has just completed a bill payment successfully,

When they tap the "⭐ Save as favourite" button on the receipt screen,

Then the biller appears in their Favourites list on the Bill Payments home within 1 second.

Notice how this AC is:

  • Specific — one action, one outcome
  • Testable — a QA can verify yes/no
  • Observable — about behaviour, not implementation ("appears in list" — not "adds a row to favourites_table")
  • Bounded — "within 1 second" quantifies it
11 One story → multiple AC

A real story needs several AC to cover the behaviour

A good rule: 3 to 7 AC per story. Fewer = probably missing edge cases. More = story is too big; split it.

Story: "As a user who pays repeat billers, I want to save billers as favourites so I can pay in 2 taps."
AC 1 · Happy path · save

Given the user completed a payment, when they tap ⭐ Save, then the biller appears in Favourites within 1 second.

AC 2 · Use the favourite

Given the user has a saved favourite, when they tap it on the home screen, then the payment form is pre-filled with biller + last amount; MPIN still required to confirm.

AC 3 · Remove a favourite

Given a saved favourite, when the user long-presses it and taps "Remove", then the favourite disappears immediately and no payment is triggered.

AC 4 · Edge · limit reached

Given the user already has 10 saved favourites, when they tap ⭐ Save, then the system shows a message "You've reached the limit of 10. Remove one to save this." and no favourite is added.

AC 5 · Edge · offline

Given the user is offline, when they tap ⭐ Save, then the save is queued locally and shows "Will sync when online"; it appears in the live list once the connection returns.

Five AC. One happy path · one inverse (remove) · two edges. That's a well-scoped story.

Part Three

What Good
AC Looks Like

Six properties · one checklist · zero ambiguity.

03
14 Six properties of good AC

Good AC are SCARCE — use this as your checklist

🎯 S · Specific

Anyone reading the AC gets the same mental picture. No room for two interpretations.

"appears within 1 second" — not "appears quickly"
📏 C · Concrete

About behaviour you can observe, not implementation or intent.

"shows a red error label" — not "validates the input"
✅ A · Atomic

One criterion = one check. If the AC contains "and" — split it into two.

One AC for save; one AC for email confirmation
🔁 R · Repeatable

Running the same test twice should give the same result. No reliance on timing luck or mystery state.

Reset the user's state before the "Given"
🧪 C · Checkable

A QA can verify yes/no after executing the When. No judgement calls.

Use exact wording, counts, timings — not feelings
⚖️ E · Equal-parties

The BA, Dev, and QA all read the same AC and all agree on what it means, with no private knowledge needed.

If only the BA knows "we meant X" — the AC is bad
15 Bad vs good AC

Same feature · two AC · one of them useful

❌ Bad
AC: The app should handle
    forgotten passwords in
    a user-friendly way and
    let users reset quickly.

Fails every SCARCE property. Vague ("user-friendly"), no observable behaviour, not atomic, not repeatable, definitely not checkable.

✅ Good
AC: Given the user taps "Forgot
    password" on the login screen,
    When they enter a registered
    phone number,
    Then the system sends an OTP
    within 10 seconds and navigates
    to the OTP entry screen.

Specific · concrete · atomic · repeatable · checkable. A QA can write a test; a dev can build it; a stakeholder can read it.

Part Four

Common AC
Mistakes

Six ways AC goes wrong — and how to catch each one.

04
17 Six common AC mistakes

Every new BA makes these. Spot them now.

1. Only the happy path

You wrote the AC for when everything goes right. You wrote none for when things don't.

Fix: for every story, write at least one AC for a limit, one for an error, one for an edge.
2. Implementation leak

"Then insert a row into the favourites table." That's how, not what. Constrains the dev for no user value.

Fix: ask "what does the user see or experience?" Describe that, not the DB.
3. Vague verbs

"The system handles errors gracefully." What does handle mean? To whom? In what form?

Fix: replace vague verbs with observable ones. "Shows" "displays" "returns" "navigates" "logs".
4. Mixing multiple behaviours

"When user X, then the system does A and B and C." Fails atomicity.

Fix: split at every "and" in the Then. Three behaviours = three AC.
5. Missing context

Jumping to "When" without a "Given". Relies on a shared assumption about starting state.

Fix: always state the precondition. "Given the user is logged in" — even if it feels obvious.
6. Untestable adjectives

"Intuitive." "Fast." "Smooth." "Delightful." Mood, not a requirement.

Fix: turn them into numbers or concrete behaviours. "Within 2 seconds." "Within 3 taps."

Part Five

AC Lifecycle

How AC flows through the team.

05
19 The AC journey

BA drafts → QA reviews → Dev builds → QA verifies

Step 1
📋 BA drafts

Writes 3–7 AC in Given/When/Then form based on stakeholder conversations and user stories.

Step 2
🔍 QA reviews

Reads AC with a QA lens. Challenges vagueness, asks about edges, adds missing scenarios. Before dev picks it up.

Step 3
💻 Dev reads + estimates

Uses AC to scope the work. If any AC is unclear or huge, goes back to BA/QA — blocks on dev start.

Step 4
💻 Dev builds to AC

Implementation confirms every AC. Writes tests alongside where possible.

Step 5
🔍 QA verifies

Walks every AC on the built software. Passes/fails each one. Files bugs for any fail.

Step 6
📋 BA confirms

Final check: does the software deliver the story, not just the AC? Signs off for release.

The most common mistake

Skipping Step 2 · QA review. Teams write AC, hand to dev, and QA only sees the AC when the build lands. By then the AC's weaknesses cost a cycle of rework. QA reviewing AC before dev starts is the single highest-leverage habit on this list.

Part Six · Hands-on

Write AC
for Password Reset

Pairs · 10 minutes · cover happy + 3 edges.

06
22 The brief

Write 4–6 Given/When/Then AC for this story

User Story
Forgotten-password reset

As a user who has forgotten my password,
I want to reset it online
so that I can log back in without contacting support.

Must cover

  • The happy path (user successfully resets)
  • At least 3 edges:
    • Unregistered email/phone
    • Expired link/OTP
    • Re-using the old password

How to work

  • Pair up (BA-track + QA-track if possible)
  • Start with the happy path
  • One of you writes Given/When/Then, the other challenges it against SCARCE
  • Cycle until both agree every AC is testable
  • Volunteer pairs share in debrief

7 min writing · 3 min pair challenge · volunteer share in debrief.

23 Debrief · a reference answer

One defensible set of AC for password reset

AC 1 · Happy path

Given the user is on the login screen and taps "Forgot password", when they enter a registered phone number and tap Submit, then the system sends a 6-digit OTP via SMS within 10 seconds and navigates to the OTP entry screen.

AC 2 · Complete reset

Given the user is on the OTP entry screen with a valid OTP, when they enter the OTP and a new password meeting policy (≥ 8 chars, not one of last 5 used), then the password is updated and they are navigated to a "Reset successful" screen with a "Log in" button.

AC 3 · Edge · unregistered

Given the user enters a phone number not registered in the system, when they tap Submit, then the system shows "If this number is registered, an OTP will arrive shortly" (same message as success — to avoid account enumeration) and takes no action.

AC 4 · Edge · expired OTP

Given an OTP was issued more than 5 minutes ago, when the user enters it, then the system shows "This code has expired. Request a new one." and provides a "Resend OTP" button.

AC 5 · Edge · re-used password

Given the user enters a new password that matches any of their last 5 passwords, when they submit, then the system shows "Please choose a password you haven't used recently" and leaves the old password unchanged.

AC 6 · Edge · rate limit

Given the user has requested 3 OTPs for the same number in the last 10 minutes, when they try again, then the system shows "Too many attempts. Try again in 10 minutes." and does not send another OTP.

6 AC · 2 happy-path steps · 4 edges. If you wrote 4–6 AC that cover the happy path + 3 edges, you're in great shape.

24 Wrap-up
🎯 Key Takeaways

What we covered today

  • ✅ AC are the specific, testable conditions a story must meet to be "done"
  • ✅ Given / When / Then is the industry-standard format
  • ✅ One story = 3–7 AC (fewer means missing edges; more means story is too big)
  • ✅ Good AC are SCARCE — Specific, Concrete, Atomic, Repeatable, Checkable, Equal-parties
  • ✅ AC is the shared artifact — BA writes, QA reviews, Dev builds, QA verifies, BA signs off
  • ✅ The highest-leverage habit: QA reviews AC before dev starts
  • ✅ Avoid the six mistakes: happy-path only, implementation leak, vague verbs, mixed behaviours, missing context, untestable adjectives

Good AC is the difference between "I thought you meant…" and "we all agreed."
