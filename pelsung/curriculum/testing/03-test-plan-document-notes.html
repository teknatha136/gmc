Part 1

What a Test Plan is · and what it is not

A Test Plan is the single document that defines testing for a defined scope — a project, a release, or a feature. It answers five questions up front, so nobody has to ask them mid-cycle:

  1. What is being tested? (items · versions · features in scope / out of scope)
  2. How will it be tested? (levels, types, techniques, tools)
  3. Who will test it? (roles, skill levels, staffing, responsibilities)
  4. When will it be tested? (milestones, entry / exit criteria, calendar)
  5. How will we know we're done? (pass/fail criteria, release readiness, metrics)

✅ A Test Plan is…

  • A strategy document — sets direction, not test cases
  • A contract between QA, Dev, and PM on what "done" means
  • Released-scoped — one per project or release
  • Signed off before testing starts; referenced throughout
  • The doc a new joiner reads to get oriented in 20 minutes

❌ A Test Plan is NOT…

  • A list of individual test cases (those live in TestRail / Xray / Zephyr)
  • A QA strategy (that's organisation-wide; the Test Plan cites it)
  • A defect report
  • A "what I did this week" status update
  • A document written AFTER testing starts
Part 2

Why written Test Plans exist · the 4 fires they put out

Teams without a Test Plan run into the same four fires, every release:

FireWhat it looks likeHow a Test Plan stops it
"Who tested that?"Mid-incident, nobody can say which team verified feature X§ 15 Responsibilities explicitly names R for each test area
"Are we done?"PM pushes for release; QA says "still testing"; nobody has a definition§ 11 Exit Criteria — a release-readiness check-list
"That was never in scope"A bug lands post-launch; dev says "not tested because nobody said to"§ 05 Features to be Tested + § 06 Features NOT to be Tested (with reason)
"We can't start testing"Build not deployed, test data not refreshed, yet QA is scheduled§ 09 Entry Criteria — pre-flight check-list

Rule of thumb

The Test Plan is for everyone, not just QA. A dev reads it to know which parts of their code will be exercised and when. A PM reads it to know when they'll have signal to ship. A support lead reads it to know when to staff up. Sponsors read the exit criteria to sign off the release.

Part 3

Project Test Plan vs Release vs Feature

Test Plans nest. The plan for a whole project is the parent; release-level plans inherit from it; feature-level plans inherit from release. Each level has a different length.

LevelScopeLengthWhen to write one
Project Test PlanWhole project / product lifecycle15–30 pagesOnce · at kickoff · updated quarterly
Release Test PlanOne major release (e.g. v3.0)10–20 pagesPer release · 4–6 weeks before cut
Feature Test PlanOne significant feature3–5 pagesWhen feature is big enough to need its own strategy
Test CaseOne concrete scenario~half a pageLives in TestRail / Xray — not in a plan document

For the rest of this note, we focus on the Release Test Plan — the most common level and the one you'll author first.

Part 4

Test Plan structure · 21 sections (IEEE 829)

IEEE 829 specifies 21 sections. Not every section needs a page of text — some are two lines — but every section exists so reviewers find what they expect where they expect it.

§§ 01–03 · Identification

Identifier · Introduction · References

Test plan ID (e.g. TP-BFAV-v1.0), status, version. Introduction (purpose, scope, audience). References (PRD, SRS, HLD/LLD, strategy docs, standards).

§§ 04–06 · What's in scope

Test Items · Features to be Tested · Features NOT to be Tested

Items = what's under test (modules · services · versions). Features-to-be-tested: traced to requirements + priority. Features-NOT-to-be-tested: explicit list with reason for each.

§§ 07–08 · Strategy

Approach · Pass/Fail criteria

Levels (unit · integration · system · acceptance) · types (functional · regression · perf · security · a11y · compat · localisation · recovery) · techniques (EP · BVA · decision tables · state transition · exploratory) · automation strategy. Pass/fail thresholds per test item.

§§ 09–11 · The gates

Entry · Suspension & Resumption · Exit criteria

When testing can start · when it must pause · when it can resume · when the release is ready to ship. Check-list style; binary yes/no per item.

§§ 12–14 · Logistics

Deliverables · Environment · Test data

What QA will produce (cases · reports · automation · traceability matrix). Environment(s) · URLs · data type · stability SLA. Test data sources, sensitive-data handling, refresh cadence.

§§ 15–17 · People & calendar

Responsibilities (RACI) · Staffing · Schedule

RACI matrix for the release (or SELISE's RCI variant). Staffing: roles, counts, names, ramp-up needs. Schedule: milestones tied to project calendar.

§§ 18–20 · Risk & ops

Risks · Metrics & Reporting · Defect management

Risks with likelihood · impact · mitigation · owner. Metrics captured daily, reporting cadence, audience. Defect taxonomy, triage cadence, SLA per severity.

§ 21 · Close

Approvals

Sign-off by QA Lead · Dev Lead · PM · Security · Release Manager. Dated and archived alongside the baselined plan.

Part 6

Test approach · levels · types · techniques

A Test Plan doesn't re-derive QA theory; it selects from the available levels / types / techniques and justifies each choice for this release.

Levels

Who tests what, at what scale

LevelWhoWhatTool(s)
UnitDevIsolated functions / classesJest · JUnit · pytest
IntegrationDev + QAService↔service · DB · cacheTestcontainers
System (E2E)QAEnd-to-end user flowsPlaywright
Acceptance (UAT)Business / QABusiness-scenario validationManual + scripted
Types

What kind of quality we're checking

Functional · Regression · Performance (load · stress · soak · spike) · Security (OWASP Top 10 · authZ matrix · deps) · Accessibility (WCAG AA · axe · keyboard · screen reader) · Compatibility (browsers · OS · devices) · Localisation · Recovery / resilience (chaos · failover) · Usability (session-based).

Techniques

How test cases are designed

Equivalence partitioning (EP) · Boundary value analysis (BVA) · Decision tables · State transition · Pairwise · Exploratory (charter-led) · Risk-based prioritisation.

Automation strategy

What is automated, what isn't, why

  • Unit: 100% of changed code paths touched (dev owns)
  • Integration: all happy paths + critical edges automated
  • E2E: top 20 user journeys automated · long-tail via exploratory
  • Perf: automated in k6 against staging · weekly baseline
  • Security: dep scan (Snyk) automated in CI · pen-test before major release
Part 7

Entry · suspension · resumption · exit criteria

Four gates govern the test cycle. Each is a check-list with binary yes/no items.

§ 09 · Entry criteria · when testing can START

  • Feature spec / SRS approved
  • Build deployed to test environment · smoke test green
  • Test data prepared & refreshed
  • Test cases reviewed & approved
  • Dev has marked ticket(s) "Ready for QA"
  • Known blocker defects resolved or flagged

§ 10 · Suspend when…

  • Smoke tests fail on a new build
  • A P0 defect blocks > 50% of remaining test cases
  • Test environment unavailable > 4 h

§ 10 · Resume when…

  • Blocker fixed & verified
  • Environment restored · data valid

§ 11 · Exit criteria · release readiness

  • 100% of P0 test cases executed · 0 P0 defects open
  • ≥ 95% of P1 test cases executed · 0 P1 defects open (or accepted by PM with mitigation)
  • Known-issue list published & reviewed with PM
  • Performance targets met (see § 05 NFRs)
  • Security scan clean OR exceptions approved by Security
  • Accessibility scan: no critical axe violations
  • Regression suite green on final build
  • Test report signed by QA Lead · Dev Lead · PM

§ 14 · Test data

Sources · sensitivity · refresh

  • Sources: synthetic generators · masked prod snapshots · fixture files
  • Sensitive data: PII masked · CID scrambled · no real card data
  • Refresh: reset-before-run for E2E · shared pool for exploratory
  • Volume (perf): 1M users · 10M transactions · 30-day history
§ 12 · Deliverables

What QA produces and hands over

  • Test strategy & plan (this doc)
  • Test cases (TestRail / Xray / Zephyr)
  • Defect reports (Jira / Linear)
  • Automation code (repo)
  • Test data scripts
  • Daily execution reports
  • Final test summary report
  • Traceability matrix (reqs → test cases → defects)

Part 10

Metrics · reporting · defect management

§ 19 · Metrics captured daily

What QA reports to the rest of the team

  • Test cases: planned · executed · passed · failed · blocked · deferred
  • Defects by severity: open · closed · reopened
  • Automation pass rate
  • Coverage vs requirements (% reqs with ≥ 1 test)
§ 19 · Reporting cadence

Who gets what, when

  • Daily — 3-line Slack summary (execution · top defect · top risk)
  • Weekly — burndown + risks to PM
  • End of cycle — full test summary report
§ 20 · Defect management

Severity · SLA · triage

Severity definitions:

  • S1 Critical — data loss · security breach · > 50% users blocked
  • S2 Major — core feature broken · no clear workaround
  • S3 Minor — feature degraded · workaround exists
  • S4 Cosmetic — UI polish · docs · non-blocking

SLA: S1 ack < 1 h · S2 < 4 h · S3 / S4 < 1 day.
Triage: daily during active testing; 2×/week otherwise.

Worked example · Biller Favourites Test Plan

TP-BFAV-v1.0 · excerpt

Condensed Test Plan for Biller Favourites · mBoB v1

§ 01 Identifier: TP-BFAV-v1.0 · Draft · 2026-04-21 · Author: QA Lead · Reviewers: Dev Lead, PM, Security, Release Mgr

§ 05 Features to be Tested (excerpt):

  • FR-01 · Save a favourite — P0 · Functional + Integration
  • FR-02 · View favourites — P0 · Functional + Perf + A11y
  • FR-03 · Use a favourite (pre-fill payment) — P0 · Functional + Security
  • NFR-Perf-01 · p95 < 1s for list — P0 · Performance
  • NFR-Sec-01 · Scope to user (no cross-account leakage) — P0 · Security
  • NFR-A11y-01 · WCAG 2.1 AA — P1 · Accessibility

§ 06 NOT to be Tested: bulk import/export (deferred v2) · legacy mBoB v4 (unsupported) · payment gateway internals (vendor responsibility).

§ 07 Approach: Unit (dev) · Integration · System (Playwright) · UAT (branch agents in Thimphu). Automation covers top 8 user journeys + all API endpoints.

§ 09 Entry: Build deployed to QA-A · smoke green · test data reset · tickets marked Ready-for-QA.

§ 11 Exit: 0 P0 defects · 0 P1 defects · p95 < 1s verified at 5k RPS · axe clean · RMA §7 compliance confirmed · sign-off by QA Lead, Dev Lead, PM.

§ 13 Environments: QA-A · QA-B (sandbox partners) · Perf (scaled prod-like) · Staging (UAT).

§ 17 Schedule: Test cases T-21 · automation T-14 · functional T-10 · perf T-5 · UAT T-2 · go/no-go T-1 · release T-0.

§ 21 Approvals: QA Lead, Dev Lead, PM, Security — all required before test entry.

The 9 most common Test Plan mistakes

MistakeWhat goes wrongFix
Writing after testing startsPlan rationalises past choices instead of driving themDraft at T-21 · baseline at T-14
No "features NOT to be tested" list"Nobody tested that" after launch§ 06 · explicit exclusions with reasons
Exit criteria are vaguePM and QA disagree on release readinessBinary check-list · "0 P0 open" not "few P0"
Entry criteria missingTesting starts on a broken build · wastes days§ 09 · smoke-test gate before any QA work
No traceabilityCan't prove a requirement was testedTraceability matrix · req → test case → result
Metrics reported only at endNo time to correct courseDaily 3-line Slack + weekly burndown
Vague risks without ownersRisks everyone knows, nobody mitigatesNamed owner per risk
One environment for everythingPerf testing breaks functional testing§ 13 · separate environments per purpose
Plan written once, never updatedReality diverges; plan becomes fictionRe-baseline per release · living doc not a museum piece
Takeaways

Key points to carry forward

What we covered today

  • A Test Plan answers what · how · who · when · when done up front — for everyone, not just QA.
  • Three levels: project · release · feature. Most plans you'll author are release-level (10–20 pages).
  • 21 sections per IEEE 829 · from identification → scope → strategy → gates → logistics → responsibilities → risk → metrics → approvals.
  • Features NOT to be tested is as important as features to be tested — with reason for each.
  • Four gates: Entry · Suspension · Resumption · Exit. Each a binary check-list.
  • Exit criteria make "are we done?" unambiguous: 0 P0 defects · 0 P1 defects (or PM-accepted with mitigation) · perf · security · a11y · regression all green.
  • RACI (or SELISE's RCI) matrix assigns every activity a single Responsible party.
  • Metrics captured daily — not saved for the post-mortem.
  • The Test Plan is a living document — written before testing, updated during, reviewed after.

If your release conversation ends with "how do we know we're done?" — your Test Plan has already failed.
