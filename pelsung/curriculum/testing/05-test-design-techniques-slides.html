03 The problem
🤯 You can't test everything

A Tashi Cell number has 8 digits.

That's 100,000,000 possible inputs to test the phone-number field.

If you tested one number per second, it would take you 3 years and 2 months — for a single field. Then there are 50 more fields in the app.

So how does QA actually work?

04 The answer

Test design techniques pick the fewest cases that catch the most bugs.

The mindset shift

Don't ask "What can I test?"
Ask "What groups of inputs behave differently — and which one example from each group will show me the truth?"

🎯

Smart, not many

5 well-chosen tests beat 50 random ones.

🧠

Think before clicking

Design first. Then test.

05
Technique 1 of 5

Equivalence
Partitioning

Group inputs that behave the same. Pick one from each group.

01
06 The idea

If two inputs make the system behave the same way, you only need to test one of them.

Equivalence Partitioning · EP

Sort the inputs into groups (called classes). Pick one example per group.

Every group is one "kind" of input. The system should treat every input in the group the exact same way. So testing one tells you about all of them.

✅ Why it helps

  • Cuts thousands of test cases down to a handful
  • Forces you to think about every kind of input, not just the easy one
  • Makes a clean, complete test list

🧠 Two types of groups

  • Valid — the system should accept it
  • Invalid — the system should reject it
  • Always test both. Most bugs hide in the invalid groups.

Field: "Age in years (0 to 120)"

Different age ranges get different treatment in our app: minor / adult / senior.

💡 Don't forget the invalid groups

Also test: −5 (negative · should reject), 121 (too high · should reject), "abc" (letters · should reject), blank (empty · should reject). 3 valid + 4 invalid = 7 test cases for a field with infinite possibilities.

💡 The EP rule of thumb

For every input field, write down the groups: valid groups + invalid groups. Pick exactly one example from each. That's your starting test list.

09
Technique 2 of 5

Boundary
Value Analysis

Bugs cluster at the edges. Test on, just below, just above each edge.

02
10 The idea

Most bugs hide at the edges of valid ranges.

Developers write code like if (age > 18) when they meant if (age >= 18). Off-by-one bugs are extremely common.

Boundary Value Analysis · BVA

For every boundary, test 3 values: just below · on · just above.

If a range is "18 to 60" then the boundaries are 18 and 60. Test: 17, 18, 19 at the lower edge and 59, 60, 61 at the upper edge.

EP and BVA work together. EP picks the groups; BVA picks the values right at the edges where the groups meet.

"Transfer amount: Nu. 100 to 50,000 per transaction"

💡 Always also test the "weird zeros"

Add: Nu. 0, negative numbers (−500), decimals (100.50), and blank. These boundary-like edge cases catch the most embarrassing bugs.

13
Technique 3 of 5

Decision
Tables

For complex business rules — multiple conditions, multiple outcomes.

03
14 The idea

When the rule has several "if this AND that" conditions, draw a table.

Decision Table

List every condition. List every combination. Write the outcome for each row.

If you have 3 yes/no conditions there are 2 × 2 × 2 = 8 combinations. Each combination becomes one test case. Nothing gets missed; nothing gets duplicated.

📋 Use it when…

  • Pricing or discount rules
  • Eligibility rules ("if X and Y, allow…")
  • Insurance, banking, ticketing rules
  • Anything described with many "if…and…then…"

🧠 Why it works

  • Easy to spot a missing combination
  • Easy to spot a contradiction in the rules
  • Same table works for BA, Dev, and QA
15 Worked example · Druk Air discount

Druk Air ticket discount rule

Discount = 30% if senior and loyalty member and booking 30+ days early; 20% if any two of those; 10% if only one; 0% if none.

Senior (60+)NNNNYYYY
Loyalty memberNNYYNNYY
Booked 30+ days earlyNYNYNYNY
→ Discount applied0%10%10%20%10%20%20%30%

8 combinations → 8 test cases. No combination forgotten.

16 Tip · how to build one

Building a decision table · 4 simple steps

Step-by-step

  1. List every condition from the spec (each with Yes / No)
  2. Calculate combinations: 2n for n conditions
  3. Fill the column for each combination — alternate Y/N like binary counting
  4. Look up the spec and write the outcome for each column

Common mistakes

  • Missing a condition → table is incomplete
  • Two columns with the same conditions but different outcomes → spec contradicts itself; flag it to the BA
  • Skipping a "boring" combination ("no one would do that") — that's exactly where bugs hide
💡 If the table gets huge

If you have 5+ conditions = 32+ rows. Group similar combinations. Or split into two smaller tables. A 2-page table is unreadable.

Technique 4 of 5

State Transition
Testing

For things that move through stages — orders, requests, accounts.

04
18 Worked example · leave request states

A leave request lives in one state at a time.

Test the arrows (transitions) — both the ones that should work, and the ones that should be blocked.

5 valid arrows. 5 valid test cases. But also — many invalid arrows that should be blocked.

19 Test the arrows that shouldn't work too

Valid transitions vs. Invalid (should be blocked)

From stateActionTo stateWhat QA tests
DraftSubmitSubmittedVALID The request appears in the manager's queue.
SubmittedManager approvesApprovedVALID Status updates · employee gets a notification.
SubmittedManager rejectsRejectedVALID Reason is captured · employee notified.
SubmittedEmployee cancelsCancelledVALID Manager queue updates immediately.
ApprovedLeave date arrivesUsedVALID Days deducted from balance.
RejectedEmployee tries to cancel—INVALID System should refuse — already rejected.
UsedManager tries to approve again—INVALID System should refuse — final state.
DraftManager tries to approve—INVALID Manager can't even see drafts — should not appear.
💡 The most-missed bugs in state testing

Junior testers test only the happy path arrows. The bugs are usually in the blocked arrows — when the system shouldn't let something happen but accidentally does. Always test both.

Technique 5 of 5

Use Case
Testing

Walk every flow of a use case — main, alternate, and exception.

05
21 Use case testing · the quick version

From a use case, write tests for main flow, alternate flows, and exception flows.

You met use cases in BA+QA Diagrams class #01. Today is just the testing use of them.

Use case · school admission form

Use case: "Parent submits a school admission form."

A use case describes a feature end-to-end from a user's point of view — the steps they take, what they need, what they get.

✅ Main flow

Parent fills all required fields, uploads CID + photo, pays the fee, sees confirmation.

Test: walk the happy path end-to-end. 1 test case.

↪ Alternate flow

Parent saves a draft and returns later to finish.

Test: leave halfway · come back · data still there. 1 test case.

⚠ Exception flow

Required field empty · uploaded file too large · payment fails · session times out.

Test each exception. 4 test cases.

One use case → 1 main + 1 alternate + 4 exceptions = 6 test cases that cover the whole feature.

22 Error Guessing · the tester's sixth sense

Sometimes the best test is the one your gut tells you to write.

All five techniques so far are systematic. Error guessing is experience-based — you use intuition and memory of past bugs to predict where new bugs are hiding.

Error Guessing · experience-based technique

"What would a careless or sneaky user do?"

EP, BVA, decision tables, state transition and use cases will catch most bugs in the spec. Error guessing catches the bugs that nobody thought to write down — the ones that come from real-world chaos.

🤔 Common error-guessing prompts

  • What if they enter something stupid? 0, negative, very large, special chars
  • What if the network drops mid-action?
  • What if the user double-taps the Submit button?
  • What if they paste a 10,000-character string?
  • What if they hit Back after paying?

🇧🇹 Bhutanese example · mBoB transfer

  • What if the user enters Nu. 0?
  • Phone number with +975 vs without?
  • What if the receiver number has a space?
  • What if they tap Send twice quickly — does it transfer twice?
  • What if the screen rotates mid-transfer?
💡 Keep a personal "bug catalog"

Bugs you've seen before are bugs that come back. Write down every bug you find — by the time you have 30+ entries, you'll have your own private error-guessing checklist that nobody else can match.

23 Pairwise · combination coverage without explosion

When inputs multiply, you can't test every combination — but you can cover every pair.

5 browsers × 3 devices × 4 user roles = 60 combinations. Most teams test 5–10. Pairwise picks a smart ~10 that cover every pair of inputs at least once.

Pairwise · combinatorial technique

Cover every pair of inputs once. Don't try to cover every full combination.

Research shows ~80% of bugs are triggered by one or two inputs together — not three or more. So pairwise gets ~80% of the coverage with ~20% of the effort.

🇧🇹 Druk Air booking example

Inputs:
· Cabin: Y / J (2)
· Payment: mBoB / Card / Cash (3)
· Loyalty: Yes / No (2)
· Departure: Paro / Bumthang / Gelephu (3)

Full combinations: 2 × 3 × 2 × 3 = 36 tests
Pairwise: ≈ 9 tests — every pair still covered.

🛠 Free tools do the math

  • PICT (Microsoft, free, command-line)
  • pairwise.teremokgames.com · online generator
  • Hexawise, ACTS (NIST) — more advanced
  • You don't need to memorize the math · just feed it your inputs
💡 What to remember from this slide

Pairwise has the steepest learning curve in this list. For now, just recognize the term and know when to reach for it: many independent inputs that multiply into too many combinations to test fully.

24 Cheat sheet · when to use which

Pick the technique that matches the situation.

Use this technique……when you're testingBhutan example
Equivalence Partitioning + Boundary ValuesAn input field with a range or a set of optionsAge field · phone number · transfer amount
Decision TableBusiness rules with several "if X and Y" conditionsDruk Air discount · loan eligibility · school admission rules
State TransitionAnything with a lifecycle that moves through stagesLeave request · order status · account suspension
Use Case TestingA feature end-to-end · main + alternate + exception flowsSubmit admission form · book a ticket · transfer money
Error GuessingAnywhere — as a final sweep using your gut + past bug experiencemBoB · "what if user enters Nu. 0?" · pasting 10,000 chars · double-tap submit
Pairwise TestingMany input combinations (browser × device × role × payment …)Druk Air booking · cabin × payment × loyalty × departure combos

In real projects you'll often combine them — EP + BVA on each input field inside a use-case flow, then a quick error-guessing sweep at the end.

27 Hints · what "good" looks like

If you're stuck, here's the shape of a good answer.

Problem 1 · target ~7 cases

  • 17 → reject (just below)
  • 18 → accept (boundary)
  • 19 → accept
  • 34 → accept
  • 35 → accept (boundary)
  • 36 → reject (just above)
  • "abc" / blank / negative → reject

Problem 2 · target 4 cases

  • Loyalty=Y · Early=Y → 20%
  • Loyalty=Y · Early=N → 10%
  • Loyalty=N · Early=Y → 5%
  • Loyalty=N · Early=N → 0%
  • (2 conditions × 2 = 4 rows · all covered)

Problem 3 · target ~8 cases

  • 5 valid arrows (one each)
  • Plus invalid attempts:
  • Approve a Draft → blocked
  • Cancel a Rejected → blocked
  • Approve a Used → blocked

Don't peek before you try! These are reference targets — your set might be slightly different and still correct.

28 Debrief · group discussion

Let's talk about what we found.

Questions for each pair

  • Which technique was clearest?
  • Which one was hardest to apply?
  • Where did you under-cover? (forget invalid? forget a boundary?)
  • Did you find a place where two techniques combined?

What good QA habits look like

  • Always ask: "What groups are there?"
  • Always test the edges
  • Always test the invalid path
  • Always think: "What can't happen?" and try to make it happen
29 Wrap-up
🎯 Key Takeaways

What we covered today

  • ✅ You can't test everything — use techniques to pick smart cases
  • ✅ Equivalence Partitioning — group inputs, pick one per group (valid + invalid)
  • ✅ Boundary Value Analysis — bugs hide at the edges, test on / just below / just above
  • ✅ Decision Tables — for "if X and Y" rules · 2n rows · cover every combination
  • ✅ State Transition — test valid arrows AND blocked arrows in a lifecycle
  • ✅ Use Case Testing — main flow + alternate flows + exception flows
  • ✅ Combine techniques · EP+BVA on inputs inside a use-case flow

A great QA tests fewer cases — but the right ones.
