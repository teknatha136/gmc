Why Web
is Your Daily Job

Most products live in a browser. Most bugs are visual. You are the human eye.

01
04 Why this is most of QA work

Almost every product you'll test runs in a browser

🌐

Web is everywhere

mBoB web · Druk Air booking · Bhutan Postal · trainee portal · government services. All a browser opens.

👁

Humans see what code can't

A button cut in half. A typo in Dzongkha. Light grey on white. Automation misses these. You don't.

🧍

You are the user

If it confuses you, it confuses every customer. Manual testing is the closest thing to "real-world feel".

Manual web testing is not a step before automation. It's the part automation will never replace.

05 Today · what we will and won't do

Stay on the human side of testing

✅ We will
  • Open real sites and look at them
  • Test on different browsers and screen sizes
  • Find visual / behaviour bugs by eye
  • Use DevTools at the simplest level (resize + watch requests)
  • Write up bugs clearly
❌ We won't
  • Write code or scripts
  • Use Selenium / Playwright / Cypress
  • Test APIs by hand
  • Read the page's source code
  • Worry about performance numbers or load testing

If you've ever opened a website and noticed something looked off — you're already 80% of a manual web tester.

Part Two

The Toolbox

Four things on your desk. Nothing more.

02
07 The four tools

You don't need fancy tools. You need these four.

🌐 1 · A modern browser

Chrome or Edge. Keep it updated. This is your main testing window.

Why: most users use Chrome. Start where they are.
🛠 2 · Browser DevTools (F12)

Press F12 (or right-click → Inspect). You'll only use two parts: Device toolbar (resize the page) and the Network tab (see if a request was sent).

Why: not for code. For shrinking the page and confirming "did the click do anything?"
🎥 3 · Screen recording

Built into Windows (Win + G), Mac (Shift + Cmd + 5), and phones. Record a 10–20 second clip when a bug is hard to describe.

Why: a developer who sees the bug fixes 3× faster than one who only reads about it.
📱 4 · Real devices

Your own phone, a tablet if available, your laptop. Don't test only in the browser emulator.

Why: only a real thumb can find a button that's too small for a real thumb.
08 DevTools · the only two things you'll touch

F12 · Device Toolbar · Network Tab. That's it.

📱 Device toolbar

Press Ctrl + Shift + M (Windows) or Cmd + Shift + M (Mac).

The page now sits inside a fake phone. You can pick a size from the dropdown — iPhone 14, iPad, Galaxy, or type your own (e.g. 375 × 812).

Use this for responsive testing without leaving your laptop.

📡 Network tab

Open DevTools → click Network. Click any button on the page.

If a row appears in the list, the click reached the server. If nothing happens — the button is dead.

You don't need to read the request. Just see if anything fires.

That's the whole DevTools curriculum for today. Resize the page, and see if a click did anything.

11 What actually differs across browsers

The same page can look and behave differently in each browser

Thing to checkWhy it differsQuick example
FontsEach browser falls back to different default fonts if the website's font fails to load.Heading looks bold in Chrome, thin in Safari.
Form autofillEach browser fills "name" / "phone" / "address" boxes differently.Safari fills a phone field with a country code; Chrome doesn't.
Date pickersBrowsers ship their own calendar UI for date inputs.iPhone shows a wheel; Chrome shows a grid; Firefox shows a dropdown.
Video / audioCodec support differs — some videos won't play.A .mov file plays on Safari but shows a black box on Firefox.
Shadows, gradients, animationsSlight CSS rendering differences.A button shadow looks soft in Chrome, hard-edged in Safari.
File upload UI"Choose file" dialogs vary by OS + browser.iPhone offers Camera, Photo Library, Files. Desktop offers Finder/Explorer.

When you test, don't just check "does the page load?" — check "does it look the same and behave the same?" in each browser.

How to switch sizes in Chrome

14 What goes wrong when the page shrinks

Six things to look for at every viewport

Visual problems

  • Text overflow — words spilling out of buttons or boxes
  • Broken layout — columns stacking weirdly, images crushed
  • Hidden buttons — important button slides off-screen on mobile

Interaction problems

  • Scroll trap — a popup you can't close on mobile
  • Tap target too small — links too close together to tap
  • Hover-only menus — works on desktop, dead on phone

Real device beats emulator when you can. The DevTools phone view doesn't have a real thumb — your phone does.

Part Five

Click-Through
Smoke Test

Five minutes. Is anything obviously broken?

05
16 The 5-minute smoke walk

Same path · every release · before any deeper test

Recall from QA #01: Smoke testing is "is this build broken in any obvious way?" Today's job: how do you actually do that on a website.

  1. Open the home page. Wait 5 seconds. Did it load? Any blank patches? Any spinner stuck forever?
  2. Click the main menu items one by one. Does each open the right page?
  3. Run the core flow — the one most users come for. (mBoB: log in → check balance. Druk Air: search → see flights.)
  4. Log out. Does the system actually log you out?
  5. Log back in. Same credentials work? Land on the right place?
  6. Refresh once on the home page. Anything change weirdly?

If any of these breaks — stop, log it, tell the team. No point doing deeper testing on a build that can't even log in.

20 The QA's eye

Scan a page the same way · every time

Random looking misses bugs. A repeated scan pattern catches them.

1 · Top to bottom

Header · main content · footer. Read the page like a story. Anything missing? Anything in the wrong order?

2 · Left to right

For each row — sidebar, content, action panel. Are columns aligned? Do widths feel right at this screen size?

3 · Edges & corners

Where buttons and labels meet the screen edge. This is where overflow / cut-off / off-screen bugs hide.

4 · Empty / extreme inputs

Submit a blank form. Type a 200-character name. What happens at the limits?

5 · Switch language / size

Toggle Dzongkha ↔ English. Resize to 375px. Re-scan. Bugs reveal themselves only at certain sizes.

6 · Refresh + back button

Reload mid-flow. Press browser back. Did the page recover, or did everything reset?

21 Logging bugs as you find them

Don't trust your memory · log in real time

When you find a bug mid-session, capture these eight lines before moving on. (Recall QA #03 Test Plan and BA+QA Quality #02 Bug Reporting — same template, applied here.)

Title: Pay Now button cut off on mBoB Bill Pay (mobile) URL: https://mbob.bt/billpay Browser: Chrome 120 / Safari iOS 17 Viewport: 375 × 812 (iPhone SE size) Steps: 1) Open mBoB web on phone 2) Log in 3) Go to Bill Pay → select any biller Expected: "Pay Now" button fully visible and tappable Actual: Right half of button cut off · cannot tap Severity: High (blocks payment flow)

Add a screenshot or 10-second screen recording when the bug is hard to describe. Always faster than typing more.

24 What to do · 30 min test + 5 min share

Find at least 5 issues from the 10 patterns

Your test plan (30 min)

  1. Pick one site
  2. Test on desktop Chrome first (5 min smoke)
  3. Switch to mobile size via DevTools (375 px) (10 min)
  4. Open the same page in Firefox · note any difference (5 min)
  5. Look specifically for the 10 UI bug patterns we just covered (10 min)
  6. Log 5+ issues in the bug template (URL · browser · viewport · steps · expected · actual · severity)

Group share (5 min)

  • Each volunteer presents one bug — the most surprising one they found
  • Show the screenshot / recording
  • Say which pattern (1–10) it matches
  • One sentence: why does this matter to a real user?

No "winner." Goal is range — see how varied real bugs are.
