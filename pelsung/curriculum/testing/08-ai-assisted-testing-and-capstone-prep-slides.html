05 Prompt Anatomy · Bad vs Good
Same goal · two prompts · very different output

The Difference is Context

✗ Bad prompt
Vague, no role, no template
give me test cases for login
  • What system? Web? Mobile?
  • How many cases? What format?
  • Positive only? Negative? Edge?
  • Result: generic, unusable, you rewrite everything
✓ Good prompt
Role, context, template, examples
I'm a QA writing manual test cases. Here's the user story and 3 acceptance criteria for [feature]. Generate 8–10 test cases (mix positive, negative, edge) using this template: ID, Title, Pre-conditions, Steps (numbered, atomic), Expected Result. Use realistic Bhutanese-context test data.
  • Tells Claude WHO it's writing for
  • Gives the template you'll paste into
  • Asks for the right MIX of cases
  • Locks the data style — no fake "John Smith"
06 Prompt Template

Five Parts of a QA Prompt

Memorise this shape. Every good test-case prompt has these five.

Claude Code · QA prompt template
# 1. ROLE I'm a manual QA on a banking app project. # 2. CONTEXT — the story & AC Story: As a Druk PNB customer, I want to transfer money to a saved beneficiary using mBoB. AC1: Amount must be between Nu. 1 and Nu. 100,000. AC2: OTP must be entered within 60 seconds. AC3: Insufficient balance shows an error before OTP. # 3. TASK Generate 8–10 manual test cases — mix of positive, negative, and edge. # 4. FORMAT Use this template per case: ID | Title | Pre-conditions | Steps (numbered) | Expected Result # 5. CONSTRAINTS Use realistic Bhutanese names and amounts in Ngultrum. Keep steps atomic — one action per step. Mark each case as POSITIVE / NEGATIVE / EDGE.
07 The Loop
The most important slide of the day

Generate → Critique → Refine

Never accept Claude's first answer. Apply your QA brain to it.

1
Generate

Send the 5-part prompt with story + AC. Get 8–10 candidate cases.

2
Critique

Apply the 7 rules from QA #04: atomic steps, concrete pre-conditions, real expected results, edge values, no hallucinations.

3
Refine

Re-prompt: "Fix case 4 — pre-condition is vague. Add 2 cases for max+1 amount." Repeat until tight.

🎯 Rule of thumb: if you would not be comfortable putting your name on the output, loop again.

10 Worked Walkthrough · Step 2 Critique
Flaws we found

What's wrong with that table?

⚠️ Issues a real QA would flag
  • TC-01: "Manager logged in" is too vague. Need: "Manager has 1 pending request from a trainee in their team."
  • TC-03: "Very long comment" — how long? 500 chars? 5000? Pin the edge value.
  • Missing: request from trainee NOT in this manager's team — should be hidden / forbidden.
  • Missing: approve a leave whose dates already passed (stale request).
  • Missing: approve while offline / no network — what happens to the comment?
  • Hallucination check: AC2 says comment required for both approve AND reject. Claude only tested reject.

12 AI Weaknesses
Where you must NOT trust it

What AI Is Bad At

Treat these as red zones. Always double-check.

🇧🇹 Real Bhutan-specific data

Currency formats, district names, school IDs, CID structure — Claude often guesses. Verify against a real example.

🔧 Your specific system's quirks

Claude doesn't know your app's button labels, error wording, or that screen #4 is broken on Tuesdays. You do.

❓ Ambiguity in your own prompt

If your AC is fuzzy, Claude will quietly fill the gap with assumptions. It will not stop and ask.

🔍 Replacing exploratory testing

Clicking around with curiosity, breaking things in unexpected ways — that's a human job. Always.

13 Reviewer's Checklist
Use this every time you read AI test cases

Reviewer's 5-Question Checklist

  • 1. UI map: Do the steps actually map to a screen / button we have? Or did Claude invent a "Bulk Approve" feature?
  • 2. Concrete pre-conditions: Is system state pinned (specific user, specific record), not just "logged in"?
  • 3. Negative coverage: Did it cover the bad paths — empty fields, wrong role, expired session?
  • 4. Specific edge values: Are edges named (0, 1, max−1, max, max+1) — not "a really big number"?
  • 5. Hallucination scan: Is anything described that does NOT exist in our app? Cross-check against the BA's spec.

If even one answer is "no", loop again before you save the file.

16 The QA's Role on a Capstone Team
What the QA does — week by week

Your Job on the Team

1. Read BA's stories & AC

Day 1, before any code. If AC is unclear, push back — that's QA's job.

2. Build mini test plan

Use the template from QA #03. One page is enough — scope, levels, types, sign-off.

3. Write cases for top features

Don't try to cover everything. Pick the 3–5 most important user journeys.

4. Manual + exploratory pass

Run scripted cases first. Then click around like a curious user. Both find different bugs.

5. Log bugs · triage with dev

Use the bug template from BA-QA Quality block. Severity, steps, expected vs actual. No drama.

6. Sign off before demo day

Make the call: ship, ship-with-known-bugs, or hold. The team trusts your judgement.

17 Capstone Test Strategy Template
Mini test strategy · 6 sections · fits on one page

Capstone Test Strategy

1
Scope

Which features ARE in. Which are explicitly OUT. Demo day is short — pick.

2
Levels

System + UAT. (Unit and integration are the dev's responsibility — ignore for now.)

3
Types

Smoke + Functional + Exploratory + Cross-browser. Skip everything else this round.

4
Tools

Manual execution. A browser. Optional: Claude for prompts and translation.

5
Sign-off criteria

What "good enough for demo" means: e.g., zero S1 bugs, ≤3 S2 bugs known.

6
Risks

Honest list: "Only 2 days for full regression." "Demo network may be slow."

18 Worked Strategy · GMC Companion App
A real-looking mini test strategy

Sample · "GMC Companion App"

Scope — visitor sign-in, daily mindfulness check-in, find-a-place map. Out: push notifications, multi-language (Phase 2).

Levels — System + UAT only. Dev team handles unit tests in their pipeline.

Types — Smoke (5 min, before every demo build), Functional (3 top journeys), Exploratory (1 hour by 2 QAs), Cross-browser (Chrome + Safari).

Tools — Manual on real laptop + phone. Claude Code for generating prompts and Dzongkha summary. Bug log: shared spreadsheet.

Sign-off — 0 S1 bugs, ≤2 S2 bugs known & documented, all 3 top journeys pass smoke twice in a row.

Risks — Only 2 days for full regression. Demo wifi may be slow. One QA on the team is also doing BA review — divided focus.

21 Hands-on · 4 Steps

Generate · Critique · Refine

30:00
  1. Strategy — write a 2-paragraph test strategy for your chosen feature. Cover scope, types, risks.
  2. Generate — use Claude Code (or any AI) to generate 6 test cases for ONE chosen story. PASTE the AI output into your doc.
  3. Critique — apply the reviewer's checklist. Mark 3 specific issues in the AI output. Propose a fix for each.
