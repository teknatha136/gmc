04
Static vs Dynamic

The most fundamental
split in testing

Before we even talk about levels or phases — there are two whole families of testing. One runs the software. The other does not.

Most beginners only know one of them.

05 Static vs Dynamic testing

Two ways to find a defect

No code runs

Static testing

We read, review and discuss the work product. We never press "run". We never click a button on the app.

  • What we look at: requirements, user stories, designs, test cases, code (by reading, not running)
  • How we look: reviews, walkthroughs, inspections, checklists
  • When: as soon as the document exists — long before any code is written
Code runs

Dynamic testing

We execute the software. We give it inputs, watch what comes out, and compare it to what we expected.

  • What we look at: the running app, the API response, the database row
  • How we look: unit tests, integration tests, system tests, UAT
  • When: only after something has been built and can be run

Both are testing. A spec review that catches a wrong rule before any developer writes code is just as valuable as catching a bug in production — usually more valuable, because it costs almost nothing to fix.

06 Why static testing is powerful

Catch the defect before any code runs

The momo restaurant analogy

You're opening a momo place in Gelephu. Before the kitchen ever cooks, you sit down with the recipe, the supplier list, the menu and the prices. You read them out loud. The chef says "we don't have a steamer big enough for that order size." The accountant says "we'll lose money at this price." You catch four problems before a single momo is cooked.

That's static testing — the kitchen never opens. We caught the defects in the plan.

What makes static so valuable

  • Cheapest possible bug fix — fixing a sentence in a Word document costs minutes; fixing the same idea once it's built into code costs days.
  • Catches different defects — missing requirements, ambiguous rules, contradictions. Dynamic testing cannot find a missing requirement.
  • You don't need code to do it — perfect for non-technical QAs. We'll show you exactly how.
  • Prevents defects, doesn't just detect them — by the time the code is written, the spec is already cleaner.
07 Static testing methods · increasing formality

Three ways to review · pick the one that fits the risk

Least formal
1 · Walkthrough

The author of a document walks the team through it, page by page, explaining as they go. Others ask questions and point out things that look wrong.

  • Led by: the author
  • Goal: shared understanding, quick feedback
  • Output: rough notes

Like the BA reading the spec out loud to the team in a coffee meeting.

Semi-formal
2 · Technical review

A small group reads the document beforehand, then meets to discuss. There's a moderator who runs the meeting. Defects are recorded properly.

  • Led by: a moderator (not the author)
  • Goal: agree on technical correctness
  • Output: a list of issues with severity

Like reviewing the BRD before signing off — multiple eyes, structured.

Most formal
3 · Inspection (Fagan)

The classic CS-textbook method. Defined roles (author, moderator, reader, recorder), a written checklist, entry and exit criteria, metrics collected on every defect.

  • Led by: trained moderator
  • Goal: highest possible defect detection
  • Output: signed defect log + metrics

Used for safety-critical systems — aviation, medical, banking core.

Same activity at heart — humans reading a document carefully. The difference is how strict the process is.

08 What a QA actually does in static testing

No code? No problem. This is your superpower.

A QA reviewing a spec, a story or a test case asks:

Is it complete? — every screen, every button, every error case described. (Spec says "user logs in" — but what if the password is wrong? What if the account is locked?)

Is it clear? — could two people read this and disagree on what it means? "Fast" is not clear. "Returns within 2 seconds" is.

Is it correct? — does the rule actually match real-world business? (Spec says "anyone over 18 can apply for pension" — but actual pension rules say 65.)

Is it consistent? — does page 5 contradict page 12? Does the spec say "Nu." in one place and "BTN" in another?

Is it testable? — could you actually write a test for this? "The system should be user-friendly" cannot be tested. "The user can complete signup in under 4 clicks" can.

Are acceptance criteria measurable? — every AC should pass or fail with no argument. If the QA and BA disagree on whether it passed, the AC was bad.

For our cohort: this is the work you can do today, on day one of any project, even before you ever look at the running app. It's the most underrated QA skill.

09
Dynamic testing — the 4 test levels

Building a house,
step by step

A house is built brick by brick, room by room. We test it at every stage — not just at the end. Software works the same way.

10 Test level 1 · Unit testing

Testing one brick before you lay it

The house analogy

One brick at a time. Before the mason lays a brick in the wall, he taps it. Is it cracked? Is it the right size? If the brick is bad, the wall will be bad. So we check each brick on its own — by itself, before it joins anything else.

A bad brick costs nothing to throw away. A bad wall costs days to rebuild.

In software

Test one tiny piece on its own. One small function — like the part that adds a number, or the part that checks a date — gets tested by itself. We do not test the whole app. We test that one piece, alone.

Who does it: developers, while they write the code. Very fast. Runs hundreds of times a day.

11 Test level 2 · Integration testing

Do the bricks bond properly with mortar?

The house analogy

Two bricks together. Each brick was fine on its own. But when we put two of them with mortar between, do they hold? Does the mortar grip? If the bond fails, the wall falls — even though every single brick was good.

The bricks were fine. The joining was wrong. That is what we are checking.

In software

Test two pieces working together. The login screen passes its check. The dashboard passes its check. But when login finishes and hands you over to the dashboard — does that handover work? That is integration testing.

Who does it: developers and QA together. Catches problems where two parts meet.

12 Test level 3 · System testing

Is the whole house livable?

The house analogy

Walk through the finished house. The walls stand. The roof is on. The doors close. The taps run. Now we walk through the whole thing. Does the front door open? Does the kitchen tap give water? Does the toilet flush? Everything together, like a real house.

Each part may be fine. The question now is: does the house work as a house?

In software

Test the whole app, end to end. Open the app. Log in. Search. Add to cart. Pay. Get receipt. Log out. The QA goes through complete journeys, just like a real user would. The whole system, all parts joined.

Who does it: the QA team. This is your bread and butter.

13 Test level 4 · User Acceptance Testing

Would the family actually move in?

The house analogy

The family visits. The builder says "the house is ready." Good. Now the family walks in. They sit on the floor. They open the cupboard. The grandmother tries to reach the top shelf — too high. The children find no place to study. The house is built right, but is it the house they wanted to live in?

A correct house is not always the right house. The family decides.

In software

Real users try the app. Real Bhutanese pension recipients try the new pension portal. Real shopkeepers try the new mBoB payment flow. They are not looking for code bugs — they are answering: "Does this actually solve my problem? Will I really use it?"

Who does it: the customer, business stakeholders, sometimes the BA. The final go-or-no-go.

14
Testing across the SDLC

Where do all
these tests fit?

The pyramid told us how many. The V-model tells us when. Shift-left tells us how early.

15 The V-model · every build step has a test step

For every brick we lay, there is a brick we check

Left side: how we build, going down. Right side: how we test, going up. Each building step has a matching test step on the other side.

Key idea: when we plan the house, we already plan how we will check it. Tests are not an afterthought — they are decided early. Static testing happens on the left side; dynamic testing on the right.

16 Iterative & Agile testing

Testing every sprint · not just at the end

The traditional way (waterfall V-model)

Build the whole house. Then test the whole house. Then hand it over. If something is wrong with the kitchen, you found out months later — and the family already moved their furniture in.

The agile / iterative way

Build a small piece — a single room. Test it. Show the family. Get feedback. Build the next room. Test it. Show again. Every two weeks (a "sprint"), the team has something testable.

  • Testing is continuous — not a final phase.
  • QA sits with the team, not after the team.
  • The V still applies — it just happens in miniature, every sprint.
  • Bugs are found in days, not months.

SELISE projects work this way. The QA you'll be is an everyday QA, not an end-of-project QA.

17 Shift-left and shift-right

Catch problems early. Watch for problems late.

⬅ Shift-left · catch bugs earlier

Don't wait until the house is built to notice the door is in the wrong place. Catch it on the drawing. Even better — catch it when the family is still describing the house.

In software: test the spec (static!). Test the design. Test the small piece. Don't wait for the whole app to be ready.

Why it matters: a bug caught in design costs almost nothing. A bug found in production can cost 100x more — and your customer's trust.

Shift-right · keep watching after launch ➡

Even after the family has moved in, the builder visits. Is the roof leaking? Are the floors cracking in winter? Real life shows things tests cannot.

In software: monitor the live app. Watch error rates. See where users get stuck. Some problems only appear in real use.

Why it matters: a lab cannot find every real-world problem. Real users on real phones in real Gelephu will surprise you.

Modern teams do both. Test early. Watch always.

20 Phase 1 · Planning
Phase 1 · Planning

What will we test, when, and with what?

The very first phase. Before any test is written, the team agrees on the scope, schedule, resources, risks and — critically — the exit criteria (when can we say "we're done testing?").

You are doing:
  • Reading the requirements and risks
  • Agreeing what's in scope and what's out
  • Estimating how long testing will take
  • Deciding what tools, environments and people are needed
  • Writing the test plan — a document everyone signs off on
Bhutan example — mBoB v2: "We will test login, fund transfer, statement download and beneficiary management on Android 10+ and iOS 14+. We will not test the kiosk version this release. Two QAs, two weeks. We're done when 95% of test cases pass and zero critical defects remain."
21 Phase 2 · Analysis
Phase 2 · Analysis

What exactly should we check?

Read the requirements line by line. Pull out every test condition — every separate thing that needs to be verified. Don't write tests yet. Just list what needs covering.

You are doing:
  • Reading every requirement, story and AC
  • Listing test conditions: "login with correct password", "login with wrong password", "login with locked account", "session timeout"...
  • This is heavy static testing — you'll spot missing rules and ambiguity here too
  • Output: a list of test conditions / coverage items, not yet steps
Bhutan example — pension portal: from one requirement ("user can apply for pension") you'll list 14 test conditions: valid CID, invalid CID, expired CID, age below 65, age 65+, missing documents, network drops mid-form, double submission… each will become a test case in Phase 3.
22 Phase 3 · Design
Phase 3 · Design

Turn each condition into a real test case

Now we write the actual test cases. Each one has steps, input data, and an expected result. Test design techniques (boundary value, equivalence partitioning, decision tables…) live here. We'll cover those in QA #04.

You are doing:
  • Writing test cases — title, preconditions, steps, expected result
  • Choosing test data: what CIDs, what amounts, what dates
  • Grouping cases into test suites
  • Reviewing each other's cases (more static testing!)
Bhutan example: Test condition "login with locked account" becomes a test case: "Pre-condition: account 'tek' is locked. Steps: open app → enter 'tek' / 'pass' → tap login. Expected: error 'Account locked, contact support', no session created."

→ Deep dive in QA #04 (Test Design Techniques).

23 Phase 4 · Implementation
Phase 4 · Implementation

Get everything ready to actually run

The test cases exist. Now make them runnable. Set up test environments, load test data, decide the order in which tests will run, prepare automation scripts if any. No tests run yet.

You are doing:
  • Building or refreshing test environments (a clone of production, but safe)
  • Loading test data — fake users, fake accounts, fake transactions
  • Ordering test cases — what depends on what?
  • Setting up access, devices, browsers, phones
  • Smoke-checking that the build is even installable
Bhutan example: for the pension portal, the team creates 50 fake CIDs, loads them with different ages and statuses, deploys the build to a test server, gives every QA login credentials, and confirms the app even opens. Now Phase 5 can begin.
24 Phase 5 · Execution
Phase 5 · Execution

Actually run the tests · log every defect

The phase most people picture when they hear "QA". You execute test cases one by one. For every one: did the actual result match the expected result? If not — log a defect.

You are doing:
  • Running each test case, recording pass / fail
  • Logging defects with proper bug reports (we covered this in Quality #02)
  • Re-testing after fixes — the bug fixed, and the area around it (regression)
  • Tracking progress: "today we ran 40 cases, 36 passed, 4 failed, 4 defects raised"
Bhutan example: while testing pension login, you find that an expired CID lets the user in for 1 second before logging out. You file: "BUG-203 · Expired CID briefly grants session · Severity: High · Steps to reproduce: ..." and the dev fixes it. You re-test. You also re-test login-with-valid-CID, in case the fix broke something nearby.

→ Bug reporting was Quality #02. The chain still applies: error → defect → failure.

25 Phase 6 · Completion
Phase 6 · Completion

Close the cycle · learn for next time

The phase most teams skip — and the one ISTQB will absolutely test you on. Once execution ends and the product ships, the test work isn't done. We close, report, archive and reflect.

You are doing:
  • Confirming all exit criteria from Phase 1 are actually met
  • Producing a test summary report — what passed, what failed, what was deferred
  • Archiving test cases, results and environments — for the next release
  • Running a retrospective: what went well, what didn't, what will we change next time?
  • Handing over: known issues, open defects, lessons learned
Bhutan example: after mBoB v2 launches, the QA lead writes: "308 test cases run, 297 passed, 11 deferred to v2.1. Two known low-severity issues documented. Lessons: test-data setup ate 3 days — we'll automate it next release." The next QA team starts where this one finished.
26 The 7 ISTQB testing principles

Seven truths every tester memorises

  1. Testing shows the presence of defects, not their absence — finding no bug doesn't prove there are none.
  2. Exhaustive testing is impossible — you can never test every possible input. Pick wisely.
  3. Early testing saves time and money — the earlier you find a bug, the cheaper it is to fix.
  4. Defects cluster together — most bugs hide in a small number of places. Find one, look around.
  5. Beware the pesticide paradox — running the same tests forever stops finding new bugs.
  6. Testing is context-dependent — testing a bank app and testing a game are not the same.
  7. Absence-of-errors is a fallacy — passing all tests does not mean the product is what users want.

We'll spend a slide each on the two most actionable for you — Principle 1 and Principle 3.

27 Principles 1 & 3 · the two you must internalise
Principle 1

Testing shows the presence of defects, not their absence

If we test and find no bugs, that does not prove the software has no bugs. It only proves our tests didn't find any. Maybe the bug is in a place we didn't test.

Bhutan example: a doctor checks your blood pressure. It is fine. That doesn't mean you are fully healthy. It only means your blood pressure is fine. Other things could be wrong that this one test did not look for.
Principle 3

Early testing saves time and money

Find the problem as early as you can. Once it has been built, fixing it costs more. Once released, fixing it costs much, much more. This is the whole reason static testing exists.

Bhutan example: if you notice the wedding hall is too small while still planning, you change venues easily. If you notice on the wedding day that 200 guests are standing outside in the rain — that is a disaster you cannot fix in time. Same with software. Catch it early.
28 QA vocabulary · part 1 · the defect chain

Error · Defect · Failure · Incident

WordWhat it meansEasy example
ErrorA human mistake. The person typed the wrong thing or thought the wrong thing.The developer typed + instead of −.
Defect / BugThe flaw left in the code because of that mistake.The wrong sign sits in the code, waiting.
FailureThe visible bad behaviour the user actually sees.The bill shows Nu. 1500 instead of Nu. 500.
IncidentAnything strange that happens during testing — may or may not be a real bug yet."The screen flashed for a second. Not sure why."

The chain: human error → defect in code → user sees failure. Use the right word — it makes you sound like a professional.

32 Wrap-up
Key Takeaways

What to carry forward from today

  • Static vs dynamic — static = no code runs (reviews, walkthroughs, inspections). Dynamic = code runs (unit / integration / system / UAT). Both are real testing.
  • Static methods, in increasing formality — walkthrough → technical review → inspection. Pick by risk.
  • Dynamic — four levels — Unit · Integration · System · UAT. Brick → wall → house → family moves in.
  • V-model — every build step has a mirror test step. Agile shrinks the V into every sprint.
  • The 6 phases — Planning → Analysis → Design → Implementation → Execution → Completion. Don't skip phase 6.
  • Two principles to live by — passing tests don't prove zero bugs. Catching early is dramatically cheaper.
  • Vocabulary — error → defect → failure. Plan vs strategy, case vs scenario, suite vs run.
  • Next class: Test design techniques — how do we actually choose what to test? (Phase 3 in detail.)

No tests can prove a thing is perfect. They can only show what isn't.
