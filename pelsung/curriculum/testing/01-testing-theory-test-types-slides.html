03
Why a taxonomy matters

Without a shared taxonomy, you can't have a real testing conversation.

Devs and POs say things like "did you smoke it?", "is regression done?", "can we send it to beta?", "is this functional or non-functional?". If you don't know which box each word belongs to, you can't follow the conversation — and worse, you'll do the wrong test for the moment.

Today we use ISTQB Foundation Level — the same taxonomy used in CS courses and standard textbooks (Ammann & Offutt, Naik & Tripathy). It's the vocabulary the rest of the industry uses.

08 The four — at a glance

Every test you'll ever do fits in one of these four boxes

Type 1
Functional

WHAT the system does. "Does the feature behave as required?" Black-box, requirement-driven.

Type 2
Non-functional

HOW WELL it does it. Speed, security, usability, compatibility — the ISO 25010 attributes.

Type 3
Structural (white-box)

HOW BUILT — testing the code paths inside. Done by devs, not QAs. Recognise the term.

Type 4
Change-related

AFTER A CHANGE — re-testing fixed bugs and re-checking everything else still works.

Source: ISTQB Foundation Level syllabus. Used by Ammann & Offutt and most undergraduate software-testing textbooks.

09 Type 1 · Functional testing

"Does the feature do what it's supposed to do?"

Functional testing

🏧 Everyday picture: the bank tells you the new ATM should let a customer enter PIN, choose "withdraw", pick an amount, and dispense the cash. Functional testing is checking each of those steps actually happens, in the right order, with the right result. Did the cash come out? Did the balance update?
Definition
Testing what the system does — against the documented functional requirements. Black-box: you treat the system as a sealed box and push buttons.
Software example
In mBoB, the requirement says: "given a valid TIN and a valid amount, the user can pay electricity." A functional test logs in, picks BPC, enters TIN + amount, taps Pay, checks the receipt screen and the new balance.
Owned by
QA — this is the bulk of what you'll do every day.
Key tip: if a test starts with "the spec says…" or "the user should be able to…", it is almost certainly a functional test.
10 Inside Functional · positive AND negative

Negative testing is a flavour of functional testing — not a separate type

Positive (happy path)
"User does what the spec said"

Valid PIN · valid amount · network OK. Does it succeed? This is what most beginners write first.

Negative
"User does the opposite"

Wrong PIN 5 times — does the card lock? Negative balance — rejected? Empty form — clear message? Real users break rules by accident, all the time.

The point: both positive and negative tests check what the system does — so both are functional. Negative is not a fifth ISTQB type. It's a sub-flavour of Type 1. Half your test cases should be negative.
11 Type 2 · Non-functional testing

"Does the feature do it well enough?"

Non-functional testing

🚌 Everyday picture: the bus reaches Phuentsholing — that's functional, it did its job. But: was it on time? Was the ride comfortable? Did the engine survive the climb? Were the seatbelts secure? The journey was OK; the quality of the journey is the non-functional question.
Definition
Testing how well the system does its job — its quality attributes. Speed, security, usability, reliability, compatibility, etc.
Recap link
These are the ISO 25010 quality attributes you saw in Quality & Compliance #01. Every non-functional test you'll ever hear of belongs under one of those eight headings.
Owned by
Mixed — performance and security usually need specialists; usability and compatibility often sit with QA.
Key tip: if the test result is a number (response in 1.4 s) or a quality score (works on 8 of 10 phones), you're almost certainly in non-functional territory.
12 Non-functional · performance · load · stress

Three cousins · same family

Concept only today — no tools, no scripts.

🚌 The Coaster bus: Performance = how fast on a normal trip? Load = built for 30 — does it still drive smoothly with 30 inside? Stress = pack 60 in — at what point does it stop, break, or refuse to start?
Performance

"How fast does it respond?"

One user, normal use. mBoB: the dashboard opens in under 2 seconds.

Load

"How does it cope with many normal users?"

Expected number, at the same time. mBoB: 5,000 customers paying bills at 10 a.m. on the 1st of the month — still smooth?

Stress

"How much before it breaks?"

Push beyond normal — find the breaking point. mBoB: 50,000 users at once. Slows down? Crashes? Recovers?

Key tip: all three live under Performance Efficiency in ISO 25010. As QA Cohort 3 you won't run them — but if a dev says "we did a load test of 5k users", you must know what they mean.
13 Other non-functional flavours

Compatibility · Usability · Security · Reliability

Recognise the names. Know the question. Know who owns it.

Compatibility

"Does it work on different devices, browsers, networks?" — mBoB on iPhone & Android, on 3G in Trashigang and 4G in Thimphu.

Often QA. We touch this in QA #04 (test environments).
Usability

"Can a regular person figure it out without help?" — Can a Druk PNB customer find statement download without calling support?

Often QA + UX. Linked to ISO 25010 usability attribute.
Security

"Can someone get in or get data they shouldn't?" — Login throttling, password rules, session timeouts on mBoB.

Specialists. Touched in BA-QA Quality #05.
Reliability

"Does it stay up and recover from problems?" — Tashi Cell network drops mid-payment; does mBoB lose your money?

Mixed; SRE / dev-ops oriented.
Reading the map: every word here is a flavour of Type 2 (non-functional). They are not separate ISTQB types.
14 Type 3 · Structural (white-box) testing

"Did every line of code get exercised?"

Structural / White-box testing

🔬 Everyday picture: a mechanic doesn't just test-drive the car (that's black-box) — they also pop the hood, check that every wire and pipe was actually inspected. Structural testing is the inside-the-engine view.
Definition
Testing the internal structure of the code — every branch, every condition, every line. Needs source-code access.
Owned by
Devs, in unit tests. QAs almost never write structural tests. But you must recognise the term when devs say "we have 80% code coverage" — that's a structural metric.
Why we mention it
Because ISTQB lists it as one of the four types. Naming it makes the taxonomy complete.
Key tip: "black-box" = testing the system as a sealed box — that's everything else we do as QA. "White-box" = looking inside the code — devs only.

16 Confirmation (re-testing)

"Is THIS specific fix actually fixed?"

Confirmation testing · also called re-testing

🔧 Everyday picture: the radio in the car was broken yesterday. Today the mechanic says "fixed". You don't drive 50 km — you only test the radio. Same broken steps, same channel, same volume. Did THIS bug go away?
Definition
Running the same test that found the bug — after the dev says "fixed" — to confirm it really is fixed. Pinpoint, single-bug focused.
Software example
Bug: in Druk PNB Mobile, the date "29-Feb" wouldn't save in the leave form. Dev fixes it. Confirmation = open the same screen, type 29-Feb, save. Did it save? Only this.
When you do it
Right after a bug is marked "fixed", before regression.
ISTQB term tip: the formal name is confirmation testing; everyone in industry calls it retest. They're the same thing.
17 Regression testing

"Did our change break anything else?"

Regression testing

🛞 Everyday picture: the mechanic replaced one tyre. Smart you doesn't only check that tyre — you also press the brakes, check the steering, check that nothing was loosened by mistake. Re-check the things that were already working.
Definition
Re-running tests on areas that were already working, to catch unintended breakage caused by a new change. Wide and deep — the safety net.
Software example
Devs added scheduled payments in mBoB. Regression = re-run the existing checks on login, balance, normal payment, statement download — to make sure none of those got broken by the new code.
When you do it
Before every release, or whenever a meaningful change goes in.
Key tip: regression is the biggest, slowest test pass — and the most important. Without it, every fix risks breaking something else.
18 Smoke testing

"Is this build alive at all? Wide and shallow."

Smoke testing

🚗 Everyday picture: you turn the car key in the morning. Before driving anywhere, you wait 5 seconds — does the engine start? Any black smoke? Lights work? You're just checking "is this thing alive?"
Definition
A short, broad check on a brand-new build — 5–10 minutes, only the most important screens. Wide, shallow. If smoke fails, send the build back.
Software example
A fresh Druk Air booking app build arrives. You open the app, log in, see the dashboard, search a flight, close. If any of these break — build rejected.
Where it sits
Smoke is often a subset of regression, run on every build as a quick gate. ISTQB lists it under change-related.
Key tip: smoke = build acceptance. It protects your time, not the user. Wide and shallow, never deep.
19 Sanity testing

"We fixed a small thing — does that small thing work, narrow and a bit deeper?"

Sanity testing

📻 Everyday picture: the mechanic fixed your car radio. You don't drive 50 km to test everything. You turn the key, switch on the radio, change two stations, check the volume. Narrow but a little deeper. Just the area that was touched.
Definition
A narrow, focused, slightly deeper check on the area of a recent fix — when there's no time for a full regression pass.
Software example
Devs changed how the mBoB date picker works. Sanity = open the date picker on 2–3 screens, pick a date, save, reopen. Don't run the entire app's tests — just check the date picker behaves.
Where it sits
Sanity is also a subset of regression — scoped to just the changed area instead of the whole app.
Smoke vs sanity in one breath: smoke = wide & shallow on a new build. Sanity = narrow & a bit deeper on a fixed area.
20 Change-related · the big four side by side

If you only remember one slide today, make it this one

NameOne-line meaningScopeWhenEveryday picture
Smoke "Is this build alive at all?" Wide · very shallow Each new build, before deeper testing Start the car — any smoke?
Sanity "Is this small change actually working?" Narrow · a little deeper After a small fix, no time for full regression Radio fixed — switch it on, change 2 stations
Retest (confirmation) "Is THIS specific bug fixed?" Pinpoint · same steps as the bug Right after dev marks a bug fixed Re-check ONLY the broken radio

Order on a release day: Smoke (build alive?) → Retest (this bug?) → Sanity (this small fix?) → Regression (everything else still OK?). All four are change-related tests — one ISTQB type, four flavours.

04
22 Alpha & Beta · operational stages, not types

"Who is allowed to use this version yet?"

Alpha & Beta testing

Categorised by audience, not by what's tested. ISTQB calls this an operational stage, not a test type.

Alpha
Internal users · controlled

Inside SELISE / inside Bank of Bhutan staff. They use the new mBoB version internally for a week before any external user sees it. Bugs come back internally only.

Beta
Limited external users · real conditions

500 willing mBoB customers get the new version for 2 weeks. Real banking, real devices, real complaints — before full public release.

Inside each stage: what those users do is still functional + non-functional + change-related testing. The "alpha" / "beta" word only tells you who is using it and when.
23 A/B testing · product experiment, not a test type

"Which version of the feature do real users prefer?"

A/B testing

🥟 Everyday picture: a momo shop in Thimphu wants to know which sells more on a weekday — chicken or veg. They put both on the menu at the same price for two weeks, count what each customer orders, pick the winner. Two real options · real users · count what wins.
What it is
Releasing two versions (A and B) of a feature to different groups of real users at the same time, then measuring which performs better on a chosen number (clicks, sign-ups, time spent).
Software example
mBoB tries two designs of "Send Money" — half users see a blue button, half see a green button. After a week, count which group tapped Send more. Keep the winner.
Where it sits
Not in the ISTQB taxonomy. It's product experimentation — a marketing / product-management activity. QA's job: make sure both A and B work, and the right user sees the right one.
Key tip: A/B is not bug-finding. Don't classify it as functional or non-functional — it's a separate concept. Place it correctly when you hear it.
24 Exploratory & Ad-hoc · techniques, not types

These are how you design tests, not what you check

Exploratory
Think while you test (with intent + notes)

One hour, charter-driven: "explore mBoB login on slow Wi-Fi". No fixed script — design the next step from what you just saw. Real work; bugs are filed.

Ad-hoc
No plan, no notes

5 minutes before a demo, click randomly. Not repeatable. Not a strategy — a quick pre-flight check.

Where they sit on the map: ISTQB classifies these as experience-based test design techniques — they belong in the Technique column from slide 06, not the Type column. We'll teach them properly in QA #05 / #06. Today: just place them correctly.

27 Wrap-up
🎯 Key takeaways

Test types · what to carry forward

  • ✅ A test type answers WHAT we check — ISTQB defines exactly four.
  • ✅ Functional = does it do what it should? · positive & negative are sub-flavours.
  • ✅ Non-functional = does it do it well enough? · performance, load, stress, compatibility, usability, security, reliability all live here (ISO 25010).
  • ✅ Structural / white-box = inside the code · devs do this · QA only needs to recognise the term.
  • ✅ Change-related = after a fix · Smoke, Sanity, Retest (Confirmation), Regression. Order: smoke → retest → sanity → regression.
  • ✅ Alpha / Beta are stages (who's using it, when), not types.
  • ✅ A/B is product experimentation, outside the ISTQB taxonomy.
  • ✅ Exploratory / Ad-hoc are techniques — how you design tests, not what you check. Taught properly in QA #05 / #06.
  • ✅ Next class: Testing Theory II — test design techniques (boundary value, equivalence partitioning, decision tables).
