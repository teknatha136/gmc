Part One

From AC

Acceptance Criteria → 1 or more test cases · always.

01
06 The story we test

Mr. Tshering applies for leave

User Story
Apply for annual leave

As an employee,
I want to submit a leave request from the HR app,
so that my manager can approve it without paperwork.

AC 1 · Happy path

Given I am logged in as an employee with at least 3 days of leave balance,
When I submit a request for 3 days of annual leave with a reason,
Then the request is saved and shown as Pending on my dashboard.

AC 2 · Not enough balance

Given I have only 2 days of leave balance,
When I submit a request for 3 days,
Then the system blocks the request and shows: "Not enough leave balance."

AC 3 · Missing reason

Given I am on the leave form,
When I click Submit with the reason field empty,
Then the form shows: "Reason is required." and does not submit.

07 Each AC becomes one or more test cases

Read AC · write the test that proves it

AC 1 · Happy path
  • Logged in · 3 days balance
  • Submit 3 days · with reason
  • Request saved · status Pending
→
TC-LEAVE-01 · Submit annual leave (3 days)
  • Pre: user logged in, balance = 12 days
  • Steps: open form · pick dates · type reason · click Submit
  • Expected: success message · request visible on dashboard, status Pending
AC 2 · Not enough balance
  • Balance = 2 days
  • Try 3-day request
  • System blocks · shows error
→
TC-LEAVE-02 · Block request when balance < days
  • Pre: user logged in, balance = 2 days
  • Steps: open form · pick 3 days · type reason · click Submit
  • Expected: red message "Not enough leave balance." · request NOT created
AC 3 · Missing reason
  • On the form
  • Submit with empty reason
  • Show "Reason is required."
→
TC-LEAVE-03 · Block submit when reason empty
  • Pre: user logged in, balance = 12 days
  • Steps: pick dates · leave reason empty · click Submit
  • Expected: error "Reason is required." · form stays open · no request created

One AC can become 1, 2 or even 5 test cases. Never zero.

10 The blank template you fill in every time
Test Case IDTC-XXXX-NN
TitleShort, action-oriented sentence — what the test verifies
Module / FeatureWhere this lives (e.g. HR App · Leave Request · Submit form)
Pre-conditionsExact starting state · user · role · data already in the system
Test DataThe exact values you will type — never "valid input"
Steps
  1. One action
  2. Next action
  3. Final action
Expected ResultThe one specific thing that MUST be true at the end
Actual Result(filled at run-time — what actually happened)
StatusPass · Fail · Blocked
Notes / TraceLinked AC · story ID · bug ticket if failed

Same shape · every time. A new joiner can read any of your test cases.

11
Part Three

The 7 Rules
of a Good Test Case

Bad → Good. Every rule.

03
12 Rule 1 of 7
1

Atomic steps · one action per step

If the step has the word "and", split it.
❌ Bad

1. Open the leave form, pick dates, type the reason and click Submit.

✅ Good

1. Open the leave form.
2. Pick start date 2026-06-01 and end date 2026-06-03.
3. Type reason: Family wedding.
4. Click Submit.

13 Rule 2 of 7
2

Numbered & sequenced

A reader should never wonder "what comes next?"
❌ Bad · bullet soup

• Open mBoB
• Transfer money
• Confirm

✅ Good · ordered

1. Open mBoB.
2. Tap Send Money.
3. Pick Sonam Lhamo from contacts.
4. Enter amount 500.
5. Tap Confirm.
6. Enter PIN 4321.

14 Rule 3 of 7
3

Concrete test data · not "valid input"

If 5 testers run this, they should all type the same thing.
❌ Bad

Steps: "Enter a valid CID and a valid amount."

✅ Good

Test Data:
CID: 10901001234
Amount: 500 Nu
Recipient: Sonam Lhamo · 17712345

15 Rule 4 of 7
4

Specific expected result · not "should work"

"Works" is not a result. A message, a value, a screen — that is a result.
❌ Bad

Expected: "The transfer should work."

✅ Good

Expected:
• Green banner: "Transfer successful · Ref BNB-2026-0419-7732"
• Sender balance reduced by 500 Nu
• Transaction shown in History with today's date

16 Rule 5 of 7
5

Independent · doesn't depend on the previous case

If TC-02 fails, TC-03 must still be runnable.
❌ Bad · chained

TC-03 Pre-condition: "TC-02 has just passed and a leave request exists."

✅ Good · stands alone

TC-03 Pre-condition: "User tshering@selisegroup.com is logged in. A pending leave request LR-2026-0512 already exists in the database (seeded)."

17 Rule 6 of 7
6

Reusable · clear enough that anyone can run it

A new joiner with no context should be able to execute it.
❌ Bad

Steps: "Use the school registration form like we did last week."

✅ Good

1. Open https://school.gov.bt/register.
2. Select Class IX · Drukgyel Higher Secondary.
3. Fill student name Pema Choden, CID 11308002211, DOB 2010-04-12.
4. Upload cid.pdf.
5. Click Register.

18 Rule 7 of 7
7

Traceable · linked to AC / story / requirement

Every test case must answer: "Why does this exist?"
❌ Bad

Notes: (empty)

✅ Good

Notes / Trace:
• Story: HR-142 · Apply for leave
• AC: AC-2 (not enough balance)
• Test Plan: TP-HR-v1.0 § 05
• Bug if failed: HR-BUG-091

No trace = nobody can prove the requirement was tested.

19 Positive vs negative · both matter

"Log in" feature · 2 positive · 2 negative

✅ Positive · the system DOES allow it
  • TC-LOGIN-01
    Valid email + correct password → user lands on dashboard.
  • TC-LOGIN-02
    Valid email + correct password + "Remember me" ON → closing & reopening browser keeps user logged in.
⛔ Negative · the system MUST block it
  • TC-LOGIN-03
    Valid email + wrong password → error "Email or password is wrong." · still on login screen.
  • TC-LOGIN-04
    Empty email + any password → form shows "Email is required." · no request sent.

A test suite with only positive cases is half a test suite.

21 The 5 mistakes we see every cohort
The mistakeWhy it hurtsFix
"Click around" steps Two testers run two different tests · results are not comparable Numbered, atomic steps with exact button names
"Verify it works" expected result "Works" is opinion, not fact · everyone passes everything One specific message, value or screen the tester must see
5 actions in 1 step If step 3 fails, you don't know which of the 5 broke Split until each step is one verb, one click, one type
Vague pre-conditions ("logged in") Logged in as whom? With what data? Bug is unreproducible "Logged in as user with role Manager and 12 days leave balance"
Skipping negative test cases Real bugs hide in the "we don't allow this" paths For every positive case, ask: "what should the system refuse?"
22 Hands-on · 30 minutes
Activity · in pairs

Approve a sick leave request as a manager

User story: As a manager, I want to approve or reject sick leave requests from my team, so that my employees can take time off without delay.

Acceptance Criteria
AC 1. Manager sees pending leave requests on the dashboard.
AC 2. Manager can approve a request and the employee is notified.
AC 3. Manager can reject a request and a reason is required.
Your task · in pairs
  • 20 min: Write 6+ test cases · mix positive and negative.
  • Use the full anatomy template · all 10 fields.
  • Cover all 3 ACs.
  • 10 min: Swap with another pair · review against the 7 rules · mark each rule ✅ or ❌.
  • Each pair shares 1 test case & 1 finding from the review.

Hint: think about empty fields · wrong roles · already-approved requests · and what happens when reason is blank.

23 Wrap-up
🎯 Key Takeaways

What we covered today

  • ✅ Every AC becomes 1 or more test cases · never zero
  • ✅ Same 10 fields every time: ID · Title · Module · Pre-conditions · Test Data · Steps · Expected · Actual · Status · Notes
  • ✅ The 7 rules: atomic · numbered · concrete data · specific expected · independent · reusable · traceable
  • ✅ Write positive AND negative test cases · always
  • ✅ Pre-conditions = state (role, balance, data) · not "logged in"
  • ✅ Expected result = a thing you can see · not "should work"
  • ✅ A good test case can be run by someone who has never seen the feature
