03 Hook
🎤 A familiar fight

PM: "We're shipping tomorrow."
QA: "We're still testing."
PM: "Which tests are left?"
QA: "…"

Without a Test Plan, "are we done?" has no answer.
Every release becomes a fight.

A Test Plan is the contract that makes "done" unambiguous.

04 What a Test Plan is (and isn't)

A strategy document · for everyone · not just QA

✅ A Test Plan is…
  • A strategy, not a list of test cases
  • A contract between QA, Dev, PM on "done"
  • Release-scoped · 10–20 pages
  • Signed off before testing starts
  • The doc a new joiner reads in 20 min
❌ A Test Plan is NOT…
  • A dump of test cases (those live in TestRail / Xray)
  • A status update
  • A defect report
  • The org-wide QA strategy
  • Written after testing starts

07 The 4 fires a Test Plan puts out

Every team without a plan · gets burned by the same four

  • 🔥 "Who tested that?" — Mid-incident, nobody can say which team verified it. → § 15 Responsibilities names R per area.
  • 🔥 "Are we done?" — PM pushes, QA still testing, nobody has a definition. → § 11 Exit criteria.
  • 🔥 "That was never in scope" — Bug lands post-launch · "nobody said to test X". → § 05 / § 06 explicit lists.
  • 🔥 "We can't start" — Build broken · data stale · ticket not ready. → § 09 Entry criteria.

02
09 § 05 & § 06 · features IN / OUT

§ 05 · Features to be tested

  • FR-01 Save favourite · P0 · Functional + Integration
  • FR-02 View favourites · P0 · Functional + Perf + A11y
  • FR-03 Use favourite (pre-fill) · P0 · Functional + Security
  • NFR-Perf-01 p95 < 1s · P0 · Performance
  • NFR-Sec-01 Scope to user · P0 · Security
  • NFR-A11y-01 WCAG AA · P1 · Accessibility

Every line traces back to the feature spec or HLD.

§ 06 · NOT to be tested · with reason

  • Bulk import/export — deferred to v2 per PM 2026-04-18
  • Legacy mBoB v4 — unsupported · product decision
  • Payment-gateway internals — vendor warranty
  • iOS 13 and below — below platform floor (iOS 14+)

Prevents "nobody tested that" fights post-launch.

Part Three

The 4 Gates

Entry · Suspend · Resume · Exit.

03
13 § 09 · Entry criteria · can testing START?

Must be TRUE before any test case is executed

  • Feature spec / SRS approved
  • Build deployed to test environment · smoke green
  • Test data prepared & refreshed
  • Test cases reviewed & approved
  • Dev has marked ticket(s) "Ready for QA"
  • Known blocker defects resolved or flagged

If the list isn't all ticked, QA does not start. Starting on a broken build burns days.

14 § 11 · Exit criteria · release readiness

All TRUE → release can ship

  • 100% of P0 test cases executed · 0 P0 defects open
  • ≥ 95% of P1 test cases executed · 0 P1 defects open (or PM-accepted with mitigation)
  • Known-issue list published & reviewed with PM
  • Performance targets met (see § 05 NFRs)
  • Security scan clean · or exceptions approved by Security
  • Accessibility · no critical axe violations
  • Regression suite green on final build
  • Sign-off: QA Lead · Dev Lead · PM

Binary. "0 P0 open" — not "few P0". "Regression green" — not "mostly green". Ambiguity here kills releases.

18 § 19 · Metrics · captured daily · not at the end

Daily capture

  • Test cases — planned · executed · passed · failed · blocked · deferred
  • Defects by severity — open · closed · reopened
  • Automation pass rate
  • Coverage vs requirements (% reqs with ≥ 1 test)

Reporting cadence

  • Daily — 3-line Slack summary (execution · top defect · top risk)
  • Weekly — burndown + risks to PM
  • End of cycle — full test summary report

Reporting only at the end = no time to correct course. Daily = early-warning system.

21 Wrap-up
🎯 Key Takeaways

What we covered today

  • ✅ Test Plan = what · how · who · when · when-done · for everyone, not just QA
  • ✅ 21 sections (IEEE 829) · from identification to approvals
  • ✅ Features NOT to be tested matters as much as features to be tested
  • ✅ 4 gates: Entry · Suspend · Resume · Exit · each a binary check-list
  • ✅ Exit = 0 P0 · 0 P1 · perf met · security clean · a11y clean · regression green
  • ✅ RACI (or RCI) = one R per activity
  • ✅ Metrics captured daily · not at the end
  • ✅ Living document · written before · updated during · reviewed after

If "are we done?" needs a meeting, the Test Plan already failed.
