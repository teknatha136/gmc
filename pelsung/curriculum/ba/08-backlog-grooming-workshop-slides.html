04 What is a backlog?

A backlog is the prioritized list of everything we MIGHT build — always more than we can.

It is not a to-do list. It is a wish list of features, fixes, and ideas — sorted top-to-bottom from "next sprint" to "maybe never". The team only builds the top.

📝 Wedding planning list

Venue, catering, costumes, music, transport, photography, gifts. 60 items. Wedding is in 8 weeks. Same idea: too much to do, not enough time.

🥟 Momo shop opening

Find shop space, get licence, buy fryer, design menu, hire helper, source pork, social media. The opening date is fixed. The list is not.

📱 mBoB feature wishlist

Bill payment, fund transfer, save favourites, dark mode, fingerprint login, group split. Bank cannot ship them all at once. Order matters.

05 Why backlogs rot

Six things that turn a clean backlog into the kitchen drawer

1. Duplicates

The same idea added twice by two different people, in different words.

"Add favourites" + "Save frequent billers"
2. Vague items

One-line wishes nobody can act on. No role, no value, no acceptance criteria.

"Make app faster" / "Improve UX"
3. Stale items

Added a year ago for a need that no longer exists. Still sitting there.

"Add COVID-19 banner"
4. No owner

Nobody to clarify, decide, or sign off. The item just floats.

"???" in the requester column
5. No acceptance criteria

Item has a title and nothing else. Dev has nothing to build to.

"Forgot password screen" — that's it
6. Way too big

An item that's actually a whole project pretending to be one story.

"Build trainee portal"

A rotting backlog slows the team — every sprint planning becomes a fight to find one ready story.

Part One

Definition
of Ready

The checklist a story must pass before it enters a sprint.

01
07 Definition of Ready · the six checks

Like a momo shop opening — you don't open the door until six things are done.

If a story fails any of these, it stays in the backlog. It does not enter the sprint.

📋 A story is READY when…
  1. 1. Story format — written as "As a … I want … so that …" with role, action, and value (see Writing User Stories).
  2. 2. INVEST checked — Independent · Negotiable · Valuable · Estimable · Small · Testable (see Writing User Stories).
  3. 3. Acceptance criteria written — at least 3 Given/When/Then statements (see the Combined Opener block on Acceptance Criteria).
  4. 4. Dependencies known — what needs to happen first? Any blockers flagged?
  5. 5. Estimable — the team has enough info to give a T-shirt size (S, M, L, XL).
  6. 6. Designs attached — if the story has a UI, the wireframe or mockup is linked (see Wireframing for Communication).

Think of this like a wedding day checklist — venue booked, food confirmed, costumes ready, transport arranged. Miss one and the day suffers.

Part Two

Six Grooming
Activities

What the BA actually does in a grooming meeting.

02
09 Six grooming activities

Each grooming meeting · pick items off the top · run them through these six

🔍 1. Clarify ambiguity

Talk to the requester. Replace vague words with concrete behaviours.

"Make it faster" → "Loads in under 3 seconds"
✅ 2. Add or refine AC

Write 3–7 Given/When/Then. Cover happy path + edges. (See the Combined Opener block on Acceptance Criteria.)

Each AC = pass/fail · testable
✂️ 3. Split too-big stories

If a story can't fit in one sprint, break it into smaller stories that can.

"Online leave request" → 3 leave-type stories
🔼 4. Re-prioritize

Move newer-higher-value items up. Use MoSCoW. (See the Combined Opener block on Estimation & Prioritization.)

Must · Should · Could · Won't
🗑️ 5. Remove dead items

Delete duplicates, stale, abandoned. Don't be afraid to drop things.

"Add COVID-19 banner" → drop
📏 6. Estimate

Team gives each top story a T-shirt size: S, M, L, or XL.

XL means split it before you build it

Grooming is not a one-time event. Most teams do it weekly · 30–60 min · just enough to keep the top of the backlog ready.

10 Splitting too-big stories

Four ways to slice a too-big story into ready ones

Story too big to fit a sprint? Pick a slice line:
1. By workflow step

Big = "user does the whole journey". Slice into one story per step.

Wedding RSVP → 1) view invite, 2) confirm attendance, 3) pick meal
2. By user type

Different users need different versions. One story per user type.

Trainee portal login → 1) trainee, 2) trainer, 3) admin
3. By happy path vs alt

Build the main flow first. Edge cases as separate stories.

Bill pay → 1) success, 2) insufficient balance, 3) timeout
4. By data variation

Same flow, different data. One story per variation.

Online leave request → 1) sick, 2) annual, 3) maternity

Rule of thumb: if a story would take more than 3–4 days to build, split it.

13 Demo · AFTER grooming

Split into 2 ready stories · AC written · DoR ticked · sized

BL-027a · sliced by data variation (sick leave first)

As a trainee, I want to submit a sick-leave request online so that I don't have to walk to the office when I'm unwell.

  • AC1: Given a logged-in trainee, when they fill in dates + reason and tap Submit, then a request is created and shown in their "My requests" list within 2 seconds.
  • AC2: Given an empty reason field, when they tap Submit, then the system shows "Reason is required" and does not submit.
  • AC3: Given a submitted request, when the trainer opens it, then they see trainee name, dates, reason, and Approve / Reject buttons.
Size · M DoR ✓ Owner · HR (Sonam) Designs · Figma link attached
BL-027b · annual leave (built next sprint)

As a trainee, I want to submit an annual-leave request with my balance shown so that I can plan how many days I have left.

  • AC1: Given a logged-in trainee with a balance > 0, when they open the form, then their remaining annual-leave days are visible at the top.
  • AC2: Given a request that exceeds the balance, when they Submit, then the system blocks submission and shows "You have only X days left."
  • AC3: Given an approved request, when the trainee views balance again, then it has been reduced by the approved days.
Size · M DoR ✓ Owner · HR (Sonam) Depends on · BL-027a

One messy XL item · two ready M stories · maternity leave parked for later. The team can pick BL-027a tomorrow.

17 Hands-on · what each groomed story looks like

Use this shape for every one of your top-5 groomed stories

Template — fill in for each story

As a [role], I want to [action] so that [value].

  • AC1 (happy path): Given …, when …, then …
  • AC2 (variation or edge): Given …, when …, then …
  • AC3 (error or limit): Given …, when …, then …
Size · S / M / L / XL DoR ✓ / ✗ MoSCoW · M / S / C / W Owner · ?
DoR check (tick each)
  • Story format with role / action / value
  • INVEST checked
  • Acceptance criteria written (3+)
  • Dependencies known
  • Estimable (T-shirt size given)
  • Designs attached if UI

If a story fails any DoR check — it goes back to the backlog · not into the sprint.

18 Debrief · what to discuss

When groups present · use these questions to challenge each other

For the presenting group

Walk us through your groomed top-5 in priority order. For each story: what does it mean? Why is it ranked here? Show one AC. T-shirt size and why.

For the listening group

Pick one story · find one AC that's not yet testable · suggest a fix. Spot any item that fails DoR. Argue one priority swap.

Reflection · trainer-led

Which item was hardest to groom? Which dead item was hardest to drop? What would you do differently if you ran a real grooming meeting next week?

What "good" looks like

BL-03 (attendance) was split. BL-12 (LMS) parked or chopped. BL-04/BL-05 deduped. BL-07 dropped (solution-leak). Top stories all have AC + size + DoR ticks.

19 Wrap-up
🎯 Key Takeaways

What we covered today

  • ✅ A backlog is the prioritized list of what we MIGHT build — always more than capacity
  • ✅ Backlogs rot from duplicates · vague items · stale items · no owner · no AC · too big
  • ✅ Definition of Ready = the 6-check gate before any story enters a sprint
  • ✅ Six grooming moves: clarify · add AC · split · re-prioritize · drop dead · estimate
  • ✅ Split too-big stories by workflow step · user type · happy/alt · or data variation
  • ✅ Grooming is weekly · 30–60 min · just enough to keep the top ready
  • ✅ A clean backlog is the BA's most visible product — protect it like a kitchen drawer that actually works

A groomed backlog beats a long backlog — every sprint, every time.
