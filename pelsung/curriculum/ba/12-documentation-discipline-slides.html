03
Part One

Docs are
a Tax

Every word you write is a word someone will need to update later.

01
04 The wedding-notes problem

Think of a family wedding planning notebook.

You write down the guest count. The venue. The menu. The car list. Every detail. Then a cousin gets added. The menu changes. The driver cancels. Every word you wrote is now a word someone has to fix.

The rule
Documentation is not free.

Every page you write costs effort to write, costs effort to read, and costs effort to keep correct. So the goal of a good BA is never "write more docs". It is: write the minimum that is maximally useful, and keep that alive.

Pay the tax deliberately. Don't write what you can't maintain.

Part Two

Write vs
Link

Two columns. One decision aid you can use every day.

02
06 A school-handbook example

A school handbook tells you what to do — but links the timetable.

If the school wrote the timetable into the handbook, every change in the period schedule would force a reprint. So they link to the live one. Same idea in software docs.

📝 Write it down
  • Decisions — and the reason behind each one
  • Acceptance criteria — what "done" looks like
  • Requirements — what the system must do
  • Sign-offs — who agreed and when
  • Glossary — what each term means here
These don't change daily. They are the handbook.
🔗 Link to it
  • Dashboards — live numbers, not screenshots
  • Code — the repo, not pasted snippets
  • Designs — Figma file, not exported PNGs
  • Schedules / rosters — the live sheet
  • Anything maintained somewhere else
These change often. They are the timetable.

The test: "Will this fact change without me knowing?" If yes — link, don't copy.

07 The doc hierarchy

Docs cascade — like a government circular.

A national policy comes down. Each ministry adds its own. Each office adds a memo. Software docs work the same way — each level narrows the scope.

Org-wideLevel 1
Brand guide, security policy, code of conduct, hiring rules.SELISE Brand Guidelines applies to every team, every project, every cohort.
Project-wideLevel 2
Vision, scope, glossary, stakeholders, the project's CLAUDE.md."GMC Portal — what it is, who uses it, what's in/out of scope."
Feature-levelLevel 3
BRD, SRS, AC, HLD/LLD for one feature."Password reset" — its own folder, its own spec.
Sprint-levelLevel 4
User stories, tasks, the day-to-day work cards."As a user I want… · Task 1: build the OTP screen."
Decision logRunning across all levels
Short notes: "We chose X because Y, on Z date." Saves the team from re-arguing the same thing six months later.One running document per project.
08 Living vs frozen docs

Some docs freeze. Some breathe. Mark which is which.

A signed contract is frozen — you don't quietly edit it later. A glossary is alive — you add terms as the project grows. If a reader can't tell which kind it is, the doc has already lost their trust.

❄️ Frozen — sign-off docs

Captured at a moment in time. Changing them needs a new version, a new approval, and a new date.

  • Signed BRD for a release
  • Approved scope statement
  • Bhutan visa rule sheet for a specific year
  • The school handbook printed for 2026
Status: APPROVED · Locked · v1.0 · 2026-04-09
🌱 Living — ongoing reference

Expected to change. Updated as the project moves. Has an owner who keeps it correct.

  • Project glossary
  • Decision log
  • Risks register
  • The team's CLAUDE.md / agent-os files

09 Why docs decay

Four reasons docs rot — every BA has seen all of them.

1. No owner

If "everyone" owns it, no one updates it. By month three the doc is wrong and no one feels responsible.

"Who keeps the requirements current?" → silence.
2. Buried in folders

Three clicks deep, in a drive only the original author knows. People give up looking — or worse, write a second one.

/Shared/Old/2024/temp/_v3/spec.docx
3. Versioned by filename

"v3_final_FINAL_FINAL2_use-this.pdf". Looks familiar? It's a confession that no one trusts the doc to live in one place.

Bhutan-visa-rules_v3 final FINAL.pdf
4. Duplicated across tools

The "real" requirements are in a Word doc, a Confluence page, a Slack thread, and someone's notebook — and they disagree.

Three sources of truth means zero sources of truth.
10 Practices that keep docs alive

Four habits that turn rotting docs into trusted ones.

1. One canonical location per topic

Every topic has one home. If you want the latest, you go there. No "is this the one?" guessing.

"GMC portal requirements live here — and only here."
2. Visible "last reviewed" + owner

Every living doc shows who owns it and when it was last checked. Same idea as the school handbook's review page.

Owner: Bhuwan · Last reviewed: 2026-04-30
3. Quarterly review ritual

Block 30 minutes every quarter. Walk every living doc. Mark stale, fix, or archive. Small effort · huge payoff.

"First Friday of the quarter — doc-cleaning day."
4. Link from where work happens

Put the doc link inside the Jira ticket, the Slack channel topic, the team's home page. People won't search — they will click what's in front of them.

Jira ticket → "Spec: link" pinned on every story.

Part Three

agent-os
The AI-Friendly Spec Format

A folder shape that turns your specs into context an AI agent can actually read.

03
12 Why AI needs structured docs

Vague docs in → vague code out.

When you ask Claude Code to build a feature, it reads what you give it. If your "spec" is a chat message saying "make a password reset thing", you get something that kind of works. If your spec has clear sections, you get something that matches.

❌ What you give the AI today
"Add password reset to the GMC
portal. Make it work like the
banking app. Should be secure
and quick. Use OTP I think."

The AI guesses. You get whatever it imagined — phone OTP or email link, 5 min expiry or 15, error wording made up. Same problem as the "two implementations" story from earlier.

✅ What a structured spec gives it
spec.md
  Overview · 2 lines
  User stories · 3 stories
  Scope · in / out
  Technical notes · OTP via SMS,
                    5-min expiry,
                    rate-limit 3/10min
tasks.md
  1. OTP request screen
  2. OTP verify + new password
  3. Rate-limit + audit log

The AI reads sections, not vibes. The output matches the spec — because the spec specified.

13 What agent-os is

A naming convention. That's it.

agent-os is an open project structure (from Builder Methods) that says: put your standards, your product context, and your per-feature specs in well-named markdown files inside well-named folders, so any AI agent — or any new teammate — can read them in order and understand the project.

No tool to install. No magic. Just opinionated folders + opinionated filenames — and a habit of writing things in those files instead of in chat.

Why a BA should care

If you write a spec in agent-os shape, a developer can hand it straight to Claude Code and the AI will produce code that matches. Your spec becomes the prompt. The clearer the spec, the better the build. That is your leverage.

14 The shape · folders & files

A typical agent-os layout

Three top-level folders inside a hidden .agent-os/ directory. Each holds a small set of named markdown files.

.agent-os/ ├── product/ # the project's "constitution" │ ├── mission.md # what we're building & for whom │ ├── roadmap.md # phases / planned features │ ├── decisions.md # why we chose what we chose │ └── tech-stack.md # what it's built on │ ├── specs/ # one folder per feature, dated │ └── 2026-05-02-password-reset/ │ ├── spec.md # the feature itself │ └── tasks.md # numbered work breakdown │ └── standards/ # house style, applies everywhere ├── code-style.md └── best-practices.md

Same shape every project. New BA, new dev, new AI session — they all know where to look.

15 Inside spec.md

Six sections. Every feature spec. Same order.

# Overview

Two or three sentences. What is this feature, and why are we building it now? A teammate or AI should understand the point in 20 seconds.

# User Stories

The "As a … I want … so that …" stories this spec covers. Usually 2 to 5 stories.

# Scope · In

A bulleted list of what this feature does. Be specific.

# Out of Scope

What this feature does not do. The single biggest gift you can give a developer or AI — most rework comes from unclear out-of-scope.

# Technical Notes

Constraints the BA already knows: "OTP via SMS", "must work offline", "uses existing user table". Not full design — just the pinned facts.

# Dependencies

What must exist first. Other specs this depends on, external services, third-party APIs, decisions still pending.

16 Sample · password-reset/spec.md

A real spec — short, structured, prompt-ready

# Overview

Let a GMC portal user reset their forgotten password without contacting support. Phone-based OTP flow, mirrors our login design.

# User Stories
  • As a user with a forgotten password, I want to reset it online, so I can log back in without help.
  • As a user, I want a clear error if my number isn't registered, so I'm not stuck guessing.
  • As a user, I want to be told if my code expired, so I can request a new one.
# Scope · In
  • "Forgot password" link on login screen
  • Phone-based OTP (6 digits, 5-min expiry)
  • New password screen with policy check (≥ 8 chars, not last 5)
  • Rate-limit: 3 OTPs per number per 10 minutes
# Out of Scope
  • Email-based reset (separate spec, later)
  • Biometric reset
  • Admin-initiated password reset
# Technical Notes
  • OTP delivery: existing SMS gateway (Bhutan Telecom)
  • Reuse login screen styling — no new design needed
  • Audit log entry for every reset attempt
# Dependencies
  • SMS gateway credentials provisioned (DevOps)
  • Spec 2026-04-15-audit-log already merged
17 The CLAUDE.md pattern

Every project gets a CLAUDE.md at its root.

Same idea as agent-os — opinionated context the AI reads before doing anything. A short file that says "this is the project, this is the audience, here's what not to touch." We use it on this very Pelsung project.

# GMC Portal # Audience GMC residents and staff. Many are first-time digital users. Plain-English copy, no jargon. Bhutanese examples preferred. # Brand SELISE blue (#0066B2). Bahnschrift headings, Open Sans body. Logo on every page (top-right). Never use red as a feature colour. # Do not touch - /auth/legacy/ — frozen, audit in progress - Existing translation keys — coordinate with content team # House rule Spec-driven only. New work needs a spec.md in .agent-os/specs/.

Five sections. The AI reads it on every session. No surprises.

Part Four · Hands-on

Convert a Casual
Spec into agent-os

Pairs · 20 minutes · produce spec.md + tasks.md

04
19 The brief

Take this casual one-paragraph spec — turn it into agent-os shape.

Casual spec — given to you in chat
Add password reset to the GMC portal

"We need a forgot-password thing on the GMC portal. Use phone OTP like the banking app. Five-minute expiry should be fine. Don't let people spam it. Block them after a few tries. Re-using the same old password should not be allowed. Make it look like the rest of the login screens — no new design. Email reset can come later."

Produce two files

  • spec.md with all six sections: Overview · User Stories · Scope In · Out of Scope · Technical Notes · Dependencies
  • tasks.md with a numbered task list (5–8 tasks)

How to work

  • Pair up
  • One person drafts spec.md · the other drafts tasks.md
  • Swap and challenge each other — is anything implied? Is anything in both?
  • Two volunteer pairs share at debrief

12 min writing · 5 min pair-challenge · 3 min debrief.

20 Debrief · one good answer

spec.md (compressed) + tasks.md

# spec.md

.agent-os/specs/2026-05-02-password-reset/spec.md

## Overview

Phone-OTP password reset for GMC portal users.

## User Stories
  • As a user with a forgotten password, I want to reset it online.
  • As a user, I want a clear error if my number isn't registered.
  • As a user, I want to know if my OTP expired.
## Scope · In
  • "Forgot password" link on login
  • 6-digit OTP, 5-min expiry
  • New-password policy: ≥ 8 chars, not last 5
  • Rate-limit: 3 OTPs / number / 10 min
## Out of Scope
  • Email reset · Biometric reset · Admin-initiated reset
## Technical Notes
  • SMS gateway: Bhutan Telecom (existing)
  • Reuse login screen styling
  • Audit-log every attempt
## Dependencies
  • SMS gateway credentials
  • Audit-log spec (already merged)
# tasks.md

.agent-os/specs/2026-05-02-password-reset/tasks.md

  • 1. Add "Forgot password?" link to login screen
  • 2. Build OTP-request screen (phone input + Submit)
  • 3. Wire SMS gateway · send 6-digit OTP · 5-min expiry
  • 4. Build OTP-verify + new-password screen
  • 5. Enforce password policy (≥ 8 chars, not last 5)
  • 6. Apply rate-limit (3 OTPs / number / 10 min)
  • 7. Write audit-log entry on every attempt
  • 8. End-to-end test: happy path + 4 edges

Notice: tasks are verbs, in order, each one a developer (or AI) can pick up alone.

Same casual paragraph · now ready to hand to Claude Code or a developer with no clarifying questions.

🎯 Key Takeaways

What we covered today

  • ✅ Docs are a tax — write the minimum that's maximally useful
  • ✅ Write decisions, AC, requirements, sign-offs, glossary · link dashboards, code, designs, schedules
  • ✅ Docs cascade in a hierarchy: org → project → feature → sprint, with a running decision log
  • ✅ Mark every doc as frozen or living, with an owner and a "last reviewed" stamp
  • ✅ Docs decay from no owner, burial, filename versions, duplication — fix each one deliberately
  • ✅ agent-os = opinionated folders (product/, specs/, standards/) with named markdown files an AI can read in order
  • ✅ A spec.md with Overview · Stories · Scope · Out of Scope · Tech Notes · Dependencies is prompt-ready
  • ✅ Every project gets a CLAUDE.md at its root — short opinionated context the AI reads first

Your spec is your prompt. Write it like one.
