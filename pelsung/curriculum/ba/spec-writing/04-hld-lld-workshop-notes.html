Part 1

Why HLD and LLD exist

A spec says what the system does. A design says how it does it. Without a written design, two problems bite:

  1. Trade-offs get re-argued. If the design only lives in the architect's head, the next engineer rediscovers every constraint — and re-litigates every trade-off.
  2. Hand-offs break. A dev who didn't attend the design meeting builds something subtly different from what the architect intended. By code review it's too late to fix cheaply.

Two documents split the load:

HLD · High-Level Design

Scope: system · boxes · integrations

Audience: architects · tech leads · SRE · security

Question answered: "How are the parts arranged, and why?"

Length: 10–25 pages

LLD · Low-Level Design

Scope: one component · classes · APIs · schema

Audience: the devs who'll write the code

Question answered: "How is this component built inside?"

Length: 5–20 pages per component

The zoom-level rule

HLD is the whiteboard photo. LLD is the photo of the sketch on the whiteboard, with every line labelled. You can't LLD without first HLD-ing; you can't write code well without first LLD-ing. Skipping a level is tempting — and expensive later.

Part 3

A BA's role on HLD and LLD

A common question: "If I'm a BA, why am I reading design docs at all?" Three reasons:

  1. Traceability. Every FR and NFR in your spec must land somewhere in the HLD / LLD. If it doesn't, something is missing.
  2. Ambiguity detection. You wrote the spec. When the architect says "I interpreted FR-03 as …", you're the one who knows if that interpretation is right.
  3. Stakeholder translation. HLDs use language sponsors don't read. Someone has to explain "we're moving to an event-driven pattern for integration resilience" in business terms.

What a BA does on the HLD/LLD review

  • Checks traceability: every FR / NFR has a home in the design.
  • Challenges interpretations: "The spec says X; the HLD seems to assume Y — which is right?"
  • Surfaces impact on users: "This choice adds 200ms — does that breach NFR-Perf-01?"
  • Flags compliance: "The HLD puts data in region A, but BRD §11 constraint C-01 requires region B."
  • Does NOT: approve tech choices, design APIs, write algorithms. Those belong to engineering.
Part 4

HLD · the 16-section structure

A well-formed HLD has 16 sections. Not every one is equally weighty on every project, but all appear — reviewers expect to find each topic in roughly the same place.

§§ 01–03 · Setup

Metadata · Purpose & Scope · Context

Who wrote it · what system this covers · where this HLD sits relative to other HLDs · links to upstream PRD / SRS. Context diagram (C4 Level 1) showing the system as one box, with every external actor and system labelled.

§§ 04–05 · The driving forces

Quality attributes · Architectural style

Architecturally-significant NFRs — availability, latency, throughput, security, data residency — and the style chosen (modular monolith · microservices · event-driven · serverless), plus why. Every later decision traces back to these.

§§ 06–07 · The structure

Container / Component diagram (C4 L2) · Data architecture

The major deployable units — services, apps, datastores — as a diagram plus a component summary table. Data architecture: logical entities, data stores with consistency models, retention & residency policies.

§§ 08–10 · The cross-cutting mechanics

Integration · Security · Deployment

How the system talks to partners (protocols · idempotency · retry · SLAs) · the security architecture (AuthN · AuthZ · network · data protection · threat model summary) · deployment architecture (environments · regions · orchestration · release strategy).

§§ 11–13 · The operational stance

Observability · Reliability · Cross-cutting concerns

Metrics · logs · traces · alerts · dashboards — golden signals per service. Failure modes considered · DR strategy · backup. Cross-cutting: I18n · accessibility · perf budgets · cost model · compliance posture.

§§ 14–16 · The thinking

ADRs · Trade-offs · Open questions · Out of scope

Architectural Decision Records (one-line summary per ADR, link to full record). Trade-offs explicitly listed. Open questions with owners. Explicit out-of-scope boundaries.

Part 5

The C4 model · 4 levels of architecture diagrams

C4 is the most widely-adopted way to draw architecture diagrams. It proposes four levels, each zooming in further. HLDs use Levels 1 & 2; LLDs use Levels 3 & 4.

Level 1
🌍 Context

The system as one box, with users and external systems around it. "What does this system touch?"

Level 2
📦 Container

Deployable units — services, apps, DBs — inside the system. "What's the shape?"

Level 3
🧩 Component

Inside one container — components, modules. "How is one service structured?"

Level 4
🔬 Code

Class / function-level detail. Often rendered as a UML class diagram.

Context diagram (Level 1) — the HLD's opening picture:

┌─────────────┐ ┌──────────────┐ │ Customer │───────▶│ │───────▶ Payment Gateway │ (mBoB app) │ │ Biller Favs │ (external) └─────────────┘ │ Service │ │ │───────▶ Core Banking Ledger ┌─────────────┐ │ │ (internal) │ Admin Tool │◀───────│ │ │ (internal) │ └──────┬───────┘───────▶ Notification Bus └─────────────┘ │ (internal) ▼ [Favourites DB]

Container diagram (Level 2) — the HLD's structural picture:

┌──────────────┐ ┌─────────────┐ ┌─────────────┐ │ Mobile App │──▶│ API GW │──▶│ Auth Svc │ └──────────────┘ └─────┬───────┘ └─────────────┘ │ ▼ ┌─────────────┐ ┌──────────────┐ │ Favs Svc │──▶│ Favourites DB│ └─────┬───────┘ └──────────────┘ │ ▼ ┌─────────────┐ │ Event Bus │ └─────────────┘
Part 6

Quality attributes drive architecture choices

Every HLD has a table of architecturally-significant quality attributes (a.k.a. NFRs that change the shape of the system). Each is a target + the architectural implication.

AttributeTargetArchitectural implication
Availability99.95% monthlyMulti-AZ active-active · no single DB primary
Latencyp95 < 250 msRead replica · cache layer · connection pooling
Throughput5 000 TPS peakHorizontal scaling · queue-based back-pressure
SecurityPCI-DSS scopedTokenisation at edge · card data never in app DB
Data residencyBhutan onlyRegion-pinned deployment · no cross-border replication
Auditability7-year retentionAppend-only event log · immutable storage (S3 + object lock)

The architecture trace rule

Every box, arrow, and datastore in the HLD must trace back to at least one quality attribute. If you can't justify "why this?" with "because of target X", the element is a guess — not a decision.

Part 7

LLD · the 16-section structure

The LLD is the developer's blueprint for one component. Two engineers reading the same LLD should produce near-identical code.

§§ 01–03 · Setup

Metadata · Purpose · Responsibilities (and non-responsibilities)

Links to parent HLD. What this component does. What this component explicitly does NOT do (delegated to other components).

§§ 04–05 · The contract

Public interface (API / methods) · Data model (physical)

Endpoint signatures, request/response schemas, HTTP status-code contract. Full DDL for any schema changes — tables, indexes, constraints, migration up/down.

§§ 06–07 · Internals

Internal structure · Algorithms & business logic

Classes · modules · their relationships (C4 Level 3). Pseudocode or prose for each non-trivial algorithm. Concurrency reasoning. Transactional boundaries.

§§ 08–10 · Error & consistency discipline

Error taxonomy · Retry, idempotency, timeouts · Concurrency & transactions

Every error code · when it fires · which HTTP status · user-facing vs generic. Idempotency strategy for writes. Isolation level per transaction. Locking strategy (optimistic / pessimistic).

§§ 11–12 · Runtime behaviour

Caching · Observability hooks · Security considerations

What is cached, key, TTL, invalidation triggers. Metrics names, log fields, trace spans, alert thresholds. Per-method AuthZ rules, audit log, rate limits.

§§ 13–16 · Validation & release

Component-level test plan · Deployment · Open questions · Out of scope

Unit · integration · load tests this LLD implies. Feature flag strategy. Schema migration order. What this LLD doesn't cover (separate LLDs).

Part 9

Error taxonomy & concurrency — often-missed sections

Two sections first-time LLD authors tend to skip — and that their first production incident punishes them for.

§ 8 · Error taxonomy

Every error code · every HTTP status · user-facing or not

A table that covers every failure mode. When each fires. What the caller sees. Without this, errors drift — two endpoints return different shapes, a mobile client can't distinguish a real error from a network blip, support can't correlate error codes across logs.

ErrorWhenHTTPUser-facing?
VALIDATIONBad input (alias > 30 chars)400Yes
UNAUTHORIZEDMissing / invalid token401Yes
FORBIDDENValid token, no permission403Yes
NOT_FOUNDID doesn't exist or not owned404Yes
LIMIT_REACHEDUser already has 10409Yes
DUPLICATE(user, biller) already exists409Yes
DB_UNAVAILABLETransient DB failure503Generic
INTERNALUnexpected500Generic
§ 9 · Concurrency & transactions

Race conditions caught on paper, not in production

Example hazard: two simultaneous create calls when user has 9 favourites. Both pass the "count < 10" check, both insert — user ends up with 11. Mitigation: serializable transaction, or DB-level constraint, or app-level lock. Which one you pick belongs in the LLD.

  • Transaction boundary: which operations are one atomic unit
  • Isolation level: READ COMMITTED / REPEATABLE READ / SERIALIZABLE (justify the choice)
  • Locking: optimistic (fail-fast) or pessimistic (row lock)
  • Idempotency: how a client can safely retry
  • At-least-once vs exactly-once for side effects (events · emails · SMS)

HLD excerpt

Availability: 99.95% monthly → multi-AZ active-active Latency: p95 < 250 ms list/save → read replica + 60s Redis cache Throughput: 5 000 RPS peak (pay-day) → horizontal scaling Security: RMA §7 compliance → region-pinned (Bhutan only) Auditability: 7-year retention → append-only event log + S3 lock

§ 6 · Container Diagram

API Gateway → Auth Service → Favourites Service → Favourites DB (Postgres). Event Bus (Kafka) receives all writes for audit.

§ 14 · ADR summary

  • ADR-001 · Modular monolith (not microservices) for MVP — reduces ops cost; revisit at 2× current load.
  • ADR-002 · Serializable transactions over CQRS — simpler; latency targets still met.
  • ADR-003 · Postgres for ledger + Redis for cache. Rejected: Cassandra (overkill), MongoDB (consistency concerns).
LLD excerpt (Favourites Service)

§ 6 · Internal structure

FavouritesController (HTTP adapter) │ ▼ FavouritesService (business rules, limit enforcement) │ ▼ FavouritesRepository (DB access, SQL) │ ▼ [ favourites table ]

§ 7 · create() — flow

function create(userId, billerId, alias): validateAlias(alias) // trim · len ≤ 30 · else ValidationError BEGIN TRANSACTION (SERIALIZABLE) existing = repo.countByUser(userId) if existing >= 10: ROLLBACK; return Err(LIMIT_REACHED) duplicate = repo.findByUserAndBiller(userId, billerId) if duplicate: ROLLBACK; return Err(DUPLICATE) row = repo.insert({ userId, billerId, alias, createdAt: now() }) COMMIT emit("favourite.created", row) // via outbox table + relay return Ok(row)

§ 9 · Concurrency

"Two concurrent create calls when count = 9 could both pass the limit check under READ COMMITTED. Mitigation: SERIALIZABLE isolation + retry on serialisation anomaly (client sees 409 DUPLICATE or LIMIT_REACHED depending on which commits first)."

Part 11

How a BA reviews an HLD / LLD

A BA is not an architect, but a BA's review adds specific value:

Review checklist

What to tick off when reading an HLD

  • □ Every FR in the upstream spec has a home — which component owns it?
  • □ Every NFR has an architectural implication listed — or a rationale for why it doesn't.
  • □ The context diagram matches the external systems I wrote in the spec.
  • □ Constraints from the BRD / spec (regulation, region, budget) appear in §4 / §9 / §13.
  • □ Out-of-scope in the HLD matches out-of-scope in the spec.
  • □ Open questions have owners and deadlines.
  • □ ADRs exist for every non-obvious tech choice.
Review checklist

What to tick off when reading an LLD

  • □ The API contract matches the behaviour in the AC.
  • □ Error taxonomy covers every failure mode the AC implies (limit, dup, offline, validation).
  • □ Data model respects constraints from the spec (max 10, unique per user).
  • □ Concurrency reasoning exists for any race condition I can imagine.
  • □ Audit logging covers everything the spec / compliance req demands.
  • □ Feature flag / rollback plan exists and is sensible.

Takeaways

Key points to carry forward

What we covered today

  • HLD = system structure · LLD = one component's internals. HLD is the whiteboard photo; LLD is the labelled sketch.
  • Design spectrum: BRD → Spec → HLD → LLD → Code. Each zooms in; each has a different author + reviewer.
  • A BA reads HLD / LLD to check traceability, detect ambiguity, translate to stakeholders, flag compliance — not to approve tech choices.
  • HLD & LLD each have 16 standard sections. Every project writes them; not every project weights them equally.
  • C4 model · Level 1 Context · Level 2 Container (HLD); Level 3 Component · Level 4 Code (LLD).
  • Every architectural element traces back to a quality attribute. If not, it's a guess.
  • LLD sections first-time authors skip: error taxonomy and concurrency. Both punish you in prod.
  • Design written after code is a post-hoc essay, not a design. Write HLD before code · LLD before component.

The best LLD reads like a textbook for the feature. Two engineers, given the same LLD, produce near-identical code.
