03 Hook
🎤 One question

What do a bank, a hospital, and an aircraft have in common?

Each one ships software that must not be wrong.
Each one is audited by regulators who cannot read your code.
Each one needs a document that is complete, unambiguous, traceable, signable.

That document is an SRS.

04 What is an SRS (and isn't)?

The contract between customer and builder

✅ An SRS is…
  • Complete — every requirement appears
  • Unambiguous — one interpretation only
  • Verifiable — each req has a test method
  • Traceable — links up (business need) and down (design + test)
  • Consistent — no contradictions
  • Ranked — priority + stability noted
❌ An SRS is NOT…
  • A design document (HLD/LLD come later)
  • A project plan
  • A user manual
  • Sales / marketing material
  • A sketch — this is 30–100+ pages
  • Optional, where regulation demands it

In regulated industries, an SRS literally is the contract. Signed, dated, versioned, legally binding.

05 When to use an SRS · the 6 signals

Reach for an SRS when one of these is true

🏛️ Regulated industry

Auditors or regulators require structured requirements.

Banking · e-health · aviation · defence
📄 Government / enterprise RFP

Procurement rules demand a standards-compliant spec.

National ID · tax portal · ministry ERP
🤝 Formal contract

Scope disputes get resolved by reading the SRS.

Fixed-price delivery to a ministry or bank
⚠️ Safety-critical

Failure costs lives, money, or reputation at scale.

Hospital monitoring · flight control · settlement
🗓️ Long lifecycle (10+ yrs)

Original team won't be around; SRS is institutional memory.

National infrastructure · telecom backbone
🔗 Cross-organisation

Multiple parties share one immutable reference.

Inter-bank settlement · cross-ministry data

If none of these apply — use a PRD or Feature Spec instead. SRSs are heavy; pay the cost only when the stakes demand it.

06 When an SRS would hurt you

Five warning signs you picked the wrong tool

🔄 Still discovering the product

You'll "baseline" reqs that change 2 sprints later. Every change becomes a 3-week review cycle.

🧑‍🤝‍🧑 Small team building iteratively

SRS overhead exceeds implementation effort. Nothing ships.

🚀 Shipping weekly or faster

The SRS can't keep up. It drifts from reality and becomes fiction.

🏢 Internal customer, same room

A shared Notion page + AC gives 80% of value for 10% of the cost.

📉 No change-control discipline

An unmaintained SRS is worse than no SRS. People treat a lie as truth.

💡 The SELISE reality

Most SELISE work uses PRDs + feature specs. We teach SRS because some engagements (core banking, govt) need it — and because learning SRS makes every smaller doc easier.

09 §1 Introduction · setting the stage

The orientation section · two pages of context

§ 1.1–1.5

Five short subsections

  • 1.1 Purpose — why this SRS exists · who should read it
  • 1.2 Scope — what the product will do · AND what it will NOT do (equally important)
  • 1.3 Definitions — every domain term a non-specialist would Google
  • 1.4 References — numbered list of every document cited (standards · regulations · interfacing SRSs)
  • 1.5 Document overview — navigation map for the rest of the doc
1.2 Scope WILL: Allow retail mBoB customers to save up to 10 recurring billers as favourites, retrieve them from the Bill Payments home, and pre-fill a new payment. MPIN re-authentication still required. WILL NOT: Allow favourite sharing between customers · support auto-pay · support bulk import/export · support legacy mBoB v4 devices.

Requirements

13 §3.2 Functional Requirements · the structured block

Numbered · atomic · testable · traceable · prioritised

FR-01 · Save a biller as a favourite Description: Customer, having completed a payment, shall save the paid biller as a favourite with a single tap. Inputs: Tap event · biller_id · alias default = biller name Processing: Enforce max 10 per user · enforce uniqueness (user, biller) Outputs: Toast within 1 s · favourite on next home render Preconditions: User authenticated · payment successful · online Postconditions: Favourite persisted · audit event emitted · cache invalid Error handling: Limit reached → "Limit reached (10)" · Duplicate → "Already saved" Offline → queue locally with retry Priority: Must Traces to: BR-02 (Customer retention through convenience) Verifies via: TC-FR01-01 (happy), TC-FR01-02 (limit), TC-FR01-03 (dup)

This is what makes an SRS long — and what makes it useful during audit.

14 §3.3 Non-Functional Requirements · five sub-types

All testable in numbers

⚡ 3.3.1 Performance

Throughput · response time · capacity · p95/p99 targets.

🛡️ 3.3.2 Safety

Prevents loss, damage, harm. References IEC 61508, ISO 26262, IEC 62304.

🔒 3.3.3 Security

AuthN · authZ · encryption · audit · compliance (PCI, HIPAA, GDPR, SOC 2).

⭐ 3.3.4 Quality attributes

Reliability · availability · maintainability · portability · usability · accessibility.

📜 3.3.5 Business Rules

"Only a manager can approve transfers over BTN 100k." Not features — constraints.

🌐 3.5 Other

Database · I18n · compliance · anything that doesn't fit above.

15 NFRs · adjectives vs numbers

The #1 reason SRSs fail audit

❌ Rejected by auditors
  • "The system shall be reliable."
  • "The UI shall be user-friendly."
  • "The system shall have high performance."
  • "Security shall be strong."
  • "Response time shall be acceptable."
✅ Accepted by auditors
  • "MTBF ≥ 720 hrs over a 90-day rolling window."
  • "Novice customer completes task T-01 in ≤ 60 s with ≤ 1 error (n=30)."
  • "p95 ≤ 1.0 s for reads at 5 000 RPS."
  • "AES-256 at rest · TLS 1.3+ in transit."
  • "95% within §3.3.1 envelope · measured hourly."

The NFR test: can a tester set up an experiment that yields yes/no? If yes → requirement. If "yes but how would we check?" → aspiration, not yet a requirement.

17 Four verification methods · T · D · I · A

Every req links to at least one

T
Test

Execute the software and check outputs against expected values.

D
Demonstration

Observe the software operating under representative conditions.

I
Inspection

Review the product / design / code without executing it.

A
Analysis

Use models, simulations, or calculations rather than execution.

🎯 Key Takeaways

What we covered today

  • ✅ SRS = formal, complete, auditable specification — IEEE 29148:2018
  • ✅ Use when: regulated · RFP · contract · safety-critical · long lifecycle · cross-org
  • ✅ Skeleton: §1 Intro · §2 Overall · §3 Specific · §4 Verification + appendices
  • ✅ §2.1 context diagram · §2.3 user classes · §2.4 constraints · §2.5 assumptions
  • ✅ §3 reqs: numbered · atomic · testable · traceable · prioritised
  • ✅ NFRs use numbers not adjectives — or the auditor rejects them
  • ✅ §4 Verification: every req → T / D / I / A method + test artifact
  • ✅ Baseline → Change-controlled → Signed-off; CCB keeps it honest
  • ✅ Next class: BRD — pre-solution business requirements

An SRS isn't longer because it's better. It's longer because the stakes require it.
