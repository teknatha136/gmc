06 What is the DOM?
Concept

Document Object Model

The browser reads your HTML and turns it into a tree of objects that JavaScript can touch.

index.html
<html> <body> <h1>Gelephu</h1> <p>Hello</p> </body> </html>
html
└── body
├── h1
│   └── "Gelephu"
└── p
    └── "Hello"

The DOM is the live version. Change a node → the page updates instantly.

07 document — your entry point
The entry point

document

Every page gives JavaScript one global variable: document. That's the root of the tree.

console
// Try it in the browser console right now console.log(document.title); // page title console.log(document.body); // <body> element console.log(document.URL); // current URL
08 Selecting elements
DOM · Read

How to grab any element

document.querySelector("h1")
First element matching the CSS selector. Returns the element or null.
document.querySelectorAll(".item")
All matching elements. Returns a NodeList (array-like).
document.getElementById("name")
Older shortcut for single-id lookup. Still fast and common.
element.querySelector(…)
Same methods work on any element — searches inside it.

If you know CSS selectors, you know DOM selection.

09 Reading & changing content
DOM · Read & write

Text, attributes, classes

app.js
const title = document.querySelector("h1"); // Read console.log(title.textContent); // "Gelephu" // Change the text title.textContent = "Gelephu Mindfulness City"; // Change an attribute const img = document.querySelector("img"); img.src = "gmc-logo.png"; img.alt = "GMC logo"; // Change CSS classes title.classList.add("highlight"); title.classList.remove("muted"); title.classList.toggle("active");
10 Creating & removing elements
DOM · Build

Grow the tree from JavaScript

app.js
// 1. Create a new element (not yet on page) const item = document.createElement("li"); item.textContent = "Tek Nath"; // 2. Add it to the page const list = document.querySelector("#trainees"); list.appendChild(item); // 3. Remove an element item.remove(); // Shortcut — append HTML or text at the end list.append("New entry");

create → set → append. Three steps, always.

11 Event listeners
DOM · React to users

How the page listens

Analogy

An event listener is like asking the element: "Please tap me on the shoulder when someone clicks you — and when you do, run this function."

app.js
const button = document.querySelector("#save"); button.addEventListener("click", () => { console.log("Button clicked!"); }); // Common events: click, input, submit, keydown, change

    15 Common DOM gotchas
    Watch out

    Three mistakes beginners make

    • Running JS before the page exists. Put your <script> at the bottom of <body>, or use defer.
    • Confusing innerHTML with textContent. innerHTML parses HTML — dangerous with user input (XSS).
    • Forgetting that querySelectorAll is not a real array. Use .forEach on it, or spread it: [...nodes].

    18 What's a browser API?
    Concept

    Tools the browser hands to JavaScript

    Think of it like this

    The browser is like a Swiss Army knife. Each blade is an "API" — storage, network, location, clipboard, audio, camera…

    19 localStorage & sessionStorage

    Remembering the user

    Programming

    How JavaScript waits without freezing

    03
    28 Why async exists
    The core problem

    JavaScript has only one counter

    Shop analogy

    Imagine a shop with one cashier. A customer asks for something from the warehouse that takes 3 minutes to fetch.

    Option A — the cashier waits in silence for 3 minutes. The whole queue freezes. 😤
    Option B — the cashier takes the next customer and comes back to the first when the item arrives. ✨

    JavaScript is Option B. That's async.

    29 Blocking vs non-blocking
    What breaks pages

    Never freeze the UI

    Slow work that "blocks" means the browser can't redraw, scroll, or respond to clicks. Async code lets JS say "I'll get back to this later" and keep the page alive.

    Blocking (bad)
    A 3-second loop runs — clicks, scrolls, animations all freeze until it finishes.
    Non-blocking (good)
    Network requests, timers, fetch — fire and continue. The page stays responsive.
    30 Callbacks — the original way
    Pattern 1

    "Call me back when you're done"

    callback.js
    // "Wait 2 seconds, THEN run this function." setTimeout(() => { console.log("Done!"); }, 2000); console.log("This prints FIRST"); // Order: "This prints FIRST" → (2s) → "Done!"

    You pass a function as an argument. JS says "OK, I'll run this later — for now, keep going."

    31 Callback hell
    Why we moved on

    Nesting gets ugly fast

    pyramid-of-doom.js
    getUser(id, (user) => { getPosts(user.id, (posts) => { getComments(posts[0].id, (comments) => { getAuthor(comments[0].id, (author) => { console.log(author); // 😵 }); }); }); });

    → This is why Promises were invented.

    32 Promises
    Pattern 2

    "An IOU for a future value"

    Real-world analogy

    You order bubble tea. The cashier hands you a receipt with a number — a promise that a drink is coming. You're free to sit down. When ready, you get the drink (fulfilled) or "sorry, out of stock" (rejected).

    ⏳ Pending ✓ Fulfilled ✕ Rejected
    33 .then / .catch / .finally
    Handling the result

    Three handlers

    promise.js
    fetch("https://api.example.com/user") .then((response) => response.json()) .then((data) => { console.log(data); // ✓ success }) .catch((err) => { console.error(err); // ✕ failure }) .finally(() => { console.log("done"); // always runs });

    Chained, not nested — much flatter than callbacks.

    34 async / await
    Pattern 3 · Modern

    Async code that reads like sync code

    async.js
    async function loadUser() { const response = await fetch("/api/user"); const data = await response.json(); console.log(data); } loadUser();
    async
    Marks a function as asynchronous. It always returns a Promise.
    await
    Pauses inside an async function until the Promise settles.
    35 Error handling with try/catch
    When things fail

    Always wrap await in try/catch

    safe-fetch.js
    async function loadUser() { try { const response = await fetch("/api/user"); if (!response.ok) throw new Error("Bad status"); const data = await response.json(); return data; } catch (err) { console.error("Failed to load:", err); return null; } }
    36 fetch — Bhutan weather
    Live demo

    Fetch real data & render to DOM

    weather.js
    async function loadGelephuWeather() { const url = "https://api.open-meteo.com/v1/forecast?latitude=26.86&longitude=90.48&current_weather=true"; try { const res = await fetch(url); const data = await res.json(); const temp = data.current_weather.temperature; document.querySelector("#temp").textContent = `Gelephu is ${temp}°C`; } catch (err) { document.querySelector("#temp").textContent = "Could not load weather"; } }
    37 Gotcha 1 · forgotten await
    Common trap

    Forgetting await

    bug.js
    const data = fetch("/api/x"); console.log(data); // Promise { <pending> }
    fix.js
    const data = await fetch("/api/x"); console.log(data); // Response { … }

    Without await, you get the Promise itself, not the value inside it.

    38 Gotcha 2 · unhandled rejections
    Common trap

    No catch = silent crash

    A Promise that rejects with no .catch or try/catch becomes an "unhandled promise rejection". Users see nothing — you see a confusing warning in the console.

    rule.js
    // Rule of thumb: every await → wrapped in try/catch // every .then chain → ended with .catch
    39 Gotcha 3 · sequential vs parallel
    Performance trap

    Promise.all for parallel work

    slow · 2s + 2s = 4s
    const a = await fetch(url1); const b = await fetch(url2);
    fast · max(2s, 2s) = 2s
    const [a, b] = await Promise.all([ fetch(url1), fetch(url2) ]);

    If two requests don't depend on each other — fire both at once.
