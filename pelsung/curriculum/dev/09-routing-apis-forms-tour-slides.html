07 Why we need routing · 9:47
The problem

React apps are single-page

One HTML file. The server gives you JS. JS draws everything.

😩 Without routing

  • One giant component with useState('list' | 'detail' | 'new')
  • Clicking the back button reloads the page
  • You can't share a link to a specific note
  • Everything renders on one URL

😎 With routing

  • URL is the source of truth for what's on screen
  • Back / forward / refresh just work
  • Links are shareable
  • Each "page" is its own component

09 The 3 building blocks · 9:53
BrowserRouter · Routes · Route

Wrap your app, list your routes

main.tsx
import { BrowserRouter, Routes, Route } from "react-router-dom"; import NotesList from "./pages/notes-list"; import NoteDetail from "./pages/note-detail"; import NoteNew from "./pages/note-new"; createRoot(document.getElementById("root")!).render( <BrowserRouter> <Routes> <Route path="/notes" element={<NotesList />} /> <Route path="/notes/new" element={<NoteNew />} /> <Route path="/notes/:id" element={<NoteDetail />} /> </Routes> </BrowserRouter> );

12 Dynamic routes · 10:08
The :id in the path

URLs can carry data — useParams()

pages/note-detail.tsx
import { useParams } from "react-router-dom"; export default function NoteDetail() { const { id } = useParams(); // id is "42" if URL is /notes/42 return <div>You're looking at note #{id}</div>; }

The :id in the route path becomes a key on the params object. This is what we'll feed into the API call in the next block.

13 Layouts with <Outlet/> · 10:14
Shared shell, different content

A header on every page, written once

layout.tsx
import { Outlet, NavLink } from "react-router-dom"; export default function Layout() { return ( <div> <header> <NavLink to="/notes">Notes</NavLink> </header> <main><Outlet /></main> </div> ); }
main.tsx
<Routes> <Route element={<Layout />}> <Route path="/notes" element={<NotesList/>}/> <Route path="/notes/:id" element={<NoteDetail/>}/> <Route path="/notes/new" element={<NoteNew/>}/> </Route> </Routes>

<Outlet/> is where the matching child route renders. Header is written once, used everywhere.

14 useNavigate() · 10:21
Redirects from code

Programmatic navigation

Link handles clicks. But after a successful form submit, you want to redirect from JavaScript. That's useNavigate().

pages/note-new.tsx
import { useNavigate } from "react-router-dom"; const navigate = useNavigate(); async function handleSubmit(data) { const res = await fetch("/api/notes", { ... }); const created = await res.json(); navigate(`/notes/${created.id}`); // ✓ user lands on the new note's page }

20 The "fire-and-forget" trap · 10:55
Common bug

What's wrong with this code?

notes-list.tsx · 🚩

  • Empty page for the first 200ms — looks broken

21 The 3-state pattern · 11:00
The mental model

Every fetch has 3 states

Render all three — never just the happy path.

⏳
Loading
Show a spinner or skeleton while we wait
⚠️
Error
Network failed or server returned a bad code
✅
Data
Render the actual list / detail / whatever

Three useStates, three early returns. This is 80% of working with APIs in React.

22 The pattern in code · 11:05
notes-list.tsx · ✓
function NotesList() { const [notes, setNotes] = useState<Note[]>([]); const [loading, setLoading] = useState(true); const [error, setError] = useState<string | null>(null); useEffect(() => { fetch("/api/notes") .then(r => { if (!r.ok) throw new Error("Failed"); return r.json(); }) .then(setNotes) .catch(e => setError(e.message)) .finally(() => setLoading(false)); }, []); if (loading) return <p>Loading…</p>; if (error) return <p className="text-red-600">{error}</p>; return <ul>{notes.map(n => <li key={n.id}>{n.title}</li>)}</ul>; }

Three states · three returns · one useEffect. Memorise the shape, not the syntax.

23 HTTP verbs beyond GET · 11:14
The four you'll actually use

REST verbs — same fetch, different method

MethodWhat it meansUsed forBody?
GETReadLoad notes list, load one noteNo
POSTCreateSubmit a new noteYes (JSON)
PUTUpdateEdit an existing noteYes (JSON)
DELETERemoveDelete a noteNo

fetch defaults to GET. For everything else, you pass an options object.

24 POST in fetch · 11:20
Sending data

Three things to remember

create-note.ts
const res = await fetch("/api/notes", { method: "POST", // 1. set the verb headers: { "Content-Type": "application/json" }, // 2. tell the server it's JSON body: JSON.stringify({ // 3. body must be a STRING title: "Class 09 notes", body: "Routing, APIs, forms…", }), }); const created = await res.json(); // → { id: 101, title: "Class 09 notes", body: "..." }

PUT works the same way (different URL like /api/notes/42). DELETE usually has no body.

25 Auth · 11:28
Telling the server who you are

Authorization: Bearer <token>

Most modern APIs hand you a token after login. You attach it to every request as a header.

authed-fetch.ts
const token = localStorage.getItem("auth_token"); const res = await fetch("/api/notes", { headers: { "Content-Type": "application/json", "Authorization": `Bearer ${token}`, // ← the magic string }, }); if (res.status === 401) { // token is missing or expired → send to /login }
26 Where API keys live · 11:34
.env files exist for a reason

Never hard-code secrets in your code

❌ Wrong
api.ts
const API_KEY = "sk_live_abc123_secret"; // 💥 anyone reading the code sees it // 💥 it ends up on GitHub // 💥 you can't change it per environment
✓ Right
api.ts
// .env.local (NOT committed to git) // VITE_API_KEY=sk_live_abc123 const key = import.meta.env.VITE_API_KEY;
⚠️
Frontend env vars are still visible in the browser. Anything in the bundle can be read by users. Only put public keys in frontend env. Real secrets stay on the server.
27 The fetch trap · 11:40
The most surprising fact about fetch

fetch() does NOT throw on 404 or 500

😱 What you expect

Server returns 500. fetch throws. Your catch handles it. ❌ It doesn't.

✓ What actually happens

fetch only throws on network errors (no DNS, no connection). For 4xx/5xx, the promise resolves — you have to check res.ok yourself.

safe-fetch.ts
const res = await fetch("/api/notes"); if (!res.ok) { throw new Error(`API error: ${res.status}`); } const data = await res.json();
28 Custom hook: useFetch · 11:46
Stop repeating yourself

Wrap the pattern in one hook

hooks/use-fetch.ts
export function useFetch<T>(url: string) { const [data, setData] = useState<T | null>(null); const [loading, setLoading] = useState(true); const [error, setError] = useState<string | null>(null); useEffect(() => { fetch(url) .then(r => { if (!r.ok) throw new Error(r.statusText); return r.json(); }) .then(setData) .catch(e => setError(e.message)) .finally(() => setLoading(false)); }, [url]); return { data, loading, error }; } // In a component: const { data, loading, error } = useFetch<Note[]>("/api/notes");
29 A peek at production · 11:54
What real teams use

TanStack Query (a.k.a. React Query)

useFetch works, but real apps need caching, refetching on focus, retries, optimistic updates. Don't build all that — use a library.

notes-list.tsx · with TanStack Query
import { useQuery } from "@tanstack/react-query"; function NotesList() { const { data, isLoading, error } = useQuery({ queryKey: ["notes"], queryFn: () => fetch("/api/notes").then(r => r.json()), }); // ✓ caches automatically // ✓ refetches when tab regains focus // ✓ shares the same data across components // ✓ retries on error }

Today: see it. Later: learn it. When your useFetch isn't enough, this is what to reach for.

34 Why hand-rolled forms hurt · 12:18
The boilerplate problem

Imagine a form with 5 fields…

register-form.tsx · 🚩
const [name, setName] = useState(""); const [email, setEmail] = useState(""); const [password, setPassword] = useState(""); const [phone, setPhone] = useState(""); const [cid, setCid] = useState(""); const [errors, setErrors] = useState({}); const [touched, setTouched] = useState({}); // + a validate() function with 5 if-statements // + onBlur handlers for "touched" // + a submit handler that re-validates everything // + reset logic

Multiply by 10 forms in your app. This is why libraries exist.

35 react-hook-form · 12:24
The form library most teams use

3 things: useForm, register, handleSubmit

note-new.tsx · with react-hook-form
import { useForm } from "react-hook-form"; function NoteNew() { const { register, handleSubmit } = useForm(); const onSubmit = (data) => console.log(data); return ( <form onSubmit={handleSubmit(onSubmit)}> <input {...register("title")} placeholder="Title" /> <textarea {...register("body")} /> <button type="submit">Save</button> </form> ); }

No useState. No onChange. The library wires the inputs for you.

36 Errors with RHF · 12:32
The fourth thing: formState.errors

Validation rules at registration

with-errors.tsx
const { register, handleSubmit, formState: { errors }, } = useForm(); <input {...register("title", { required: "Title is required", minLength: { value: 3, message: "Min 3 characters" }, })} /> {errors.title && <p className="text-red-600">{errors.title.message}</p>}

This works for simple cases. But your validation rules end up scattered across every input. Enter zod.

37 zod · 12:42
Schema-first validation

One schema · all your rules in one place

zod describes the shape of valid data. The schema is the single source of truth for validation.

schemas/note.ts
import { z } from "zod"; export const noteSchema = z.object({ title: z.string().min(3, "Min 3 characters").max(100), body: z.string().min(10, "Body is too short"), tags: z.array(z.string()).optional(), }); export type Note = z.infer<typeof noteSchema>; // ↑ Note is { title: string; body: string; tags?: string[] } // — one schema gives you both validation AND types
38 zod + react-hook-form · 12:50
The bridge

@hookform/resolvers/zod

One line connects them.

note-new.tsx · the real version
import { useForm } from "react-hook-form"; import { zodResolver } from "@hookform/resolvers/zod"; import { noteSchema, Note } from "./schemas/note"; const { register, handleSubmit, formState: { errors, isSubmitting }, } = useForm<Note>({ resolver: zodResolver(noteSchema), }); <input {...register("title")} /> {errors.title && <p>{errors.title.message}</p>}

Now your form gets type safety + validation + error messages, all from one schema.
