04 The DOM way — imperative
Recap from Class 04

Step by step, you tell the browser what to do

counter.js — the manual way
let count = 0; const btn = document.querySelector("#btn"); const label = document.querySelector("#label"); btn.addEventListener("click", () => { count++; label.textContent = `Count: ${count}`; // you update the DOM yourself });

For a counter, fine. For a screen with 30 things that change, this becomes a nightmare.

05 The React way — declarative
React

You describe the screen, React updates it

counter.tsx — the React way
function Counter() { const [count, setCount] = useState(0); return ( <button onClick={() => setCount(count + 1)}> Count: {count} </button> ); }

No querySelector. No textContent. You change count, the screen updates.

06 Imperative vs Declarative

Same outcome, different mindset

AspectVanilla DOMReact
Mindset"Click → find element → change it""What should the screen look like for this state?"
UpdatesYou write every textContent changeReact diffs the old & new, updates only what changed
ReusabilityCopy-paste blocks of HTML + JSDefine a component once, use it anywhere
ScalingTangled at ~10 featuresStays manageable across hundreds

React isn't faster magic — it's a better mental model.

09 What is a component?

A component is a function that returns UI

Mental model
props go in  →  the component runs  →  JSX comes out  →  React turns it into DOM.
welcome.tsx
function Welcome() { return <h1>Hello, Pelsung!</h1>; }

That's it. Capital-letter name + returns JSX = component. Functional components only — class components are legacy, you won't write them.

10 Using a component

Once defined, use it like an HTML tag

Define

function Welcome() { return <h1>Hello!</h1>; }

Use

function App() { return ( <div> <Welcome /> <Welcome /> </div> ); }

One definition, many uses. That's the whole point.

11 JSX — looks like HTML, is JavaScript
JSX

It's JavaScript wearing an HTML costume

JSX vs. plain JS
// What you write const el = <h1 className="title">Hello</h1>; // What it compiles to (Vite handles this) const el = React.createElement("h1", { className: "title" }, "Hello");

You write the top. The bundler converts it to the bottom. You'll never have to write the bottom yourself.

12 JSX rules to remember

Five small differences from HTML

className, not class
class is reserved in JS. Use className="card".
camelCase events
onclick → onClick, onchange → onChange.
Self-close empty tags
<img />, <input />, <br /> — always with the slash.
One parent element
Wrap siblings in <div> or a Fragment <>…</>.
{ } for JS expressions
Anything in curly braces is JS — variables, calls, math.
style is an object
style={{ color: "red" }} — double braces, camelCase.
13 Embedding JS in JSX
{ } is your friend

Curly braces let any expression in

app.tsx
function App() { const name = "Karma"; const isOnline = true; const topics = ["JSX", "Props", "State"]; return ( <div> <h1>Hello, {name}!</h1> <p>{isOnline ? "🟢 Online" : "🔴 Offline"}</p> <ul> {topics.map(t => <li key={t}>{t}</li>)} </ul> </div> ); }

Variables, ternaries, .map() — all just JS in { }.

Props

19 What are props?

Props are arguments to a component

Same idea you already know
A function takes arguments  →  a component takes props.
greet("Karma")  ⟷  <Greet name="Karma" />

Props are read-only. A component never changes its own props — that's what state is for (after the break).

20 Passing props

Pass them like HTML attributes

greet.tsx
// 1. The component receives props as its argument function Greet(props) { return <h1>Hello, {props.name}!</h1>; } // 2. The parent passes data when using the component <Greet name="Karma" /> <Greet name="Pema" /> <Greet name="Tashi" />

Three uses, three different greetings. One component definition.

21 Typing props with TypeScript
From Class 05 — type, not interface

Add a Props type

greet.tsx — typed
type GreetProps = { name: string; role?: string; // optional }; function Greet({ name, role }: GreetProps) { return ( <div> <h1>Hello, {name}!</h1> {role && <p>You are a {role}.</p>} </div> ); }

Now the editor catches <Greet name={42} /> instantly. Optional props use ?.

22 Composition — same component, many uses

Render a list from data

app.tsx
const trainees = [ { name: "Karma", role: "Dev" }, { name: "Pema", role: "BA" }, { name: "Tashi", role: "QA" }, ]; function App() { return ( <div> {trainees.map(t => ( <Greet key={t.name} name={t.name} role={t.role} /> ))} </div> ); }

key helps React track items in a list. We'll cover it deeply in Class 07. For now: use a unique value.

23 Default values for props

Use destructuring defaults

type ButtonProps = { label: string; variant?: "primary" | "secondary"; }; function Button({ label, variant = "primary" }: ButtonProps) { return <button className={variant}>{label}</button>; } <Button label="Save" /> // uses primary <Button label="Cancel" variant="secondary" />
24 The children prop
Special prop

Whatever's between your tags

card.tsx
type CardProps = { children: React.ReactNode }; function Card({ children }: CardProps) { return <div className="card">{children}</div>; } <Card> <h2>Hello</h2> <p>This whole block is the children prop.</p> </Card>

Great for wrappers — modals, panels, layout boxes.

25 Common props gotchas

Three traps to avoid

Strings vs expressions
count="3" → string "3".
count={3} → number 3.
Don't mutate props
props.name = "X" is a bug. Props flow in only.
Forgot to pass it
Required prop missing → TS error. Optional prop missing → it's undefined.
key is not a prop you read
key is for React's diffing. You can't access it inside the component.

State

Data that changes over time — and tells React to redraw.

04
29 Why state exists

Props are read-only — but pages aren't

The problem
A counter clicks. A form types. A toggle flips. Something changes, and the screen has to follow.

Plain variables won't trigger a redraw. useState does.
30 useState — the basics
Your first hook

A value + a way to change it

counter.tsx
import { useState } from "react"; function Counter() { const [count, setCount] = useState(0); // ↑ value ↑ setter ↑ initial value return ( <button onClick={() => setCount(count + 1)}> Clicked {count} times </button> ); }

useState(initial) returns a pair — the current value, and a function to set it.

31 The re-render mental model

setState → component runs again → screen updates

setCount(1)
you call the setter
→
Counter() runs
function called again
→
new JSX
with count = 1
→
React diffs
updates only what changed

Don't mutate state directly (count++ won't work). Always go through the setter.

32 State — three rules to remember

Hold these three in your head

Use the setter
setCount(count + 1) — never count++ or count = count + 1.
Setting → re-render
Calling the setter schedules React to run your component again with the new value.
Each component, its own state
Two <Counter />s on the page count independently. State is per-instance.
