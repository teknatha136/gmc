Block 1 · 50 minutes

🧭 Routing & Navigation

Multiple URLs in a single-page app. The library: react-router-dom.

🤔 Why routing exists

A React app is a single-page application (SPA). The server hands the browser one HTML file and a bundle of JavaScript. JS then draws everything on screen. There's no "second page" the way there was in old PHP / multi-page sites.

So how do we get multiple "pages" with different URLs — a list page, a detail page, a settings page? We fake it. JavaScript watches the URL and swaps which component is rendered. That's all routing is.

😩 Without routing

  • One giant component with useState('list' | 'detail' | 'new')
  • Browser back / forward buttons reload the whole app
  • You can't share a link to a specific note
  • Refreshing the page sends you back to the start

😎 With routing

  • The URL is the source of truth for what's on screen
  • Back / forward / refresh just work
  • Links are shareable — /notes/42 opens note 42
  • Each "page" is its own component file

📦 Install & the 3 building blocks

Terminal
npm install react-router-dom

Three things from the library, used together:

🌍 BrowserRouter

  • Wraps your whole app once
  • Watches the URL bar
  • Goes near the root, in main.tsx

🔀 Routes

  • Like a giant switch statement
  • Picks the first matching <Route>

🎯 Route

  • One path → one component
  • path="/notes/:id" with the colon for params
main.tsx
import { BrowserRouter, Routes, Route } from "react-router-dom"; import NotesList from "./pages/notes-list"; import NoteDetail from "./pages/note-detail"; import NoteNew from "./pages/note-new"; createRoot(document.getElementById("root")!).render( <BrowserRouter> <Routes> <Route path="/notes" element={<NotesList />} /> <Route path="/notes/new" element={<NoteNew />} /> <Route path="/notes/:id" element={<NoteDetail />} /> </Routes> </BrowserRouter> );
⚠️
Order matters. List /notes/new before /notes/:id, otherwise React Router thinks "new" is an id. (Modern versions are smarter, but it's still a habit worth keeping.)

🔗 <Link>, never <a>

The browser's native <a href> reloads the page. That kills your SPA — React unmounts everything, the bundle reloads, your state is gone. Always use <Link> from react-router-dom inside your app.

links.tsx
import { Link } from "react-router-dom"; // ❌ Wrong — full page reload <a href="/notes/42">Open</a> // ✓ Right — SPA navigation <Link to="/notes/42">Open</Link>

For navigation menus where you want to highlight the current page, use NavLink:

nav-bar.tsx
import { NavLink } from "react-router-dom"; <NavLink to="/notes" className={({ isActive }) => isActive ? "text-blue-600 font-bold" : "text-gray-600" } >All Notes</NavLink>
💡
You can still use <a href> for links to other websites (https://...). Just not for routes inside your own app.

🔢 Dynamic routes & useParams()

Real apps have URLs like /notes/42, /users/bhuwan, /products/laptop-x1. The variable bit (:id) is called a route parameter. You read it with the useParams() hook.

pages/note-detail.tsx
import { useParams } from "react-router-dom"; export default function NoteDetail() { const { id } = useParams(); // If URL is /notes/42, then id === "42" (always a string) return <div>You're looking at note #{id}</div>; }
⚠️
Params are always strings. Even if your URL says /notes/42, id is the string "42", not the number. If you need a number, do Number(id).

Why this matters today: in Block 2, we use this id as the input to our API call — fetch(`/api/notes/${id}`). The URL feeds the data layer.

🪟 Layouts with <Outlet/>

Most apps share a header / sidebar / footer across pages. You don't want to copy-paste the header into every page. The pattern: a layout component with a placeholder slot called <Outlet/>, and child routes that render into it.

layout.tsx
import { Outlet, NavLink } from "react-router-dom"; export default function Layout() { return ( <div className="min-h-screen bg-gray-50"> <header className="border-b p-4 flex gap-4"> <NavLink to="/notes">All notes</NavLink> <NavLink to="/notes/new">+ New</NavLink> </header> <main className="p-6 max-w-3xl mx-auto"> <Outlet /> {/* child routes render here */} </main> </div> ); }
main.tsx · using nested routes
<Routes> <Route element={<Layout />}> <Route path="/notes" element={<NotesList/>} /> <Route path="/notes/new" element={<NoteNew/>} /> <Route path="/notes/:id" element={<NoteDetail/>} /> </Route> </Routes>

🚀 useNavigate() — redirect from code

<Link> handles user clicks. But sometimes you want to navigate from JavaScript — for example, after a form submits successfully, send the user to the new note's detail page. That's useNavigate().

pages/note-new.tsx
import { useNavigate } from "react-router-dom"; function NoteNew() { const navigate = useNavigate(); async function handleSubmit(data) { const res = await fetch("/api/notes", { ... }); const created = await res.json(); navigate(`/notes/${created.id}`); // User is now on the new note's detail page } }

You can also pass navigate(-1) to go back, or navigate('/login', { replace: true }) to replace the current entry in browser history (so back doesn't return to the protected page).

🛡️ Protected routes (route guards)

"If the user isn't logged in, send them to /login". The pattern is just a tiny wrapper component that checks auth and either renders its children or redirects.

routes/protected.tsx
import { Navigate } from "react-router-dom"; export function Protected({ children }) { const user = useCurrentUser(); // some auth hook you write if (!user) { return <Navigate to="/login" replace />; } return children; } // In your routes: <Route path="/notes" element={<Protected><NotesList/></Protected>} />
🚨
Important: route guards on the client only hide the UI. They are not security. The actual data still must be protected by the API server — otherwise anyone can call the API directly. Frontend guards = good UX. Backend guards = real security.

Block 1 takeaways

  • BrowserRouter wraps once · Routes is the switch · Route is one URL → one component.
  • Use <Link>, never <a href>, inside the app — otherwise the page reloads.
  • useParams() reads route params like :id. Always returns strings.
  • A layout with <Outlet/> lets you write the header / nav once.
  • useNavigate() redirects from code. Use it after a successful action.
Block 2 · 70 minutes

🔌 Working with APIs

Past basic fetch — into the production patterns.

😱 The "fire-and-forget" trap

You already know fetch. The naive way to use it in React looks like this:

notes-list.tsx · 🚩 the bug everyone writes
function NotesList() { const [notes, setNotes] = useState([]); useEffect(() => { fetch("/api/notes") .then(r => r.json()) .then(setNotes); }, []); return notes.map(n => <li>{n.title}</li>); }

What's wrong here?

  • The first 200ms shows an empty page — users think it's broken.
  • If the network fails, the user sees an empty list with no message.
  • If the API is slow, no spinner — users click again and again.
  • If the API returns a 404 or 500, .json() may explode silently with a confusing console error.

🎯 The 3-state pattern

Every fetch in a UI has three possible states. You should render all three — never just the happy path.

⏳ Loading

  • Initial state, while the request is in flight
  • Show a spinner, skeleton, or "Loading…" text

⚠️ Error

  • Network failed or the server returned a bad code
  • Show a user-facing message; offer a retry

✅ Data

  • The happy path — render the real list / detail
  • Always handle the empty-list case here
notes-list.tsx · ✓
function NotesList() { const [notes, setNotes] = useState<Note[]>([]); const [loading, setLoading] = useState(true); const [error, setError] = useState<string | null>(null); useEffect(() => { fetch("/api/notes") .then(r => { if (!r.ok) throw new Error("Failed to load notes"); return r.json(); }) .then(setNotes) .catch(e => setError(e.message)) .finally(() => setLoading(false)); }, []); if (loading) return <p>Loading notes…</p>; if (error) return <p className="text-red-600">{error}</p>; if (notes.length === 0) return <p>No notes yet.</p>; return <ul>{notes.map(n => <li key={n.id}>{n.title}</li>)}</ul>; }

Memorise the shape: three useStates, three early returns. This is 80% of working with APIs in React.

🔤 HTTP verbs — same fetch, different method

fetch defaults to GET. For POST, PUT, or DELETE you pass an options object as the second argument.

MethodWhat it doesUsed forBody?
GETReadLoad notes list, load one noteNo
POSTCreateSubmit a new noteYes (JSON)
PUTUpdate (replace)Edit an existing noteYes (JSON)
DELETERemoveDelete a noteNo
create-note.ts · POST in fetch
const res = await fetch("/api/notes", { method: "POST", // 1. set the verb headers: { "Content-Type": "application/json" }, // 2. tell the server it's JSON body: JSON.stringify({ // 3. body must be a STRING title: "Class 09 notes", body: "Routing, APIs, forms…", }), }); const created = await res.json(); // → { id: 101, title: "Class 09 notes", body: "..." }
⚠️
Body must be a string. Never pass a JS object directly. Always wrap it in JSON.stringify(...). And the matching Content-Type: application/json header tells the server how to parse it.

🔐 Auth — just a header

Modern APIs hand you a token after a successful login (often a long random-looking string called a JWT). You attach it to every request as the Authorization header.

authed-fetch.ts
const token = localStorage.getItem("auth_token"); const res = await fetch("/api/notes", { headers: { "Content-Type": "application/json", "Authorization": `Bearer ${token}`, // ← the magic string }, }); if (res.status === 401) { // token is missing or expired → send to /login } if (res.status === 403) { // authenticated, but not allowed → show "no permission" }

🗝️ Where API keys live

Hard-coding API keys in your code is a classic mistake. They end up in your git history, on GitHub, and in your shipped JS bundle. Use environment variables instead.

.env.local · NOT committed to git
VITE_API_URL=https://api.example.com VITE_PUBLIC_KEY=pk_live_abc123
api.ts · using env vars
const apiUrl = import.meta.env.VITE_API_URL; const res = await fetch(`${apiUrl}/notes`);
🚨
Frontend env vars are NOT secret. Anything in your bundle ships to the browser, where any user can read it. Only put public keys (like Stripe publishable keys, public API endpoints) in frontend env. Real secrets — database passwords, private API keys — live on the server.

Make sure .env.local is in your .gitignore. Vite already does this by default. In Vite, frontend env vars must start with VITE_ to be exposed to your code.

🪤 The fetch trap — res.ok

The most surprising fact about fetch: it does not throw on 4xx or 5xx responses. The promise resolves, even if the server returned 404 or 500. Only true network errors (no DNS, no connection) throw.

You have to check res.ok yourself:

safe-fetch.ts
const res = await fetch("/api/notes"); if (!res.ok) { // Status is 4xx or 5xx — treat as an error throw new Error(`API error: ${res.status} ${res.statusText}`); } const data = await res.json();

res.ok is true when the status is in the 200–299 range. For everything else, you need to throw or handle it explicitly.

🪝 A custom hook: useFetch

You'll write the 3-state pattern dozens of times. The DRY (don't-repeat-yourself) move is to wrap it in a custom hook.

hooks/use-fetch.ts
import { useEffect, useState } from "react"; export function useFetch<T>(url: string) { const [data, setData] = useState<T | null>(null); const [loading, setLoading] = useState(true); const [error, setError] = useState<string | null>(null); useEffect(() => { setLoading(true); fetch(url) .then(r => { if (!r.ok) throw new Error(r.statusText); return r.json(); }) .then(setData) .catch(e => setError(e.message)) .finally(() => setLoading(false)); }, [url]); return { data, loading, error }; } // Use it in any component: const { data, loading, error } = useFetch<Note[]>("/api/notes");

Now every component that fetches data is one line. The pattern lives in one file. If you want to add retries, abort-on-unmount, or caching later, you change one place.

🚀 A peek at production: TanStack Query

Once your app grows, your useFetch won't be enough. Real apps need:

  • Caching — don't re-fetch the same data twice
  • Refetch on focus — user comes back to the tab? show fresh data
  • Retries with backoff on transient failures
  • Optimistic updates — show the user's change instantly, roll back if the server rejects
  • Sharing data across components without prop-drilling

Don't build all this yourself — use a library. The industry default is TanStack Query (formerly React Query).

notes-list.tsx · with TanStack Query
import { useQuery } from "@tanstack/react-query"; function NotesList() { const { data, isLoading, error } = useQuery({ queryKey: ["notes"], queryFn: () => fetch("/api/notes").then(r => r.json()), }); // You get caching, refetching, retries, deduping — all free. }
💡
Today: see it. Later: learn it. When your useFetch isn't enough, TanStack Query is what you reach for. Don't worry about it for the first month or two of practice — the simple pattern is enough.

Block 2 takeaways

  • Always render three states: loading, error, data.
  • res.ok is your friend — fetch doesn't throw on 4xx/5xx.
  • POST/PUT need method, Content-Type, and JSON.stringify'd body.
  • Auth is a header: Authorization: Bearer <token>.
  • Secrets live in .env — and frontend env vars are NOT secret.
  • Wrap repeated patterns in custom hooks. When you outgrow them, reach for TanStack Query.
Block 3 · 70 minutes

📝 Forms & Validation

Take input from users, trust nothing, show good errors. react-hook-form + zod.

🎛️ Controlled inputs — quick recap

You saw this in classes 06 and 08. In React, the input's value is owned by component state — you bind value and onChange:

controlled-input.tsx
const [title, setTitle] = useState(""); <input value={title} onChange={e => setTitle(e.target.value)} className="border rounded p-2" />

This works for one input. The problem starts when your form has 5, 10, 15 fields with their own validation rules.

😩 Why hand-rolled forms hurt

Imagine a registration form with 5 fields. Done by hand, you end up with:

register-form.tsx · 🚩 boilerplate explosion
const [name, setName] = useState(""); const [email, setEmail] = useState(""); const [password, setPassword] = useState(""); const [phone, setPhone] = useState(""); const [cid, setCid] = useState(""); const [errors, setErrors] = useState({}); const [touched, setTouched] = useState({}); // + a validate() function with 5 if-statements // + onBlur handlers to mark "touched" // + a submit handler that re-validates everything // + reset / clear logic // + repeated for every form in your app

Multiply by 10 forms. This is why form libraries exist.

🪝 react-hook-form — the form library most teams use

Three things to learn: useForm, register, handleSubmit. The library wires the inputs — you don't manage state per field.

note-new.tsx · with RHF, no validation yet
import { useForm } from "react-hook-form"; function NoteNew() { const { register, handleSubmit } = useForm(); const onSubmit = (data) => console.log(data); return ( <form onSubmit={handleSubmit(onSubmit)} className="space-y-3"> <input {...register("title")} placeholder="Title" className="border rounded p-2 w-full" /> <textarea {...register("body")} placeholder="Body" className="border rounded p-2 w-full" /> <button type="submit" className="bg-blue-600 text-white rounded p-2">Save</button> </form> ); }

No useState. No onChange. The spread syntax {...register("title")} attaches the right props (ref, onChange, name, etc.) for you.

💡
Why "uncontrolled" is faster: RHF uses refs instead of React state for input values. Each keystroke doesn't re-render the whole form. For a form with 20 inputs, this is a noticeable speed difference.

🛡️ zod — one schema, validation + types

You could write validation rules inline in register("title", { required: true, minLength: 3 }). But that scatters your rules across the JSX. zod describes the shape of valid data in one place.

schemas/note.ts
import { z } from "zod"; export const noteSchema = z.object({ title: z.string() .min(3, "Title must be at least 3 characters") .max(100, "Title must be under 100 characters"), body: z.string().min(10, "Body is too short"), tags: z.array(z.string()).optional(), }); export type Note = z.infer<typeof noteSchema>; // Note is { title: string; body: string; tags?: string[] } // One schema gives you BOTH validation AND TypeScript types.

z.infer<typeof schema> turns your schema into a TypeScript type. So you write the rules once, and your code is type-checked against the same rules.

🪢 Connecting zod and react-hook-form

The bridge is one line: resolver: zodResolver(noteSchema). Now RHF uses your zod schema for validation.

Terminal
npm install react-hook-form zod @hookform/resolvers
note-new.tsx · the real version
import { useForm } from "react-hook-form"; import { zodResolver } from "@hookform/resolvers/zod"; import { noteSchema, Note } from "./schemas/note"; function NoteNew() { const { register, handleSubmit, formState: { errors, isSubmitting }, } = useForm<Note>({ resolver: zodResolver(noteSchema), }); return ( <form onSubmit={handleSubmit(onSubmit)} className="space-y-3"> <div> <input {...register("title")} className="border rounded p-2 w-full" /> {errors.title && ( <p className="text-red-600 text-sm mt-1"> {errors.title.message} </p> )} </div> <button type="submit" disabled={isSubmitting} className="bg-blue-600 text-white rounded p-2 disabled:opacity-50"> {isSubmitting ? "Saving…" : "Save"} </button> </form> ); }

errors.title is filled in by zod with the message you wrote in the schema. isSubmitting is provided by RHF — perfect for disabling the button while the network request is in flight.

Block 3 takeaways

  • Hand-rolled forms produce a boilerplate explosion as fields grow.
  • react-hook-form: useForm, register, handleSubmit — three things, no per-field state.
  • zod describes the shape of valid data in one schema.
  • z.infer<typeof schema> turns the schema into a TypeScript type for free.
  • zodResolver(schema) connects RHF to zod — one line.
  • formState.isSubmitting + button disabled = no double-submits.
Section · Bringing it all home

🪢 Tying it all together

All three blocks meet in the create-note page.

Here's the final NoteNew component. Watch how routing, APIs, and forms all show up:

pages/note-new.tsx · final version
import { useNavigate } from "react-router-dom"; // ← Block 1 import { useForm } from "react-hook-form"; // ← Block 3 import { zodResolver } from "@hookform/resolvers/zod"; // ← Block 3 import { noteSchema, Note } from "../schemas/note"; export default function NoteNew() { const navigate = useNavigate(); const { register, handleSubmit, formState: { errors, isSubmitting } } = useForm<Note>({ resolver: zodResolver(noteSchema) }); const onSubmit = async (data: Note) => { const res = await fetch("/api/notes", { // ← Block 2 method: "POST", headers: { "Content-Type": "application/json" }, body: JSON.stringify(data), }); if (!res.ok) return alert("Save failed"); const created = await res.json(); navigate(`/notes/${created.id}`); // ← Block 1 }; return ( <form onSubmit={handleSubmit(onSubmit)} className="space-y-3"> <input {...register("title")} className="border rounded p-2 w-full" /> {errors.title && <p className="text-red-600 text-sm">{errors.title.message}</p>} <textarea {...register("body")} className="border rounded p-2 w-full" /> {errors.body && <p className="text-red-600 text-sm">{errors.body.message}</p>} <button type="submit" disabled={isSubmitting} className="bg-blue-600 text-white rounded p-2"> {isSubmitting ? "Saving…" : "Save"} </button> </form> ); }

One screen, three concepts. Routing takes the user there and out again. API persists the data. Forms capture and validate it. This is the working triangle of every single CRUD app you'll ever build.

At-home practice · ~60 minutes

🛠️ Practice

Fork the notes app from class. Pick one easier task and one stretch task.

Easier 1Add an About page

Add a new route /about that renders a static page describing the app. Add a NavLink for it in the layout's header. Make sure it shows up as "active" when you're on it.

  1. Create src/pages/about.tsx with a simple page (just an <h1> and a paragraph).
  2. Add a <Route path="/about" element={<About/>}/> inside your <Layout/> route block.
  3. Add <NavLink to="/about">About</NavLink> in the header.
  4. Style it with Tailwind — text-blue-600 font-bold for active, text-gray-600 otherwise.

Easier 2Make the loading state look nicer

Replace the plain "Loading…" text with a Tailwind spinner or skeleton. (Ask Claude Code: "show me a Tailwind skeleton loader for a list of cards".)

Stretch 1Add a tags field

Extend the note schema with an optional tags array of strings. Update the form to accept a comma-separated input ("react, typescript, vite") and split it into an array before submitting.

  1. In schemas/note.ts, add tags: z.array(z.string()).optional().
  2. In the form, register a tagsInput field as a string.
  3. In onSubmit, split data.tagsInput by comma and trim each piece.
  4. On the detail page, render each tag as a small Tailwind chip (bg-blue-100 text-blue-700 rounded px-2 py-0.5).

Stretch 2Add a delete button

On the detail page, add a "Delete note" button. On click, send a DELETE request to the API and navigate back to /notes on success.

  1. Add a button on the detail page styled with red Tailwind (bg-red-600 text-white rounded p-2).
  2. On click, call fetch(`/api/notes/${id}`, { method: "DELETE" }).
  3. Check res.ok. If true, call navigate("/notes").
  4. Add a window.confirm("Delete this note?") guard so users don't delete by accident.
💡
Note: JSONPlaceholder will pretend to delete — the response succeeds but the data isn't actually changed on their server. That's fine for practice. In a real app you'd use a backend you control.
