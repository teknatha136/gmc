04 Three patterns for today

Three problems → three patterns

Composition
Problem: Two cards look the same but show different things.
Pattern: children prop.
Lift state up
Problem: Two siblings need the same value.
Pattern: Move state to parent.
Context
Problem: A value needs to reach far down the tree.
Pattern: Provider + useContext.
05 Reach for them in this order

Simplest first

  • Composition first — it solves most reuse problems.
  • Lift state up when two close siblings need to share a value.
  • Context only when lifting means too many layers in between.

08 The children prop

Whatever you put between the tags arrives as children

card.tsx
type CardProps = Readonly<{ children: React.ReactNode; }>; export function Card({ children }: CardProps) { return ( <div className="rounded-xl border border-gray-200 bg-white p-5 shadow-sm"> {children} </div> ); }

children is just a prop with a special name. React.ReactNode means "anything React can render".

09 Using a Card

Same component, three different appearances

app.tsx
<Card> <h3>Site Tour</h3> <p>Tuesday · 9 AM</p> </Card> <Card> <img src="/cohort.jpg" /> </Card> <Card> <EventsCounter /> // even another component! </Card>

One Card definition, used three times. The Card doesn't know or care what's inside.

10 When to use children vs props

A simple rule

If you need…Use
Different content inside the same shellComposition with children
Different data, same structure (e.g. a list of names)Regular props
A bit of both — text and a custom layout insideMix them

Most "wrapper" components (Card, Modal, Layout, Page) use children. Most "data" components (UserBadge, EventRow) use props.

Section 03 · 11:15 – 12:00

Lifting
State Up

Two siblings, one shared value.

03
19 The problem

Two siblings need the same value

A real example
A SearchBar component types a query.
An EventList component shows the matching events.
Both need the same query value. Where does it live?

Components can pass data down to children. They can't talk sideways to siblings.

20 The solution — move state up

Put state in the closest common parent

EventsTracker [query]
SearchBar value, onChange
EventList query

The parent owns query. It passes the value to SearchBar + a setter to update it. It passes the value to EventList for filtering.

21 What it looks like in code
events-tracker.tsx
function EventsTracker() { // state lives HERE — in the parent const [query, setQuery] = useState(""); return ( <> <SearchBar value={query} onChange={setQuery} /> <EventList query={query} /> </> ); }

Both children now see the same value, owned by the parent.

22 The child becomes "controlled"

Parent owns the value · child is the view

search-bar.tsx
type SearchBarProps = Readonly<{ value: string; onChange: (v: string) => void; }>; export function SearchBar({ value, onChange }: SearchBarProps) { return ( <input className="w-full rounded-md border border-gray-300 px-3 py-2" value={value} onChange={e => onChange(e.target.value)} /> ); }

SearchBar has no useState of its own. The parent decides everything.

23 Summary — when to lift

Three steps

1. Notice
Two siblings need the same value.
2. Move
Take the useState out of the child. Put it in the parent.
3. Pass
Give value & setter to children as props.

25 Hands-on 2 — what we're building
Goal

A search box that filters a list of events

Components

  • EventsTracker — the parent. Holds the query state and the events list.
  • SearchBar — controlled input. Receives value and onChange.
  • EventList — receives the (filtered) events. Renders them.

Behaviour

  • Type "tour" → only events with "tour" in the title show.
  • Empty box → all events show.
  • Filter is case-insensitive.

Wire up the parent

9

10

Context

A quick look — for when lifting goes too far.

04
32 Why we need Context

Sometimes lifting means too many layers

App [user]
Layout user
Header user
Avatar user

Prop drilling: middle components carry props they don't use. Annoying, repetitive.

33 Context — three steps
user-context.tsx
// 1. Create — once, somewhere central const UserContext = createContext<User | null>(null); // 2. Provide — wrap the part of the tree that needs it function App() { return ( <UserContext.Provider value={currentUser}> <Layout /> </UserContext.Provider> ); } // 3. Consume — anywhere inside, no prop chain function Avatar() { const user = useContext(UserContext); return <img src={user?.avatar} />; }

Three pieces: createContext, <Provider value=…>, and useContext in any descendant.

34 When to use Context
SituationUse
Two close siblings need the same valueLift to parent (no Context)
Value is needed 1–2 layers belowJust pass props
Value crosses 3+ layers, used way down deepContext
Theme · current user · languageContext — almost always

Don't reach for Context just because you can. Try lifting first.
