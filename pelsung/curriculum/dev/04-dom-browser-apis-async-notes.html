Section 01

The DOM — Document Object Model

The page you see, as an object that JavaScript can touch.

What the DOM is

When the browser loads your HTML, it reads the text and builds a tree of objects in memory. Every tag becomes a "node". JavaScript gets one variable, document, that is the root of this tree.

This tree is live. If you change a node in JavaScript, the page on the screen updates right away.

🌳 HTML (static text)

  • What you type in index.html
  • Parsed once, when the page loads
  • The browser reads it top to bottom

⚡ DOM (live object)

  • Built from the HTML
  • JavaScript reads and changes it
  • The screen reflects it in real time
html
└── body
    ├── h1 → "Gelephu Mindfulness City"
    ├── p → "A new city in Bhutan"
    └── ul
        ├── li → "Tek Nath"
        └── li → "Karma"
💡
The entry point is always document. Open any page, hit F12, and in the Console try document.title or document.body — you're talking to the DOM.
Section 02

Selecting & changing elements

The four things you'll do on the DOM every day.

🎯 Selecting elements

To change an element, you first need to grab it. The browser gives us querySelector and querySelectorAll — they accept any CSS selector you already know.

select.js
// Grab the FIRST match const heading = document.querySelector("h1"); const firstCard = document.querySelector(".card"); const saveBtn = document.querySelector("#save"); // Grab ALL matches (a NodeList — loop with forEach) const allCards = document.querySelectorAll(".card"); allCards.forEach((card) => { console.log(card.textContent); });
⚠️
Returns null if nothing matches. Always check that your selector is correct — otherwise you'll get Cannot read property of null.

✏️ Reading & writing

Once you have an element, you can read its text, change it, adjust attributes, or toggle CSS classes.

modify.js
const title = document.querySelector("h1"); // Read console.log(title.textContent); // Write title.textContent = "Gelephu Mindfulness City"; // Attributes const img = document.querySelector("img"); img.src = "gmc.png"; img.alt = "GMC logo"; // Form inputs const input = document.querySelector("#name"); console.log(input.value); // CSS classes title.classList.add("highlight"); title.classList.remove("muted"); title.classList.toggle("active");
⚠️
Avoid innerHTML with user input. It parses its contents as HTML — an attacker can inject <script> tags. Use textContent for text.

🌱 Creating & removing elements

To grow the tree, create an element, set its contents, then append it to a parent. To remove it, call .remove() on it.

create.js
// 1. Create — not yet on the page const li = document.createElement("li"); li.textContent = "Tek Nath"; // 2. Add to the page const list = document.querySelector("#trainees"); list.appendChild(li); // 3. Remove li.remove();

Section takeaways

  • querySelector → first match; querySelectorAll → every match
  • textContent for reading/writing text; avoid innerHTML with user input
  • classList.add / remove / toggle for CSS classes
  • Build new elements with createElement → set properties → appendChild

🧪 Complete live demo — GMC attendance

This is a single, self-contained HTML file that uses every DOM technique from this section: selecting, creating, appending, removing, toggling classes, and listening to click + keyboard events.

You can run it yourself — open pelsung/curriculum/dev/04-dom-demo.html in a browser. Type a name, press Enter, click a name to mark them present, press ✕ to remove.

The HTML

04-dom-demo.html
<h1>GMC Attendance</h1> <p id="count">0 trainees</p> <input type="text" id="name" placeholder="Trainee name…"> <button id="add">Add</button> <ul id="list"></ul> <button id="clear">Clear all</button>

The JavaScript (with comments explaining each step)

script · inside 04-dom-demo.html
// ── 1. SELECT the elements we'll use ───────────────────── const input = document.querySelector("#name"); const addBtn = document.querySelector("#add"); const clearBtn = document.querySelector("#clear"); const list = document.querySelector("#list"); const count = document.querySelector("#count"); // ── 2. Helper: keep the count text in sync ─────────────── function updateCount() { const n = list.querySelectorAll("li").length; count.textContent = n + " trainee" + (n === 1 ? "" : "s"); } // ── 3. ADD a trainee ───────────────────────────────────── function addTrainee(name) { if (!name) return; // ignore empty input // Create the <li> and fill its text const li = document.createElement("li"); li.textContent = name; // Click the name to toggle "present" class (CSS handles the look) li.addEventListener("click", () => { li.classList.toggle("present"); }); // Create the remove button and nest it inside the <li> const x = document.createElement("button"); x.textContent = "✕"; x.addEventListener("click", (e) => { e.stopPropagation(); // don't trigger the li click li.remove(); // detach from the DOM updateCount(); }); li.appendChild(x); // Put the <li> on the page list.appendChild(li); updateCount(); } // ── 4. Wire up the "Add" button ────────────────────────── addBtn.addEventListener("click", () => { addTrainee(input.value.trim()); input.value = ""; // reset the input input.focus(); // bring cursor back }); // Pressing Enter in the input also adds input.addEventListener("keydown", (e) => { if (e.key === "Enter") addBtn.click(); }); // ── 5. Wire up the "Clear all" button ──────────────────── clearBtn.addEventListener("click", () => { if (!confirm("Clear everyone?")) return; list.querySelectorAll("li").forEach((li) => li.remove()); updateCount(); }); updateCount(); // initialise

Key CSS that changes when the "present" class toggles

style
li { padding: 0.75rem 1rem; background: #F9F9F9; cursor: pointer; } li.present { background: #F0F7EC; color: #2A4D14; text-decoration: line-through; }
🎯
What to point out while teaching: the HTML gives us the skeleton, the CSS describes the two states (plain / present), and the JavaScript's only job is to move elements between those states by toggling a class. No DOM trickery — just classList.toggle.
Section 03

Event listeners

How the page responds to clicks, typing, and form submissions.

👂 addEventListener

An event listener is a function you attach to an element, saying: "When this event happens, please run this function." The most common events are click, input, submit, keydown, and change.

events.js
const btn = document.querySelector("#save"); btn.addEventListener("click", () => { console.log("Clicked!"); }); // Reading what the user typed, on every keystroke const input = document.querySelector("#name"); input.addEventListener("input", (e) => { console.log(e.target.value); }); // Submitting a form without reloading the page const form = document.querySelector("form"); form.addEventListener("submit", (e) => { e.preventDefault(); // stop default page reload // … handle data … });
💡
The argument passed to your handler is an event object. It has useful things like e.target (the element that triggered it) and e.preventDefault().
Section 04

Browser APIs — a tour

The other tools the browser hands to JavaScript for free.

🧰 What is a browser API?

The DOM is one API the browser gives you. There are dozens more — for saving data, timing things, getting the user's location, reading the clipboard, playing audio, and so on. The browser builds the hard parts; JavaScript just calls them.

💾 localStorage & sessionStorage

Two simple key-value stores built into every browser. Both only hold strings — to save an object, convert it with JSON.stringify first, and JSON.parse when reading back.

📦 localStorage

  • Survives closing the browser
  • Good for user settings, drafts
  • About 5MB of storage

⏱ sessionStorage

  • Cleared when the tab closes
  • Good for short-lived state
  • Same API, same size limit
storage.js
// Save a string localStorage.setItem("theme", "dark"); // Read const theme = localStorage.getItem("theme"); // Save an object — stringify it first const user = { name: "Karma", age: 24 }; localStorage.setItem("user", JSON.stringify(user)); // Read and parse const saved = JSON.parse(localStorage.getItem("user")); // Remove one item, or clear everything localStorage.removeItem("theme"); localStorage.clear();

🔍 console — more than just log

The console is your best friend during development. Beyond console.log:

console.warn / .error

  • Colored messages
  • Easier to spot issues

console.table

  • Prints arrays of objects as a table
  • Very readable

console.time / timeEnd

  • Measure how long code takes
  • Simple performance check

📱 navigator — the user's device

navigator tells you things about where the user is and what their browser can do.

navigator.onLine

  • true or false
  • Good for showing "You're offline" banners

navigator.geolocation

  • Request GPS coordinates
  • Asks the user for permission

navigator.clipboard

  • Copy text with writeText
  • Paste text with readText

navigator.language

  • Preferred language code
  • e.g. "en-US", "dz"

⏲ setTimeout, setInterval, dialogs

These live on window, the global browser object — you can call them without prefixing anything.

timers.js
// Run ONCE after 2 seconds setTimeout(() => { console.log("2 seconds passed"); }, 2000); // Run EVERY second const id = setInterval(() => { console.log("tick"); }, 1000); // Stop it clearInterval(id); // Classic browser dialogs — quick but ugly alert("Hello"); const ok = confirm("Are you sure?"); // true/false const nm = prompt("Your name?"); // string or null

🌐 location & history

location describes the URL bar; history is how you navigate. Later, when we learn React Router, it builds on history to change the URL without reloading the page.

nav.js
location.href; // full URL location.pathname; // "/page" location.search; // "?id=5" // Send the user somewhere location.href = "/dashboard"; // Go back (like the back button) history.back();

Browser API takeaways

  • localStorage persists across restarts; sessionStorage is per-tab
  • Storage holds strings only — use JSON.stringify / parse for objects
  • console.table / time / timeEnd are underused debugging gems
  • navigator, location, history are always ready to use
Section 05

Why async exists

The big idea: JavaScript is single-threaded, so slow work has to be clever.

🏪 The shop with one cashier

JavaScript runs on a single thread — like a shop with exactly one cashier. If a customer asks for something that takes 3 minutes to fetch from the warehouse, the cashier has two choices:

❌ Block

  • Cashier freezes, queue waits
  • In a browser: clicks, scrolls, animations all stop
  • Page looks broken

✓ Async

  • Cashier starts the fetch, takes next customer
  • When the item arrives, they return to the first person
  • Queue keeps moving
💡
Any slow operation is async in JS: network calls (fetch), timers (setTimeout), reading files, user events. The rest of your code keeps running while they do their thing.
Section 06

Promises & async/await

The three patterns JavaScript uses to handle async work.

📞 Pattern 1 — Callbacks

The oldest way. You pass a function as an argument, and that function gets called when the slow work finishes.

callback.js
console.log("A"); setTimeout(() => { console.log("B (after 1s)"); }, 1000); console.log("C"); // Prints: A, C, B

Callbacks still exist (events are callbacks!), but when you need to chain many async steps, callbacks nest deeper and deeper — the "pyramid of doom". That's why we got Promises.

🎟️ Pattern 2 — Promises

A Promise is an object that represents a future value. Think of it as a receipt: you hand it to JavaScript, and eventually it'll either give you the goods (fulfilled) or an apology (rejected).

⏳ Pending — still waiting ✓ Fulfilled — we got a value ✕ Rejected — something went wrong

You handle a Promise with three methods:

promise.js
fetch("/api/user") .then((response) => response.json()) // got a response .then((data) => { console.log(data); // success path }) .catch((err) => { console.error(err); // failure path }) .finally(() => { console.log("done either way"); });

Each .then runs when the previous step succeeds. .catch runs if any step in the chain rejects. .finally runs no matter what — great for cleanup like hiding a loading spinner.

✨ Pattern 3 — async / await (modern)

Chained .then calls still get messy when you have many steps. async and await let you write async code that looks exactly like normal code, one line after another.

async.js
async function loadUser() { const response = await fetch("/api/user"); const data = await response.json(); console.log(data); } loadUser();

async function

  • Marks a function as asynchronous
  • It always returns a Promise
  • Required before you can use await inside

await keyword

  • Pauses the function until the Promise settles
  • Unwraps the value
  • Only valid inside an async function

For errors with async/await, wrap the calls in a try / catch block — the same way you'd handle any other error.

safe-fetch.js
async function loadUser() { try { const res = await fetch("/api/user"); if (!res.ok) throw new Error("Bad status: " + res.status); const data = await res.json(); return data; } catch (err) { console.error("Failed:", err); return null; } }
Section 07

fetch in practice

Get real data from the internet — and put it on the page.

🌐 A complete fetch example

This example fetches the current weather for Gelephu from the free Open-Meteo API, and writes the temperature into an <h1> on the page.

weather.js
async function showWeather() { const h1 = document.querySelector("#temp"); h1.textContent = "Loading…"; try { const url = "https://api.open-meteo.com/v1/forecast" + "?latitude=26.86&longitude=90.48&current_weather=true"; const res = await fetch(url); const data = await res.json(); const temp = data.current_weather.temperature; h1.textContent = `Gelephu is ${temp}°C`; } catch (err) { h1.textContent = "Could not load weather"; } } showWeather();
💡
fetch returns a Response object. You need a second await res.json() to actually get the data — the body arrives separately.
Section 08

Common gotchas

Three mistakes that trip up almost everyone.

1. Forgetting await

Without await, fetch gives you the Promise itself, not the value inside it. Your variable ends up as Promise { <pending> }.

bug.js — missing await
const data = fetch("/api/x"); console.log(data); // Promise { <pending> }

2. Unhandled promise rejections

If a Promise rejects and you haven't caught it, the browser logs a warning but the user sees nothing. Every await should live inside a try / catch, or every .then chain should end with a .catch.

3. Sequential awaits when parallel would do

If two requests don't depend on each other, don't await them one after another — fire both at once with Promise.all. You get the total time of the slower request, not the sum.

parallel.js
// 🐢 Sequential — total time is a + b const a = await fetch(urlA); const b = await fetch(urlB); // ⚡ Parallel — total time is max(a, b) const [a, b] = await Promise.all([ fetch(urlA), fetch(urlB) ]);

Today's big ideas

Three tools. Same underlying idea: JavaScript talking to the world outside itself.

🌳 DOM

The page as a live tree. Select with querySelector, change with textContent / classList, listen with addEventListener.

🧰 Browser APIs

Free tools: localStorage for saving, navigator for device info, setInterval for timing, location for URLs.

⏳ Async

Promises + async/await. Wait for slow work without freezing the page. Always handle errors with try / catch.
