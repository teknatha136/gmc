Section 01 · Mindset

Three patterns for today

Three problems show up when components scale — three patterns solve them.

🧩 Composition

  • Problem: two cards look the same but show different things.
  • Pattern: the children prop.

⬆️ Lift state up

  • Problem: two siblings need the same value.
  • Pattern: move the state to their parent.

🌐 Context

  • Problem: a value needs to reach far down the tree.
  • Pattern: Provider + useContext.

Reach for them in this order: composition first (it solves the most). Lifting second. Context only when the other two fall short.

Section 02 · Composition

Composition with children

Same shell, different content.

🧩 The children prop

Whatever you put between a component's opening and closing tags arrives as a prop named children. It's not magic — just a prop with a reserved name. You can render it wherever you want inside the component.

card.tsx
type CardProps = Readonly<{ children: React.ReactNode; }>; export function Card({ children }: CardProps) { return ( <div className="rounded-xl border border-gray-200 bg-white p-5 shadow-sm"> {children} </div> ); }

React.ReactNode is the type for "anything React can render" — strings, numbers, JSX, arrays of JSX, even null.

🎨 Same Card, three appearances

app.tsx
<Card> <h3>Site Tour</h3> <p>Tuesday · 9 AM</p> </Card> <Card> <img src="/cohort.jpg" /> </Card> <Card> <EventsCounter /> // even another component! </Card>

One Card definition, three different appearances. The Card doesn't know — or care — what's inside. That's the point.

🤔 children vs props

If you need…Use
Different content inside the same shellComposition with children
Different data, same structure (a list of names)Regular props
Both — text and a custom layoutMix them

Wrappers (Card, Modal, Layout, Page) usually use children. Data components (UserBadge, EventRow) usually use props.

Section takeaways

  • children is just a prop with a special name.
  • Type it as React.ReactNode.
  • Use it for "same shell, different content" components.
  • Wrappers use children; data components use props.
Section 03 · Hands-on 1

Hands-on 1 — Build a Card (30 min)

A reusable Card with three different appearances.

Goal Build a Card and use it three times

One Card component. Three usages: an event, a photo, a quote. Each looks different but uses the same shell.

  1. In src/, create a new file card.tsx.
  2. Define the type: type CardProps = Readonly<{ children: React.ReactNode }>;
  3. Write the function: export function Card({ children }: CardProps) { return <div className="rounded-xl border border-gray-200 bg-white p-5 shadow-sm">{children}</div>; }
  4. That's it for styling — Tailwind utilities give it padding, a rounded border, white background, and a soft shadow. No CSS file needed.
  5. In app.tsx, import the Card: import { Card } from "./card";
  6. First Card — event: put <h3>Site Tour</h3><p>Tuesday · 9 AM</p> inside.
  7. Second Card — photo: put <img src="…" alt="" /> inside (any image URL).
  8. Third Card — quote: put <blockquote>…</blockquote><cite>— …</cite> inside.
  9. Run npm run dev. You should see three differently-shaped cards on screen.

📄 The final code

card.tsx
type CardProps = Readonly<{ children: React.ReactNode }>; export function Card({ children }: CardProps) { return ( <div className="rounded-xl border border-gray-200 bg-white p-5 shadow-sm"> {children} </div> ); }

Section 04 · Lifting state

Lifting state up

When two siblings need the same value, move the state to their parent.

⬆️ The pattern

Components can pass data down via props. They can not talk sideways to siblings. So when two siblings need the same value, you move it up to their nearest common parent — and the parent passes it back down to both children.

Concrete example: a SearchBar with a query, and an EventList that filters events by that query. Both need the query. So we lift it to their parent, EventsTracker.

events-tracker.tsx
function EventsTracker() { // state lives here — in the parent const [query, setQuery] = useState(""); return ( <> <SearchBar value={query} onChange={setQuery} /> <EventList query={query} /> </> ); }

📐 Three steps

1. Notice

  • Two siblings need the same value.

2. Move

  • Take the useState out of the child.
  • Put it in the parent.

3. Pass

  • Pass the value to children that read it.
  • Pass the setter to children that change it.
Section 05 · Controlled components

Controlled components

When the parent owns the value, the child becomes "controlled".

🎛️ Parent owns the value · child is the view

Once state is lifted, the children don't have their own useState. They receive value and onChange from the parent and reflect what they're told.

search-bar.tsx
type SearchBarProps = Readonly<{ value: string; onChange: (v: string) => void; }>; export function SearchBar({ value, onChange }: SearchBarProps) { return ( <input className="w-full rounded-md border border-gray-300 px-3 py-2" value={value} onChange={e => onChange(e.target.value)} placeholder="Search GMC events…" /> ); }
💡
You've seen this shape before. The native HTML <input> works exactly this way — that's the model. Once you see "parent owns the value, child reflects it," every form in React becomes simpler.

Section takeaways

  • Siblings can't share state directly — lift to a common parent.
  • The child becomes "controlled" — it has no useState of its own.
  • Pass value for reading and onChange for changing.
  • Same shape as native <input>.
Section 06 · Hands-on 2

Hands-on 2 — Search filter (60 min)

Type in a box → the list filters live. Lifting state in action.

Goal Search filter for GMC events

Three components: EventsTracker (parent, holds the query), SearchBar (controlled input), EventList (renders the filtered events). Type into the box → matching events show.

  1. In src/, create events-tracker.tsx.
  2. Add the type: type GmcEvent = Readonly<{ id: number; title: string; date: string }>;
  3. Create another file search-bar.tsx. Type its props: Readonly<{ value: string; onChange: (v: string) => void }>.
  4. Inside SearchBar, return <input className="w-full rounded-md border border-gray-300 px-3 py-2" value={value} onChange={e => onChange(e.target.value)} />.
  5. Create event-list.tsx. Type its props: Readonly<{ events: GmcEvent[] }>.
  6. Inside EventList, return a <ul> mapping over events: {events.map(e => <li key={e.id}>{e.title} — {e.date}</li>)}
  7. Show a friendly message when empty: if (events.length === 0) return <p>No matches</p>;
  8. Back in events-tracker.tsx, lift the query: const [query, setQuery] = useState("");
  9. Make a small array: const allEvents: GmcEvent[] = [{ id: 1, title: "GMC Site Tour", date: "Apr 28" }, …]; — at least 4 events.
  10. Compute the filtered list: const filtered = allEvents.filter(e => e.title.toLowerCase().includes(query.toLowerCase()));
  11. Return: <SearchBar value={query} onChange={setQuery} /> followed by <EventList events={filtered} />.
  12. In app.tsx, render <EventsTracker />. Run the dev server. Type "tour" → only matching events show.

📄 The final code (parent)

events-tracker.tsx
import { useState } from "react"; import { SearchBar } from "./search-bar"; import { EventList } from "./event-list"; export type GmcEvent = Readonly<{ id: number; title: string; date: string }>; const allEvents: GmcEvent[] = [ { id: 1, title: "GMC Site Tour", date: "Apr 28" }, { id: 2, title: "SELISE Open House", date: "Apr 30" }, { id: 3, title: "Cohort 3 Showcase", date: "May 02" }, { id: 4, title: "Pelsung Site Visit", date: "May 05" }, ]; export function EventsTracker() { const [query, setQuery] = useState(""); const filtered = allEvents.filter(e => e.title.toLowerCase().includes(query.toLowerCase()) ); return ( <> <SearchBar value={query} onChange={setQuery} /> <EventList events={filtered} /> </> ); }

Section 07 · Context

Context — a peek

For when lifting goes too far and props travel through layers that don't use them.

🌐 The problem

Lifting works beautifully when components are close together. But sometimes a value lives at the very top (App), and is needed deep down (Avatar) — five layers below. Every middle layer carries a user prop just to pass it along. That's prop drilling, and it's annoying.

📐 Context — three steps

user-context.tsx
import { createContext, useContext } from "react"; // 1. Create — once, somewhere central export const UserContext = createContext<User | null>(null); // 2. Provide — wrap the part of the tree that needs it function App() { return ( <UserContext.Provider value={currentUser}> <Layout /> </UserContext.Provider> ); } // 3. Consume — anywhere inside, no prop chain function Avatar() { const user = useContext(UserContext); return <img src={user?.avatar} />; }

Three pieces — createContext, the Provider with a value, and useContext in any descendant. The middle layers don't have to know about it.

🤔 When to use Context

SituationUse
Two close siblings need the same valueLift to parent (no Context)
Value needed 1–2 layers belowJust pass props
Value crosses 3+ layers, used way downContext
Theme · current user · languageContext — almost always
💡
Don't reach for Context just because you can. Try lifting first. Only switch to Context when prop drilling actually becomes annoying — usually 3+ layers.

Section takeaways

  • Context lets you skip the prop chain.
  • Three steps: createContext, wrap with Provider, read with useContext.
  • Use it for things like the current user, theme, or language.
  • Don't use it just to avoid one prop. Lifting is simpler when it works.
