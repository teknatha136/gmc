Section 01 · Why

Why React?

A short answer: it's the right mental model for screens that change.

💡 The DOM way (what you've been doing) vs. the React way

In Class 04 you built a counter the manual way — find an element with querySelector, listen for clicks, update textContent yourself. It works for one counter. It does not scale to a real product with dozens of moving parts. You spend more time keeping the screen in sync with your data than writing the actual feature.

Vanilla JS — imperative
let count = 0; const btn = document.querySelector("#btn"); const label = document.querySelector("#label"); btn.addEventListener("click", () => { count++; label.textContent = `Count: ${count}`; });
React — declarative
function Counter() { const [count, setCount] = useState(0); return ( <button onClick={() => setCount(count + 1)}> Count: {count} </button> ); }

In React you describe what the screen looks like for a given state. When state changes, React figures out the smallest update needed and applies it to the DOM. You don't touch textContent ever again.

Section takeaways

  • Vanilla JS is imperative — you tell the browser every step.
  • React is declarative — you describe the shape, React handles updates.
  • Components are reusable: define once, use anywhere with different data.
  • React + TypeScript catches a whole class of bugs before you hit "save".
Section 02 · Setup

Setting up a Vite project

The same tool you used in earlier classes — now with a React + TypeScript template.

⚙️ Scaffold and run

Vite is a fast dev server. The react-ts template gives you a working React + TypeScript starter in seconds.

terminal
npm create vite@latest hello-react -- --template react-ts cd hello-react npm install npm run dev

Open the URL Vite prints (default http://localhost:5173) and you'll see the starter page. Edit a file → save → the page updates automatically.

📄 src/main.tsx

  • The entry file — runs first.
  • Mounts <App /> into <div id="root">.
  • You usually don't touch this much.

📄 src/app.tsx

  • Your top-level component.
  • Where you start writing today.
  • Imports other components from src/.

📄 index.html

  • Just one <div id="root"></div>.
  • React fills the page from there.
  • You almost never edit it.
Section 03 · Components

Components

A component is a function that returns UI. That's it.

🧱 Your first component

A React component is a JavaScript function whose name starts with a capital letter and whose return value is JSX. Capital letter is mandatory — React uses it to tell components apart from regular HTML tags.

welcome.tsx
function Welcome() { return <h1>Hello, Pelsung!</h1>; } export default Welcome;

Once defined, you use it like an HTML tag:

app.tsx
import Welcome from "./welcome"; function App() { return ( <div> <Welcome /> <Welcome /> </div> ); }
⚠️
Functional components only. You may run into "class components" online — they look like class App extends React.Component. They're legacy. We don't write them in 2026, and we won't cover them in this course.

Section takeaways

  • Components are just functions returning JSX.
  • Names start with a capital letter — always.
  • Use them like custom HTML tags: <Welcome />.
  • One file per component is the convention.
Section 04 · JSX

JSX

It looks like HTML. It's actually JavaScript.

🏷️ What JSX really is

JSX is a syntax extension on top of JavaScript. The <h1> you write is not HTML — it's JS that gets compiled to a function call. Vite handles the conversion. You'll never write the compiled version yourself, but it's worth seeing once.

JSX vs compiled
// What you write const el = <h1 className="title">Hello</h1>; // What it becomes after Vite compiles it const el = React.createElement("h1", { className: "title" }, "Hello");

🔧 JSX rules to remember

JSX has small but important differences from HTML. Memorise these — they catch every beginner.

1. className, not class

  • class is reserved in JS.
  • Use <div className="card">.

2. camelCase events

  • onclick → onClick
  • onchange → onChange
  • onsubmit → onSubmit

3. Self-close empty tags

  • <img />, <input />, <br />
  • The slash before > is required.

4. One parent element

  • JSX must return a single root.
  • Wrap siblings in <div> or a Fragment <>...</>.

5. { } for JS expressions

  • Anything inside curly braces is JS.
  • Variables, calls, ternaries, math.

6. style is an object

  • style={{ color: "red" }} — double braces.
  • camelCase property names: backgroundColor.

🔣 Embedding JS in JSX

Curly braces inside JSX are an escape hatch back into JavaScript. Anything that evaluates to a value goes there.

app.tsx
function App() { const name = "Karma"; const isOnline = true; const topics = ["JSX", "Props", "State"]; return ( <div> <h1>Hello, {name}!</h1> // variable <p>{isOnline ? "🟢 Online" : "🔴 Offline"}</p> // ternary <ul> {topics.map(t => <li key={t}>{t}</li>)} // list </ul> </div> ); }

Section takeaways

  • JSX is JavaScript with HTML-shaped syntax — Vite compiles it to function calls.
  • Six rules: className, camelCase events, self-close, one parent, { }, style is an object.
  • Inside { } you have full JavaScript — variables, ternaries, .map().
  • JSX can't run statements like if or loops directly — use ternaries and .map() instead.
Section 05 · Props

Props — passing data into components

Props turn one component definition into many different appearances.

📦 What props are

A component without props is rigid — it always shows the same thing. Real apps need flexibility. Props are arguments to a component, the same way function arguments work. The parent passes data down; the child renders based on it.

greet.tsx
// Receive props as an argument function Greet(props) { return <h1>Hello, {props.name}!</h1>; } // Pass them like attributes <Greet name="Karma" /> <Greet name="Pema" /> <Greet name="Tashi" />
🛑
Props are read-only. Inside a component, never write props.name = "X". Props flow downward — if the value needs to change, that's a job for state (next section), and the change is owned by the parent that passed the prop.
Section 06 · Typing

Typing props with TypeScript

Use type for the props shape — it's the SELISE convention you saw in Class 05.

🔤 Define a Props type, destructure in the signature

greet.tsx — typed
type GreetProps = { name: string; role?: string; // optional }; function Greet({ name, role }: GreetProps) { return ( <div> <h1>Hello, {name}!</h1> {role && <p>You are a {role}.</p>} </div> ); }

Two things to notice:

Destructuring in the signature

  • { name, role } pulls those keys out directly.
  • Cleaner than writing props.name everywhere.

Optional props with ?

  • role?: string means the parent doesn't have to pass it.
  • Inside the component, role may be undefined — guard with &&.
💡
Default values use destructuring defaults: function Button({ label, variant = "primary" }). If the parent doesn't pass variant, you get "primary".
Section 07 · Composition

Composition & the children prop

Render a list. Wrap content. Build screens out of small parts.

🧩 Render a list from data

The most common React pattern: an array of data, mapped into an array of components.

app.tsx
const trainees = [ { name: "Karma", role: "Dev" }, { name: "Pema", role: "BA" }, { name: "Tashi", role: "QA" }, ]; function App() { return ( <div> {trainees.map(t => ( <Greet key={t.name} name={t.name} role={t.role} /> ))} </div> ); }
🔑
The key prop. When you render a list, give each item a unique key. React uses it to track items efficiently when the list changes. key must be unique among siblings — usually an id from your data. We'll go deeper in Class 07.

📥 The children prop

children is a special prop — whatever you put between a component's opening and closing tags is its children. It's how you build wrappers like cards, modals, or layout boxes.

card.tsx
type CardProps = { children: React.ReactNode }; function Card({ children }: CardProps) { return <div className="card">{children}</div>; } // Used like this: <Card> <h2>Hello</h2> <p>Anything here becomes the children prop.</p> </Card>

🐛 Common props mistakes

MistakeWhat goes wrong
count="3"Passes the string "3", not the number. Use count={3}.
props.name = "X"Mutating props is a bug. Props are read-only.
Forgetting a required propTypeScript error — the editor catches it before run.
Reading props.keykey is for React's diff algorithm, not accessible inside.
Same key twice in a listReact warning + buggy updates. Each key must be unique among siblings.

Section takeaways

  • Props are arguments to a component. Read-only inside.
  • Type them with type Props = { ... } and destructure in the signature.
  • Use ? for optional, = in destructuring for defaults.
  • Lists use .map() + a unique key on each item.
  • children is the special prop for whatever's between the tags.
Section 08 · State

State with useState

Data that changes — and tells React to redraw.

🔁 Why state exists

Props come from outside and don't change inside the component. But pages aren't static — counters click, forms type, toggles flip. Plain variables won't trigger a redraw, because React doesn't know they changed. State is data React watches: when it changes, React runs your component again and updates the screen.

useState is the simplest hook. It gives you a value, and a way to update it.

counter.tsx
import { useState } from "react"; function Counter() { const [count, setCount] = useState(0); // ↑ value ↑ setter ↑ initial value return ( <button onClick={() => setCount(count + 1)}> Clicked {count} times </button> ); }

useState(0) returns a pair: the current value (count) and a setter (setCount). The names are yours — it's just array destructuring.

Section 09 · Re-renders

The re-render mental model

Understand this and useState stops being magic.

🔄 What happens when you call the setter

  1. You call setCount(1).
  2. React schedules a re-render of Counter.
  3. React calls your Counter() function again — top to bottom.
  4. This time, useState hands back 1 instead of 0.
  5. Your function returns new JSX. React diffs it against the old JSX.
  6. Only the parts that actually changed get applied to the real DOM.
🛑
Don't mutate state directly. Writing count++ or count = count + 1 doesn't tell React anything changed. Always call the setter: setCount(count + 1).
📐
For arrays and objects in state, make a new one. Don't topics.push("X") — do setTopics([...topics, "X"]). React detects "this is a different array" and re-renders. Mutating in place won't trigger anything.

📌 Three rules for state

Use the setter

  • Always go through the setter function.
  • count++ is a silent bug.

Setting → re-render

  • Calling the setter schedules React to re-run the component.
  • You don't need to update the DOM yourself.

Per-instance

  • Two <Counter />s have two separate counts.
  • Each component instance has its own state.

Section takeaways

  • useState(initial) returns [value, setter].
  • Calling the setter triggers a re-render.
  • Never mutate state in place — for arrays/objects, create new ones.
  • Each component instance has its own independent state.
  • We'll do more hooks (useEffect, custom hooks, etc.) in Class 07.
