Section 01

Why TypeScript?

The short version: it's JavaScript plus a type checker that catches mistakes before you run the code.

🛡 Catching bugs before the user does

Plain JavaScript only finds errors at runtime — meaning your code has to actually execute the broken line for you to notice. TypeScript runs a check when you save the file, and tells you about problems right there in the editor.

plain JS — silent bug
function greet(user) { return "Hello, " + user.name; } greet(42); // "Hello, undefined" — no error, just wrong output
TypeScript — caught instantly
function greet(user: { name: string }) { return "Hello, " + user.name; } greet(42); // ✕ Argument of type 'number' is not assignable

🎯 Benefits

  • Bugs caught at compile time
  • Editor autocomplete everywhere
  • Safer refactors across files
  • Self-documenting function signatures

⚙️ How it works

  • You write .ts files
  • The TS compiler checks types
  • Types are stripped out
  • Plain JS is what actually runs
💡
Key mental model: types are erased at build time. At runtime, your code is just JavaScript — the browser and Node never see the types. TypeScript is a tool for you, not for the computer.

For this class, we'll use the TypeScript Playground — a browser-based editor where you type TS on the left and see the compiled JS (and any errors) on the right. No install needed.

Section 02

Basic types & annotations

The building blocks — primitives, arrays, objects.

🔤 Primitives

TypeScript has names for each of JavaScript's built-in value types. You write them after a colon — that's a type annotation.

primitives.ts
const name: string = "Karma"; const age: number = 24; const active: boolean = true; const none: null = null; const notSet: undefined = undefined;

📋 Arrays & objects

Arrays have a type too. You can write them as string[] (the common style) or Array<string> (equivalent, slightly more formal).

arrays-objects.ts
// Arrays const tags: string[] = ["dev", "qa"]; const scores: Array<number> = [88, 92]; // Objects — inline type const trainee: { name: string; age: number } = { name: "Karma", age: 24 };

Inline object types work fine for one-off uses. When the same shape shows up in two or more places, give it a name with a type alias — that's the next section.

🔍 Type inference — let TS figure it out

In most places, TypeScript can tell the type from the value. You don't need to write it.

inference.ts
// TS infers these automatically const name = "Karma"; // : string const age = 24; // : number const tags = ["dev", "qa"]; // : string[] // But annotate when inference isn't enough: const items: string[] = []; // empty array → TS infers any[] without annotation
💡
Rule of thumb: let TS infer values, but always annotate function parameters. Parameters are the one place TS can't guess what you meant.
Section 03

Typing functions

Two annotations: one per parameter, one for the return type.

⚡ Function signatures

functions.ts
// Regular function — params + return type function add(a: number, b: number): number { return a + b; } // Arrow function — same idea const greet = (name: string): string => { return `Hello, ${name}`; }; // No return value? Use void function log(msg: string): void { console.log(msg); } // Optional parameter (notice the ?) function greetMaybe(name?: string): string { return name ? `Hi ${name}` : "Hi!"; } // Default value — type is inferred from the default function greetDefault(name = "friend") { return `Hi ${name}`; }

The return type is often inferred from the return statement, but it's a good habit to annotate it for any function someone else might call. That way, if you accidentally change the return shape, TS tells you immediately.

Section 04

any · unknown · void

Three special types. One to avoid, one to use carefully, one for return types.

🪤 The three escape hatches

any

  • Turns off all type checks
  • You can do anything with it
  • Avoid it. Every any is a future bug

unknown

  • "Could be anything"
  • You must check the type first
  • Use for API responses, JSON

void

  • "Returns nothing"
  • For side-effect-only functions
  • Common on event handlers
safe-unknown.ts
// unknown is like any — but safer function handle(data: unknown) { // You CAN'T just call methods: // data.toUpperCase(); // ✕ error // You MUST narrow the type first: if (typeof data === "string") { data.toUpperCase(); // ✓ inside the check, TS knows it's a string } }

Section takeaways

  • Annotate function parameters, let TS infer values
  • string[], number[] for arrays; inline { } for one-off object shapes
  • Annotate return types on functions you export / reuse
  • Avoid any. Use unknown when you genuinely don't know the type yet.
Section 05

Naming shapes with type

Give names to the shapes you reuse — objects, unions, functions, all with one keyword.

📦 type — name an object shape

trainee.ts
type Trainee = { name: string; age: number; vertical: string; }; const t: Trainee = { name: "Karma", age: 24, vertical: "Dev" }; // Reuse everywhere function describe(t: Trainee): string { return `${t.name} is in ${t.vertical}`; } const all: Trainee[] = [t];

Read it as: "Trainee is another name for this shape." Once named, you can use Trainee anywhere you'd otherwise repeat the object shape.

🔀 type does more than just objects

The real win of type is that one keyword handles everything — object shapes, unions, literal unions, primitives, function signatures. You use the same grammar for every kind of type definition.

type-everywhere.ts
// Object shape type Trainee = { name: string; age: number; }; // Unions type ID = string | number; // Literal unions type Status = "loading" | "success" | "error"; type Vertical = "Dev" | "QA" | "BA"; // Function signatures type Click = (e: MouseEvent) => void;

🏢 SELISE convention — type everywhere

TypeScript also has an interface keyword. For object shapes, interface and type work equivalently — but interface can't describe unions, primitives, or function aliases, and it silently allows "declaration merging" which can surprise readers.

At SELISE, we use type for everything. One keyword, no surprises, consistent grammar across all files.

Featuretypeinterface
Object shapes✓✓
Unions & primitives✓✕
Function typescleanvia method syntax
One keyword for every kind of type✓✕
Declaration merging (surprising)—yes
🏢
SELISE convention: use type for all type definitions — objects (including React props), unions, literal unions, function types. You'll see interface in external libraries and other codebases — that's fine, but in your own code at SELISE, always reach for type.
Section 06

Optional & readonly

Two small modifiers that do a lot of work.

🎯 ? — optional properties

A property followed by ? is allowed to be missing. Whenever you access it, TS treats the value as T | undefined — so you're forced to handle both cases.

optional.ts
type Trainee = { name: string; email?: string; // optional age?: number; }; const t: Trainee = { name: "Karma" }; // ✓ no email/age // You must handle the undefined case const upper = t.email?.toUpperCase(); // optional chaining const display = t.email ?? "no email"; // nullish coalescing

🔒 readonly — can't reassign

readonly.ts
type Trainee = { readonly id: number; // can't change after set name: string; }; const t: Trainee = { id: 1, name: "Karma" }; t.name = "Sonam"; // ✓ t.id = 2; // ✕ Cannot assign to readonly property
Section 07

Unions & literal types

"One of several" — the most powerful everyday TypeScript pattern.

🔀 Union types

A union type uses the | operator to say "the value can be any one of these types."

union.ts
type ID = string | number; function format(id: ID): string { return `ID: ${id}`; } format(42); // ✓ format("abc"); // ✓ format(true); // ✕ // Extremely common: value-or-null while loading let user: Trainee | null = null;

🎯 Literal types

Instead of saying "any string", you can say "only this exact string". Combined with unions, this is how you build enums-on-the-cheap.

literal.ts
type Align = "left" | "right" | "center"; type Status = "loading" | "success" | "error"; type Vertical = "Dev" | "QA" | "BA" | "Design"; function setAlign(a: Align) { /* … */ } setAlign("center"); // ✓ setAlign("centre"); // ✕ catches the typo
💡
Your editor will autocomplete the allowed values — so you literally cannot typo them. This is one of the most loved features of TS.
Section 08

Type narrowing

How you teach TypeScript what you know.

🎯 typeof, in, truthy

When a value is a union, TypeScript won't let you use methods that exist on only one side. You have to narrow the type first — usually with a check.

narrowing.ts
// typeof narrowing — for primitives function printId(id: string | number) { if (typeof id === "string") { console.log(id.toUpperCase()); // TS: id is string here } else { console.log(id.toFixed(2)); // TS: id is number here } } // "in" narrowing — for objects type Dog = { bark: () => void }; type Fish = { swim: () => void }; function move(a: Dog | Fish) { if ("bark" in a) a.bark(); else a.swim(); } // Truthy narrowing — user | null function greet(user: Trainee | null) { if (!user) return "hello, stranger"; return `hello, ${user.name}`; // TS knows user is Trainee here }

⭐ Discriminated unions

One of the most powerful patterns. Give each shape a distinct "tag" property (often called status, kind, or type). TS uses the tag to figure out which shape you have.

discriminated.ts
type Request = | { status: "loading" } | { status: "success"; data: string } | { status: "error"; error: string }; function render(r: Request) { if (r.status === "success") return r.data; // ✓ if (r.status === "error") return r.error; // ✓ return "Loading…"; }
🚀
This is exactly how you'll model API loading states in React. Instead of three booleans (isLoading, isError, isSuccess), you have one tagged type — and impossible states become impossible.
Section 09

Generics

Types as parameters — one function, many types.

🧬 The <T> syntax

Sometimes you want to write a function that works for any type, but should preserve the specific type in the return. Generics let you do that.

generic.ts
function first<T>(arr: T[]): T | undefined { return arr[0]; } const n = first([1, 2, 3]); // number | undefined const s = first(["a", "b", "c"]); // string | undefined // T is a placeholder. TS fills it in from the argument.

You've already used generics without realising it:

everyday-generics.ts
// Array<T> is a generic built into TS const names: Array<string> = ["Karma"]; // Promise<T> — resolves to a value of type T async function load(): Promise<number> { return 42; } // In React, you'll write: // const [count, setCount] = useState<number>(0); // const [user, setUser] = useState<User | null>(null);
Section 10

Utility types

Built-in generic types that transform other types.

🛠 Partial · Required · Pick · Omit

Once you have a Trainee type, utility types let you derive related types from it instead of duplicating. Fewer types to maintain, no drift between them.

Partial<T>

  • Every property becomes optional
  • Great for "update" operations

Required<T>

  • Every optional becomes required
  • Opposite of Partial

Pick<T, K>

  • Keep only the listed properties
  • e.g. Pick<User, "name" | "email">

Omit<T, K>

  • Keep everything EXCEPT the listed
  • e.g. Omit<User, "password">
utilities.ts
type Trainee = { id: number; name: string; email: string; vertical: string; }; // Form for creating — no id yet type NewTrainee = Omit<Trainee, "id">; // Patch request — anything optional type TraineeUpdate = Partial<Trainee>; // Public view — only safe fields type TraineePublic = Pick<Trainee, "name" | "vertical">;

⚛️ Where this all leads — React props

TraineeCard.tsx
type TraineeCardProps = { name: string; // primitive vertical: "Dev" | "QA" | "BA"; // literal union tags?: string[]; // optional array onClick?: () => void; // optional function variant?: "default" | "compact"; // literal union }; function TraineeCard(props: TraineeCardProps) { return ( <div> <strong>{props.name}</strong> <small>{props.vertical}</small> </div> ); }

Every piece of this component props type — primitives, literal unions, optional properties, function signatures — is something you learned today. That's the point.

Today's four big ideas

Everything you need to start writing typed React components.

🔤 Basic types

Primitives, arrays, function signatures. Let TS infer values; annotate parameters. Avoid any.

📦 Naming shapes with type

One keyword for everything — objects, unions, primitives, functions. Use ? for optional, readonly for immutable. SELISE convention: type everywhere.

🔀 Unions & narrowing

A | B for "one of several". Literal strings. Narrow with typeof, in, truthy checks, and discriminator tags.

🧬 Generics & utilities

<T> makes functions work for any type. Partial, Pick, Omit derive new types without duplication.

Practice

Home exercises

Six problems you already solved in plain JavaScript during Class 03b. Now solve them again in TypeScript. Use TS Playground or a .ts file in VS Code.

🎯
Focus on parameters and return types. Most other types will be inferred. When you're done, look at the compiled JS tab — notice that all the types disappear. That's normal.

Ex 1 · from 1.2 Grade classifier

Write a function that takes a score (0–100) and returns a grade letter.

  • Signature: classify(score: number): "A" | "B" | "C" | "D" | "F"
  • 90+ → A · 80–89 → B · 70–79 → C · 60–69 → D · below 60 → F
  • Why this matters: the literal union return type prevents you from accidentally returning "E" or a lowercase "a".

Ex 2 · from 1.5 Celsius ↔ Fahrenheit

Two typed arrow functions.

  • const cToF = (c: number): number => …
  • const fToC = (f: number): number => …
  • Formulas: F = C × 9/5 + 32 · C = (F − 32) × 5/9

Ex 3 · from 2.2 Largest in array

Find the largest number in an array.

  • Signature: largest(arr: number[]): number
  • What should happen if the array is empty? Try throwing, or changing the signature to number | undefined and handling it.

Ex 4 · from 2.6 Palindrome checker

Return true if the string reads the same forwards and backwards.

  • Signature: isPalindrome(s: string): boolean
  • Ignore case: "Racecar" should count as a palindrome.

Ex 5 · from 3.1 Build a Trainee type

Define a proper type alias and create two example trainees.

  • Define type Trainee = { name: string; age: number; vertical: string; }
  • Create two variables typed as Trainee, each with all three fields filled in.
  • Bonus: add an optional email?: string and only set it on one of them.

Ex 6 · from 3.4 Find trainee by name

Given an array of trainees, find one by name.

  • Signature: findByName(ts: Trainee[], name: string): Trainee | undefined
  • Why | undefined? Because the name might not exist. TS forces the caller to handle that case.
  • Hint: use the array method .find.
