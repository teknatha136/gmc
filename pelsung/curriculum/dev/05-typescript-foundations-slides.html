04 Why TypeScript?
Motivation

JavaScript + types = confidence

plain JS · crashes at runtime
function greet(user) { return "Hello, " + user.name; } greet(42); // 💥 "Hello, undefined" — no warning
TypeScript · caught before you run
function greet(user: { name: string }) { return "Hello, " + user.name; } greet(42); // ✕ Argument of type 'number' is not…
  • Catch bugs at compile time — not when a user complains.
  • Editor autocomplete — you always know what a variable can do.
  • Safer refactors — rename a property, TS shows every place to fix.
  • Self-documenting code — types tell the next reader what to expect.

Mental model

Key insight

Types are erased at build time. At runtime, your program is just JavaScript. Types are a tool for you — not for the browser.

08 Primitive types
Building blocks

The six you'll use all the time

primitives.ts
const name: string = "Karma"; const age: number = 24; const isActive: boolean = true; const nothing: null = null; const notSet: undefined = undefined; // Arrays — pick ONE of these two styles const tags: string[] = ["dev", "qa"]; const scores: Array<number> = [88, 92];

The : type after a variable is called a type annotation.

09 Type inference
Less typing, same safety

TS figures out most types

inference.ts
// No annotation needed — TS infers these const name = "Karma"; // : string const age = 24; // : number const tags = ["dev", "qa"]; // : string[] // But you DO want to annotate: // • function parameters // • when the initial value is wrong (e.g. empty array) const items: string[] = []; // without annotation: any[]

Rule of thumb: let TS infer values, but always annotate function parameters.

10 Typing functions
Parameters + return

Every function, two parts

functions.ts
// Regular function function add(a: number, b: number): number { return a + b; } // Arrow function — same idea const greet = (name: string): string => { return `Hello, ${name}`; }; // No return value? Use void function log(msg: string): void { console.log(msg); }

Return type is often inferred, but annotating it is a good habit for public functions.

11 Object types
Inline shapes

Describe an object inside the type

objects.ts
function showTrainee(t: { name: string; age: number }) { console.log(t.name, t.age); } showTrainee({ name: "Karma", age: 24 }); // ✓ showTrainee({ name: "Tek" }); // ✕ missing age showTrainee({ name: "Tek", age: "22" }); // ✕ age is string

Inline types are fine for one-off uses. When you reuse the same shape, give it a name — which is the next section.

12 any · unknown · void
Special types

When you don't know the type yet

any
Turns off type checking. Avoid. It's an escape hatch — every any is a future bug.
unknown
"Could be anything" — but TS forces you to check before using it. Use for API responses, JSON parsing.
void
"Returns nothing" — use as the return type of functions that only side-effect (log, save, etc).
safe.ts
function handle(data: unknown) { if (typeof data === "string") { data.toUpperCase(); // ✓ inside the check, TS knows it's a string } }

02
16 type
Name an object shape

type

trainee.ts
type Trainee = { name: string; age: number; vertical: string; }; const t: Trainee = { name: "Karma", age: 24, vertical: "Dev" }; function describe(t: Trainee): string { return `${t.name} is in ${t.vertical}`; }

Now Trainee is a reusable shape. Every place that accepts a Trainee uses the same definition.

17 type does more
One keyword, every kind of type

Why we use type at SELISE

type-everywhere.ts
// Object shapes type Trainee = { name: string; age: number; }; // Unions & primitives type ID = string | number; // Literal unions type Vertical = "Dev" | "QA" | "BA"; // Function types type Click = (e: MouseEvent) => void;
typeinterface
Object shapes✓✓
Unions / primitives / function types✓✕
One keyword for everything✓✕
Declaration merging (can be surprising)—yes

SELISE convention: use type for everything — object shapes, unions, props, all of it.

interface exists and is valid TypeScript — you'll see it in other codebases. At SELISE we prefer the consistency of one keyword.

18 Optional & readonly
Two modifiers you'll use daily

? and readonly

modifiers.ts
type Trainee = { readonly id: number; // can't change after set name: string; email?: string; // optional — may or may not exist age?: number; }; const t: Trainee = { id: 1, name: "Karma" }; // ✓ no email/age t.id = 2; // ✕ Cannot assign to readonly t.email?.toLowerCase(); // ✓ safe access (optional chaining)

Optional props combine with optional chaining (?.) — TS forces you to handle undefined.

Unions
& Narrowing

"Either this, or that — here's how to tell"

03
26 Union types
"One of several types"

The | operator

unions.ts
// An ID could be a number or a string type ID = string | number; function format(id: ID): string { return `ID is ${id}`; } format(42); // ✓ format("u-007"); // ✓ format(true); // ✕ boolean not allowed

Unions are everywhere — especially in React state, where a value might be the data OR null while loading.

27 Literal types
Exact string/number values

Pin it down to specific values

literals.ts
type Align = "left" | "right" | "center"; type Status = "loading" | "success" | "error"; type Vertical = "Dev" | "QA" | "BA" | "Design"; function setAlign(a: Align): void { // … } setAlign("center"); // ✓ setAlign("centre"); // ✕ not allowed — TS catches the typo

The editor autocompletes the allowed values — very useful for UI components.

28 Narrowing with typeof
Teaching TS what you know

Type narrowing

narrowing.ts
function printId(id: string | number) { if (typeof id === "string") { // inside this block, TS knows id is a string console.log(id.toUpperCase()); } else { // TS knows id must be a number here console.log(id.toFixed(2)); } }

typeof, in, and truthy checks all narrow the type. TS follows your checks and updates the type inside each branch.

29 Narrowing with "in"
Object discriminators

Check for a property

in-operator.ts
type Dog = { bark: () => void }; type Fish = { swim: () => void }; function move(animal: Dog | Fish) { if ("bark" in animal) { animal.bark(); // TS: it's a Dog } else { animal.swim(); // TS: it's a Fish } }
30 Discriminated unions
Common in React state

A "tag" property picks the shape

discriminated.ts
type Request = | { status: "loading" } | { status: "success"; data: string } | { status: "error"; error: string }; function render(r: Request) { if (r.status === "success") { return r.data; // ✓ TS knows data exists here } if (r.status === "error") { return r.error; // ✓ } return "Loading…"; }

This pattern is how you'll model loading/success/error states in React.

34 Why generics?
The motivation

One function, many types

Analogy

Imagine a box. You'd like a box that holds anything — books, apples, phones — but once you put something in, TypeScript should remember what's inside. Generics let you describe that.

without generics
// We'd need to write this for every type we store function firstString(arr: string[]): string { return arr[0]; } function firstNumber(arr: number[]): number { return arr[0]; } // 🙃
35 A first generic
The <T> syntax

Type as a parameter

generic.ts
function first<T>(arr: T[]): T | undefined { return arr[0]; } const n = first([1, 2, 3]); // : number | undefined const s = first(["a", "b", "c"]); // : string | undefined // T was filled in automatically based on the argument

T is a placeholder. TS fills it in each time you call the function — keeping the return type linked to the input type.

36 Where you've already seen this
React connection

Array<T> · Promise<T> · useState<T>

everyday-generics.ts
// Array of strings — two ways to write it const names: string[] = ["Karma"]; const also: Array<string> = ["Karma"]; // A promise that resolves to a number async function count(): Promise<number> { return 42; } // Coming in React: // const [count, setCount] = useState<number>(0); // const [user, setUser] = useState<User | null>(null);
37 Utility types
Built-in generic types

Transform a type into a new one

Partial<T>
Every property of T becomes optional. Great for "update" functions.
Required<T>
Opposite of Partial — every optional becomes required.
Pick<T, K>
Keep only the listed properties. Pick<Trainee, "name" | "age">
Omit<T, K>
Keep everything EXCEPT the listed properties.
38 Utility types in action
Real example

Reuse, don't rewrite

utilities.ts
type Trainee = { id: number; name: string; email: string; vertical: string; }; // Create form: no id yet — derive the type type NewTrainee = Omit<Trainee, "id">; // Update form: everything optional type TraineeUpdate = Partial<Trainee>; // Public summary: only name + vertical type TraineeSummary = Pick<Trainee, "name" | "vertical">;
