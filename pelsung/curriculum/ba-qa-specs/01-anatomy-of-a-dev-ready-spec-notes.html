Part 1

Why specs matter — the dev-ready test

Here's the simplest test for a spec.

Hand the spec to a mid-level engineer. Watch what they do next.

  • If they open their editor and start coding — the spec is dev-ready.
  • If they schedule a meeting — the spec is not.

Most specs fail this test. Not because the author was lazy, but because the structure wasn't there. This class is about that structure — the 12 sections that show up in every solid spec across the industry, whether it's a formal IEEE-style SRS or a lightweight feature ticket in Linear.

Learn the 12 sections and the three traditions underneath, and you'll be able to write specs that engineers — and AI — can actually build from.

Part 2

Spec · Story · PRD · SRS — what's the difference?

The words sound confusingly similar. They're actually differences of scope and depth, not structure. Under the hood, they share the same 12 sections.

NameScopeLengthWhen used
User Story One user-visible feature < 1 page Sprint-level planning; lives in Jira / Linear
Feature Spec One feature, fully specified (story + UX + data + edges + NFRs) 2–5 pages Day-to-day dev work; our main teaching focus
PRD A product area or major release — multiple features + rationale + metrics 10–30 pages Quarterly planning, new product launches
SRS A whole system, exhaustive 30–100+ pages Formal / regulated environments (banking, medical, aviation, defence)

Throughout this class, when we say "spec" we mean the feature spec — the level you'll use in daily sprint work. But everything we teach scales up to PRD and SRS; you just add more depth per section.

Part 3

Which format when? — the decision table

Before you pick up a template, know which one you're reaching for. The wrong format at the wrong time is the top-one cause of wasted spec effort — a 50-page SRS for a one-sprint feature is as broken as a one-paragraph story for a nationwide banking system.

Pick by what you're trying to ship

If you're……use thisWhy
Shipping one user-visible feature next sprint Feature Spec Sprint-sized. 2–5 pages. Dev can start coding in 20 minutes.
Planning a release or product area (multiple features) PRD Aligns PM · Eng · Design · GTM. Includes metrics, rollout, launch plan.
Building for a regulated industry (bank, medical, aviation), or responding to a government RFP SRS (IEEE 29148) Formal, exhaustive, audit-traceable. Required by regulation or contract.
Getting leadership to approve the idea before any solution exists BRD Business language only. No systems named. Pre-solution artifact.
Designing the architecture of a new system or module HLD Boxes, arrows, integrations, NFR-driven choices. Architect-level.
Designing the internals of one component before coding LLD Class signatures, SQL, error taxonomy, concurrency. Implementation-ready.
Planning testing for a project / release Test Plan (IEEE 829) Scope of testing, entry/exit, environments, responsibilities, schedule.

Rule of thumb

Start from the smallest document that captures what you need. A BRD is not "better" than a Feature Spec — it answers a different question (should we? vs how exactly?). Pick by the decision you're trying to unblock.

Part 4

Industry standards · three traditions

Every shop draws from one of three traditions. They look different on the surface — the underlying skeleton is the same.

Formal · since 1984
IEEE 830 / ISO/IEC/IEEE 29148

The formal SRS (Software Requirements Specification) standard. Heavy structure: introduction → overall description → specific requirements → appendices. The current version is ISO/IEC/IEEE 29148:2018, which supersedes IEEE 830.

You'll see it in: banks, medical device makers, aviation, defence contractors, government tenders. Often mandated by regulation.

Modern tech · since ~2005
PRD (Product Requirements Document)

The tech-industry convention. Lighter than IEEE, more user-focused. Popularised by Google, Atlassian, Intercom, Shape Up (Basecamp), and templates circulating in product-management circles. Same core sections, friendlier tone, more emphasis on users and metrics.

You'll see it in: SELISE, startups, most product-led companies. The default outside regulated industries.

Agile · since ~2010
Feature Spec / BDD

Per-feature, story-sized. Given/When/Then acceptance criteria at the core (borrowed from Behaviour-Driven Development). Often lives inside a Jira or Linear ticket, a Markdown file in the repo, or a Notion page.

You'll see it in: daily sprint work, anywhere the team writes many small specs a week rather than a few big ones.

The same skeleton underneath

Strip any of these down and you find the same 12 sections — they just differ in tone, depth, and length. Learn the 12 sections and you can write fluently in all three traditions.

Trace one concept across all three

Take "non-functional requirements". In an SRS it's §3.3 ("Performance · Safety · Security · Quality attributes · Business rules"). In a PRD it's §9 ("Requirements · release-level NFRs"). In a Feature Spec it's §08 ("NFRs relevant to this feature"). Same concept. Different depth. Same idea: name the NFRs explicitly so silence doesn't bite you.

Part 7

Every dev-ready spec has the same 12 sections, in the same order. Think of them as four zones:

  • Header (1–5) — who/what/why: metadata, TL;DR, problem, goals, users
  • Requirements (6–8) — the contract: functional, acceptance, non-functional

  • Scope (12) — the "we're not doing this" list

Part 8

Header zone · metadata to personas

§ 01 · Metadata

The spec's "ID card"

A short key-value block at the very top. Any reader should know in five seconds what they're looking at and whether it's current. Skimping on metadata is the cause of at least half of all "oh, I was reading an old version" conversations.

Include:

  • Title — feature name, product, version. "Biller Favourites · mBoB · v1"
  • Status — Draft / Review / Approved / Implemented / Archived
  • Version — semantic (0.1, 0.3, 0.7, 1.0). Bump on every meaningful change.
  • Author(s) — who owns this document
  • Reviewers — dev lead, QA lead, design lead, stakeholder
  • Last updated — single date that changes on every edit
  • Ticket / epic link — Jira, Linear, or GitHub issue
§ 02 · TL;DR

Two or three sentences. That's it.

A senior leader should be able to read only the TL;DR and know what's being built, for whom, and why. If someone skips the whole spec and reads only the TL;DR, they should still be able to have a correct hallway conversation about it.

TL;DR — We let mBoB users save up to 10 repeat billers as favourites on the Bill Payments home. One-tap pre-fill, MPIN still required. Solves the 36-per-year re-entry problem for the 62% of users who pay the same billers every month. Ships v1 end of May.

Three sentences. Names the feature, the user, the mechanism, the problem, the evidence, and the timing. A good TL;DR is dense.

§ 03 · Problem Statement

Whose pain, how big, with evidence

One tight paragraph. Name the user segment, describe the pain concretely, quantify it if possible, cite evidence (analytics, support tickets, research). Then say why this is worth solving now.

62% of mBoB users pay the same 3+ billers every month (Q1 2026 analytics). Each payment requires re-typing a 12-digit account number. Support volume: 8% of all tickets reference "can I save billers". Pain compounds: ~36 re-entries per active user per year.

The most common problem-statement sin

Describing the solution ("users need favourites") instead of the pain ("users re-type account numbers 36× a year"). If your problem statement names a solution, you've already narrowed the design space — probably accidentally.

§ 04 · Goals & Non-goals

What success looks like — and what it doesn't

Goals are the measurable outcomes that will tell you the feature worked. Non-goals are the features adjacent stakeholders might expect to be included but aren't. Non-goals are as important as goals — without them, scope quietly balloons.

  • Goals: Reduce re-entry time for repeat billers by ≥80%. Hit 40% weekly active use within 2 months. Support volume on "save billers" drops to <1%.
  • Non-goals: Sharing favourites between users. Auto-pay from a favourite. Moving money between bank accounts. Support for legacy app versions.
§ 05 · Users & Personas

Design for specific humans

Name 2–4 concrete personas. Give them names, details, context, constraints. If a feature works for your three named personas, it works for millions. If you can only name "the user", your spec is wearing a blindfold.

Primary — Tashi (power user) Thimphu · 34 · shopkeeper · pays 30+ bills/month Wants speed. Will learn shortcuts. Annoyed by extra taps. Secondary — Aum Pema (conservative user) Trashigang · 58 · retired civil servant Dzongkha. Big buttons. Uses mBoB for exactly 2 things. Must not be surprised by new UI. Edge — Karma (student abroad) Australia · 22 · may not have Bhutan SIM Logs in over Wi-Fi; performance on slow networks matters.

Once named, these personas should appear in acceptance criteria, UX reviews, and trade-off decisions. "Would Aum Pema get confused by this?" is the kind of question a persona section makes possible.

Part 9

Requirements zone · FR · AC · NFR

§ 06 · Functional Requirements

User stories, flows, behaviour

A numbered list of user stories the feature delivers. Each is "As a … I want … so that …", with the so that explaining the real goal so dev and AI can make smart trade-offs. One feature = multiple FRs; if a single FR takes more than a sentence, split it.

  • FR-01 · Save a paid biller as a favourite (from receipt screen)
  • FR-02 · View favourites on Bill Payments home
  • FR-03 · Tap a favourite to pre-fill a payment
  • FR-04 · Remove a favourite
  • FR-05 · Edit alias / nickname on a favourite
§ 07 · Acceptance Criteria

Given / When / Then · pass or fail

Every FR gets 1–3 AC, written in Given/When/Then form. Each AC must be specific, observable, testable. AC is covered in depth in Combined Class 04 — Acceptance Criteria. In the spec itself, AC lives here.

AC-01 · Given user completed a payment, When they tap ⭐ Save, Then biller appears in Favourites within 1 second. AC-02 · Given user has 10 favourites already, When they tap ⭐ Save, Then system shows "Limit reached" and adds nothing. AC-03 · Given user is offline, When they tap ⭐ Save, Then save is queued locally and shows "Will sync" toast; syncs when connection returns.
§ 08 · Non-Functional Requirements

How well the system must do it

Covered in depth in Combined Class 03 — Functional vs Non-Functional Requirements. In the spec, list the NFRs relevant to this feature — not the whole system. Use numbers, not adjectives.

  • Performance — Favourites list renders <1s on 4G, p95.
  • Security — Favourites scoped to logged-in user; no cross-account leakage in audit logs.
  • Reliability — Offline save queues locally; syncs when online; never silently drops.
  • Accessibility — ⭐ button labelled for screen readers; colour-independent cue (star shape).
  • Scalability — Supports 10k concurrent users on pay-day without degraded UX.

The sin here is silence

The common NFR failure isn't bad NFRs — it's silent NFRs. If an attribute (performance, security, a11y) isn't named in the spec, it doesn't exist as a requirement. Silent NFRs get discovered by users, at the worst possible time.

Part 10

Supporting zone · UX · data · dependencies

§ 09 · UX & UI

Where it lives and how it flows

Wireframes or a link to Figma. Low-fi is fine — the job is alignment on placement and flow, not final visual design. List every state explicitly (empty, populated, loading, error, offline) — each missing state becomes a bug later.

  • Entry point: Receipt screen after a successful payment
  • Primary screen: Bill Payments home, new "Favourites" row above "Recent"
  • States to cover: empty · populated · loading · error · offline
  • Figma: mbob-favs-v1
§ 10 · Data Model

Entities · fields · relationships · volumes

Sketch the new or changed data structures. Field names, types, relationships, constraints. Not a full ERD — just enough for dev to match on existing schema and catch naming collisions.

Favourite { id: uuid userId: fk → users.id billerId: fk → billers.id alias: string (optional, max 30) lastUsedAt: timestamp lastAmount: decimal (optional) createdAt: timestamp } Volume: avg 4 favourites per active user Max: 10 per user (enforced at app layer)
§ 11 · Dependencies, Risks, Open Questions

What could block, bite, or surprise

Three short lists. Keeping them visible is how teams avoid the "oh, we forgot about that" moment a week before ship.

  • Dependencies: biller-list API needs a new lastUsedAt sort field; depends on Design finalising the star icon.
  • Risks: some users may exceed 10 favourites before launch if we're not careful with data migration; plan a backfill cleanup script.
  • Open questions: does a favourite persist across device reinstall? (Answer: yes, server-side — confirm with Support before launch.)
Part 11

Scope · § 12 · Out of Scope

§ 12 · Out of Scope

The deadline-saving section

Every spec needs an explicit "we are not doing this" section. Without it, stakeholders quietly assume their favourite idea is included — and discover it isn't the week before ship, when it's too late to fix.

  • Sharing favourites between users
  • Categories / custom tags / icons
  • Auto-pay from a favourite
  • Support for legacy mBoB v4
  • Bulk import / export

A good out-of-scope list is a deadline-saving device. Each line potentially prevents a week of rework.

Part 12

Status lifecycle & versioning

A spec is a living document. It moves through five statuses, each with its own bar for what's allowed.

📝 Draft (versions 0.1–0.3)

Being written. Sections missing. Nobody should build from a Draft. Review comments welcome but not expected.

👀 Review (versions 0.4–0.9)

All sections present. Reviewers (dev lead, QA lead, design lead, stakeholder) are actively reading. Questions fly. Multiple minor version bumps.

✅ Approved (version 1.0)

Reviewers signed off. Dev can start. Any change now requires a version bump and at minimum a notification to reviewers. Substantive changes need re-review.

🔨 Implemented (versions 1.x)

Feature is built, tested, shipped. Spec gets "as-built" updates if reality diverged during implementation. The spec now serves as reference documentation.

📦 Archived

Feature has been superseded, retired, or replaced by a newer spec. Kept for historical context; marked clearly so nobody mistakes it for current.

Versioning convention

Use major.minor:

  • Minor bumps (0.4, 0.5, 0.6) during drafting and review — for each meaningful content change.
  • Major bumps (1.0, 2.0) when the status changes or the scope materially expands.
  • Changelog at the bottom — one line per bump, dated, with the key change. This is how a reviewer who hasn't looked at the spec for a week knows what's moved.
Part 13

Length discipline · the 20-minute rule

The 20-minute rule

A dev-ready feature spec should take a mid-level engineer 20 minutes or less to read end-to-end. If yours takes longer, either it's the wrong type (probably should be a PRD) or it's got fat to trim.

Common bloat

  • Paragraphs where a bullet list would do
  • Repeating the TL;DR three times in slightly different words
  • Embedding whole meeting transcripts
  • Speculating about future versions ("and later we might…")
  • Re-explaining company or product context everyone already knows

Discipline moves

  • Bullets over prose whenever possible
  • Diagrams where 200 words would be needed
  • Link out to background documents — don't quote them inline
  • One sentence per sentence — no nested subordinate clauses
  • Cut every "it should be noted that" and "as mentioned above"

Targets: feature spec 2–5 pages · PRD 10–30 pages · SRS larger, but never longer than necessary. Length is a cost, not a signal of quality.

Part 15

Spec author's checklist

Before you mark a spec as Ready for Review, walk this list. Each unchecked item is a future bug or rework cycle.

All 12 sections present

  • Metadata block complete — title, status, version, author, reviewers, date, ticket link
  • TL;DR is 2–3 sentences and stands alone
  • Problem names whose pain, how big, with evidence
  • Goals measurable; non-goals explicit
  • 2–4 named personas with concrete details
  • Functional requirements listed (FR-01, FR-02…)
  • Acceptance criteria in Given/When/Then; at least one edge per FR
  • NFRs named in numbers, not adjectives
  • UX/UI section names every state (including error, loading, offline)
  • Data model sketched — entities, fields, volumes, constraints
  • Dependencies, risks, open questions each listed separately
  • Out of scope is explicit and specific

    • Quality gates

  • A mid-level engineer could start coding after reading this
  • Reads end-to-end in <20 minutes
  • No adjectives ("fast", "intuitive", "user-friendly") that a QA can't test
  • No implementation leaks in AC ("inserts a row" — describe behaviour instead)
  • Every "and" in an AC Then has been split
  • Changelog present at the bottom
