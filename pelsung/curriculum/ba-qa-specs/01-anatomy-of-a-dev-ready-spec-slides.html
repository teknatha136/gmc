04 Spec, Story, PRD, SRS · what's the difference?

Different names. Different scope. Same DNA.

Story
User Story

One user-visible feature. 1 page max. "As a … I want … so that …" + 3–7 AC.

Scope: one feature
Spec
Feature Spec

A story plus UX, data, edges, NFRs, out-of-scope. 2–5 pages. Today's focus.

Scope: one feature, fully specified
PRD
Product Requirements Document

A product area or major release. Multiple features + rationale + metrics. 10–30 pages.

Scope: a release or product area
SRS
Software Requirements Specification

Formal, exhaustive. IEEE/ISO standard. Used in regulated industries (banking, medical, aviation). 30–100+ pages.

Scope: the whole system

The sections are the same; the depth grows with scope. Master the feature spec and you can scale up or down.

Industry
Standards

Three traditions. One shared anatomy.

01
07 The three traditions

Every shop draws from one of these

Formal · since 1984
IEEE 830 / ISO/IEC/IEEE 29148

The formal SRS (Software Requirements Specification) standard. Heavy structure: introduction · overall description · specific requirements · appendices. Still mandatory in banking, medical, aviation, defence.

You'll see it in regulated industries and government RFPs.
Modern tech · since ~2005
PRD (Product Requirements Document)

The tech-industry convention. Lighter than IEEE, more user-focused. Popularised by Google, Atlassian, Intercom, Lenny's Newsletter templates. Same core sections, friendlier tone.

You'll see it at SELISE, startups, most product companies.
Agile · since ~2010
Feature Spec / BDD

Per-feature, story-sized. Given/When/Then acceptance criteria at the core. Often lives inside a Jira/Linear ticket or a lightweight Markdown file.

You'll see it in daily sprint work — our main teaching focus.

Same skeleton underneath

Strip any of these down to its bones and you find the same 12 sections. The differences are tone, depth, and length — not structure. Learn the 12 sections and you can write in any of the three conventions.

10 §§ 01–02 · Metadata + TL;DR

The part that looks trivial but saves weeks

§ 01 · Metadata

The spec's "ID card"

A short block of key-value fields at the very top so any reader knows — in 5 seconds — what they're looking at and whether it's current.
  • Title — "Biller Favourites · mBoB · v1"
  • Status — Draft / Review / Approved / Implemented / Archived
  • Version — 0.3 (semantic; bump on every meaningful change)
  • Author(s) — the BA / PM / owner
  • Reviewers — dev lead, QA lead, design lead, stakeholder
  • Last updated — date; single place that changes on every edit
  • Link to ticket / epic — Jira / Linear / GitHub issue
§ 02 · TL;DR

Two or three sentences. That's it.

A senior leader should be able to read only this and know what's being built, for whom, and why. If someone skips the whole spec and only reads the TL;DR, they should still leave able to have a correct hallway conversation about it.
TL;DR — We let mBoB users save up to 10 repeat billers as favourites on the Bill Payments home. One-tap pre-fill, MPIN still required. Solves the 36-per-year re-entry problem for the 62% of users who pay the same billers every month. Ships v1 end of May.
11 §§ 03–04 · Problem + Goals & Non-goals

Direction before implementation

§ 03 · Problem Statement

Whose pain, how big, with evidence

One tight paragraph. Name the user segment, describe the pain concretely, quantify it if possible, and say why now. Evidence matters — usage data, support tickets, research.
62% of mBoB users pay the same 3+ billers every month (data: Q1 2026 usage analytics). Each payment requires re-typing a 12-digit account number. Support volume: 8% of all tickets reference "can I save billers". Pain compounds: ~36 re-entries per year per active user.
Problem statements fail when they describe the solution ("users need favourites") instead of the pain ("users re-type account numbers 36×/year").
§ 04 · Goals & Non-goals

What success looks like — and what it doesn't

✅ Goals
  • Reduce re-entry time for repeat billers by ≥ 80%
  • Hit 40% weekly active use of favourites within 2 months
  • Support volume on "save billers" drops to <1%
❌ Non-goals
  • Not: sharing favourites between users
  • Not: auto-pay from a favourite
  • Not: moving money between bank accounts
Non-goals are just as important as goals. Every stakeholder has their own "what about X?" — a visible non-goals list protects scope without requiring a meeting.
12 § 05 · Users & Personas

Design for specific humans

§ 05 · Users & Personas

"Everyone" is not a user

Name 2–4 concrete personas. Real details — age, context, constraints. If a feature works for your three named personas, it works for millions. If you can only name "the user", your spec is wearing a blindfold.
Primary persona — Tashi (power user) Thimphu · 34 · shopkeeper · pays 30+ bills/month Wants speed. Will learn shortcuts. Annoyed by extra taps. Secondary persona — Aum Pema (conservative user) Trashigang · 58 · retired civil servant Dzongkha. Big buttons. Uses mBoB for exactly 2 things. Must not be surprised by new UI. Edge persona — Karma (student abroad) Australia · 22 · may not have Bhutan SIM Logs in over Wi-Fi; performance on slow networks matters.
Stories, AC, and UX decisions should all reference these personas by name — "Can Aum Pema use this in Dzongkha?" is the kind of question a persona section makes possible.
13 §§ 06–07 · Functional Requirements + AC

What the system does — and how we'll know it does

§ 06 · Functional Requirements

User stories + flows + behaviour

A list of user stories the feature delivers. Each story is "As a … I want … so that …" — with the so that explaining the real goal, so dev and AI can make smart trade-offs.
  • FR-01 · Save a paid biller as a favourite
  • FR-02 · View favourites on Bill Payments home
  • FR-03 · Tap a favourite to pre-fill a payment
  • FR-04 · Remove a favourite
  • FR-05 · Edit alias / nickname on a favourite
One feature = multiple FRs. If a single FR takes more than a sentence to describe, it's two FRs.
§ 07 · Acceptance Criteria

Given / When / Then · pass or fail

Every FR gets 1–3 AC. Taught in depth in Combined Class 04 — here, they live inside the spec. 3–7 total AC across a feature spec is normal.
AC-01 · Given user completed a payment, When they tap ⭐ Save, Then biller appears in Favourites within 1 second. AC-02 · Given user has 10 favourites already, When they tap ⭐ Save, Then system shows "Limit reached" and adds nothing.
14 § 08 · Non-Functional Requirements

How well the system must do it

§ 08 · NFRs

Performance · Security · Reliability · Usability · Accessibility · Scalability

Taught in depth in Combined Class 03. In the spec itself, list the NFRs that apply to this feature — not the whole system. Use numbers, not adjectives.
  • Performance — Favourites list renders < 1s on 4G, p95
  • Security — Favourites scoped to logged-in user; no cross-account leakage in audit
  • Reliability — Offline save queues locally; syncs when online; never silently drops
  • Accessibility — ⭐ button labelled for screen readers; colour-independent cue
  • Scalability — Supports 10k concurrent users on pay-day without degraded UX
The sin here is silence, not vagueness. If an NFR isn't named, it doesn't exist. Silent NFRs get discovered by users.
15 §§ 09–10 · UX & Data Model

What it looks like · what gets stored

§ 09 · UX & UI

Where it lives and how it flows

Wireframes or a link to Figma. Low-fi is fine. The job is to align the team on placement and flow, not final visual design.
  • Entry point — Receipt screen after a successful payment
  • Primary screen — Bill Payments home, new "Favourites" row above "Recent"
  • States to cover — empty · populated · loading · error · offline
Every state is a separate UX consideration. Miss one and QA will file a bug.
§ 10 · Data Model

Entities · fields · relationships · volumes

Sketch the new or changed data structures. Field names, types, relationships. Not a full ERD — just enough for dev to match on existing schema.
Favourite { id: uuid userId: fk → users.id billerId: fk → billers.id alias: string (optional, max 30) lastUsedAt: timestamp lastAmount: decimal (optional) createdAt: timestamp } Volume: avg 4 favourites per active user Max 10 per user (enforced at app layer)
16 §§ 11–12 · Dependencies, Risks, Out of Scope

The "what could hurt us" sections

§ 11 · Dependencies, Risks, Open Questions

What could block, bite, or surprise

Three short lists. Keeping them visible is how teams avoid the "oh, we forgot about that" moment a week before ship.
  • Dependencies — the biller-list API needs a new lastUsedAt sort; depends on Design finalising the star icon
  • Risks — ≥ 10 favourites per user may happen before launch if we're not careful with the migration; plan a data backfill
  • Open questions — does "favourite" persist across device re-install? Answer: yes, server-side — confirm with Support before launch
§ 12 · Out of Scope

The deadline-saving list

Every spec needs an explicit "we're not doing this" section. Without it, stakeholders quietly assume their favourite idea is included — until it isn't.
  • Sharing favourites between users
  • Categories / custom tags / icons
  • Auto-pay from a favourite
  • Support for legacy mBoB v4
  • Bulk import / export

Versioning, Length

A spec is a living document.

03
18 Status lifecycle

Five statuses · every spec flows through them

0.1–0.3
📝 Draft

Being written. Sections missing. Nobody builds from this.

→
0.4–0.9
👀 Review

All sections present. QA + dev + stakeholder review. Questions fly.

→
1.0
✅ Approved

Signed off by reviewers. Dev can start. Changes need a bump + re-review.

→
1.x
🔨 Implemented

Built, tested, shipped. Spec gets "as-built" updates if reality diverged.

→
final
📦 Archived

Feature superseded or retired. Kept for historical context.

Versioning convention

Use major.minor. Minor bumps (0.4, 0.5) during drafting and review. Major bumps (1.0, 2.0) when status changes state. Keep a changelog at the bottom — one line per bump — so anyone re-opening the spec can see what moved.

19 Length discipline

A spec you can't read in 20 minutes doesn't get read

🕐 The 20-minute rule

A dev-ready feature spec should take a mid-level engineer 20 minutes or less to read end-to-end. If yours takes longer, it's either the wrong type (should probably be a PRD) or it's got fat to trim.

Common bloat

  • Paragraphs where a bullet list would do
  • Repeating the TL;DR three times
  • Embedding whole meeting transcripts
  • Speculating about future versions
  • Re-explaining company context everyone knows

Discipline moves

  • Bullets over prose wherever possible
  • Diagrams where 200 words would be needed
  • Link out to background docs — don't quote them
  • One sentence per sentence — no nested clauses
  • Cut every "it should be noted that"

Target: feature spec 2–5 pages · PRD 10–30 · SRS larger but never longer than necessary.
