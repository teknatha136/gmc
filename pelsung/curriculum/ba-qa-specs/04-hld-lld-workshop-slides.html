03 Hook
🎤 Scenario

An architect whiteboards a design.
The next day, two devs start building.
Three weeks later they realise they built different things.

Why? The design only lived in one person's head.
HLD and LLD turn that head-knowledge into a document two people can build from identically.

04 HLD vs LLD · what each covers

Two zoom levels · two purposes

🏗️ HLD · High-Level Design

  • Scope: system · boxes · integrations
  • Audience: architects · SRE · security · ARB
  • Answers: "How are the parts arranged, and why?"
  • Length: 10–25 pages
  • Diagrams: C4 Level 1 & 2

🔧 LLD · Low-Level Design

  • Scope: one component · classes · APIs · schema
  • Audience: the devs who'll write the code
  • Answers: "How is this component built inside?"
  • Length: 5–20 pages per component
  • Diagrams: C4 Level 3 & 4

HLD = the whiteboard photo. LLD = the labelled sketch on the whiteboard. Skipping a level is tempting — and expensive later.

07 § 04 · Quality attributes · the driver

Every architectural choice traces back to one of these

AttributeTargetArchitectural implication
Availability99.95% monthlyMulti-AZ active-active · no single DB primary
Latencyp95 < 250 msRead replica · cache layer · connection pooling
Throughput5 000 TPS peakHorizontal scaling · queue-based back-pressure
SecurityPCI-DSS scopedTokenisation at edge · card data never in app DB
Data residencyBhutan onlyRegion-pinned deployment · no cross-border replication
Auditability7-year retentionAppend-only event log · immutable storage + object lock

The architecture trace rule: every box, arrow, and datastore must justify itself by "because of target X". Can't justify? Not a decision — a guess.

11 § 14 · ADRs · decision records

Capture trade-offs on paper · don't re-argue them in 6 months

Biller Favourites · ADR summary

  • ADR-001 · Modular monolith (not microservices) for MVP — reduces ops cost; revisit at 2× current load.
  • ADR-002 · Serializable transactions over CQRS — simpler; latency targets still met.
  • ADR-003 · Postgres for ledger · Redis for cache. Rejected: Cassandra (overkill) · MongoDB (consistency).
  • ADR-004 · Region-pinned to Bhutan per RMA §7; no multi-region DR because of residency constraint.

Each ADR is a ~1-page document. The HLD links to it; nothing gets re-argued because the record exists.

14 § 04 · API contract & § 05 · schema
POST /api/v1/favourites -> 201 Favourite | 409 LimitReached | 409 Duplicate GET /api/v1/favourites -> 200 Favourite[] DELETE /api/v1/favourites/{id} -> 204 | 404 PATCH /api/v1/favourites/{id} -> 200 | 404 | 400 ValidationError CREATE TABLE favourites ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE, biller_id UUID NOT NULL REFERENCES billers(id), alias VARCHAR(30), created_at TIMESTAMPTZ NOT NULL DEFAULT now(), CONSTRAINT favourites_user_biller_unique UNIQUE (user_id, biller_id) ); CREATE INDEX favourites_user_idx ON favourites (user_id, last_used_at DESC);

Concrete enough that two engineers would produce near-identical code.

15 § 08 · Error taxonomy · the often-missed section

Every failure mode · every HTTP code · every caller can distinguish

ErrorWhenHTTPUser-facing?
VALIDATIONBad input (alias > 30)400Yes
UNAUTHORIZEDMissing / invalid token401Yes
FORBIDDENValid token · no permission403Yes
NOT_FOUNDID doesn't exist or not owned404Yes
LIMIT_REACHEDUser already has 10409Yes
DUPLICATE(user, biller) already exists409Yes
DB_UNAVAILABLETransient DB failure503Generic
INTERNALUnexpected500Generic

Without this, every endpoint returns a different shape · mobile clients can't distinguish errors · support can't correlate logs.

16 § 09 · Concurrency · race conditions on paper

Two concurrent creates at count = 9 → could both commit

function create(userId, billerId, alias): validateAlias(alias) // trim · len ≤ 30 · else ValidationError BEGIN TRANSACTION (SERIALIZABLE) existing = repo.countByUser(userId) if existing >= 10: ROLLBACK; return Err(LIMIT_REACHED) duplicate = repo.findByUserAndBiller(userId, billerId) if duplicate: ROLLBACK; return Err(DUPLICATE) row = repo.insert({ userId, billerId, alias, createdAt: now() }) COMMIT emit("favourite.created", row) // via outbox + relay (at-least-once) return Ok(row)

On paper: serializable isolation catches the race · one client sees 409 · retry-safe via idempotency key. In production: no 11th-favourite surprise.

17 How a BA reviews an HLD / LLD

Not approving tech choices — adding a different lens

HLD review · tick list

  • Every FR has a home — which component owns it?
  • Every NFR → architectural implication or a stated rationale
  • Context diagram matches the spec's external systems
  • BRD / spec constraints appear (region · regulation · budget)
  • Out-of-scope matches spec's out-of-scope
  • Open questions have owners + deadlines
  • ADRs exist for every non-obvious tech choice

LLD review · tick list

  • API contract matches AC behaviour
  • Error taxonomy covers all AC failure paths
  • Data model respects spec constraints (max 10 · unique)
  • Concurrency reasoning exists for races I can think of
  • Audit logging meets compliance reqs
  • Feature flag / rollback plan exists
  • Tests cover every FR + NFR in this component

  • ✅ Design after code is a post-hoc essay — not a design.
