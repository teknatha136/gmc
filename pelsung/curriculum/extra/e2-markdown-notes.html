Part 1

Why Markdown matters

Everyone in software writes. Developers write READMEs. QAs write bug reports. BAs write specs and user stories. Designers write annotations. Everyone writes prompts to AI tools.

For all of these, Word documents are the wrong tool. They don't work inside GitHub, inside Jira, inside Slack, or inside Claude Code. Markdown does, and Markdown is everywhere in software. Learning it once pays off for the rest of your career.

The one-minute argument

Markdown is plain text with a handful of tiny marks. Any text editor can make it. GitHub, Jira, Linear, Slack, Notion, Obsidian, Claude Code, ChatGPT — all read and write it. You learn it once. You use it everywhere.

Part 2

What Markdown actually is

A rule book for how plain text turns into formatted text. You write a # and the tool turns it into a heading. You write **word** and the tool turns "word" bold. You don't click buttons — you type the marks inline.

You write
# My Report I visited **Gelephu** last week. Three things I noticed: - Fast internet - New buildings - [Friendly people](https://gmc.bt)
Readers see

My Report

I visited Gelephu last week.

Three things I noticed:

  • Fast internet
  • New buildings
  • Friendly people

The whole language fits on one page. You can learn the entire thing today, and be faster with it by next week.

Part 3

The syntax, one feature at a time

1. Headings

One # is the biggest heading. More # marks = smaller. You can go up to six. Always put a space after the #.

Write
# Project plan ## Phase 1 ### Week 1 goals #### Daily stand-ups
Shown

Project plan

Phase 1

Week 1 goals

Daily stand-ups

2. Bold, italic & strikethrough

Wrap the word in stars. One star = italic. Two stars = bold. Three = both. Two tildes (~~) = strikethrough.

Write
*tired* — italic **important** — bold ***urgent*** — bold italic ~~cancelled~~ — strikethrough
Shown

tired — italic

important — bold

urgent — bold italic

cancelled — strikethrough

3. Lists

Use - for bullets or 1. for numbered. Indent 2 spaces to nest.

Write
Groceries: - Rice - Chili - Red - Green - Oil Steps: 1. Open laptop 2. Open browser 3. Visit gmail.com
Shown

Groceries:

  • Rice
  • Chili
    • Red
    • Green
  • Oil

Steps:

  1. Open laptop
  2. Open browser
  3. Visit gmail.com

Tip: for numbered lists, you can write 1. 1. 1. and it still renders as 1. 2. 3. — the tool fixes numbering for you.

4. Links & images

Square brackets for the text, round brackets for the URL. For images, put a ! in front.

Write
Visit [SELISE](https://selisegroup.com). Logo: ![SELISE logo](logo.png)
Shown

Visit SELISE.

Logo:
📷 SELISE logo

5. Code

For a short bit of code inside a sentence, wrap it in single backticks: `npm install`. For a multi-line block, use three backticks on their own line. Optionally add the language after the opening backticks (```js, ```python) so the code gets coloured.

Write
Run `npm install` first. ```js function greet(name) { return "Hello, " + name; } ```
Shown

Run npm install first.

function greet(name) {
  return "Hello, " + name;
}

6. Blockquotes & horizontal rules

> at the start of a line makes it a quote. --- on its own line draws a divider.

Write
> The best way to predict > the future is to invent it. > — Alan Kay --- Next topic below the line.
Shown
The best way to predict the future is to invent it.
— Alan Kay

Next topic below the line.

7. Tables

Columns are split by |. The second row must be --- (this is what tells Markdown "these are headers"). You don't need to line up the pipes; the tool fixes it.

Write
| Name | Vertical | Status | |-------|----------|--------| | Karma | Dev | Active | | Pema | QA | Active | | Tashi | BA | Review |
Shown
NameVerticalStatus
KarmaDevActive
PemaQAActive
TashiBAReview

8. Line breaks — the gotcha

Markdown ignores single line breaks. If you hit Enter once, the two lines join into one paragraph. To split paragraphs, leave a blank line between them.

This is the #1 thing beginners get wrong.

If your text isn't splitting into paragraphs, you forgot the blank line.

9. Escaping — showing the marks literally

If you want to show a star or a hash without formatting, put a backslash \ in front.

Write
This becomes italic: *hello* This stays: \*hello\* Not a heading: \# hello
Shown

This becomes italic: hello

This stays: *hello*

Not a heading: # hello

Part 5

GitHub-Flavored Markdown

GitHub extended standard Markdown with extras that you will use constantly at SELISE. Most modern tools (Linear, Jira, Notion, Obsidian) now support these too.

Write
Today's tasks: - [x] Write spec - [ ] Review with Tek - [ ] Share with team Tag people: @bhuwan Link an issue: #42 Emoji: :rocket: :bug:
Shown on GitHub

Today's tasks:

  • Write spec
  • Review with Tek
  • Share with team

Tag people: @bhuwan
Link an issue: #42
Emoji: 🚀 🐛

Task lists (checkboxes) are a big deal — they're how teams track progress inside issues and PRs.

Part 6

How each vertical uses Markdown every day

Same language, three very different documents.

👨‍💻 Dev — a project README

Every GitHub project has a README.md. It's the first thing a reviewer, a teammate, or your future self sees. A clear README is the single biggest signal that a developer is professional.

# Weather App

A simple weather app for GMC residents.

## Setup

```bash
npm install
npm start
```

## Features

- Current temperature
- 7-day forecast
- Works offline

## Built with

- React
- OpenWeather API
🐛 QA — a bug report

A developer reads your bug report alone, at their desk, often hours after you filed it. Your Markdown structure is doing the explaining. A shapeless paragraph wastes their time; a structured report gets fixed fast.

# Bug: Login fails on Bhutanese numbers

**Environment:** Production, iPhone 14, Safari
**Severity:** High

## Steps to reproduce

1. Go to login page
2. Enter phone `+97517123456`
3. Tap "Send OTP"

## Expected
OTP arrives within 30 seconds.

## Actual
Screen freezes. No error shown.

## Evidence
![screenshot](freeze.png)
📋 BA — a user story / mini-spec

A BA's spec is read by developers, designers, and QAs. All of them should take the same thing away from it. Markdown's structure (headings, lists, checkboxes) makes that much more likely.

# Story: Forgot password

## Who
Any registered user.

## Why
They forgot their password and can't log in.

## Acceptance criteria

- [ ] "Forgot password?" link on the login screen
- [ ] User enters email → gets reset link
- [ ] Link expires after 30 minutes
- [ ] Link can only be used once

## Out of scope

- SMS-based reset
- Security questions
Part 7

Markdown & AI — why structured prompts win

AI tools like Claude Code, ChatGPT, and Gemini all read Markdown natively. They use its structure to understand what you want. This means the way you lay out a prompt directly affects the quality of the answer.

A structured prompt gets a structured answer. A shapeless prompt gets a shapeless answer.

❌ Shapeless prompt
make a react counter with buttons +1 and -1 also reset and use tailwind keep it in one file
✅ Markdown prompt
# Task Build a small React counter component. # Requirements - Start at 0 - Buttons: +1, -1, reset - Use Tailwind for styles # Constraint Keep it in one file.

The second one takes 20 extra seconds to write and gives you a dramatically better answer. Every time.

This is why Pelsung uses Claude Code

Claude Code and similar AI tools read Markdown by default — CLAUDE.md files, specs, prompts, responses. If you can write good Markdown, you can work well with AI. The two skills are basically one skill.

Part 9 · Do the work

Three exercises — no shortcuts

You cannot learn Markdown by reading about Markdown. Do these three exercises in the playground above. Don't copy-paste from the starter templates — type every character yourself. That's how your hands learn the language.

Exercise 1 · ~15 min

Write your "About Me"

Create a short personal profile in Markdown. It must include:

  • A big heading with your name
  • A smaller "About Me" heading and a paragraph about yourself
  • A bulleted list of 3 things you like
  • A numbered list of 3 goals for Pelsung
  • A link to a website you visit daily
  • Something in italic and something in bold
  • A blockquote with a favourite saying

Done when: the right pane looks like a real profile page, not a dump of stars and hashes.

Exercise 2 · ~15 min

File a fake bug report

Pretend you found a bug in WhatsApp (or any app you use daily). Write the report in Markdown as if you were filing it in Jira or GitHub. Must include:

  • A clear one-line title (H1) — not "bug bug bug"
  • Environment — device, browser/OS, app version (use bold labels)
  • Steps to reproduce — a numbered list of 3+ steps
  • Expected and Actual as two H2 sub-headings
  • A code snippet (e.g. a URL or error message) in backticks
  • A task list (checkboxes) of things still to verify

Tip: click the "Bug report starter (QA)" button above and change every field. Keep nothing the same.

Exercise 3 · ~15 min

Build a comparison table

Pick 3 apps you use every day (WhatsApp, Instagram, Druk Pay, mBoB, etc.). Build a Markdown table comparing them. Columns:

  • App name
  • What you use it for
  • One thing you like
  • One thing that annoys you

Then, under the table, add a blockquote (>) saying which one you'd keep if you could only have one — and why.

This is the kind of document a BA might produce after a discovery interview.

Part 10

Common mistakes

❌ Missing space after #

#Hello does nothing. # Hello becomes a heading. Always put one space after the hash.

❌ No blank lines between blocks

Two lines of text with no blank line between them join into one paragraph. A table right after a paragraph with no blank line between often fails. When in doubt, add the blank line.

❌ Curly quotes in code

If your text editor turned your backticks into curly quotes (', "), code formatting breaks. Use plain straight backticks — the key below Esc on most keyboards.

❌ Tables without the --- row

Markdown tables need that second row of dashes. Without it, you just get ugly pipes and text.

❌ Using 1 space for nested lists

Indent with 2 spaces (or a tab) to nest a list. One space often fails silently.

✅ The fix for almost all of these

Look at the rendered preview before you send. If it doesn't look right, it isn't right. Fix it first.

Part 11

Tools & further reading

Where to write Markdown

  • VS Code (free) — the editor you'll use for everything else too. Press Ctrl+Shift+V (Mac: Cmd+Shift+V) to open a preview. Install the "Markdown All in One" extension for more power.
  • Obsidian (free) — keeps your notes as plain Markdown files on your laptop. Excellent for personal knowledge notes.
  • GitHub — every .md file auto-previews when you click on it. This is where you'll end up writing most Markdown in your career.
  • dillinger.io, stackedit.io, markdownlivepreview.com — browser-only playgrounds, great for quick edits.
